## 1) 가장 흔한 전이 흐름 (포그라운드 → 백그라운드 → 서스펜드)

### A. 사용자가 홈으로 나가거나 앱 스위처로 나감

일반적인 흐름: **Active → Inactive → Background → (잠시 후) Suspended**

- `Inactive`는 “전환 애니메이션/시스템 인터럽트” 같은 완충지대
    
- `Background`에서 앱이 할 일을 정리하면
    
- OS가 곧 `Suspended`로 얼립니다(자동)

---

## 2) “서스펜드에서 백그라운드”는 보통 **없다**

여기 질문에서 핵심:  **Suspended → Background** 는 “일반적인 상태 전이”로는 거의 발생하지 않습니다.

왜냐면 Suspended는 **코드 실행이 0**인 “얼린 상태”라서,  앱이 스스로 “백그라운드로 가서 뭐 할게요”라고 움직일 수가 없어요.

다만 예외적으로 OS가 앱을 깨우는 경우가 있어요(아래 “깨우는 이벤트”).

---

## 3) 백그라운드에서 서스펜드로 가는 건 **OS가 자동으로** 한다

**Background → Suspended** 는 앱이 “전이 요청”해서 되는 게 아니라:

- 앱이 background 진입 후 정리 끝내고 더 이상 실행할 게 없고 백그라운드 모드 권한/작업이 없으면  
  → OS가 잠깐 뒤에 **자동으로 얼림(suspend)**

이 전이에 대해 “콜백이 따로” 오지 않는 경우가 많아서(혹은 Scene delegate에서 체감이 약해서) 개발자들이 “언제 서스펜드 됐지?” 하고 헷갈려요.

---

## 4) 다시 포그라운드로 돌아올 때 흐름

### B. 서스펜드/백그라운드 상태에서 앱 아이콘을 눌러 복귀

흐름은 보통: **(Suspended 또는 Background) → Inactive → Active**

- 이미 메모리에 살아있으면 “재개”
    
- OS가 이미 종료해버렸다면 → “완전 재실행(Cold start)”

---

## 5) “깨우는 이벤트” (Suspended 상태여도 OS가 깨울 수 있는 경우)

이게 예외 케이스예요. 아래는 iOS가 앱을 잠깐 깨울 수 있는 대표 상황들:

- **푸시 알림**(특정 유형: content-available 등)
    
- **Background fetch**(시스템이 타이밍 결정)
    
- **BGTaskScheduler**(백그라운드 작업 스케줄)
    
- **위치 업데이트**, **블루투스(central/peripheral 상태 복원)** 같은 “Background Modes”
    
- **VoIP**, 오디오 재생 등(해당 모드 권한 필요)

이 경우엔 “서스펜드였다가” 시스템이 깨워서 **Background 실행 시간**을 잠깐 줍니다.

즉, 사용자가 앱을 열지 않아도: **Suspended → (OS가 깨움) → Background(잠깐 실행) → 다시 Suspended/종료** 가 가능해요.

---

## 6) 실제로 어디서 뭘 처리해야 하나? (실무 체크리스트)

### 포그라운드 → 백그라운드 들어갈 때 꼭 하는 것

- 화면/입력 중단 대응(일시정지)
    
- 미완료 작업 저장(임시 저장, CoreData save 등)
    
- 네트워크/스트림/오디오/센서 정리
    
- BLE 같은 경우: 백그라운드 모드 유무에 따라 연결 유지 전략 분기

### 백그라운드에서 “조금 더 일해야 할 때”

- `beginBackgroundTask`로 **짧은 시간 연장**(무한 작업은 안 됨)
    
- 장기/정기 작업은 **BGTaskScheduler** 설계가 정석

### “서스펜드/종료” 대비

- “다음에 돌아올 때 이어서 복원”을 기본 전략으로
    
- 상태 복원 정보는 background 진입 시점에 저장해두는 습관이 중요

## 7) (UIKit Scene) 콜백/노티 매핑

### SceneDelegate 기준으로 대표 콜백

- `sceneDidBecomeActive` : Active 됨
    
- `sceneWillResignActive` : Active → Inactive로 가기 직전
    
- `sceneWillEnterForeground` : Background → Foreground 진입 직전
    
- `sceneDidEnterBackground` : Foreground → Background 진입 완료
    

> “Suspended 됨”을 직접 알려주는 콜백은 일반적으로 없습니다.  
> 보통 `sceneDidEnterBackground` 이후 OS가 얼립니다.

### AppDelegate(구형/추가)

- `applicationDidBecomeActive`
    
- `applicationWillResignActive`
    
- `applicationDidEnterBackground`
    
- `applicationWillEnterForeground`
    
- `applicationWillTerminate` (항상 보장 X)

---

## 8) 전이 관계를 한 장으로 요약

- **Active → Inactive**: 인터럽트/전환 시작
    
- **Inactive → Active**: 인터럽트 끝 / 다시 사용 가능
    
- **Inactive → Background**: 홈/앱전환으로 화면에서 사라짐
    
- **Background → Suspended**: OS가 자동 “얼림”
    
- **(Suspended/Background) → Inactive → Active**: 사용자가 다시 앱을 열어 복귀
    
- **Suspended → Background**: 일반 전이는 거의 없고, **OS가 깨우는 이벤트가 있을 때만 가능**
