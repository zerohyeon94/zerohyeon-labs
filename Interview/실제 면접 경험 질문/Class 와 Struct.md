## 1) Struct (구조체) 설명

- **값 타입(Value Type)** → 대입/전달 시 “값이 복사”되는 의미(성능은 COW/최적화가 개입할 수 있음).
    
- **상속 불가**
    
- **ARC(참조 카운팅) 대상 아님** (참조 타입이 아니니까)
    
- Swift 표준 라이브러리의 대부분(`String`, `Array`, `Dictionary`, `Set`)이 struct 기반

```swift
struct Point {
    var x: Int
    var y: Int
}

var p1 = Point(x: 0, y: 0)
var p2 = p1      // 값 복사(의미상)
p2.x = 10
// p1.x는 0, p2.x는 10
```

### 언제 struct를 쓰나?

- “**데이터 모델**”처럼 값 자체가 의미인 경우
    
- **공유 상태를 피하고 싶을 때** (안전)
    
- 불변(immutable)하게 쓰기 좋음 (`let`)

---

## 2) Class (클래스) 설명

- **참조 타입(Reference Type)** → 대입/전달 시 “같은 인스턴스(주소)를 공유”
    
- **상속 가능** (다형성/오버라이딩 가능)
    
- **ARC 대상** (strong/weak/unowned)
    
- `deinit`으로 해제 시점에 작업 가능

```swift
class Person {
    var name: String
    init(name: String) { self.name = name }
}

let a = Person(name: "A")
let b = a         // 같은 인스턴스 참조
b.name = "B"
// a.name도 "B"
```

### 언제 class를 쓰나?

- “**정체성(identity)**”이 중요한 경우 예) 같은 객체를 여러 곳에서 공유하면서 상태가 바뀌어야 함
    
- **상속/다형성**이 필요한 경우
    
- **라이프사이클**을 가져야 하는 경우(예: `UIViewController`, `UIView`)
---

## 3) 가장 중요한 차이: Value vs Reference

| 구분      | Struct          | Class       |
| ------- | --------------- | ----------- |
| 타입      | 값 타입            | 참조 타입       |
| 대입/전달   | 복사(의미상)         | 참조 공유       |
| 변경 영향   | 복사본만 변경         | 모두에게 영향     |
| 메모리 관리  | ARC 아님          | ARC(참조 카운팅) |
| 상속      | 불가              | 가능          |
| deinit  | 없음              | 있음          |
| 스레드 안전성 | 상대적으로 유리(공유 적음) | 공유 상태 주의 필요 |

---

## 4) 추가 차이들 (면접에서 가끔 나오는 꼬리질문)

### 4-1. Mutability (let/var)

- struct는 `let`이면 **내부 프로퍼티 변경 불가**

```swift
let p = Point(x: 1, y: 1)
// p.x = 2  // ❌ 불가 (p 자체가 상수)
```

- class는 `let`이어도 **참조는 고정, 내부 값 변경 가능**

```swift
let a = Person(name: "A")
a.name = "B" // ✅ 가능 (a가 가리키는 인스턴스 내부 변경)
```

### 4-2. 동등성(==) vs 동일성(===)

- struct는 보통 `Equatable`로 “값이 같은지” 비교
    
- class는 “값이 같은지(==)” + “같은 인스턴스인지(===)” 비교 가능

```swift
a === b // 같은 객체(주소)인가?
```

### 4-3. 초기화/상속 관련

- class는 상속 때문에 **designated / convenience init**, `required`, `override` 등 규칙이 더 많음
    
- struct는 비교적 단순하고, 멤버와이즈 초기화(자동 init)가 제공됨

---

## 5) 실전 선택 기준 (면접 답변에 가장 유용)

### 기본 원칙(애플도 자주 권장하는 방향)

- **“기본은 struct, 필요할 때만 class”**
    
- 아래에 해당하면 struct가 보통 더 적합:
    
    - 단순 데이터
        
    - 공유 상태 없어야 함
        
    - 값이 복사되어도 의미가 자연스러움
        
    - 스레드/동시성 안전성을 높이고 싶음
        
- 아래에 해당하면 class가 적합:
    
    - 객체의 **정체성**이 중요 (여러 곳이 같은 인스턴스를 바라봐야 함)
        
    - 공유/변경되는 상태를 한 곳에서 관리해야 함
        
    - 상속/다형성이 필요
        
    - UIKit/AppKit 타입들과 연동(UIView, VC 등)

---

## 6) 면접용 20초 답변 템플릿

Struct는 값 타입이라 대입/전달 시 값이 복사되는 의미이고, 공유 상태가 적어 안전하며 상속이 안 됩니다. 
Class는 참조 타입이라 대입/전달 시 같은 인스턴스를 공유하고, ARC로 메모리 관리가 되며 상속과 다형성이 가능합니다. 
그래서 보통 데이터 모델처럼 값 의미가 중요하면 struct를, 
여러 곳에서 동일 객체를 공유하거나 라이프사이클/상속이 필요한 경우엔 class를 선택합니다.
