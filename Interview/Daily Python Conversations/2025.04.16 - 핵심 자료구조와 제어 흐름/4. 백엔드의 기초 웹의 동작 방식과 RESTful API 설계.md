## 📌 핵심 요약 (웹과 API의 본질)

- **클라이언트와 서버의 분리**: 화면을 보여주고 사용자 입력을 받는 역할(클라이언트, 예: iOS 앱)과 데이터를 저장하고 비즈니스 로직을 처리하는 역할(서버)을 명확히 나눕니다.
    
- **HTTP 프로토콜**: 이 둘이 서로 데이터를 주고받기 위해 사용하는 전 세계 공통의 통신 규약입니다. 요청(Request)을 보내면 응답(Response)을 받는 단순하고 확실한 구조입니다.
    
- **RESTful API**: 서버가 가진 기능이나 데이터를 클라이언트가 쉽게 사용할 수 있도록, URL(주소)과 HTTP 메서드(행위)를 조합하여 알아보기 쉽게 만든 일관된 규칙(인터페이스)입니다.

## 📖 상세 설명 (실무에서 쓰이는 방식)

### Step 1. 웹의 기본 통신 흐름 (Request & Response)

iOS 기기에서 스터디 타이머의 '시작' 버튼을 누르는 상황을 상상해 보세요.

1. **요청(Request)**: 클라이언트(앱)가 서버로 "이 사용자의 새로운 집중 시간 기록을 시작해 줘"라는 메시지와 필요한 데이터(JSON)를 보냅니다.
    
2. **처리**: 백엔드 서버는 파이썬(FastAPI 등)으로 작성된 로직을 실행하여 데이터베이스에 기록을 생성합니다.
    
3. **응답(Response)**: 서버가 클라이언트에게 "성공적으로 기록되었어"라는 상태 코드와 함께 결과를 돌려줍니다.

### Step 2. 무질서한 주소를 깔끔하게, RESTful API

서버의 기능을 주소로 만들 때, `도메인.com/create_data`, `도메인.com/get_user_info`처럼 중구난방으로 만들면 개발자들끼리 소통하기 힘듭니다. 그래서 **자원(명사)**과 **행위(HTTP 메서드)**를 분리하는 RESTful 설계 방식을 따릅니다.

- **행위 (HTTP Method)**:
    
    - `GET`: 데이터 조회 (Read)
        
    - `POST`: 새로운 데이터 생성 (Create)
        
    - `PUT` / `PATCH`: 기존 데이터 수정 (Update)
        
    - `DELETE`: 데이터 삭제 (Delete)
        
- **자원 (URL)**: 행위를 나타내는 동사는 빼고, 오직 명사로만 구성합니다.
    
    - 좋은 예: `GET /users/123` (123번 유저 정보 조회)
        
    - 나쁜 예: `GET /get_user?id=123` (URL에 행위가 들어감)

### Step 3. 대화의 마침표, HTTP 상태 코드 (Status Codes)

서버는 응답할 때, 작업이 잘 끝났는지 문제가 생겼는지를 3자리 숫자로 클라이언트에게 알려줍니다.

- **200번대 (성공)**: `200 OK` (정상 처리), `201 Created` (데이터 생성 성공)
    
- **400번대 (클라이언트 잘못)**: `400 Bad Request` (잘못된 요청 형식), `401 Unauthorized` (로그인 안 됨), `404 Not Found` (없는 주소나 데이터)
    
- **500번대 (서버 잘못)**: `500 Internal Server Error` (서버 내부 코드 에러 발생)

### Step 4. FastAPI를 활용한 초간단 REST API 구현 예시

파이썬의 최신 웹 프레임워크인 FastAPI를 사용하면 이런 RESTful API를 매우 직관적으로 만들 수 있습니다.

```Python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# 클라이언트가 보낼 JSON 데이터 구조(스키마) 정의
class StudyRecord(BaseModel):
    user_id: str
    focus_time_minutes: int

# GET 메서드: 특정 사용자의 기록 조회
@app.get("/records/{user_id}")
def get_user_records(user_id: str):
    # 실제로는 DB에서 user_id로 데이터를 찾아옵니다.
    return {"user_id": user_id, "status": "success", "message": "기록을 성공적으로 불러왔습니다."}

# POST 메서드: 새로운 집중 시간 기록 생성
@app.post("/records")
def create_study_record(record: StudyRecord):
    # 전달받은 record 객체(딕셔너리 형태)를 DB에 저장하는 로직이 들어갑니다.
    return {"status": "created", "saved_time": record.focus_time_minutes}
```

## 🆚 상황별 API 메서드 선택 가이드 (Cheatsheet)

|**해결하려는 문제 상황 (클라이언트의 요청)**|**필요한 HTTP 메서드 및 URL 설계 가이드**|
|---|---|
|"내 프로필 화면에 띄울 사용자 기본 정보를 가져와야 해."|**GET** `/users/{user_id}`|
|"앱에서 방금 측정한 새로운 스터디 세션 결과를 서버에 저장할래."|**POST** `/sessions`|
|"사용자가 닉네임을 변경하고 '저장'을 눌렀어."|**PUT** 또는 **PATCH** `/users/{user_id}/nickname`|
|"더 이상 사용하지 않는 예전 타이머 알람 설정을 삭제할래."|**DELETE** `/alarms/{alarm_id}`|

## ❓ 확인 질문 (면접 대비용 최종 보스)

**Q. 사용자의 집중도를 측정하여 기록하는 타이머 앱의 백엔드를 설계하고 있습니다. 사용자가 새로운 집중 세션을 시작할 때, 그리고 세션이 끝난 뒤 그날의 전체 누적 집중 시간을 조회할 때 클라이언트가 호출해야 할 RESTful API의 HTTP 메서드와 URL 주소를 각각 어떻게 설계하시겠습니까?**

**A. >**

> 1. 새로운 집중 세션을 시작(생성)할 때는 데이터를 서버에 새로 만들어야 하므로 **POST** 메서드를 사용하겠습니다. 주소는 자원을 명시적으로 나타내는 명사형을 사용하여 **`POST /sessions`** 로 설계하고, 시작 시간 등의 데이터를 JSON 바디에 담아 보내도록 하겠습니다.
>     
> 2. 세션이 끝난 후 그날의 누적 집중 시간을 조회할 때는 서버의 상태를 변경하지 않고 데이터를 가져오기만 하므로 **GET** 메서드를 사용하겠습니다. 주소는 특정 사용자의 통계 데이터임을 명확히 하기 위해 **`GET /users/{user_id}/sessions/stats`** (또는 쿼리 파라미터를 조합해 날짜 지정) 와 같이 직관적인 RESTful 원칙에 맞추어 설계하겠습니다.
