# IEC 62304 — 의료기기 소프트웨어 생애주기 프로세스

> 관련: [[MIL-STD-810-461 핵심 개념 정리]], [[IEC 61010-1 전기안전 표준]]

---

## 1. 표준 개요

| 항목 | 내용 |
|------|------|
| 정식 명칭 | IEC 62304:2006+AMD1:2015 — Medical device software: Software life cycle processes |
| 적용 대상 | 의료기기에 탑재되거나 의료기기 자체인 소프트웨어 |
| 핵심 목적 | 의료기기 SW 개발·유지보수 전 생애주기에 걸쳐 안전성 보장 |
| 관련 표준 | IEC 62443, ISO 14971(위험관리), IEC 60601-1(전기안전) |

---

## 2. 소프트웨어 안전 등급 (Safety Class)

IEC 62304의 핵심 개념. **소프트웨어 결함이 환자에게 미치는 위해 수준**으로 등급 결정.

| 등급          | 정의              | 예시                   |
| ----------- | --------------- | -------------------- |
| **Class A** | 부상 위험 없음        | 통계 보고 SW, 관리자 UI     |
| **Class B** | 심각하지 않은 부상 가능   | 알림 표시 SW, 운동 가이드 앱   |
| **Class C** | 사망 또는 심각한 부상 가능 | 인슐린 펌프 제어, 방사선 치료 SW |

> **핵심**: 등급이 높을수록 요구하는 문서화·검증 수준이 올라간다. Class C는 모든 프로세스를 완전 이행해야 한다.

등급 결정 방법: **ISO 14971(위험관리)** 결과물을 입력으로 사용한다.

---

## 3. 생애주기 프로세스 구조

```
SW 개발 계획
    ↓
SW 요구사항 분석
    ↓
SW 아키텍처 설계
    ↓
SW 세부 설계          ← Class C만 필수
    ↓
SW 구현 및 단위 검증
    ↓
SW 통합 및 통합 시험
    ↓
SW 시스템 시험
    ↓
SW 릴리즈
    ↓
SW 유지보수 (별도 프로세스)
```

---

## 4. 주요 프로세스별 요구사항

### 4.1 SW 개발 계획 (5.1절)
- 생애주기 모델 정의 (폭포수, 반복형 등)
- 개발 표준·방법론 명시
- SW 형상 관리 계획
- SW 위험관리 계획 포함

### 4.2 SW 요구사항 분석 (5.2절)
- 기능·성능·인터페이스 요구사항 문서화
- 안전 관련 요구사항을 **명시적으로 식별** (Safety Requirement)
- IEC 62304 Class에 따른 문서 수준 차이:
  - Class A: 요구사항 목록
  - Class B/C: 추적성(traceability) 포함

### 4.3 SW 아키텍처 설계 (5.3절)
- SW 아이템(item)으로 분할
- 각 아이템의 안전 등급 재평가 가능
- **SOUP(Software of Unknown Provenance)** 식별 — 오픈소스·서드파티 라이브러리 포함

### 4.4 SW 구현 (5.5절)
- 코딩 표준 적용
- Class B/C: 단위 시험 계획 및 실행 필수
- Class C: 추가적인 코드 리뷰 요구

### 4.5 SW 시험 (5.6~5.7절)
| 시험 유형 | Class A | Class B | Class C |
|-----------|---------|---------|---------|
| 통합 시험 | 필요 | 필요 | 필요 |
| 시스템 시험 | 필요 | 필요 | 필요 |
| 단위 시험 | 불필요 | 필요 | 필요 |
| 시험 추적성 | 불필요 | 권장 | 필수 |

---

## 5. SOUP (Software of Unknown Provenance)

의료기기 SW에서 **외부 라이브러리·오픈소스를 사용할 때의 핵심 개념**.

### 정의
- 개발 이력이 IEC 62304 프로세스에 따라 관리되지 않은 모든 외부 SW
- 예: SwiftUI, Kotlin stdlib, TensorFlow Lite, CoreML, React Native

### SOUP 관리 요구사항
1. **목록 작성**: 이름, 버전, 제조사, 기능 설명
2. **기능 요구사항 정의**: SOUP이 해야 하는 동작 명시
3. **알려진 이상 목록**: 버그 트래커, 릴리즈 노트 검토
4. **검증**: SOUP이 실제로 요구사항대로 동작하는지 확인

> 실무 팁: iOS 앱이라면 CoreML, HealthKit 전부 SOUP으로 등록해야 한다.

---

## 6. SW 형상 관리 (8절)

- SW 아이템의 버전 식별 체계 수립
- 변경 제어 절차 (Change Control)
- 빌드 재현성 보장 — 동일 버전 태그로 동일 바이너리 생성 가능해야 함
- Git 기반 프로젝트에서는 태그 + 빌드 환경 고정(lock 파일)으로 대응

---

## 7. SW 유지보수 프로세스 (6절)

의료기기 출시 후의 SW 변경도 동일한 생애주기를 따른다.

- **결함 수정**: 결함 영향 분석 → 등급 재평가 → 수정 → 재검증
- **레거시 SW**: 출시 이전에 개발된 SW도 62304 적용 시 소급 분석 필요 (gap 분석)

---

## 8. 문제 해결 프로세스 (9절)

- 이상(anomaly) 발생 시 문서화 → 조사 → 원인 분석 → 조치
- 안전 관련 이상은 ISO 14971 위험관리 프로세스와 연동

---

## 9. 핵심 문서 산출물 요약

| 산출물 | 필수 등급 |
|--------|-----------|
| SW 개발 계획 | A/B/C |
| SW 요구사항 명세 | A/B/C |
| SW 아키텍처 설계서 | B/C |
| SW 세부 설계서 | C |
| SOUP 목록 | B/C |
| 단위 시험 계획/결과 | B/C |
| 통합·시스템 시험 계획/결과 | A/B/C |
| 추적성 매트릭스 | C (B는 권장) |

## 10. 면접 포인트

**자주 나오는 질문과 핵심 답변**

Q. IEC 62304에서 Safety Class를 어떻게 결정하나요?
> ISO 14971 위험 분석 결과를 기반으로, 소프트웨어 결함이 환자에게 미칠 수 있는 최대 위해 수준으로 결정합니다. 잔류 위험이 허용 가능하면 Class A, 심각하지 않은 부상이면 Class B, 사망/심각한 부상이면 Class C입니다.

Q. SOUP을 어떻게 관리했나요?
> 사용한 라이브러리의 이름·버전·기능 요구사항을 목록화하고, 각 버전의 릴리즈 노트와 이슈 트래커를 검토해 알려진 이상 여부를 확인했습니다.

Q. IEC 62304와 애자일 개발은 어떻게 병행하나요?
> IEC 62304는 특정 개발 방법론을 강요하지 않습니다. 반복 개발 모델을 명시적으로 계획서에 기술하고, 각 스프린트마다 요구사항 추적성과 시험 결과를 유지하는 방식으로 병행 가능합니다.
