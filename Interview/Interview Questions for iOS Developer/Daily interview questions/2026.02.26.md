### 1. iOS 앱의 생명 주기(App Lifecycle)에 대해 설명해주세요.
답변: iOS 앱의 상태는 크게 Not Running, Inactive, Active, Background, Suspended로 나뉩니다. 사용자가 앱을 실행하면 Inactive를 거쳐 Active 상태가 되어 이벤트를 받을 수 있습니다. 홈 버튼을 누르거나 다른 앱으로 전환하면 Background 상태로 들어가며, 여기서 추가 작업이 없으면 곧바로 메모리에만 유지되는 Suspended 상태로 전환됩니다.

### 2. Class와 Struct의 차이점에 대해 설명해주세요. (추가 질문: Struct는 상속이 안되나요?)
답변: Class는 참조 타입(Reference Type)이고 주로 힙(Heap) 메모리 영역에 할당되며 상속이 가능합니다. 반면 Struct는 값 타입(Value Type)이고 주로 스택(Stack) 영역에 할당되며 상속이 불가능합니다. Struct에서 상속과 유사한 다형성을 구현하려면 프로토콜(Protocol)을 활용해야 합니다.

### 3. static 키워드는 무엇인가요? class 키워드와는 어떤 차이가 있나요? 
답변: static 키워드는 타입 프로퍼티나 타입 메서드를 정의할 때 사용하며, 인스턴스화하지 않고도 타입 자체에서 접근할 수 있습니다. class 키워드 역시 클래스 내에서 타입 메서드를 정의할 때 사용되지만, static과 달리 서브클래스에서 오버라이딩(재정의)이 가능하다는 차이점이 있습니다.

### 4. Swift의 Collection Type들에 대해 설명해주세요. 
답변: Swift의 대표적인 컬렉션 타입에는 Array, Dictionary, Set이 있습니다. Array는 순서가 있는 데이터의 집합이고, Dictionary는 순서 없이 키와 값의 쌍으로 이루어진 데이터 집합입니다. Set은 순서가 없으며 중복되지 않는 유일한 값들의 집합을 의미합니다.

### 5. ARC(Automatic Reference Counting)에 대해 설명하고, 강한 순환 참조를 해결하는 방법을 말씀해주세요. 
답변: ARC는 런타임에 메모리를 관리하는 방식으로, 객체를 참조하는 횟수를 추적하여 참조 횟수가 0이 되면 메모리에서 해제합니다. 두 객체가 서로를 강하게 참조하여 메모리 누수가 발생하는 강한 순환 참조를 피하기 위해, 참조의 한쪽을 weak(약한 참조)나 unowned(미소유 참조)로 선언하여 해결할 수 있습니다.

### 6. UIKit 기반의 프로젝트를 SwiftUI로 전환할 때 느끼는 두 프레임워크의 가장 큰 차이점은 무엇이라고 생각하시나요?
답변: UIKit은 명령형 프로그래밍 방식으로 개발자가 뷰의 상태 변화와 UI 업데이트 과정을 직접 지시해야 합니다. 반면 SwiftUI는 선언형 프로그래밍 방식으로, 상태(State)를 선언하고 그 상태에 따라 뷰가 어떻게 보여야 하는지를 정의하면 시스템이 알아서 UI를 업데이트한다는 점이 가장 큰 차이입니다.

### 7. Optional이란 무엇이며, 안전하게 값을 추출하는 방법들을 설명해주세요. 
답변: Optional은 값이 있을 수도 있고, 없을 수도 있는(nil) 상태를 나타내는 타입입니다. 안전하게 값을 추출하기 위해서는 if let 이나 guard let 을 사용하는 Optional Binding을 주로 활용하며, 기본값을 제공하는 Nil-Coalescing 연산자(??)를 사용할 수도 있습니다. 강제 추출(!)은 런타임 에러를 유발할 수 있어 지양해야 합니다.

### 8. Main Queue와 Global Queue의 차이점은 무엇이며, UI 업데이트는 어디서 처리해야 하나요? 
답변: Main Queue는 단일 스레드인 메인 스레드에서 순차적으로 작업을 처리하며 화면의 UI 업데이트를 전담합니다. Global Queue는 시스템이 관리하는 동시성 큐로 백그라운드에서 네트워크 요청이나 무거운 연산을 처리할 때 사용합니다. 따라서 UI 업데이트 작업은 반드시 Main Queue에서 비동기적으로 처리해야 합니다.

### 9. 클로저(Closure)에서 캡처 리스트(Capture List)를 사용하는 이유는 무엇인가요? 
답변: 클로저는 주변 환경의 변수나 상수를 참조할 때 강한 참조로 캡처하는 성질이 있습니다. 만약 클래스 인스턴스가 클로저를 프로퍼티로 가지고 있고, 그 클로저가 다시 해당 인스턴스를 캡처하면 강한 순환 참조가 발생합니다. 이를 방지하기 위해 캡처 리스트에 weak self 등을 명시하여 참조 카운트가 올라가지 않도록 만들어줍니다.

### 10. MVC 패턴과 MVVM 패턴의 차이점에 대해 설명해주세요. 
답변: MVC는 Model, View, Controller로 구성되며, Controller가 View와 Model 사이의 중재자 역할을 합니다. iOS에서는 ViewController가 View의 역할까지 일부 떠안아 코드가 비대해지는 단점이 있습니다. MVVM은 Model, View, ViewModel로 구성되며, View는 ViewModel의 상태를 관찰하여 UI를 갱신합니다. 이를 통해 View와 비즈니스 로직을 철저히 분리하고 테스트 용이성을 높일 수 있습니다.
