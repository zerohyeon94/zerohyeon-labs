### [복습] SwiftUI에서 `some View` (불투명 타입, Opaque Type)를 사용하는 이유와 컴파일러 관점에서의 이점을 설명해주세요.

개발자에게는 뷰가 결합될 때마다 길어지는 복잡한 반환 타입을 숨겨 코드를 간결하게 유지하게 해줍니다. 동시에 컴파일러에게는 반환되는 뷰가 단일하고 명확한 구체적 타입이라는 것을 보장해주어, 컴파일 시점에 메모리 크기를 정확히 유추하고 렌더링 성능을 극대화할 수 있도록 돕습니다. 제네릭의 유연성과 정적 타입의 안전성을 모두 챙길 수 있는 문법입니다.

### [복습] 비동기 처리 클로저 내부에서 `[unowned self]` 대신 `[weak self]`를 권장하는 기술적인 원인(포인터 관점)은 무엇인가요?

뷰 컨트롤러가 먼저 해제되었을 때 `unowned self`는 파괴된 메모리 주소를 여전히 가리키는 댕글링 포인터(Dangling Pointer)가 되어, 여기에 접근할 시 강제 종료(Crash)를 유발합니다. 반면 `weak self`는 참조하던 인스턴스가 해제되면 ARC가 자동으로 `nil`을 할당해주기 때문에, 옵셔널 바인딩을 통해 앱의 크래시 없이 안전하게 실행을 종료시킬 수 있습니다.

### [Architecture] MVVM 패턴을 구현할 때, 뷰모델에 Input/Output (I/O) 패턴을 적용하는 이유는 무엇인가요?

뷰에서 발생하는 사용자 액션(Input)과 그에 따라 뷰에 그려져야 할 데이터 상태(Output)를 명확히 분리하여 뷰모델의 인터페이스를 규격화하기 위함입니다. 이를 통해 뷰와 뷰모델 간의 데이터 흐름을 단방향으로 철저하게 통제할 수 있고, 코드의 가독성이 높아지며, Input 값에 따른 Output의 결과만 검증하면 되므로 유닛 테스트(Unit Test)를 작성하기가 훨씬 수월해집니다.

### [Core ML / AVFoundation] 실시간 카메라 화면을 통해 사용자의 상태를 분석하는 앱을 만들 때, `AVCaptureSession`에서 들어오는 무수한 비디오 프레임 처리 시 주의할 점은 무엇인가요?

카메라 프레임은 초당 수십 장씩 매우 빠르게 들어오므로, 이전 프레임의 추론 연산이 끝나기 전에 다음 프레임이 계속 큐에 쌓이게 되면 기기에 심한 발열이 생기고 메모리가 폭발할 수 있습니다. 따라서 백그라운드 직렬 큐(Serial Queue)를 사용하여 연산이 겹치지 않게 제어하고, 처리 중일 때는 새로 들어오는 프레임을 과감하게 버리는(Drop) 전략을 세워 파이프라인의 병목 현상을 방지해야 합니다.

### [Swift / Combine] 최근 iOS 비동기 이벤트 처리에 자주 쓰이는 Combine 프레임워크에서 Publisher와 Subscriber의 역할에 대해 설명해주세요.

`Publisher`는 시간의 흐름에 따라 데이터나 이벤트(성공, 실패 등)를 생성하고 방출하는 역할을 합니다. `Subscriber`는 이 Publisher를 구독하여 방출된 데이터를 받아 UI를 뷰에 업데이트하거나 비즈니스 로직을 처리합니다. 이 둘 사이에 `map`, `filter`와 같은 Operator를 두어 데이터를 원하는 형태로 변형하여 전달할 수도 있습니다.

### [Network] 기본 `URLSession` 대신 `Alamofire`와 같은 서드파티 라이브러리를 프로젝트에 도입하는 이유는 무엇이라고 생각하시나요?

`URLSession`만으로도 충분히 네트워킹이 가능하지만, 반복적인 에러 처리, 멀티파트(Multipart) 이미지 업로드, 그리고 토큰 만료 시 재시도하는 인터셉터(Interceptor) 등을 직접 구현하려면 보일러플레이트 코드가 많아집니다. `Alamofire`는 이러한 기능들을 간결한 체이닝 문법으로 제공하며, JSON 디코딩과 라우터(Router) 패턴 적용을 쉽게 만들어 주어 개발 생산성과 코드 유지보수성을 크게 높여주기 때문입니다.

### [SwiftUI] SwiftUI에서 뷰에 `.id()` 모디파이어를 부여하는 역할은 무엇이며, 이를 통해 어떤 UI 효과를 얻을 수 있나요?

`.id()`는 뷰에 고유한 식별값을 부여합니다. SwiftUI의 렌더링 시스템은 이 식별값이 변경되면 기존 뷰의 상태가 바뀐 것이 아니라 '완전히 새로운 뷰'로 교체된 것으로 인식합니다. 이를 활용하면 `ScrollViewReader`를 통해 리스트의 특정 위치로 스크롤을 이동시키거나, 뷰를 강제로 초기화하여 명시적인 트랜지션(Transition) 애니메이션을 다시 발생시킬 때 유용하게 쓰입니다.

### [Swift] `Any`와 `AnyObject`의 차이점은 무엇이며, 실무에서는 주로 언제 사용하나요?

`Any`는 함수, 구조체, 열거형, 클래스 등 Swift의 모든 타입을 담을 수 있는 범용 타입으로, 주로 타입이 혼재된 복잡한 JSON 딕셔너리를 파싱할 때 `[String: Any]` 형태로 사용됩니다. 반면 `AnyObject`는 오직 '클래스' 타입만 담을 수 있는 프로토콜입니다. 실무에서는 Delegate 패턴을 구현할 때, 강한 순환 참조를 막기 위해 해당 프로토콜을 `AnyObject`로 제한하여 델리게이트 변수에 `weak` 키워드를 쓸 수 있도록 만들 때 주로 사용합니다.

### [Memory] 앱 실행 중 메모리 사용량이 계속 증가하는 메모리 누수(Memory Leak)를 디버깅할 때 사용하는 Xcode 도구들에 대해 설명해주세요.

첫 번째로 `Debug Memory Graph`를 사용하여 객체 간의 참조 관계를 시각적으로 확인하고, 화면을 닫았음에도 강한 참조(Retain Cycle)로 인해 메모리에 남아있는 뷰 컨트롤러가 있는지 빠르게 찾습니다. 더 동적인 분석이 필요할 때는 `Instruments` 앱의 `Leaks` 도구를 사용하여 런타임에 발생하는 메모리 누수를 자동으로 탐지하거나, `Allocations` 도구로 특정 객체의 메모리 할당 및 해제 생명주기를 상세히 프로파일링합니다.

### [Git] 이미 원격 저장소에 푸시(Push)된 커밋에 심각한 버그가 있어 이전 상태로 되돌려야 합니다. `git reset`과 `git revert` 중 어떤 것을 사용해야 하며, 그 이유는 무엇인가요?

이미 팀원들과 공유된 원격 저장소의 커밋이라면 반드시 `git revert`를 사용해야 합니다. `git reset`은 커밋 히스토리를 아예 삭제하고 시간을 되돌리므로 다른 팀원들의 로컬 저장소와 심각한 충돌을 일으킵니다. 반면 `git revert`는 이전 커밋의 내용을 역으로 취소하는 '새로운 커밋'을 생성하여 히스토리에 덧붙이는 방식이므로, 안전하게 과거의 상태로 되돌리면서 팀원들과의 협업 흐름도 꼬이지 않게 보호할 수 있습니다.
