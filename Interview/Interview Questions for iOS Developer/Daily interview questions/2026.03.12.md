### [복습] 프로토콜 초기 구현(Protocol Extension)의 핵심 이점 세 가지를 설명해주세요.

첫째, 여러 타입에서 동일한 기능을 구현할 때 발생하는 **코드 중복을 제거**하여 유지보수성을 높입니다. 
둘째, 값 타입(구조체, 열거형)에도 상속 없이 **다형성을 부여**하여 필요한 기능만 유연하게 조합할 수 있게 해줍니다. 
셋째, 기본 구현이 제공되더라도 이를 채택하는 타입에서 필요에 따라 자유롭게 **오버라이딩(재정의)**하여 커스터마이징할 수 있다는 점입니다.

### [복습] Main Queue와 Global Queue의 차이점은 무엇이며, 네트워크 통신 후 UI를 업데이트할 때 주의할 점은 무엇인가요?

Main Queue는 단일 스레드에서 직렬로 동작하며 앱의 UI 업데이트를 전담합니다. 
Global Queue는 동시성 큐로 백그라운드에서 무거운 연산이나 네트워크 통신을 처리합니다. 
네트워크 통신은 Global Queue에서 비동기적으로 처리되지만, 통신 완료 후 화면의 텍스트나 이미지를 변경하는 등의 UI 업데이트 작업은 반드시 `DispatchQueue.main.async`를 통해 Main Queue로 보내서 처리해야 화면 멈춤이나 크래시를 방지할 수 있습니다.

### SwiftUI 기반의 앱에서 기존 UIKit의 뷰나 뷰 컨트롤러를 재사용하려면 어떤 프로토콜을 사용해야 하며, 데이터 동기화는 어떻게 처리하나요?

UIView를 띄우려면 `UIViewRepresentable`을, UIViewController를 띄우려면 `UIViewControllerRepresentable` 프로토콜을 사용합니다. 
필수 메서드인 `makeUIView(context:)`에서 UIKit 뷰를 초기화하고, `updateUIView(_:context:)`에서 SwiftUI의 상태 변화를 UIKit 뷰에 반영합니다. 
반대로 UIKit 뷰에서 발생한 이벤트나 데이터를 SwiftUI로 전달하려면 `Coordinator` 객체를 만들어 Delegate 패턴 등을 통해 데이터를 동기화합니다.

### 앱 내에서 Core ML 모델을 사용하여 디바이스 자체적(On-Device)으로 추론을 수행할 때, 앱의 용량이나 메모리 측면에서 고려해야 할 최적화 방안은 무엇인가요?

모델 파일 자체의 크기가 크면 앱 번들 용량이 커지므로, 가중치를 16비트나 8비트로 줄이는 양자화(Quantization)를 고려할 수 있습니다. 
또한, 실시간 추론 시 매 프레임마다 모델을 새로 로드하면 메모리와 배터리 소모가 심하므로, 모델 인스턴스를 한 번만 생성하여 재사용해야 합니다. 
필요하다면 앱 실행 시점이 아니라 사용자가 해당 기능을 쓰는 시점에 백그라운드에서 모델을 컴파일하고 로드하는 지연 로딩 방식을 적용할 수 있습니다.

### Git을 사용하여 협업할 때, Merge와 Rebase의 차이점은 무엇이며 각각 어떤 상황에서 사용하는 것이 좋나요?

`Merge`는 두 브랜치의 변경 사항을 하나로 합치면서 새로운 병합 커밋(Merge Commit)을 생성합니다. 
브랜치의 역사(History)가 그대로 보존되지만 커밋 트리가 복잡해질 수 있습니다. 
`Rebase`는 현재 브랜치의 커밋들을 대상 브랜치의 최신 커밋 위로 옮겨 붙여, 마치 일직선으로 작업한 것처럼 히스토리를 깔끔하게 만듭니다. 
개인 작업 브랜치를 최신 상태로 유지할 때는 Rebase가 좋고, 팀원들과 공유된 공용 브랜치(예: main)에 기능을 완전히 합칠 때는 히스토리 추적을 위해 Merge를 사용하는 것이 안전합니다

### 값 타입인 구조체(Struct) 내부에서 자신의 프로퍼티 값을 변경하는 메서드를 작성할 때 `mutating` 키워드를 붙이는 이유는 무엇인가요?

구조체는 값 타입이기 때문에, 인스턴스가 상수로 선언되면 내부 프로퍼티도 모두 상수가 되어 수정할 수 없습니다.
`mutating` 키워드는 해당 메서드가 구조체의 상태(프로퍼티)를 변경할 수 있다는 것을 컴파일러에게 명시적으로 알려주는 역할을 합니다. 
내부적으로는 `mutating` 메서드가 호출될 때, 기존 구조체를 변경하는 것이 아니라 변경된 값을 가진 새로운 구조체를 생성하여 현재 인스턴스에 다시 할당(Reassignment)하는 방식으로 동작합니다

### 의존성 주입(Dependency Injection, DI)을 구현하는 방법들에는 어떤 것들이 있으며, 이로 인해 얻는 가장 큰 이점은 무엇인가요?

객체를 내부에서 직접 생성하지 않고 외부에서 주입받는 방식으로, 
주로 이니셜라이저를 통한 주입(Initializer Injection), 
프로퍼티를 통한 주입(Property Injection), 
메서드를 통한 주입(Method Injection)을 사용합니다. 
가장 큰 이점은 객체 간의 결합도를 낮춰 유연한 코드를 만들 수 있다는 점이며, 특히 테스트 시 실제 네트워크나 데이터베이스 대신 가짜 객체(Mock)를 쉽게 주입할 수 있어 단위 테스트(Unit Test)를 작성하기가 매우 수월해집니다

### 앱 내에서 상태를 관리하기 위해 싱글톤(Singleton) 패턴을 남용할 경우 발생할 수 있는 안티 패턴(Anti-pattern)의 문제점들은 무엇인가요?

싱글톤은 전역에서 접근 가능하므로 어디서든 상태를 변경할 수 있어 데이터의 흐름을 추적하기 매우 어려워집니다. 
또한, 싱글톤 객체는 앱이 종료될 때까지 메모리에 남아있어 메모리 낭비의 원인이 될 수 있습니다. 
멀티 스레드 환경에서 동시에 접근할 때 데이터 경합(Data Race) 문제가 발생하기 쉬우며, 의존성이 강하게 결합되어 독립적인 단위 테스트를 수행하기 어렵다는 치명적인 단점이 있습니다

### **SwiftUI에서 뷰의 상태 변화에 애니메이션을 줄 때 `withAnimation` 블록을 사용하는 것과 `.animation()` 모디파이어를 사용하는 것의 차이는 무엇인가요?**

`withAnimation` 블록은 상태 값(State)을 변경하는 이벤트 자체를 감싸서, 그 상태 변화로 인해 영향을 받는 모든 뷰에 명시적으로 애니메이션을 적용합니다 (명령형 접근). 
반면 `.animation()` 모디파이어는 특정 뷰나 계층에 붙여서, 해당 뷰의 특정 값(예: 크기, 투명도)이 변경될 때마다 자동으로 애니메이션이 반응하도록 설정합니다 (선언형 접근). 
SwiftUI 최신 버전에서는 의도치 않은 애니메이션을 막기 위해 `.animation(_:value:)` 형태로 특정 값이 변할 때만 동작하도록 명시하는 것을 권장합니다.

### 앱을 테스트하다가 뷰 컨트롤러가 메모리에서 해제(deinit)되지 않고 남아있는 메모리 누수(Memory Leak)를 발견했습니다. 이를 디버깅하기 위해 어떤 도구나 방법을 사용하시나요?

가장 기본적인 방법은 뷰 컨트롤러의 `deinit` 블록에 `print` 문을 작성하여 화면을 닫았을 때 정상적으로 호출되는지 확인하는 것입니다. 
도구로는 Xcode의 `Debug Memory Graph`를 켜서 뷰 컨트롤러를 강하게 참조(Strong Reference)하고 있는 객체가 있는지 시각적으로 확인합니다. 
더 상세한 분석이 필요할 때는 Xcode의 `Instruments` 도구 중 `Leaks`나 `Allocations`를 사용하여 앱 실행 중 발생하는 메모리 누수 지점과 객체의 생명주기를 프로파일링합니다.
