### **1. [복습] iOS 앱의 생명 주기(App Lifecycle) 중 Background 상태와 Suspended 상태의 차이를 설명해주세요.** 

**답변:** Background 상태는 앱이 화면에 보이지 않지만 여전히 코드를 실행하고 있는 상태입니다. 음악 재생이나 위치 업데이트 등 특정 작업을 수행할 수 있습니다. 반면 Suspended 상태는 앱이 백그라운드에 있지만 코드를 실행하지 않고 메모리에만 머물러 있는 상태입니다. 시스템 메모리가 부족해지면 OS는 Suspended 상태의 앱을 예고 없이 종료시켜 메모리를 확보합니다.

### **2. [복습] Class와 Struct의 차이를 설명하고, 프로젝트에서 이 둘을 선택하는 기준을 말씀해주세요.** 

**답변:** Class는 참조 타입(Reference Type)으로 상속이 가능하고 메모리의 Heap 영역에 저장되며, Struct는 값 타입(Value Type)으로 상속이 불가능하고 주로 Stack 영역에 저장됩니다. 애플은 기본적으로 Struct 사용을 권장합니다. 상속이 필요하거나, Objective-C와 상호 운용성이 필요하거나, 데이터의 단일 인스턴스를 여러 곳에서 공유하며 상태를 제어해야 할 때만 Class를 선택하는 것이 좋습니다.

### **3. JSON 파일이나 UserDefaults 대신 SwiftData(또는 Core Data)를 사용하여 로컬 데이터를 관리할 때의 장점과 단점은 무엇인가요?** 

**답변:** SwiftData나 Core Data는 단순한 저장을 넘어 객체 간의 관계성(Relationship)을 관리하고, 조건부 검색 및 정렬에 최적화되어 있어 대용량 데이터를 다룰 때 성능이 뛰어납니다. 또한 데이터 모델의 버전 마이그레이션을 지원합니다. 단점으로는 JSON 기반 저장이나 UserDefaults에 비해 초기 설정이 복잡하고 학습 곡선이 높다는 점이 있습니다.

### **4. SwiftUI에서 `@State`와 `@Binding`의 차이점은 무엇이며, 데이터 흐름이 어떻게 이루어지는지 설명해주세요.** 

**답변:** `@State`는 뷰 내부에서 생성되고 관리되는 로컬 상태 값을 의미하며, 값이 변경되면 뷰를 다시 렌더링합니다. `@Binding`은 자체적으로 데이터를 소유하지 않고, 상위 뷰에서 선언된 `@State` 데이터에 대한 읽기 및 쓰기 권한을 전달받아 연결하는 역할을 합니다. 이를 통해 단일 진실 공급원(Single Source of Truth)을 유지하며 양방향 데이터 흐름을 만듭니다.

### **5. iOS 기기 내부에서 Core ML을 사용하여 머신러닝 모델을 구동(On-Device)할 때, 서버 통신 방식과 비교하여 얻을 수 있는 장점은 무엇인가요?

** **답변:** 기기 내부에서 처리하므로 사용자의 민감한 데이터가 외부 서버로 전송되지 않아 개인정보 보호에 유리합니다. 또한 네트워크 연결이 없는 오프라인 상태에서도 동작하며, 서버 통신에 따른 지연 시간(Latency)이 없어 실시간 데이터 처리에 매우 적합합니다. 서버 유지보수 비용이 절감되는 효과도 있습니다.

### **6. 메모리 관리 측면에서 `weak`와 `unowned`의 공통점과 차이점은 무엇이며, 언제 `unowned`를 사용하는 것이 적절한가요?

** **답변:** 둘 다 ARC의 참조 카운트를 증가시키지 않아 강한 순환 참조를 방지한다는 공통점이 있습니다. 차이점은 `weak`는 참조하던 인스턴스가 해제되면 자동으로 `nil`이 할당되므로 반드시 Optional 타입이어야 합니다. `unowned`는 참조하던 인스턴스가 해제되어도 `nil`이 할당되지 않으므로, 해제된 후 접근하면 런타임 에러가 발생합니다. 따라서 `unowned`는 참조하는 객체의 생명주기가 자신과 같거나 더 길다는 것이 확실할 때만 사용해야 합니다.

### **7. 앱이 백그라운드로 내려갔을 때도 타이머나 특정 연산이 멈추지 않고 동작하게 하려면 어떤 처리가 필요한가요?** 

**답변:** 일반적인 타이머나 연산은 앱이 Background를 거쳐 Suspended 상태가 되면 동작을 멈춥니다. 이를 유지하려면 `UIBackgroundTaskIdentifier`를 사용하여 시스템에 백그라운드 작업 시간을 요청해야 합니다. 하지만 이 시간도 제한적(약 30초 내외)이므로, 오디오 재생, 위치 업데이트 등 특정 백그라운드 모드를 활성화하거나, `BackgroundTasks` 프레임워크를 활용해 시스템이 적절한 시점에 작업을 스케줄링하도록 해야 합니다.

### **8. 기존의 탈출 클로저(@escaping)를 활용한 비동기 처리와 비교하여, Swift Concurrency(async/await)가 가지는 장점은 무엇인가요?** 

**답변:** 콜백 함수(클로저)를 중첩해서 사용할 때 발생하는 콜백 지옥(Callback Hell)을 해결하여 코드를 동기 코드처럼 위에서 아래로 직관적으로 읽을 수 있게 해줍니다. 또한 제어 흐름이 명확해져 에러 처리(`throws`, `try`)가 훨씬 자연스러워지며, 컴파일러가 비동기 흐름을 더 잘 이해하므로 데이터 경합(Data Race)과 같은 동시성 문제를 컴파일 단에서 방지하는 데 도움을 줍니다

### **9. SwiftUI의 생명주기 수식어인 `.onAppear`와 `.task`의 차이점은 무엇인가요?** 

**답변:** `.onAppear`는 뷰가 화면에 나타날 때 동기적인 액션을 수행합니다. 비동기 작업을 하려면 내부에서 `Task {}` 블록을 열어주어야 합니다. 반면 `.task`는 뷰가 나타날 때 비동기 작업을 바로 실행할 수 있도록 비동기 컨텍스트를 제공합니다. 가장 큰 차이점은 `.task`로 실행된 비동기 작업은 해당 뷰가 화면에서 사라질 때 자동으로 취소(Cancel) 처리가 된다는 점입니다.

### **10. 의존성 주입(Dependency Injection, DI)의 개념과, 이를 iOS 프로젝트에 적용했을 때 얻을 수 있는 이점을 설명해주세요.** 

**답변:** 의존성 주입은 내부에서 객체를 직접 생성하는 것이 아니라, 외부에서 생성된 객체를 주입받는 디자인 패턴입니다. 객체 간의 결합도를 낮추고 유연성을 높이는 것이 핵심입니다. 이를 통해 코드를 수정하지 않고도 동작을 변경할 수 있으며, 특히 테스트 더블(Mock 객체 등)을 쉽게 주입할 수 있어 유닛 테스트(Unit Test)를 작성하기가 매우 수월해집니다.
