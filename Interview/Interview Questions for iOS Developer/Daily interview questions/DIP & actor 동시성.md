### 1. [복습] 의존성 역전 원칙(DIP)과 의존성 주입(DI)의 관계를 '추상화'라는 키워드를 사용하여 설명해주세요.

의존성 역전 원칙(DIP)은 상위 모듈과 하위 모듈이 서로 직접 의존하지 않고, 둘 다 '추상화(프로토콜)'에 의존해야 한다는 설계 원칙입니다. 이 원칙을 실제 코드로 구현하기 위한 방법론이 바로 의존성 주입(DI)입니다. 즉, 상위 모듈(뷰모델)이 추상화된 프로토콜만 바라보고 있을 때, 외부에서 그 프로토콜을 준수하는 구체적인 객체(하위 모듈)를 생성하여 주입(DI)해 줌으로써 코드를 수정하지 않고도 부품을 교체할 수 있는 유연한 구조를 완성합니다.

### 2. **[복습] 동시성 처리를 위해 `actor`를 사용할 때, 외부에서 `actor`의 내부 프로퍼티나 메서드에 접근하려면 왜 반드시 `await` 키워드를 써야 하나요?**

`actor`는 멀티 스레드 환경에서 데이터 경합을 막기 위해 한 번에 하나의 작업만 내부 상태에 접근하도록 자체적인 격리 환경(우편함 큐)을 가지고 있습니다. 외부에서 `actor`에 접근하려고 할 때 이미 다른 스레드의 작업이 처리 중이라면, 내 순서가 올 때까지 기다려야 합니다. `await` 키워드는 시스템에게 "이 작업은 `actor`의 상태 접근 권한을 얻을 때까지 잠시 대기(Suspend)하겠다"고 명시적으로 알려주는 역할을 하기 때문입니다.

### 3. [Swift] SwiftUI에서 뷰를 반환할 때 주로 `some View`라고 작성합니다. 여기서 `some` 키워드(Opaque Type)의 역할은 무엇인가요?

`some` 키워드는 '불투명 타입(Opaque Type)'을 나타냅니다. 함수를 호출하는 외부(개발자)에게는 정확히 어떤 뷰(예: `VStack`, `Text`)가 반환되는지 숨기고 오직 `View` 프로토콜을 따른다는 것만 알려줍니다. 하지만, 일반적인 프로토콜 반환과 달리 **컴파일러는 반환되는 실제 구체적인 타입을 하나로 명확하게 알 수 있습니다.** 이를 통해 제네릭의 유연성을 챙기면서도 컴파일 시점에 타입이 확정되므로 성능 최적화가 가능해집니다.

### 4. [Memory] 네트워크 통신 완료 클로저(Completion Handler) 내부에서 `[unowned self]` 대신 `[weak self]`를 권장하는 이유는 무엇인가요?

네트워크 요청은 비동기적으로 처리되므로 응답이 도착하기 전에 사용자가 화면을 뒤로가기 하여 뷰 컨트롤러(`self`)가 메모리에서 먼저 해제될 수 있습니다. 이때 `[unowned self]`를 사용했다면 해제된 메모리에 접근하려다 앱이 크래시(강제 종료)됩니다. 반면 `[weak self]`를 사용하면 인스턴스가 해제되었을 때 자동으로 `nil`이 되므로, 옵셔널 바인딩(`guard let self = self else { return }`)을 통해 안전하게 클로저의 실행을 종료시킬 수 있기 때문입니다.

### 5. [Architecture] MVVM 패턴에서 View와 ViewModel 간의 데이터 바인딩(Binding)은 어떻게 구현하나요? (UIKit과 SwiftUI를 비교해서 설명해주세요)

UIKit 환경에서는 기본적으로 바인딩 기능이 없기 때문에, 클로저를 활용하거나 KVO, 혹은 Combine/RxSwift와 같은 리액티브 프레임워크를 도입하여 뷰가 뷰모델의 상태 변화를 구독(Subscribe)하도록 개발자가 직접 구현해야 합니다. 반면 SwiftUI 환경에서는 프레임워크 자체가 선언형 및 데이터 주도적(Data-driven)이므로, `@StateObject`, `@ObservedObject`, `@Published` 등의 프로퍼티 래퍼를 사용하기만 하면 뷰와 뷰모델 간의 강력한 양방향 바인딩이 자동으로 이루어집니다.

### [Core ML] `.mlmodel` 파일을 Xcode 프로젝트에 추가했을 때 내부적으로 어떤 일이 일어나며, 이 모델을 코드에서 어떻게 활용하나요?

Xcode 프로젝트에 `.mlmodel` 파일을 추가하면, Xcode는 컴파일 시점에 해당 모델의 입력(Input)과 출력(Output) 규격에 맞춘 Swift 래퍼 클래스(Wrapper Class)를 자동으로 생성해 줍니다. 개발자는 이 자동 생성된 클래스의 인스턴스를 만들고, 파라미터로 필요한 데이터(예: 이미지나 센서 데이터)를 넣은 뒤 `prediction(input:)` 메서드를 호출하기만 하면 매우 직관적이고 쉽게 추론(Inference) 결과를 받아볼 수 있습니다.

### [SwiftUI] SwiftUI에서 `@EnvironmentObject`는 어떤 상황에서 사용하며, 의존성 주입 관점에서 볼 때 어떤 특징이 있나요?

뷰 계층 구조가 매우 깊을 때, 하위 뷰에서 필요한 데이터를 상위 뷰부터 하나하나 매번 생성자로 전달(Prop Drilling)하는 불편함을 해결하기 위해 사용합니다. 최상위 뷰에 `.environmentObject()`로 객체를 주입해두면, 하위 뷰 어디서든 `@EnvironmentObject`를 선언하여 전역 변수처럼 쉽게 꺼내 쓸 수 있습니다. 이는 뷰 트리 전체에 데이터를 공급하는 '암시적인 의존성 주입(Implicit DI)' 방식으로 볼 수 있습니다.

### [Architecture] SOLID 원칙 중 단일 책임 원칙(SRP, Single Responsibility Principle)을 iOS 개발에 어떻게 적용할 수 있나요?

단일 책임 원칙은 "하나의 클래스나 모듈은 변경되어야 할 이유가 단 하나뿐이어야 한다"는 원칙입니다. iOS에서는 하나의 뷰 컨트롤러(Massive VC)가 화면도 그리고 통신도 하고 데이터도 다루는 것을 지양해야 합니다. 네트워크 통신 로직은 `NetworkManager`로, 화면 그리기는 `View`로, 비즈니스 로직과 데이터 포맷팅은 `ViewModel`로 철저히 분리하여 각 객체가 오직 한 가지 역할만 수행하도록 설계하는 것이 대표적인 SRP 적용 사례입니다.

### [Swift] Swift의 `Error` 프로토콜을 활용하여 커스텀 에러 열거형(Enum)을 만들 때의 장점은 무엇인가요?

앱 내에서 발생할 수 있는 에러 상황들(예: `networkTimeout`, `invalidToken`)을 열거형의 케이스로 명확하게 정의할 수 있습니다. `do-catch` 구문을 사용할 때 패턴 매칭을 통해 각 에러 케이스별로 세밀한 분기 처리가 가능해집니다. 또한, 열거형의 연관 값(Associated Value)을 활용하면 에러가 발생한 구체적인 이유나 HTTP 상태 코드 같은 추가적인 문맥(Context)을 에러 객체에 담아 호출자에게 전달할 수 있다는 큰 장점이 있습니다.

### [Git] 현재 브랜치에서 기능 개발을 하고 있는데 수정이 완료되지 않았습니다. 이때 다른 브랜치로 급하게 넘어가서 버그를 고쳐야 하는 상황이라면 커밋 대신 어떤 명령어를 사용해야 하나요?

`git stash` 명령어를 사용합니다. 현재 워킹 디렉토리에서 작업 중이던 커밋하지 않은 변경 사항들을 임시 저장소(스택)에 보관해 두고, 현재 브랜치를 깔끔한 상태(HEAD)로 되돌려 줍니다. 덕분에 충돌 없이 다른 브랜치로 이동해 버그를 수정할 수 있습니다. 작업을 마친 후 다시 원래 브랜치로 돌아와서 `git stash pop` 명령어를 실행하면, 임시 저장해 두었던 작업 내역을 꺼내와서 개발을 그대로 이어나갈 수 있습니다.

# 애플이 Swift 5.1에서 SwiftUI와 함께 `some` 키워드를 도입한 이유
### 1. `some`이 없다면? (구체적인 타입을 명시해야 하는 지옥)

SwiftUI의 뷰들은 레고 블록처럼 겹겹이 쌓이는 구조입니다. 만약 `some` 키워드가 없다면, 우리는 반환되는 뷰의 **정확한 타입**을 모두 적어주어야 합니다.

```Swift
// ❌ some 키워드가 없다면 이렇게 작성해야 합니다.
struct MyView: View {
    // 뷰가 조금만 복잡해져도 반환 타입이 끔찍하게 길어집니다.
    var body: VStack<TupleView<(Text, Button<Text>)>> {
        VStack {
            Text("안녕하세요")
            Button("확인") { }
        }
    }
}
```

버튼 하나, 텍스트 하나만 추가해도 반환 타입(`VStack<TupleView<...>>`)이 눈덩이처럼 불어나기 때문에 개발자가 이를 매번 정확히 명시하는 것은 사실상 불가능합니다.

### 2. 그렇다면 그냥 `View` 프로토콜만 반환하면 안 되나요?

"어차피 다 View 프로토콜을 따르니까, `-> View` 라고만 적으면 안 되나?"라는 의문이 생길 수 있습니다.

```Swift
// ❌ 컴파일 에러 발생
struct MyView: View {
    var body: View { // 에러: Protocol 'View' can only be used as a generic constraint
        Text("안녕하세요")
    }
}
```

Swift의 `View` 프로토콜 내부에는 `associatedtype Body`라는 제네릭 요소가 들어있습니다. Swift 컴파일러는 메모리를 미리 할당하기 위해 컴파일 시점에 반환되는 객체의 **정확한 크기(메모리 사이즈)**를 알아야 하는데, 뭉뚱그려서 `View`라고만 해버리면 그 안에 텍스트가 들어있는지, 무거운 이미지가 들어있는지 알 수 없어 메모리 할당을 할 수 없습니다.

### 3. 해결책: `some View` (불투명 타입, Opaque Type)의 마법

이 두 가지 문제를 완벽하게 해결하기 위해 등장한 것이 `some` 키워드입니다.

```Swift
// ✅ 정상 동작 (우리가 항상 쓰는 방식)
struct MyView: View {
    var body: some View {
        VStack {
            Text("안녕하세요")
            Button("확인") { }
        }
    }
}
```

- **개발자 입장 (외부):** "이 함수가 정확히 무슨 타입(`VStack<TupleView...`)을 반환하는지는 복잡하니까 안 적을게. 그냥 `View` 프로토콜을 준수하는 **어떤(some) 뷰**라는 것만 알아둬!"
    
- **컴파일러 입장 (내부):** "알겠어. 네가 안 적어줘도 내가 컴파일할 때 코드(VStack 안의 Text와 Button)를 싹 훑어보고 **정확한 실제 타입과 메모리 크기를 스스로 유추해서 확정** 지을게."

즉, 개발자의 코드는 깔끔하게 만들어주면서도, 컴파일러는 정확한 타입을 추론할 수 있게 하여 **앱의 렌더링 성능을 극대화**하는 일석이조의 효과를 가져옵니다.

### 🎙️ 면접 꼬리 질문 대비: `some` 사용 시 주의할 점

면접관이 _"그럼 `some View`를 사용할 때 컴파일 에러가 발생하는 경우는 언제인가요?"_ 라고 물어볼 수 있습니다.

**답변 포인트:** "`some` 키워드는 컴파일러가 단 1개의 명확한 타입을 추론할 수 있어야 하므로, 조건문(`if-else`)을 통해 서로 완전히 다른 타입을 반환하려고 하면 에러가 발생합니다."

```Swift
// ❌ 컴파일 에러: 컴파일러가 하나의 타입으로 확정 지을 수 없음
var body: some View {
    if isTrue {
        return Text("참") // Text 타입
    } else {
        return Image("거짓") // Image 타입
    }
}
```

_(참고: SwiftUI에서는 이런 경우 상위에 `@ViewBuilder` 속성이 적용된 `VStack`이나 `Group`으로 묶거나, 뷰를 지워버리는 `AnyView`를 사용해 해결하지만, 성능상 `AnyView`는 지양하는 것이 좋습니다.)_
