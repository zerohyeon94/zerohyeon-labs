### **1. [복습] 클로저(Closure)에서 캡처 리스트(Capture List)를 사용하는 이유와 강한 순환 참조를 방지하는 방법에 대해 설명해주세요.** 

**답변:** 클로저는 주변 환경의 변수나 상수를 캡처할 때 기본적으로 강한 참조(Strong Reference)를 사용합니다. 클래스의 인스턴스가 클로저를 프로퍼티로 가지고 있고, 그 클로저 내부에서 `self`를 참조하게 되면 서로가 서로를 강하게 참조하는 순환 참조가 발생해 메모리 누수가 생깁니다. 이를 방지하기 위해 캡처 리스트에 `[weak self]` 또는 `[unowned self]`를 명시하여 참조 카운트가 올라가지 않도록 처리해야 합니다.

### **2. [복습/심화] SwiftUI와 UIKit을 하나의 프로젝트에서 함께 사용해야 할 때, 두 프레임워크를 어떻게 연동할 수 있나요?** 

**답변:** UIKit 기반 프로젝트에 SwiftUI 뷰를 띄우려면 `UIHostingController`를 사용하여 SwiftUI 뷰를 감싸서 UIKit의 뷰 계층에 추가합니다. 반대로 SwiftUI 기반 프로젝트에서 기존 UIKit 뷰나 뷰 컨트롤러를 사용하려면 `UIViewRepresentable` 또는 `UIViewControllerRepresentable` 프로토콜을 채택한 래퍼(Wrapper) 구조체를 만들어 SwiftUI 환경에 통합할 수 있습니다.

### **3. Core ML 모델을 활용한 실시간 추론이나 복잡한 데이터 연산 등 무거운 작업을 메인 스레드에서 실행하면 어떤 현상이 발생하며, 어떻게 해결해야 하나요?** 

**답변:** 메인 스레드는 UI 업데이트와 사용자의 터치 이벤트를 처리하는 곳입니다. 여기서 무거운 연산을 수행하면 작업이 끝날 때까지 메인 스레드가 차단(Block)되어 앱의 화면이 멈추거나 버벅거리는 프리즈(Freeze) 현상이 발생합니다. 이를 해결하려면 무거운 작업은 `DispatchQueue.global()`이나 Swift Concurrency의 `Task`를 활용해 백그라운드 스레드로 넘기고, 연산이 끝난 후 UI를 갱신할 때만 다시 메인 스레드(`DispatchQueue.main`)로 돌아와 처리해야 합니다.

### **4. Swift는 객체지향 프로그래밍(OOP) 언어이면서 동시에 프로토콜 지향 프로그래밍(POP) 언어입니다. POP의 장점은 무엇인가요?**

**답변:** 객체지향(OOP)에서는 다형성을 위해 주로 클래스의 '상속'을 사용하지만, 다중 상속이 불가능하고 불필요한 기능까지 물려받아야 하는 단점이 있습니다. 프로토콜 지향 프로그래밍(POP)은 프로토콜을 통해 수평적인 기능 확장이 가능하며, 값 타입(Struct, Enum)에도 다형성을 부여할 수 있습니다. 또한, 프로토콜 초기 구현(Protocol Extension)을 통해 코드 중복을 줄이고 모듈화와 테스트 용이성을 크게 높일 수 있습니다.

### **5. 테이블 뷰(UITableView)나 컬렉션 뷰(UICollectionView)에서 셀(Cell)을 재사용하는 원리와 `prepareForReuse()` 메서드의 역할에 대해 설명해주세요.** 

**답변:** 리스트에 수백 개의 데이터가 있더라도 화면에 보이는 만큼의 셀만 메모리에 할당하고, 스크롤 시 화면 밖으로 벗어난 셀은 '재사용 큐(Reuse Queue)'에 넣습니다. 그리고 새로 화면에 나타날 데이터는 큐에 있는 셀을 꺼내어 데이터만 덮어씌워 보여줌으로써 메모리를 최적화합니다. `prepareForReuse()`는 셀이 재사용되기 직전에 호출되는 메서드로, 이전 셀에 남아있던 이미지나 텍스트, 상태(State) 값 등을 초기화하여 데이터가 겹쳐 보이는 UI 버그를 방지하는 역할을 합니다.

### **6. Swift에서 제네릭(Generics)을 사용하는 이유는 무엇인가요?** 

**답변:** 제네릭은 타입에 의존하지 않고 범용적이고 재사용 가능한 코드를 작성하기 위해 사용합니다. 함수나 타입을 정의할 때 실제 타입을 명시하는 대신 타입 매개변수(Type Parameter, 예: `<T>`)를 사용하여, 호출되는 시점에 실제 타입이 결정되도록 합니다. 이를 통해 코드의 중복을 막고, 타입 안정성(Type Safety)을 유지하면서 유연한 프로그래밍이 가능해집니다. Array나 Dictionary 같은 컬렉션 타입이 제네릭으로 구현된 대표적인 예입니다.

### **7. 사용자의 자동 로그인 정보나 토큰 같은 민감한 데이터를 저장할 때, UserDefaults 대신 KeyChain을 사용해야 하는 이유는 무엇인가요?** 

**답변:** `UserDefaults`는 데이터를 평문(Plain Text) 형태의 plist 파일로 저장하므로, 탈옥된 기기나 특정 툴을 통해 쉽게 내용이 노출될 위험이 있어 단순한 사용자 설정값을 저장하는 데 적합합니다. 반면 `KeyChain`은 데이터를 암호화하여 OS 내부의 안전한 공간에 저장하며, 앱을 삭제해도 데이터가 유지되도록 설정할 수 있어 보안이 중요한 인증 토큰이나 비밀번호 저장에 필수적입니다.

### **8. Swift에서 에러 처리를 위해 사용하는 `do-catch` 구문과 `Result` 타입의 차이점과 각각의 장단점을 설명해주세요.** 

**답변:** `do-catch`는 에러를 던질 수 있는(`throws`) 함수를 동기적인 흐름에서 직관적으로 처리할 때 유용하며, 코드가 깔끔해집니다. 
하지만 비동기 작업의 완료 클로저 내부에서는 에러를 던지기 어렵다는 단점이 있습니다. 
`Result` 타입은 성공(`success`)과 실패(`failure`)를 명확하게 캡슐화한 열거형으로, 비동기 작업의 결과를 반환하거나 에러를 값처럼 전달해야 할 때 매우 유용하게 쓰입니다.

### **9. JSON 데이터를 파싱하기 위해 사용하는 `Codable` 프로토콜은 내부적으로 어떻게 구성되어 있나요?** 

**답변:** `Codable`은 외부 표현(JSON 등)을 Swift 모델로 변환하는 `Decodable` 프로토콜과, Swift 모델을 외부 표현으로 변환하는 `Encodable` 프로토콜을 합친 타입 에일리어스(Type Alias)입니다. 속성 이름과 JSON의 키 값이 다를 경우 `CodingKeys` 열거형을 정의하여 매핑해 줄 수 있습니다.

### **10. 앱 실행 중 시스템으로부터 메모리 부족 경고(Memory Warning)를 받았을 때, 앱 내부에서는 어떤 일이 발생하며 어떻게 대응해야 하나요?** 

**답변:** 시스템 메모리가 부족해지면 iOS는 실행 중인 앱들에게 `didReceiveMemoryWarning()` 메서드나 `UIApplicationDidReceiveMemoryWarningNotification` 알림을 보냅니다. 이때 앱은 현재 화면에 보이지 않는 뷰, 다시 로드할 수 있는 이미지 캐시나 대용량 데이터 파일 등 불필요한 메모리를 즉각적으로 해제하여 앱이 강제 종료(OOM, Out of Memory)되는 것을 막아야 합니다.
