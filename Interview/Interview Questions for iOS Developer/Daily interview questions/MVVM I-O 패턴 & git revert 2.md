### [복습] MVVM 패턴을 구현할 때, 뷰모델에 Input/Output (I/O) 패턴을 적용하는 이유는 무엇인가요?

뷰에서 발생하는 사용자 액션(Input)과 뷰에 그려질 데이터 상태(Output)를 명확히 분리하여, 뷰와 뷰모델 간의 데이터 흐름을 단방향으로 철저하게 통제하기 위함입니다. 이를 통해 뷰모델의 역할이 규격화되어 코드 가독성이 높아지고, Input에 따른 Output 결과만 검증하면 되므로 유닛 테스트(Unit Test)를 작성하기가 매우 수월해집니다.

### [복습] 이미 원격 저장소에 푸시(Push)된 커밋을 이전 상태로 되돌려야 할 때, `git reset` 대신 `git revert`를 사용해야 하는 이유는 무엇인가요?

`git reset`은 커밋 히스토리 자체를 삭제하고 시간을 되돌리기 때문에, 이미 원격 저장소를 공유하고 있는 다른 팀원들의 로컬 저장소와 심각한 충돌을 유발합니다. 반면 `git revert`는 이전 커밋의 변경 사항을 역으로 취소하는 '새로운 커밋'을 히스토리에 안전하게 덧붙이는 방식이므로, 팀원들과의 협업 흐름을 깨지 않고 과거 상태로 롤백할 수 있습니다.

### [SwiftUI] 앱의 설정값이나 간단한 데이터를 저장할 때 사용하는 `@AppStorage`는 내부적으로 어떻게 동작하며, `UserDefaults`와 어떤 차이가 있나요?

`@AppStorage`는 내부적으로 `UserDefaults`를 래핑(Wrapping)한 SwiftUI의 프로퍼티 래퍼입니다. 기존 `UserDefaults`는 값이 변경되었을 때 UI를 업데이트하려면 개발자가 직접 Notification이나 KVO, Combine 등을 연동해야 했습니다. 하지만 `@AppStorage`는 값이 변경되는 즉시 뷰를 자동으로 다시 그리도록(Re-render) 트리거해주어 선언형 UI 프레임워크에 최적화되어 있습니다.

### [Core ML / Vision] 카메라 프레임(CMSampleBuffer)을 Core ML 모델이나 Vision 프레임워크에 전달할 때, 기기의 회전(Orientation) 방향을 따로 처리해 주어야 하는 이유는 무엇인가요?

기기 내부의 카메라 센서는 물리적으로 고정되어 있기 때문에, 렌즈를 통해 들어온 원본 이미지 버퍼의 방향은 사용자가 기기를 들고 있는 화면 방향(Portrait, Landscape 등)과 일치하지 않을 수 있습니다. 따라서 모델이 얼굴이나 사물을 똑바로 인식할 수 있도록, 기기의 현재 방향이나 카메라 커넥션 방향을 확인하여 `CGImagePropertyOrientation` 값으로 변환한 뒤 프레임과 함께 모델에 전달해 주어야 정확한 추론 결과를 얻을 수 있습니다.

### [Combine / Memory] Combine에서 `sink`를 통해 구독(Subscribe)을 시작하면 `AnyCancellable` 타입이 반환됩니다. 이것의 역할은 무엇이며 어떻게 메모리를 관리해야 하나요?

`AnyCancellable`은 해당 구독 상태를 유지하고 필요 시 취소할 수 있는 토큰(Token) 역할을 합니다. 만약 이를 변수에 저장하지 않으면 함수 종료 시 구독이 즉시 해제되어 이벤트 전달이 끊깁니다. 따라서 뷰모델 내부에 `Set<AnyCancellable>` 타입의 저장소를 만들고 `.store(in:)` 메서드로 담아두어, 뷰모델이 메모리에서 해제될 때(deinit) 구독들도 함께 안전하게 해제되도록 객체의 생명주기와 동기화해야 합니다.

### [SwiftUI vs UIKit] UIKit의 `UINavigationController`와 SwiftUI의 `NavigationStack`을 통한 화면 이동 로직의 가장 큰 구조적 차이는 무엇인가요?

UIKit은 개발자가 직접 `ViewController` 인스턴스를 생성하여 스택에 밀어 넣거나 빼는(Push/Pop) **명령형(Imperative)** 방식입니다. 반면 최신 SwiftUI의 `NavigationStack`은 이동할 화면의 '데이터(상태값)'를 배열(path)에 추가하거나 제거하면, 시스템이 그 데이터에 매핑된 화면을 자동으로 그려주는 **데이터 주도적인 선언형(Declarative)** 방식입니다.

### [Swift / Memory] `Timer`를 사용할 때 발생하기 쉬운 강한 순환 참조(Retain Cycle) 문제는 무엇이며, 어떻게 해결할 수 있나요?

`Timer`는 실행을 위해 타겟 객체(보통 뷰 컨트롤러의 `self`)를 강하게 참조합니다. 타이머가 명시적으로 멈추지 않으면 뷰 컨트롤러를 닫아도 메모리에서 해제되지 않아 누수가 발생합니다. 이를 해결하려면 뷰가 사라질 때(`viewWillDisappear` 등) 반드시 `timer.invalidate()`를 호출하여 타이머를 중지시키고 강한 참조를 끊어주거나, iOS 10 이후 추가된 블록(클로저) 기반의 Timer API를 사용하여 내부에서 `[weak self]`를 캡처하도록 구현해야 합니다.

### [Network / Security] 앱의 보안을 강화하기 위해 적용하는 SSL Pinning (Certificate Pinning) 기법은 무엇이며, 어떤 해킹 공격을 방지할 수 있나요?

클라이언트(앱) 내부에 통신할 서버의 인증서나 공개키(Public Key)를 미리 내장해 두고, 네트워크 통신 시 서버가 내려주는 인증서가 내장된 것과 일치하는지 엄격하게 검증하는 기법입니다. 이를 통해 해커가 공공 와이파이 등에서 기기를 조작해 가짜 인증서를 심고 패킷을 가로채는 '중간자 공격(MITM, Man-in-the-Middle Attack)'을 효과적으로 방지할 수 있습니다.

### [UX / Performance] 네트워크 로딩 중에 사용자에게 단순히 빙글빙글 도는 스피너(Activity Indicator) 대신, 스켈레톤 UI(Skeleton UI)를 보여주는 것이 UX 측면에서 어떤 장점이 있나요?

스켈레톤 UI는 콘텐츠가 표시될 대략적인 레이아웃 구조를 먼저 보여주어 사용자의 심리적인 체감 대기 시간을 줄여줍니다. 또한, 로딩이 완료된 후 데이터가 화면에 뿌려질 때 기존 UI 요소들이 갑자기 밀려나는 레이아웃 시프트(Layout Shift) 현상을 방지하여, 화면 전환이 훨씬 부드럽고 완성도 높은 앱이라는 인상을 줍니다.

### [CI/CD] iOS 프로젝트에서 GitHub Actions나 Fastlane을 활용한 CI/CD(지속적 통합/지속적 배포) 파이프라인을 구축하면 어떤 실무적인 이점이 있나요?

팀원들이 코드를 푸시하거나 PR을 올릴 때마다 자동으로 빌드와 유닛 테스트를 수행하여 버그를 조기에 발견할 수 있습니다(CI). 또한, 복잡하고 번거로운 인증서 매칭, TestFlight 업로드, App Store 심사 제출 등의 일련의 배포 과정을 명령어 하나로 자동화하여 개발자가 단순 반복 작업 대신 핵심 기능 개발에만 집중할 수 있게 해줍니다(CD).
