### [Daily iOS Chat #7 - Bonus] Static Dispatch의 진짜 맛: Inlining

**1. "코드 주소가 정해진다" (O)** 네, 컴파일 타임에 **"이 함수는 0x1004 번지에 있다"**라고 딱 정해집니다. 런타임에 "어? 자식 클래스가 오버라이딩했나?" 하고 테이블을 뒤적거릴 필요가 없죠.

**2. "순서가 정해진다" → "코드를 아예 복사해버린다 (Inlining)"** 사용자님이 말씀하신 "순서가 정해진다"는 것이 아마 **"함수 호출 흐름이 고정된다"**는 의미로 말씀하신 것 같은데요, 여기서 컴파일러는 한 발 더 나아갑니다.

**Dynamic Dispatch**일 때는 함수가 바뀔 수도 있으니 함부로 손을 못 댑니다. 하지만 **Static Dispatch(final)**가 되면 컴파일러는 이런 생각을 합니다.

> _"어차피 무조건 저 0x1004번지 함수가 실행될 거잖아? 굳이 점프(JMP/CALL)해서 갔다 올 필요가 있나? 그냥 그 함수 내용을 여기에 **복붙(Copy-Paste)** 해버리자!"_

이것을 **함수 인라이닝(Inlining)**이라고 합니다.

### 🖥️ 펌웨어 개발자의 시각 (Assembly)

이해를 돕기 위해 의사 코드(Pseudo Assembly)로 비교해볼게요.

#### 상황: `calculate()`라는 함수를 호출할 때

**A. Dynamic Dispatch (final 아님, 일반 class)**

```
; 1. v-table에서 주소 찾기 (시간 걸림)
MOV R1, [Object + Offset] 
; 2. 찾은 주소로 이동 (Context Switching 비용 발생 가능)
CALL R1 
; 3. 다시 돌아오기
RET
```

**B. Static Dispatch (final class)**

```
; 1. 그냥 주소로 바로 점프 (빠름!)
CALL 0x1004 
```

**C. Static Dispatch + Inlining (최적화 끝판왕)**

```
; 컴파일러: "어차피 더하기 하는 함수네? 점프도 하지 마."
ADD R1, R2  ; <-- 함수 내용이 그냥 호출하는 곳에 박혀버림
```

-> 이렇게 되면 함수를 호출하는 오버헤드(스택 프레임 생성, 레지스터 저장 등)가 **0**이 됩니다.

### 💡 오늘의 결론

사용자님의 이해를 정리하면:

1. **어떤 메서드인지 정해짐:** 맞습니다. (컴파일 타임 확정)
    
2. **주소가 정해짐:** 맞습니다. (Hard-coded Address)
    
3. **순서가 정해짐:** 맞습니다. 더 나아가 아예 **호출 과정을 생략(Inlining)**하고 코드를 합쳐버릴 수도 있습니다.

**한 줄 요약:**

> "`final`을 붙이면 컴파일러가 **'의심'**을 하지 않고 **'확신'**을 가지고 코드를 최적화(단축)합니다."
