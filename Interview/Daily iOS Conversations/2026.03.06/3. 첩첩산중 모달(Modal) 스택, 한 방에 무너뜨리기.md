**🗣️ 가볍게 시작하기 (Simple)**

모달로 화면을 계속 덮어씌우는 상황(A ➔ B ➔ C ➔ D ➔ E ➔ F)을 **Deep Modal Stack**이라고 부릅니다. 유저가 가입 프로세스나 복잡한 결제 단계를 거칠 때 종종 발생하죠. 이때 끝단인 F에서 첫 화면 A로 한 번에 돌아가는 마법은 두 프레임워크 모두 존재합니다.

---

**🤿 딥하게 들어가기 (Deep: UIKit의 진실)**

사용자님이 상상하신 *"F dismiss, E dismiss... 5번 호출해야 한다"*는 아주 흔한 오해입니다!

UIKit은 화면들이 꼬리를 물고 있는 구조를 모두 기억하고 있습니다. 이때 **'이 모달 파티를 처음 주최한 사람(Root)'**에게 "이제 파티 끝내!"라고 명령하면, 중간에 있는 화면들은 애니메이션 한 번에 싹 다 치워집니다.

**💻 UIKit 실전 해결책:** F 화면에서 버튼을 눌렀을 때, 최상단 부모(A)를 찾아서 딱 한 번만 `dismiss`를 부르면 됩니다.

```Swift
// F 화면의 뷰 컨트롤러 안에서
self.view.window?.rootViewController?.dismiss(animated: true, completion: nil)

// 또는 A 화면을 참조할 수 있다면
// viewControllerA.dismiss(animated: true)
```

이렇게 하면 B, C, D, E, F가 메모리에서 한꺼번에 우르르 해제되면서 깔끔하게 A 화면이 나타납니다.

---

**🤿 더 딥하게 (Deep+: SwiftUI의 진실)**

SwiftUI도 마찬가지로 B가 C를 띄우고, C가 D를 띄우는 식으로 꼬리를 뭅니다. (애초에 SwiftUI는 한 뷰(A)에서 동시에 여러 개의 `.fullScreenCover`를 병렬로 띄우는 것을 허용하지 않습니다. 에러가 나거나 뷰가 꼬입니다.)

따라서 F 화면 내부에서 단순히 `@Environment(\.dismiss)`를 호출하면 **E 화면으로 돌아갑니다.** A로 가지 않습니다!

**💻 SwiftUI 실전 해결책: "상태(State) 스위치 끄기"**

SwiftUI에서 F ➔ A로 한 방에 가는 방법은 **[Daily iOS Chat #19]**에서 배웠던 데이터 흐름을 이용하는 것입니다. A 화면에 있는 **'스위치(Boolean)'** 하나만 끄면 모달 스택 전체가 증발합니다.

1. **A 화면 (스위치 소유자):**
    
    ```Swift
    @State private var isFlowActive = false
    
    // 버튼을 누르면 B를 띄움
    .fullScreenCover(isPresented: $isFlowActive) {
        ViewB(isFlowActive: $isFlowActive) 
    }
    ```
    
2. **B ➔ C ➔ D ➔ E ➔ F 화면 (스위치 릴레이):** 이 `@Binding` 값을 F까지 쭉쭉 전달합니다. (혹은 `@EnvironmentObject`로 띄워둡니다.)
    
3. **F 화면 (스위치 끄기):**
    
    ```Swift
    @Binding var isFlowActive: Bool
    
    Button("처음으로 돌아가기") {
        // 이 값을 false로 바꾸는 순간!
        isFlowActive = false 
    }
    ```

`isFlowActive`가 `false`가 되는 순간, SwiftUI는 **"어? 최상단 A 화면이 더 이상 모달을 안 띄운다고 상태가 바뀌었네?"**라고 판단하고, 그 아래에 주렁주렁 매달려 있던 B, C, D, E, F를 단숨에 파괴(Deallocate)해 버립니다.

---

**💡 오늘의 요약**

> **UIKit**은 계층의 뿌리(Root)를 찾아가 **물리적으로 닫으라고 명령(명령형)**하는 방식이고, **SwiftUI**는 계층을 열어주었던 **상태(State) 값을 초기화시켜 알아서 닫히게(선언형)** 하는 방식입니다.
