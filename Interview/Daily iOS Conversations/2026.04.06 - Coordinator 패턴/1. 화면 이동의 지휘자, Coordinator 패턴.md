**🗣️ 가볍게 시작하기 (Simple): "관광객과 패키지 가이드"**

- **Coordinator가 없을 때:** 관광객(ViewController)이 다음 목적지의 지도, 버스 티켓, 호텔 예약(ViewModel 조립)까지 직접 다 해야 합니다. 관광객은 관광(화면 그리기)에만 집중할 수가 없습니다.
    
- **Coordinator가 있을 때:** 관광객은 아무것도 몰라도 됩니다! 패키지 가이드(Coordinator)가 **"자~ 다음은 기록 통계 화면으로 이동하겠습니다! 버스(NavigationController) 타세요!"** 하고 모든 이동과 준비를 알아서 통제합니다.

---

**🤿 딥하게 들어가기 (Deep: 코디네이터가 없는 세계의 끔찍함)**

먼저 코디네이터가 없을 때, 집중 타이머 화면에서 '기록 보기' 버튼을 눌렀을 때 벌어지는 참사를 보겠습니다.

```Swift
// ❌ 기존의 방식 (뷰 컨트롤러가 다음 화면을 알게 됨 = 강한 결합)
class FocusTimerViewController: UIViewController {
    
    @objc func historyButtonTapped() {
        // 1. 내가 직접 다음 화면의 뷰모델을 꺼내고...
        let historyVM: HistoryViewModel = DIContainer.shared.resolve()
        
        // 2. 다음 화면(VC)을 직접 생성하고...
        let historyVC = HistoryViewController(viewModel: historyVM)
        
        // 3. 직접 화면을 이동시킴!
        self.navigationController?.pushViewController(historyVC, animated: true)
    }
}
```

이러면 `FocusTimerViewController`는 `HistoryViewController`라는 존재를 평생 알게 됩니다. 나중에 타이머 화면만 똑 떼서 다른 앱에 재사용하려고 해도, 기록 화면까지 줄줄이 딸려오게 되죠.

---

**✅ Coordinator의 구원: 화면은 화면만 그리게 하라!**

이제 이 화면 이동 권한을 모두 빼앗아서 전담 매니저인 Coordinator에게 넘겨주겠습니다.

**Step 1: 지휘자 규격(Protocol) 만들기** 모든 코디네이터는 기본적으로 `start()`라는 시작 버튼을 가져야 합니다.

```Swift
protocol Coordinator: AnyObject {
    var navigationController: UINavigationController { get set }
    func start()
}
```

**Step 2: 메인 코디네이터 생성하기** 이 녀석이 바로 화면의 흐름을 통제하고, DI 컨테이너에서 뷰모델을 조립해 주는 진짜 '가이드'입니다.

```Swift
class MainCoordinator: Coordinator {
    var navigationController: UINavigationController
    
    init(navigationController: UINavigationController) {
        self.navigationController = navigationController
    }
    
    // 1. 첫 화면(타이머 화면)을 띄우는 역할
    func start() {
        let timerVM: FocusTimerViewModel = DIContainer.shared.resolve()
        let timerVC = FocusTimerViewController(viewModel: timerVM)
        
        // ⭐️ 핵심: "타이머 화면아, 기록 화면으로 가고 싶으면 나(Coordinator)한테 말해!"
        timerVC.onHistoryButtonTapped = { [weak self] in
            self?.showHistoryScreen()
        }
        
        navigationController.pushViewController(timerVC, animated: false)
    }
    
    // 2. 두 번째 화면(기록 화면)을 조립하고 띄우는 역할
    func showHistoryScreen() {
        let historyVM: HistoryViewModel = DIContainer.shared.resolve()
        let historyVC = HistoryViewController(viewModel: historyVM)
        
        // 타이머 화면은 기록 화면의 존재를 1도 모르지만, 코디네이터가 알아서 화면을 넘겨줍니다!
        navigationController.pushViewController(historyVC, animated: true)
    }
}
```

**Step 3: 완전히 자유로워진 뷰 컨트롤러** 이제 타이머 화면은 다음 화면이 뭔지, 어떻게 만드는지 전혀 모릅니다. 그냥 **"나 버튼 눌렸어! 가이드님 알아서 해주세요!"** 하고 소리치기(Closure 호출)만 하면 끝입니다.

```Swift
class FocusTimerViewController: UIViewController {
    
    private let viewModel: FocusTimerViewModel
    
    // 코디네이터에게 알릴 신호탄 (클로저)
    var onHistoryButtonTapped: (() -> Void)?
    
    init(viewModel: FocusTimerViewModel) { ... }
    
    @objc func historyButtonTapped() {
        // 다음 화면을 직접 안 만들고, 코디네이터를 부름!
        onHistoryButtonTapped?() 
    }
}
```

---

### 🚨 대규모 앱(Multi-Module)과 Coordinator의 미친 시너지

우리가 이전에 Tuist로 앱을 조립식(모듈화)으로 만들었던 것 기억하시나요? `TimerFeature` 모듈과 `HistoryFeature` 모듈은 서로 남남이어야 합니다.

이때 Coordinator 패턴을 쓰면, **`TimerFeature` 모듈 안에는 `History`라는 단어가 단 한 줄도 등장하지 않게 됩니다!** 오직 가장 최상단 껍데기 계층인 `App` 타겟 안에 있는 Coordinator만이 두 모듈을 불러와서 서로 부드럽게 이어줄 뿐이죠. 모듈화 아키텍처의 퍼즐이 여기서 완벽하게 완성되는 것입니다!

---

**💡 오늘의 핵심 요약**

> **Coordinator 패턴**은 뷰 컨트롤러에서 화면 이동(Navigation)과 의존성 조립 코드를 완전히 분리해 내는 설계 기법입니다. 이를 통해 뷰 컨트롤러를 **어디서든 재사용 가능한 완벽한 순수 UI 컴포넌트**로 만들 수 있습니다.

MVVM으로 로직(뇌)을 분리하고, DI Container로 부품 조립을 분리하고, Coordinator로 화면 이동마저 분리해 냈습니다. 현업에서 요구하는 가장 단단하고 이상적인 앱 아키텍처(MVVM-C)의 모든 그림을 그려내셨네요! 🚀
