## 🗣️ 가볍게 시작하기 (Simple): "미니멀리즘 철학" vs "미니멀한 인테리어"

- **REST:** 웹 생태계를 깔끔하게 유지하기 위해 만들어진 **'미니멀리즘(명사)'**이라는 아주 엄격한 설계 철학이자 규칙입니다.
    
- **RESTful API:** 그 미니멀리즘 철학을 아주 철저하게 지켜서 예쁘게 만들어낸 **'미니멀한(형용사) 방(API)'**입니다.

---

## 🤿 딥하게 들어가기 (Deep: 왜 이름이 두 개로 불릴까?)

### 1. REST (Representational State Transfer)

이것은 API 자체가 아닙니다. 로이 필딩(Roy Fielding)이라는 사람이 "웹에서 데이터를 주고받을 때는 제발 이렇게 깔끔한 구조(아키텍처)로 주고받자!"라고 정해둔 **6가지의 깐깐한 제약 조건(가이드라인)**입니다.

- 예: 서버와 클라이언트는 완전히 분리되어야 한다, 주소(URL)만 봐도 무슨 데이터인지 직관적으로 알아야 한다 등.

### 2. RESTful (REST + ful)

이름 그대로 "REST의 원칙을 가득 채운(ful)"이라는 뜻의 **'형용사'**입니다. 우리가 흔히 쓰는 무수히 많은 API들 중에서도, 앞서 말한 REST의 깐깐한 원칙들을 교과서처럼 완벽하게 다 지켜서 설계된 API를 가리켜 **"이 API는 참 RESTful 하네!"**라고 부르는 것입니다.

## 🚨 핵심 포인트: 실무에서는 왜 똑같이 부를까?

학문적으로 보면 완벽하게 원칙을 지킨 것만 'RESTful API'라고 불러야 맞습니다. 하지만 현실의 개발 세계에서는 이 6가지 원칙을 100% 전부 지키기가 (특히 HATEOAS 같은 복잡한 조건은) 너무 까다롭고 비효율적일 때가 많습니다.

그래서 현업에서는 원칙을 적당히(실용적으로)만 지켜도, **"우리 REST API로 통신하자~"** 혹은 **"이번 백엔드는 RESTful하게 짜주세요~"**라며 아무런 구분 없이 동의어처럼 사용합니다.

단어의 생김새 때문에 헷갈리셨겠지만, 결국 **"URL 주소와 GET/POST 방식을 깔끔하고 직관적으로 설계한 API 통신 방식"**이라는 하나의 똑같은 덩어리로 이해하시면 완벽합니다!

## 💡 오늘의 요약 처방

- **REST**는 웹을 위한 통신 **설계 원칙(명사)**입니다.
    
- **RESTful**은 그 원칙을 잘 준수하고 있다는 **상태(형용사)**를 뜻합니다.
    
- 개발자들의 일상 대화나 실무에서는 **완전히 똑같은 개념**으로 취급하셔도 아무런 문제가 없습니다!
