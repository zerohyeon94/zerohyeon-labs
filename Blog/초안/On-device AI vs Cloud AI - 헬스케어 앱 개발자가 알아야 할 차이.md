## 들어가며

헬스케어 앱에 AI를 붙이기로 결정한 순간, 가장 먼저 마주치는 질문이 있다.

> "모델을 서버에 올릴까, 아니면 앱 안에 직접 넣을까?"

이 선택은 단순한 기술 취향의 문제가 아니다. 헬스케어라는 도메인에서는 이 결정이 **개인정보 규제 준수 여부**, **실시간 응답 가능 여부**, 심지어 **의료기기 인허가 난이도**까지 바꿔놓는다.

나는 졸음 감지 iOS 앱 ZipJoong을 개발하면서 이 선택을 직접 해야 했다. 이 글은 그 과정에서 배운 것들을 정리한 것이다.

---

## 1. 두 방식의 핵심 차이

먼저 구조부터 보자.

**Cloud AI 흐름:**
```
기기에서 데이터 수집 (카메라, 센서)
    ↓
데이터를 서버로 전송 (네트워크)
    ↓
서버에서 AI 모델 추론
    ↓
결과를 기기로 응답
```

**On-device AI 흐름:**
```
기기에서 데이터 수집 (카메라, 센서)
    ↓
기기 내 AI 모델 추론 (Core ML / Neural Engine)
    ↓
즉시 결과 사용
```

| 항목 | Cloud AI | On-device AI |
|------|----------|--------------|
| 인터넷 필수 | ✅ 필수 | ❌ 불필요 |
| 응답 지연 | 수십ms ~ 수초 | 수ms ~ 수십ms |
| 개인정보 | 서버로 전송됨 | 기기 밖으로 나가지 않음 |
| 모델 크기 | 제한 없음 | 기기 저장공간 내 |
| 운영 비용 | API 호출당 발생 | 없음 |
| 모델 업데이트 | 서버에서 즉시 | 앱 업데이트 필요 |

표만 보면 On-device가 무조건 좋아 보인다. 하지만 실제로는 그렇지 않다. **헬스케어에서는 시나리오에 따라 정답이 완전히 달라진다.**

---

## 2. 헬스케어에서 On-device가 특별히 중요한 세 가지 이유

### 2-1. 개인정보보호 규제

헬스케어 데이터는 일반 데이터와 다르다.

한국 **개인정보보호법**은 건강정보, 바이오정보(얼굴 이미지 포함)를 민감정보로 분류한다. 서버로 전송하려면 별도 동의 절차와 암호화가 필요하다.

미국 **HIPAA**는 더 엄격하다. 얼굴 이미지, 바이탈 데이터 같은 PHI(Protected Health Information)를 다루는 서버와는 Business Associate Agreement(BAA)를 맺어야 한다. 스타트업에게 이 절차는 상당한 부담이다.

**On-device로 처리하면?** 데이터가 기기 밖으로 나가지 않으므로 이 규제 부담이 대폭 줄어든다. ZipJoong이 카메라로 얼굴을 분석하면서도 어떤 이미지도 서버로 보내지 않는 이유가 여기 있다.

### 2-2. 실시간성

졸음 감지는 0.1초 내로 판단이 이루어져야 한다. 눈이 감기는 순간을 놓치면 의미가 없기 때문이다.

Cloud AI를 사용한다면:
- 카메라 프레임 캡처 → 서버 전송 (30~100ms)
- 서버 추론 (10~50ms)
- 결과 응답 (30~100ms)
- **총 지연: 70ms ~ 250ms 이상**

네트워크 상태가 나쁘면 수초까지 늘어난다.

On-device Core ML로 처리하면:
- A15 Bionic 기준 MobileNetV3-Small 추론: **약 5~15ms**

실시간 감지, 낙상 알림, 심방세동 경보처럼 **즉각적인 반응이 생명인 헬스케어 시나리오**에서 Cloud AI는 구조적으로 불리하다.

### 2-3. 오프라인 환경

병원은 생각보다 인터넷이 불안정한 환경이 많다. 특히 수술실, 중환자실, 지방 의원 등은 망분리 정책이나 보안 규정으로 인터넷이 제한될 수 있다.

헬스케어 앱이 병원 내에서도 동작해야 한다면, On-device AI가 유일한 선택지다.

## 3. 실제 사례 — ZipJoong의 선택

> **[이 섹션은 오늘 직접 채울 것]**
> 아래 내용을 토대로 본인 언어로 구체화해주세요.

ZipJoong은 카메라로 운전자/학습자의 얼굴을 분석해 졸음 여부를 실시간으로 감지하는 iOS 앱이다.

**현재 구조: Vision(70%) + CoreML(30%) 하이브리드**

처음부터 이 비율이었던 것은 아니다. 이 선택에는 이유가 있다.

**Vision Framework를 70%로 선택한 이유:**
- Apple이 직접 최적화한 얼굴 랜드마크 감지 (VNDetectFaceLandmarksRequest)
- 눈 종횡비(EAR, Eye Aspect Ratio)를 좌표로 직접 계산 가능
- 별도 모델 없이도 안정적으로 동작 → 신뢰할 수 있는 기반

*(여기에 본인이 Vision 프레임워크를 선택하면서 느꼈던 점, 어떤 문제가 있었는지 추가해주세요)*

**CoreML MobileNetV3-Small을 30%로 선택한 이유:**
- Vision만으로는 졸음과 "그냥 눈을 가늘게 뜬 상태"를 구분하기 어려웠음
- 커스텀 학습 데이터로 경계 케이스를 보완
- MobileNetV3-Small: 모델 크기 ~5MB, iPhone 12 이상에서 실시간 추론 가능

## 4. 그래서 언제 어떤 걸 써야 할까?

내가 내린 의사결정 기준은 이렇다:

**On-device AI를 선택해야 하는 조건:**
- 개인정보(얼굴, 바이탈, 건강 기록)를 실시간으로 처리해야 할 때
- 응답 지연이 100ms 이하여야 할 때
- 인터넷 없이도 동작해야 할 때
- 모델이 5~50MB 이내로 충분히 압축 가능할 때

- 전체 환자 집단 데이터로 개인을 비교 분석해야 할 때
- 모델을 자주 업데이트하고 즉시 반영해야 할 때

## 5. 앞으로의 흐름 — Apple Intelligence 시대

iOS 18에서 Apple은 `FoundationModels` 프레임워크를 공개했다. 앱 내에서 온디바이스 LLM(대규모 언어 모델)을 직접 호출할 수 있게 된 것이다.

```swift
import FoundationModels

let session = LanguageModelSession()
let response = try await session.respond(to: "오늘 집중 패턴을 요약해줘")
```

ZipJoong 같은 헬스케어 앱이라면, 집중/졸음 데이터를 FoundationModels로 자연어 리포트로 변환하는 것도 가능해진다. **개인 건강 데이터가 서버로 나가지 않으면서도 지능적인 분석을 제공하는 것** — On-device AI의 미래가 여기에 있다.

---

## 마치며

헬스케어 앱에서 AI를 선택하는 건 기술만의 문제가 아니다. 개인정보, 실시간성, 규제, 그리고 사용자의 신뢰 모두가 엮여 있다.

On-device AI는 이 모든 조건을 동시에 충족할 수 있는 몇 안 되는 방법이다. 물론 모델 크기와 정확도의 한계는 있다. 하지만 Apple Silicon의 Neural Engine이 빠르게 발전하는 지금, 그 한계선은 매년 올라가고 있다.

iOS 개발자이면서 헬스케어 도메인을 이해하는 사람이라면, 이 흐름을 가장 잘 탈 수 있는 위치에 있다.
