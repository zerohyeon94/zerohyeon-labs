## 목적

VC가 보유한 객체의 **클로저**가 다시 `self`를 강하게 캡처해 **VC가 해제되지 않는** 상황 재현/해결.

## 증상(LEAKY)

- VC를 Pop/Dismiss 후에도 `deinit` 로그가 안 찍힘.
    
- Memory Graph에서 VC 인스턴스가 계속 남음.

## 원인

- 클로저는 기본이 `strong` 캡처.  VC → DataProvider (strong) → onUpdate 클로저 → `self`(VC) (strong) → **순환**.

## 해결(FIXED)

- 캡처 리스트로 약한 참조:

```swift
dataProvider.onUpdate = { [weak self] values in
    self?.render(values)
}
```

- 가능하면 **self 미캡처 구조로 설계**(handler가 VC에 의존하지 않게 분리).

## 검증 체크

-  Pop/Dimiss 직후 `deinit` 로그 출력
    
-  Memory Graph에서 VC 미존재

## 자주 하는 실수

- `guard let self = self else { return }` 쓰면서 **굳이 강한 self 유지**
    
- 비동기 큐에서 지연 실행이 **수명 초과**를 더 쉽게 유발

# 결론

- VC가 **dataProviderLeaky**를 `strong`으로 가지고 있고
    
- `dataProviderLeaky.onUpdate`에 저장된 **클로저가 `self`(= VC)**를 **강하게 캡처**하니까
    
- **VC → provider → 클로저 → VC**로 고리가 생겨서 `dismiss`해도 `deinit`이 안 불려요.

깨는 방법 3가지(실전 정리):

1. 가장 정석: 
```swift
dataProviderLeaky.onUpdate = { [weak self] values in
   self?.log("fixed: \(values)")
}
```
2. 화면 떠날 때 고리 끊기(보조책):
```swift
override func viewDidDisappear(_ animated: Bool) {
    super.viewDidDisappear(animated)
    dataProviderLeaky.onUpdate = nil
}

```
3. 설계로 회피: VC가 provider를 소유하되, provider 쪽 API는 **VC를 캡처하지 않는** 형태(이벤트만 내보내고 VC는 구독 시 `[weak self]`).
