## 오늘 이 주제를 보는 이유

2026-05-19 회고에서 내일로 넘긴 핵심은 `난임 환자 대상 해커톤 모델링`을 먼저 정리하는 것이었다.

모델링은 바로 알고리즘을 고르는 일이 아니라, 먼저 무엇을 예측할지와 어떤 데이터가 들어오는지를 좁히는 일이다. 특히 헬스케어 데이터는 예측 목표가 흐리거나 데이터 누수가 있으면 성능이 높아 보여도 실제 설명력이 떨어질 수 있다.

## 학습 목표

- 난임 환자 대상 문제를 한 문장으로 정의한다.
- 입력 데이터가 환자 기본 정보, 검사 정보, 치료 과정 정보, 결과 정보 중 어디에 속하는지 나눈다.
- 예측 목표가 분류인지 회귀인지 먼저 판단한다.
- 모델 결과를 모바일 앱 또는 헬스케어 서비스와 어떻게 연결할지 1줄로 적는다.

## 핵심 개념

### 1. 문제 정의가 먼저다

좋은 모델링 질문은 아래처럼 적는다.

```text
누구에게:
어떤 상황에서:
무엇을 예측하거나 분류할 것인가:
그 결과를 왜 쓰는가:
```

예를 들어 `난임 환자의 치료 성공 여부를 예측한다`는 너무 넓다. 치료 단계, 예측 시점, 결과 정의가 같이 있어야 한다.

```text
예측 시점: 치료 시작 전 / 시술 전 / 배아 이식 전 / 치료 후
예측 대상: 임신 성공 여부 / 착상 여부 / 특정 치료군 추천 / 위험군 분류
결과 활용: 상담 보조 / 위험도 안내 / 치료 과정 기록 / 의료진 의사결정 보조
```

### 2. 입력 데이터와 정답 데이터는 분리해서 본다

모델 입력에 정답 이후에만 알 수 있는 정보가 들어가면 데이터 누수가 생긴다.

```text
입력 후보:
- 환자 기본 정보
- 과거 병력 또는 치료 이력
- 검사 수치
- 시술 전 상태
- 치료 과정에서 얻은 정보

주의할 것:
- 결과 이후에 기록된 값
- 성공 여부를 사실상 직접 알려주는 컬럼
- 특정 병원이나 시점에만 존재하는 편향된 값
```

### 3. 평가지표는 정확도만 보면 안 된다

헬스케어 데이터는 클래스 불균형이 있을 가능성이 높다. 성공/실패 비율이 한쪽으로 치우치면 정확도만 높아도 실제로 쓸 수 없는 모델이 될 수 있다.

```text
분류 문제 후보 지표:
- ROC-AUC: 전체 분류 경향 확인
- PR-AUC: 양성 클래스가 적을 때 유용
- Recall: 놓치면 안 되는 케이스를 줄일 때 중요
- F1-score: Precision과 Recall 균형 확인
```

오늘은 지표를 확정하기보다, 목표가 무엇인지에 따라 어떤 지표를 볼지 후보만 잡는다.

## 실무 연결

오늘 남길 최소 산출물은 아래 4줄이면 충분하다.

```text
문제 정의:
입력 데이터:
예측 목표:
모바일 연결 가능성:
```

모바일 연결은 모델을 반드시 iPhone 안에 넣는다는 뜻이 아니다. 처음에는 `예측 결과를 사용자가 이해할 수 있게 보여주는 화면`, `치료 과정 기록`, `위험도 변화 시각화`, `상담 준비용 요약`처럼 서비스 경험으로 연결해도 된다.
