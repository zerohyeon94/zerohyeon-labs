# 4-1. ResultSnapshot으로 MainActor 경계 줄이기

> Daily: [[Daily/2026-06-07]]
> 분야: Swift Study
> 대주제: 성능 최적화
> 원 질문: "CameraService와 ViewModel 사이에 어떤 타입을 두면 MainActor 범위를 줄이기 쉬운가요?"
> 관련 학습 파일: [[4. CameraService와 ViewModel 사이에 ResultSnapshot 타입 두기]]

## 짧은 답

CameraService와 ViewModel 사이에는 원본 프레임이나 서비스 객체를 넘기지 말고, UI 판단에 필요한 값만 담은 작은 `ResultSnapshot`을 둔다.

예시는 아래처럼 잡을 수 있다.

```swift
struct ResultSnapshot: Sendable, Equatable {
    let frameID: Int64
    let capturedAt: Date
    let finishedAt: Date

    let focusScore: Double
    let confidence: Double
    let shouldShowWarning: Bool
}
```

이 타입은 `CMSampleBuffer`, `CVPixelBuffer`, `VNRequest`, 모델 원시 출력 같은 무거운 값을 들지 않는다. 그래서 CameraService 내부 계산은 백그라운드나 별도 actor에 남기고, ViewModel은 이 작은 값만 받아서 UI 상태를 갱신할 수 있다.

## 왜 MainActor 범위가 줄어드는가

`@MainActor` 범위가 커지는 흔한 이유는 ViewModel이 원본 처리까지 같이 맡기 때문이다.

```text
ViewModel
- 카메라 프레임 받기
- 추론 실행
- 결과 후처리
- UI 상태 갱신
```

이 구조에서는 ViewModel 전체가 MainActor가 되면 무거운 계산까지 메인 격리 안으로 끌려오기 쉽다.

사이에 `ResultSnapshot`을 두면 흐름을 이렇게 나눌 수 있다.

```text
CameraService 또는 Analyzer
- sampleBuffer 처리
- 모델 추론
- ResultSnapshot 생성

@MainActor ViewModel
- snapshot 비교
- UI 반영 여부 판단
- @Published 상태 갱신
```

즉, ViewModel이 받는 것은 "계산할 재료"가 아니라 "이미 계산된 작은 결과"가 된다.

## ViewModel에서 한 번 더 UIState로 바꾸기

`ResultSnapshot`을 그대로 View에 노출해도 되지만, 화면 표현이 많아지면 `CameraUIState`로 한 번 더 바꾸는 편이 좋다.

```swift
struct CameraUIState: Equatable {
    let scoreText: String
    let warningMessage: String?
    let isWarningVisible: Bool
}
```

그러면 책임은 아래처럼 나뉜다.

```text
ResultSnapshot
- 서비스가 만든 분석 결과
- 숫자, boolean, timestamp 중심

CameraUIState
- View가 바로 그릴 표현 상태
- 문구, 표시 여부, 색상 선택 기준
```

문자열을 어디서 만들지는 기준이 있다.

- 분석 의미 자체인 문장: `ResultSnapshot`에 넣어도 됨
- UI 표현 문구: ViewModel에서 `CameraUIState`로 조합

예를 들어 `"집중도가 낮습니다"` 같은 사용자 표시 문구는 ViewModel이나 별도 formatter에서 만드는 편이 낫다.

## 코드 흐름 예시

```swift
actor CameraAnalyzer {
    func analyze(frame: CameraFrame) async throws -> ResultSnapshot {
        let modelOutput = try await runModel(frame)

        return ResultSnapshot(
            frameID: frame.id,
            capturedAt: frame.capturedAt,
            finishedAt: Date(),
            focusScore: modelOutput.focusScore,
            confidence: modelOutput.confidence,
            shouldShowWarning: modelOutput.focusScore < 0.4
        )
    }
}

@MainActor
final class CameraViewModel: ObservableObject {
    @Published private(set) var uiState = CameraUIState(
        scoreText: "-",
        warningMessage: nil,
        isWarningVisible: false
    )

    private var latestSnapshot: ResultSnapshot?

    func apply(_ snapshot: ResultSnapshot) {
        guard shouldApply(snapshot) else {
            return
        }

        latestSnapshot = snapshot
        uiState = makeUIState(from: snapshot)
    }
}
```

이 구조에서 `CameraAnalyzer`는 MainActor가 아니어도 된다. ViewModel의 `apply(_:)`만 MainActor에서 실행되면 된다.

## ResultSnapshot에 넣으면 좋은 필드

처음에는 아래 정도면 충분하다.

```text
frameID
capturedAt
finishedAt
focusScore
confidence
shouldShowWarning
```

필요하면 이후에 추가할 수 있다.

```text
latencyMs
category
debugReason
```

단, 디버깅용 필드는 운영 UI에 계속 필요한 값인지 확인한 뒤 넣는 것이 좋다. Snapshot이 커지면 다시 경계가 흐려진다.

## 넣지 않는 것이 좋은 필드

아래 값은 CameraService 또는 Analyzer 내부에 남기는 편이 좋다.

- `CMSampleBuffer`
- `CVPixelBuffer`
- `UIImage` / `CGImage`
- 모델 원시 출력 전체
- `VNRequest`
- ViewModel이 직접 호출할 서비스 참조

이 값들을 Snapshot에 넣으면 타입이 무거워지고, `Sendable`로 다루기도 어려워지며, ViewModel이 원본 처리 세부사항을 알게 된다.

## 오늘 기억할 문장

```text
MainActor 범위를 줄이려면
ViewModel로 원본 데이터를 넘기지 말고,
이미 계산된 작은 Sendable 값 타입을 넘긴다.
```
