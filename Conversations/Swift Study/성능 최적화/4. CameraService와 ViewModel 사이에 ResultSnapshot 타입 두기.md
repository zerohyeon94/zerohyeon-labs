# 4. CameraService와 ViewModel 사이에 ResultSnapshot 타입 두기

> Daily: [[Daily/2026-06-07]]
> 분야: Swift Study
> 대주제: 성능 최적화
> 멘토링 시작 문장: "Swift 멘토링을 시작하겠습니다. 질문: CameraService와 ViewModel 사이에 어떤 타입을 두면 MainActor 범위를 줄이기 쉬운가요?"

## 오늘 이 주제를 보는 이유

어제는 `ViewModel 전체 @MainActor`와 `메서드 단위 @MainActor`를 비교했다.

오늘은 그 판단을 더 실무적으로 좁혀서, 어떤 데이터를 사이에 두면 메인 격리 범위를 줄이기 쉬운지 본다.

## 학습 목표

- 서비스 내부 원본 데이터와 UI 전달용 스냅샷을 구분한다.
- `ResultSnapshot` 같은 작은 `struct`가 왜 유리한지 설명할 수 있다.
- CameraService와 ViewModel의 책임을 더 선명하게 나눈다.

## 핵심 개념

### 1. 사이 타입은 "작고 복사 가능한 요약값"이 좋다

CameraService와 ViewModel 사이에는 아래 같은 값이 잘 맞는다.

```text
ResultSnapshot
- frameID
- updatedAt
- focusScore
- shouldShowWarning
```

이 타입은 원본 버퍼나 프레임 객체를 직접 들지 않고, UI 판단에 필요한 값만 갖는다.

### 2. `struct` + 단순 필드 조합이 MainActor 범위를 줄인다

이유는 단순하다.

- 메인 스레드에서 원본 프레임을 들고 있을 필요가 없다
- 비교와 상태 반영 로직을 분리하기 쉽다
- `Sendable`로 다루기 좋은 형태에 가깝다

즉, 큰 참조 타입보다 작은 값 타입이 경계 설계에 유리하다.

### 3. CameraService는 만들고, ViewModel은 받는다

역할을 아래처럼 나누면 된다.

```text
CameraService
- 원본 프레임 처리
- 추론 수행
- ResultSnapshot 생성

ViewModel
- 마지막 스냅샷과 비교
- 반영 여부 판단
- UI 상태 갱신
```

이렇게 두면 `전체 ViewModel @MainActor`를 써도 서비스 쪽 계산이 메인으로 끌려오지 않는다.

### 4. 헷갈리면 질문을 바꿔 본다

질문은 "어디에 @MainActor를 붙일까?"보다 아래가 먼저다.

- 메인 스레드에서 꼭 필요한 값은 무엇인가?
- 그 값을 가장 작은 타입으로 줄이면 어떤 모양인가?

이 답이 `ResultSnapshot` 같은 작은 구조체라면 격리 경계도 따라오기 쉽다.

## 실무 연결

실시간 분석이나 네트워킹 결과 처리에서도 같은 방식이 통한다.

```text
원본 처리 객체
-> 작은 스냅샷 struct
-> MainActor 상태 관리자
```

이 흐름을 먼저 만들면 격리 범위를 감으로 정하지 않게 된다.
