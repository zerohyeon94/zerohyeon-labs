# 3. @MainActor 범위를 ViewModel 전체와 메서드 단위로 나누기

> Daily: [[Daily/2026-06-06]]
> 분야: Swift Study
> 대주제: 성능 최적화
> 멘토링 시작 문장: "Swift 멘토링을 시작하겠습니다. 질문: ViewModel 전체에 @MainActor를 붙일지, 일부 메서드에만 붙일지 어떤 기준으로 나눠야 하나요?"

## 오늘 이 주제를 보는 이유

오늘의 핵심 축은 iOS 카메라 파이프라인 복습이다.

그 흐름에서 헷갈리기 쉬운 지점이 `어디까지를 MainActor 격리 범위로 둘 것인가`다. 이 파일은 범위를 넓히지 않고, 격리 단위를 짧게 판단하기 위한 선택 복습이다.

## 학습 목표

- ViewModel 전체 격리와 메서드 단위 격리의 차이를 구분한다.
- UI 상태와 계산 도우미를 같은 격리 범위에 넣지 않는 이유를 정리한다.
- 지금 코드에 더 가까운 기준을 하나 고른다.

## 핵심 개념

### 1. ViewModel 전체를 `@MainActor`로 두기 좋은 경우

아래 조건이 많으면 타입 전체 격리가 단순하다.

```text
- 대부분의 프로퍼티가 @Published UI 상태다
- 외부에서 호출하는 메서드도 거의 상태 갱신 목적이다
- 계산은 별도 서비스나 helper로 이미 내려가 있다
```

이 경우에는 "이 타입은 UI 상태 관리용"이라는 의미가 분명하다.

### 2. 메서드 단위로만 `@MainActor`를 두기 좋은 경우

아래처럼 타입 안에 계산/조립 로직이 아직 남아 있으면, 전체 격리는 과할 수 있다.

```text
- 일부 메서드는 무거운 계산이나 데이터 가공을 한다
- 같은 타입 안에 네트워크/추론 결과 후처리 로직이 섞여 있다
- 점진적으로 구조를 나누는 중이다
```

이때는 최소한 `상태 반영 메서드만 MainActor`, 계산은 바깥이나 백그라운드로 두는 쪽이 덜 막힌다.

### 3. 핵심 판단 기준은 "이 타입의 주 역할"이다

질문은 단순하다.

- 이 타입은 UI 상태 관리자에 가까운가?
- 아니면 계산과 상태 갱신이 아직 섞여 있는가?

첫 번째면 타입 전체 격리가 자연스럽고, 두 번째면 메서드 단위 격리가 과도한 메인 스레드 점유를 줄이기 쉽다.

## 실무 연결

카메라나 실시간 추론 코드에서는 아래 문장을 먼저 적어 두면 좋다.

```text
이 타입은 상태 반영 전용인가?
무거운 계산 메서드가 남아 있는가?
```

두 번째 질문에 `예`가 나오면, 전체 `@MainActor`를 붙이기 전에 책임 분리가 먼저다.
