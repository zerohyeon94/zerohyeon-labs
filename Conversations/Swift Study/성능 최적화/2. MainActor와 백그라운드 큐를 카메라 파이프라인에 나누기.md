# 2. MainActor와 백그라운드 큐를 카메라 파이프라인에 나누기

> Daily: [[Daily/2026-06-05]]
> 분야: Swift Study
> 대주제: 성능 최적화
> 멘토링 시작 문장: "Swift 멘토링을 시작하겠습니다. 질문: MainActor와 백그라운드 큐를 카메라 파이프라인에 나눌 때 상태 변경과 계산 작업을 어떤 기준으로 분리해야 하나요?"

## 오늘 이 주제를 보는 이유

오늘의 핵심 축은 iOS 카메라 파이프라인이지만, 실제 병목은 Swift 동시성 경계를 어떻게 두느냐에서 자주 생긴다.

그래서 이 파일은 새 범위를 넓히기보다 `무엇을 MainActor에 두고 무엇을 백그라운드로 내려야 하는가`를 언어 관점으로 짧게 다시 고정하기 위한 선택 복습이다.

## 학습 목표

- `MainActor`가 의미하는 바를 UI 전용 실행 맥락으로 이해한다.
- 계산 작업과 상태 변경 작업을 서로 다른 실행 맥락에 배치한다.
- 카메라/추론 코드에서 메인 스레드 병목이 나는 전형적인 패턴을 구분한다.
- 오늘 iOS 파일과 연결되는 질문 1개를 남긴다.

## 핵심 개념

### 1. MainActor는 "UI 상태 변경 장소"에 가깝다

`MainActor`는 무거운 계산을 얹는 장소가 아니라, 사용자에게 보이는 상태를 일관되게 갱신하는 장소다.

```text
MainActor에 두기 좋은 것:
- @Published 상태 갱신
- SwiftUI 바인딩 반영
- 경고 문구, 화면 표시 여부 결정 결과 반영

MainActor에 두기 나쁜 것:
- Vision/Core ML 추론
- 프레임 후처리 반복 계산
- 매 프레임 이미지 변환
```

### 2. 계산은 백그라운드에서 끝내고 결과만 건넨다

카메라 파이프라인에서는 아래 흐름이 기본이다.

```text
frame input
-> background calculation
-> small result
-> MainActor state update
```

중간 계산 결과나 원본 프레임을 메인으로 올리기 시작하면 앱은 쉽게 버벅인다.

### 3. 성능 문제는 "어디서 실행되는지 모르는 상태"에서 커진다

오늘 체크할 포인트는 단순하다.

- 이 메서드는 어느 큐/액터에서 실행되는가?
- 이 값은 UI 상태인가 계산 중간값인가?
- 이 작업은 매 프레임마다 돌아도 괜찮은가?

이 세 질문에 답하지 못하면, 성능 이슈가 생겼을 때 원인을 좁히기 어렵다.

## 실무 연결

실무에서는 "동작은 하는데 가끔 버벅이는 카메라 화면"이 자주 나온다. 이럴 때는 알고리즘보다 먼저 실행 경계를 정리해야 한다.

오늘은 아래 두 문장만 남겨도 충분하다.

```text
1. 이 상태 변경은 MainActor에서만 한다.
2. 이 계산은 MainActor 밖에서 끝낸 뒤 결과만 넘긴다.
```
