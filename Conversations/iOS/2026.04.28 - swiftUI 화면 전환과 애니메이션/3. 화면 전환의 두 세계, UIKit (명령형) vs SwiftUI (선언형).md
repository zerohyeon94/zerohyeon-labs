**🗣️ 가볍게 시작하기 - 수동 운전 vs 자율 주행 목적지 설정**

- **UIKit (명령형 - Imperative)** "내비게이션 컨트롤러야, 지금 당장 B 화면 객체를 생성해서 내 화면 위로 밀어 넣어(Push)!" 개발자가 직접 핸들을 잡고 **'어떻게(How)'** 이동할지 하나하나 지시하는 수동 운전입니다.
    
- **SwiftUI (선언형 - Declarative)** "나는 지금 '상세 화면 띄움' 상태를 true로 바꿨어. 프레임워크야, 네가 알아서 이 상태에 맞는 화면(B 화면)을 보여줘!" 개발자는 **'무엇을(What)'** 보여줄지만 선언하고, 실제 화면 전환은 프레임워크가 알아서 처리하는 자율 주행입니다.

---

**🤿 딥하게 들어가기 - 3가지 결정적 차이점**

**1. 화면을 띄우는 주체와 시점**

**UIKit**은 특정 이벤트(버튼 클릭 등)가 발생했을 때, 그 함수 안에서 다음 화면을 **'직접 생성하고 강제로 밀어 넣습니다'**.

```Swift
// UIKit 코드 예시
@objc func nextButtonTapped() {
    let nextVC = DetailViewController() // 직접 객체 생성
    self.navigationController?.pushViewController(nextVC, animated: true) // 강제 이동 명령
}
```

**SwiftUI**는 함수 안에서 화면을 띄우는 명령을 내리지 않습니다. 오직 '상태 변수(State)'만 변경하며, 화면 전환 코드는 뷰 구조(Body) 안에 미리 다 선언되어 있습니다.

```Swift
// SwiftUI 코드 예시
@State private var showDetail = false // 상태 스위치

var body: some View {
    Button("다음 화면으로") {
        showDetail = true // 상태만 변경함!
    }
    // 스위치가 true가 되면 프레임워크가 알아서 띄워줌
    .sheet(isPresented: $showDetail) { 
        DetailView()
    }
}
```

**2. 데이터 전달(의존성 주입) 방식**

- **UIKit** 뷰 컨트롤러가 다른 뷰 컨트롤러를 띄우기 직전에, 프로퍼티에 직접 접근해서 데이터를 꽂아 넣어줘야 합니다. (`nextVC.data = myData`) 화면 간의 결합도가 높아지기 쉽습니다.
    
- **SwiftUI** 상태 변수를 바인딩(`$`)으로 넘기거나, 상위 뷰에서 환경 변수(`@EnvironmentObject`)를 세팅해 두면 하위 뷰들이 알아서 꺼내 씁니다. 화면끼리 서로의 존재를 모른 채 데이터만 공유하기가 훨씬 수월합니다.

**3. 화면의 라이프사이클 (생성과 소멸)**

- **UIKit** 한 번 화면에 띄워진 뷰 컨트롤러는 메모리에 무겁게 계속 상주합니다. (스택에 쌓임)
    
- **SwiftUI** 상태(State)가 `false`가 되는 순간, 프레임워크가 뷰를 흔적도 없이 파괴하고 메모리에서 날려버립니다. 뷰 자체가 극도로 가벼운 구조체(`struct`)이기 때문에 언제든 부수고 새로 렌더링하는 것에 부담이 없습니다.

---

**💡 오늘의 핵심 요약**

> **UIKit**은 개발자가 뷰의 객체를 직접 쥐고 흔들며 화면을 덮어씌우는 물리적인 방식이고, **SwiftUI**는 데이터(State)의 상태가 변함에 따라 프레임워크가 적절한 뷰를 알아서 렌더링해 주는 논리적인 방식입니다.

이러한 철학의 차이 때문에, 앞선 수다에서 언급되었던 Coordinator 패턴 역시 UIKit에서는 '직접 화면을 생성해서 Push 하는 매니저' 역할을 했지만, SwiftUI에서는 '경로 배열(NavigationPath) 상태만 조작하는 리모컨' 역할로 완전히 다르게 구현되는 것입니다.
