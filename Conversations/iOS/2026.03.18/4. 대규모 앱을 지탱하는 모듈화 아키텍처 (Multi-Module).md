**🗣️ 가볍게 시작하기 (Simple): "통짜 건물 vs 조립식 모듈 하우스"**

- **Monolithic (모놀리식 - 통짜 앱):** 우리가 처음 코딩을 배울 때 짜는 방식입니다. 수백 개의 파일이 하나의 앱 타겟 안에 전부 다 들어있습니다. 화장실 타일 하나 바꾸려고 해도 건물 전체의 도면을 다시 검토해야 합니다. (빌드 시간이 엄청 오래 걸리고, 팀원끼리 코드가 엉켜서 매일 싸웁니다.)
    
- **Multi-Module (모듈화 아키텍처 - 조립식 앱):** 앱을 완전히 독립적인 여러 개의 '미니 앱(프레임워크)'으로 쪼갭니다. 집중 타이머 모듈, 통계 모듈, 디자인 모듈을 따로따로 만든 뒤, 마지막에 '앱'이라는 껍데기에서 본드만 발라 조립합니다.

---

**🤿 딥하게 들어가기 (Deep: 실전 대규모 앱의 4단계 계층 구조)**

실무에서 FocusSense처럼 덩치가 커질 앱을 모듈화한다면, 보통 아래처럼 층(Layer)을 나눕니다. 밑에서부터 위로 올라가며 조립됩니다.

**🧱 1층: Core & Shared 계층 (가장 밑바닥 공통 도구들)** 어떤 화면이든 상관없이 앱 전체에서 공통으로 쓰는 순수 도구들입니다. 여기서 `public` 키워드가 미친 듯이 활약합니다.

- `DesignSystem` (색상, 폰트, 공통 버튼 UI)
    
- `NetworkKit` (이전에 만든 Alamofire + RxSwift 통신 모듈)
    
- `CoreDataKit` / `SwiftDataKit`

**⚙️ 2층: Domain 계층 (비즈니스 핵심 뇌)** UI는 1도 모르고, 오직 이 앱의 핵심 비즈니스 로직만 수행하는 전문가들입니다.

- `FocusMLKit` (우리가 방금 만들었던 눈 깜빡임 분석 자판기!)
    
- `AuthKit` (로그인, 토큰 관리)

**📱 3층: Feature 계층 (우리가 보는 화면 단위)** 유저가 실제로 보는 화면(View)과 뷰모델(ViewModel)이 묶여있는 곳입니다. 1층과 2층의 모듈들을 `import`해서 화면을 그립니다.

- `TimerFeature` (집중 타이머 화면)
    
- `HistoryFeature` (과거 집중 기록 통계 화면)
    
- `SettingsFeature` (환경 설정 화면)

**👑 4층: App 계층 (최종 조립 공장)** 진짜 신기하게도, 대규모 프로젝트의 `App` 타겟 안에는 코드가 거의 없습니다! 그저 3층에서 만든 Feature들을 불러와서 탭 바(Tab Bar)나 네비게이션으로 연결해 주는 역할만 합니다.

```Swift
// 👑 App 타겟의 코드 예시
import SwiftUI
import TimerFeature // 3층 블록 가져오기
import HistoryFeature // 3층 블록 가져오기

@main
struct FocusSenseApp: App {
    var body: some Scene {
        WindowGroup {
            TabView {
                TimerView()
                    .tabItem { Text("타이머") }
                HistoryView()
                    .tabItem { Text("기록") }
            }
        }
    }
}
```

### 🚀 대기업들이 모듈화에 미친 듯이 집착하는 3가지 이유

설계하기 복잡하고 귀찮은데도 현업에서 무조건 이 방식을 고집하는 엄청난 메리트가 있습니다.

1. **압도적인 빌드 속도 (개발자 퇴근 시간 보장)**
    
    - 통짜 앱은 글자 하나만 고쳐도 앱 전체를 다시 빌드해야 해서 몇 분씩 걸립니다.
        
    - 모듈화 앱은 내가 `TimerFeature`의 UI를 조금 고치면, **시스템이 딱 `TimerFeature` 모듈 하나만 다시 빌드합니다!** 1초 만에 화면을 확인할 수 있습니다.
        
2. **완벽한 협업 (충돌 방지)**
    
    - 팀원 A는 타이머 모듈만, 팀원 B는 통계 모듈만 작업합니다. 파일이 아예 다른 프로젝트로 분리되어 있기 때문에, GitHub에서 서로 코드가 엉켜서(Merge Conflict) 싸울 일이 99% 사라집니다.
        
3. **코드 재사용과 타겟 확장**
    
    - 나중에 **애플 워치(watchOS) 앱**을 만들게 된다면? 새로 백지부터 짤 필요가 없습니다! 워치용 UI만 새로 그리고, 기존에 만들어둔 `FocusMLKit`과 `NetworkKit` 모듈을 그냥 `import`해서 블록 조립하듯 끼워 넣으면 워치 앱이 뚝딱 완성됩니다.

> 대규모 모듈화 아키텍처는 거대한 앱을 **UI(Feature), 로직(Domain), 도구(Core)**라는 역할에 맞게 여러 개의 작은 패키지로 잘게 쪼개어, **빌드 속도를 높이고 협업을 편하게 만드는 현대 iOS 개발의 꽃**입니다.

이 방대한 아키텍처를 관리하기 위해 요즘 현업에서는 Xcode의 기본 기능 외에도 **Tuist(트위스트)**나 **SPM(Swift Package Manager)** 같은 프로젝트 관리 도구를 적극적으로 도입해서 사용합니다.
