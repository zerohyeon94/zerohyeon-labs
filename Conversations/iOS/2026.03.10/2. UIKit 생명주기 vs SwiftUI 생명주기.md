**🗣️ 1. SwiftUI에서는 `UIViewController` 대신 `View`의 생명주기로 봐야 하나요?**

**네, 맞습니다! 하지만 여기서 가장 큰 '개념의 차이(Paradigm Shift)'를 인정하셔야 합니다.**

- **UIKit (`UIViewController` - Class):** 메모리에 한 번 무겁게 올라가면, 자신이 언제 태어나고(`viewDidLoad`), 언제 화면에 나타날 준비를 하고(`viewWillAppear`), 언제 화면에 띄워졌는지(`viewDidAppear`)를 세세하게 보고하는 **'상태를 가진 객체'**입니다.
    
- **SwiftUI (`View` - Struct):** 그저 화면이 어떻게 생겼는지 묘사하는 **'가벼운 종이 쪼가리(설계도)'**에 불과합니다. 데이터가 바뀌면 이 종이는 미련 없이 버려지고 수십, 수백 번 새로 그려집니다. 그래서 "언제 태어났다"는 개념 자체가 굉장히 희미하고, 그저 **"화면(View Tree)에 붙었다(.onAppear)"**와 **"화면에서 떨어졌다(.onDisappear)"**라는 직관적인 상태만 존재합니다.

---

**🤿 2. UIKit ➔ SwiftUI 작업 시 1:1 매칭이 가능한가요?**

단호하게 말씀드리면, **"억지로 1:1 매칭을 하려고 하면 SwiftUI의 늪에 빠지게 됩니다."**

SwiftUI에는 `viewWillAppear`와 `viewDidAppear`를 구분하는 미세한 타이밍이 없습니다. 하지만 마이그레이션을 해야 하는 개발자들을 위해, 가장 합리적인 **대체(Mapping) 가이드**를 표로 정리해 드립니다.

| **UIKit 메서드**                                                    | **SwiftUI 대체제**                              | **실전 마이그레이션 팁 (어떻게 바꾸나?)**                                                                                                                           |
| ---------------------------------------------------------------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`viewDidLoad`**<br>(최초 1회 실행)                                  | **없음** (완벽한 1:1 대응 불가)                       | 1. 뷰모델의 `@StateObject`가 생성되는 시점(`init`)을 활용합니다.<br><br>2. 꼭 한 번만 실행해야 하는 UI 로직이라면 `.onAppear` 안에서 `@State` 플래그(예: `isFirstLoad`)를 체크해서 한 번만 돌게 만듭니다. |
| `viewWillAppear`<br><br>`viewIsAppearing`<br><br>`viewDidAppear` | **`.onAppear`**<br><br>또는<br><br>**`.task`** | SwiftUI는 이 셋을 굳이 구분하지 않습니다. 화면에 뷰가 렌더링될 때 `.onAppear`가 불립니다.<br><br>**비동기 작업(네트워크 등)**을 하던 곳이라면 무조건 **`.task`**로 옮기시면 됩니다.                           |
| `viewWillDisappear`<br><br>`viewDidDisappear`                    | **`.onDisappear`**                           | 화면에서 뷰가 사라질 때 호출됩니다. 타이머를 멈추거나, Notification 관찰을 해제할 때 사용합니다. (참고로 `.task`를 썼다면 비동기 취소는 알아서 해줍니다!)                                                   |
| **`viewWillLayoutSubviews`**                                     | **`GeometryReader`**                         | UIKit에서 화면 크기가 바뀔 때(회전 등) 뷰 크기를 재계산하던 곳이죠? SwiftUI에서는 `GeometryReader`를 써서 남은 공간을 동적으로 읽어오게 선언형으로 바꿔야 합니다.                                           |

---

### 🚨 가장 많이 하는 실수 (주의사항)

UIKit 개발자들이 SwiftUI로 넘어올 때 제일 많이 하는 실수가 바로 **`init()`에 무거운 작업을 넣는 것**입니다.

- **UIKit 마인드:** "뷰 컨트롤러가 만들어질 때(`init`)나 `viewDidLoad`에서 API 통신 시작해야지!"
    
- **SwiftUI 현실:** 구조체 뷰의 `init()`은 상위 뷰의 데이터가 바뀔 때마다 무자비하게 수십 번 다시 호출될 수 있습니다. 여기서 네트워크 통신을 하면 서버에 API 요청을 수십 번 날리는 대참사가 벌어집니다.
    
- **해결책:** 무거운 작업은 반드시 뷰가 완전히 화면에 붙은 뒤인 **`.task`**나 **`.onAppear`**로 미루셔야 합니다.

**💡 오늘의 요약**

> SwiftUI로 넘어올 때는 **"언제(When) 이 코드를 실행할까?"**라는 시간의 흐름(생명주기)에서 벗어나, **"어떤 데이터(State)가 변할 때 화면이 바뀌어야 할까?"**라는 상태의 흐름으로 뇌 구조를 살짝 리모델링하셔야 합니다!
