### Q1. ATS가 강제하는 핵심 보안 요구사항은 무엇이며, 서버가 충족하지 못할 때 iOS에서 어떤 증상이 나타나나요?

_힌트: TLS 버전, FS(ECDHE), 인증서/서명, -1022_

### 내 답변

```text
HTTP 통신을 제한하며 https로 시작하는 보안 URL만을 허용합니다. 이때 서버가 충족하지 못하면 이 네트워크에 연결을 하지 못하게 됩니다.
```

## Q1 피드백 - ATS 핵심 요구사항

**당신의 요지**: “HTTPS만 허용, 서버가 기준 미달이면 연결 불가.”  
**강점**: 핵심 의도 파악 👍  
**보완 포인트**: ATS의 구체 요건을 함께 언급하면 완성도가 크게 올라갑니다.

- 최소 **TLS 1.2+**
    
- **Forward Secrecy(ECDHE)** 권장
    
- 유효 인증서(+ SHA-256 이상 서명)
    
- 실패 시 흔한 에러: **NSURLErrorDomain(-1022)**
    

**업그레이드 답변 예시**

> “ATS는 iOS가 `URLSession` 요청에 **HTTPS(TLS 1.2+)** 를 강제하는 정책입니다. 서버가 약한 TLS/인증서를 쓰거나 HTTP면 **-1022**로 차단되고, 가능하면 **서버를 먼저 개선**하며 불가피하면 도메인 한정 예외를 씁니다.”

### Q2. WKWebView만 ATS 완화를 하고 네이티브 `URLSession`에는 영향이 없게 하려면 Info.plist에 무엇을 넣나요?

_정확한 키 이름과 효과를 설명하세요._

### 내 답변

```text
실제로 WKWebView를 써보진 않았고 앱내에서 사파리 브라우저를 띄우는 import SafariServices를 통해 이것은 사용해봤습니다. 단순히 저는 URL로 Safari를 모달로 띄우는 작업만 진행해보았습니다.
```

## Q2 피드백 - WebView만 예외

**당신의 요지**: SFSafariViewController 경험 공유.  
**정확성**: SFSafariViewController는 **앱 외부(사파리 뷰)** 로 열리는 브라우저로, **앱 내 네트워킹/ATS 예외와 별개**입니다.  
**핵심 정답**: “웹뷰만 완화하려면 `Info.plist`에서  `NSAppTransportSecurity` → `NSAllowsArbitraryLoadsInWebContent = true` 설정. ”

- 이 설정은 **WKWebView** 로드에만 영향을 주고, **네이티브 URLSession에는 적용되지 않음**.

**업그레이드 답변 예시**

> “웹 콘텐츠만 예외: `NSAllowsArbitraryLoadsInWebContent`를 켭니다. 네이티브 요청은 ATS를 그대로 따릅니다.”

### Q3. 사내 IoT 장비(사설 IP, `.local`)와 통신이 필요할 때 ATS를 깨지 않으면서 접근하는 설정은?

_키 이름과 적용 범위를 설명하세요._

### 내 답변

```text
NSAppTransportSecurity 키에다가 로컬 주소에서는 HTTP를 허용하게 구현하려고 합니다. 하지만 이것은 Apple에서 예외적으로 허용하는 것이라 실제로 앱을 출시할 때는 적용하면 안되는 것으로 알고 있습니다.
```

## Q3 피드백 - 로컬 네트워킹

**당신의 요지**: 로컬 HTTP를 예외로 허용하려 한다. 출시에선 지양.  
**정확성 보완**:

- 로컬/사설망 접속을 위해선 `NSAllowsLocalNetworking = true` 를 사용합니다.
    
- 이 키는 **사설 IP·.local 등 로컬 네트워크 대상을 허용**할 때 쓰는 **공식적이고 합법적인 방법**이에요. 출시 앱에서도 **정당한 사유**가 있다면 사용 가능합니다.
    
- 전역 허용(`NSAllowsArbitraryLoads`)은 지양하지만, **로컬 한정 허용**은 현실적인 옵션입니다.

**업그레이드 답변 예시**

> “사설 IP나 `.local` 기기 접근은 `NSAllowsLocalNetworking`을 사용합니다. 외부 인터넷 도메인엔 영향이 없고, 필요 시 도메인 예외와 조합해 최소 범위로 관리합니다.”

### Q4. `NSExceptionDomains`로 HTTP를 임시 허용했다고 가정. 장기적으로 **보안 수준을 유지**하면서 사용자를 덜 불편하게 만들 전략을 2가지 제시하세요.

_예: 서버/인프라, 릴리즈 전략, 모니터링 관점_

### 내 답변

```text
방법으로는 우선적으로 https URL이 접근하도록 합니다. 이는 조건문으로 구현을 합니다. 그리고 그게 아니라 http의 경우에는 팝업창으로 보안이 불안정한 서버에 접속하다는 안내문과 주의 메시지를 넘겨주고 그래도 접속하겠다면 연결하도록 구현해야될 것 같습니다.
```

## Q4 피드백 - HTTP 경고 UX

**당신의 제안**: http면 경고 후 계속 진행 선택  
**중요 교정**:

- **ATS는 런타임 팝업으로 우회할 수 없습니다.** HTTP 요청은 **아예 생성/완료가 차단**되기 때문에, “경고 후 진행” 같은 브라우저 스타일 처리는 **네이티브 네트워킹에서 불가능**합니다.
    
- 해결책은 둘 중 하나입니다.
    
    1. **서버를 HTTPS/TLS 1.2+로 전환**(정석)
        
    2. **`NSExceptionDomains` 로 해당 도메인만 임시 HTTP 허용**(최소 범위, 제거 계획 포함)

**업그레이드 답변 예시**

> “HTTP 강제 허용은 UI로 풀 수 없고, **서버 TLS 업그레이드** 또는 `NSExceptionDomains`로 **도메인 한정 예외**가 필요합니다. 예외는 임시로 쓰고 모니터링하며 제거합니다.”
