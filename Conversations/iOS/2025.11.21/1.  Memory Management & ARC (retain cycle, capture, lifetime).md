**Q1. ARC 동작 원리와 참조 그래프**  
Swift ARC가 **언제 reference count를 +1/-1** 하는지 설명하세요. 아래 케이스를 각각 어떻게 처리하는지도 덧붙이세요:

- 지역 변수/함수 반환값/옵셔널 바인딩
- 배열/딕셔너리에 class 인스턴스를 저장·제거
- struct 안에 class, class 안에 struct가 있을 때
- `deinit` 호출 시점 정의

**당신의 요지**

- 참조할 때 RC가 증가, dismiss 등으로 사용 끝나면 RC 감소, 0이면 해제.
    
- 강한 참조만 남으면 순환 참조 가능.

**Review (strengths/gaps)**

- ✅ 큰 흐름 이해 OK.
    
- ⚠️ 세부 규칙이 빠졌습니다: 언제 **증가/감소**가 일어나는지 각 케이스별로 명확히.

**Ideal points (핵심 체크리스트)**

- **대입/소유**: `strong var/let`에 **대입할 때 +1**, 다른 변수에서 **참조 해제될 때 -1**.
    
- **옵셔널 바인딩/반환**: 바인딩은 사실상 **대입**(필요 시 +1). 함수 **반환값**을 강한 변수에 할당하면 +1.
    
- **컬렉션(Array/Dictionary/Set)**: 요소를 넣으면 컬렉션이 **strong 보유(+1)**, 제거/컬렉션 해제 시 -1.
    
- **클로저 캡처**:
    
    - **non-escaping**: 호출 범위를 벗어나지 않아 self 수명 연장 X(추가 보유 없음).
        
    - **@escaping**: 클로저가 저장되면 **self를 캡처하면 +1** → `[weak self]` 등 필요.
        
- **값 타입 안의 참조**: `struct`가 `class`를 프로퍼티로 가지면, 그 프로퍼티에 **대입 시 +1**. COW(`Array/Data`)는 내부 버퍼 참조 카운트로 **쓰기 시 복사**.
    
- **deinit**: RC = 0일 때만 호출. **강제 종료/크래시/순환참조** 상황에선 호출 보장 없음.

**Q2. 강한 순환 참조(Retain Cycle) 4종 & 해법**  
다음 시나리오별로 **문제 원인**과 **정확한 해결책(weak/unowned/capture list/설계 변경)** 을 제시하세요. 가능한 한 **코드 한 줄**로 보여주세요.

- (a) **ViewController ↔ ViewModel(클로저 콜백)**
- (b) **delegate 패턴**
- (c) **Timer/DisplayLink**
- (d) **NotificationCenter** 구독

**당신의 요지**

- `[weak self]` 사용, delegate는 간접 접근.

**Review**

- ✅ 방향은 맞지만 **시나리오별 정답 문장/한 줄 코드**가 필요합니다.

**Ideal points (시나리오별 한 줄 해법)**

- (a) **VC ↔ ViewModel(콜백 클로저)**
    
    - 원인: VM이 보관한 클로저가 `self(VC)`를 strong 캡처.
        
    - 해법: `vm.onUpdate = { [weak self] in self?.render() }`

- (b) **delegate 패턴**
    
    - 원인: child가 owner(delegate)를 strong 보유 → owner가 child strong 보유와 순환.
        
    - 해법: `weak var delegate: SomeDelegate?`

- (c) **Timer / CADisplayLink**
    
    - 원인: 타이머 블록이 self를 strong 캡처, 타이머가 self에 의해 유지.
        
    - 해법:
        
        - 블록형: `Timer.scheduledTimer(...){ [weak self] _ in self?.tick() }` + **`timer.invalidate()`**
            
        - 타깃형: `Timer(target: proxy, selector: ..., repeats: ...)`(proxy 약한참조)
            
- (d) **NotificationCenter / Combine**
    
    - 원인: 블록 옵저버/AnyCancellable 보관 방식 착오.
        
    - 해법:
        
        - NC: 토큰을 보관 후 `deinit`에서 `NotificationCenter.default.removeObserver(token)`
            
        - Combine: `store(in: &bag)` + 클로저는 `[weak self]`

**Q3. @escaping / @MainActor / 클로저 캡처**  
`@escaping` 클로저가 **왜 순환 참조를 더 쉽게 만든다**고 하는지 설명하고, 다음 코드에서 **누수가 생기는지**와 **고치는 방법**을 말해보세요.  
힌트: 비동기 작업, 클로저 내부에서 `self` 캡처, UI 업데이트 위치.

```swift
final class ProfileVC: UIViewController {
  let api: API
  init(api: API) { self.api = api; super.init(nibName: nil, bundle: nil) }
  required init?(coder: NSCoder) { fatalError("init(coder:)") }

  func load() {
    api.fetchUser { user in
      self.title = user.name   // UI 업데이트
      self.startTimer()        // 타이머 시작
    }
  }

  func startTimer() {
    Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { _ in
      self.view.alpha.toggle()
    }
  }
}
```

**당신의 요지**

- `@escaping`은 외부 저장으로 위험. 타이머가 동기로 돌아가니 비동기로 하자.

**Review**

- ⚠️ 포인트가 빗나갔어요: 타이머 동기/비동기와 누수는 별개입니다.
    
- 핵심은 **비동기 경계(클로저/Task)** 가 **self를 오래 붙잡을** 수 있다는 점.

**Ideal points (핵심 패턴)**

- `Task { [weak self] in guard let self else { return } ... }`
    
- `@MainActor` 메서드라도 **@escaping 클로저/Task** 내부에서 self를 strong 캡처하면 누수 위험.
    
- `Task.detached`는 **actor-inheritance 없음** → 더 주의(캡처/동기화).
    
- **취소**는 메모리 해제와 동일하지 않음: 캡처가 끊기지 않으면 누수 지속.
    
- UI 변경은 `@MainActor` 또는 `await MainActor.run { ... }`.

**교정 예시**

```swift
@MainActor
func load() {
  Task { [weak self] in
    guard let self else { return }
    let user = try await api.fetchUser()
    self.title = user.name
    self.startTimer() // Timer는 invalidate 필요
  }
}

// Timer
private var timer: Timer?
func startTimer() {
  timer?.invalidate()
  timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] _ in
    self?.view.alpha.toggle()
  }
}
deinit { timer?.invalidate() }
```

**Q4. 누수 진단 루틴(실전 Instruments & Memory Graph)**  
Xcode **Memory Graph**와 **Instruments–Leaks/Allocations**로 누수를 찾는 본인 루틴을 단계별로 말해보세요.

- 최소 재현 케이스 만들기
    
- 의심 노드 식별(소유 경로/강한 엣지)
    
- “의도된 보유” vs “실제 누수” 구분법
    
- 흔한 함정 2가지와 회피법

**당신의 요지**

- (미답)

**Ideal points (나만의 루틴)**

1. **재현 시나리오 축소**: 화면 전환 N회, 특정 기능 반복 등.
    
2. **Memory Graph(Debug) 스냅샷**: 누수 의심 객체 찾아 **Owner Path**(강한 엣지) 추적.
    
3. **의도된 보유 vs 누수** 구분: 싱글톤/캐시/전역 큐에 의도적으로 보관 중인지 확인.
    
4. **핫스팟 패턴 점검**: Timer/DisplayLink 미무효화, NC 옵저버 미해제, delegate strong, `@escaping` 클로저 strong self.
    
5. **Instruments(Leaks/Allocations)**: 시간축에서 **증가하는 인스턴스 그래프** 확인, 책임 스택 추적.
    
6. **수정→재검증**: `[weak/unowned]`, 구조 변경, 해제 시점/무효화 추가 → 다시 측정.

**False positives(헷갈리기 쉬운 사례)**

- UIKit 내부 캐시/지연 해제, 디버그 빌드 최적화 차이, 한 번만 증가하고 안정화되는 패턴.
