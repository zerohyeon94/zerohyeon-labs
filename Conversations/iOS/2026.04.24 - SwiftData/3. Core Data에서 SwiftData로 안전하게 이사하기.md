**🗣️ 가볍게 시작하기 (Simple): "알고 보면 같은 집, 인테리어만 바꾸기"**

가장 중요한 핵심은 **SwiftData와 Core Data는 밑바탕(SQLite 파일)이 완전히 똑같다는 것**입니다! 유저의 핸드폰에 저장된 데이터 창고 자체를 부수고 새로 짓는 게 아니라, 창고의 문을 여는 '열쇠와 관리 시스템'만 최신식(SwiftData)으로 교체하는 작업입니다. 그래서 생각보다 이사가 아주 매끄럽게 진행됩니다.

---

**🤿 딥하게 들어가기 (Deep: 마이그레이션 3단계 실전 가이드)**

기존에 `.xcdatamodeld` 파일을 만들고 긴 코드로 Core Data를 세팅해 두셨다고 가정하고, 이를 어떻게 갈아엎는지 순서대로 살펴보겠습니다.

**Step 1: 똑같은 이름으로 `@Model` 클래스 찍어내기** 가장 먼저 할 일은 기존 Core Data의 Entity 이름과 속성(Attribute) 이름을 **토시 하나 틀리지 않고 똑같이** SwiftData의 `@Model` 클래스로 만드는 것입니다.

_기존 Core Data에 `FocusRecord`라는 Entity가 있었다면:_

```Swift
import SwiftData
import Foundation

// 기존 Entity 이름과 완벽하게 똑같아야 기존 데이터를 인식합니다!
@Model
class FocusRecord {
    var id: UUID
    var focusDate: Date
    var durationMinutes: Int
    var finalScore: Int

    init(focusDate: Date, durationMinutes: Int, finalScore: Int) {
        self.id = UUID()
        self.focusDate = focusDate
        self.durationMinutes = durationMinutes
        self.finalScore = finalScore
    }
}
```

**Step 2: 낡은 Core Data 코드 과감하게 지우기** 기존에 작성하셨던 수십 줄짜리 `NSPersistentContainer` 세팅 코드, 그리고 시각적 에디터인 `.xcdatamodeld` 파일은 이제 프로젝트에서 과감하게 삭제(Delete)하셔도 됩니다. 속이 다 시원해지는 순간입니다!

**Step 3: 기존 창고(SQLite) 주소를 찾아가도록 세팅하기** 새로 빈 창고를 만드는 게 아니라, 유저 핸드폰에 이미 존재하는 옛날 Core Data 창고 파일을 SwiftData가 읽어오도록 연결해 줘야 합니다. 앱의 가장 첫 진입점(`@main`)에 아래 코드를 작성합니다.

```Swift
import SwiftUI
import SwiftData

@main
struct ZipJoongApp: App {
    
    // 이사 관리자(Container)를 수동으로 세팅합니다.
    var sharedModelContainer: ModelContainer = {
        do {
            // 1. 기존 Core Data가 쓰던 저장소 이름과 완벽히 똑같이 적습니다.
            let storeURL = URL.documentsDirectory.appending(path: "ZipJoongDataModel.sqlite")
            
            // 2. "이 주소에 있는 파일을 열어줘!" 라고 설정합니다.
            let config = ModelConfiguration(url: storeURL)
            
            // 3. SwiftData용 새 캐비닛을 생성합니다.
            return try ModelContainer(for: FocusRecord.self, configurations: config)
        } catch {
            fatalError("데이터베이스 이사 실패: \(error)")
        }
    }()

    var body: some Scene {
        WindowGroup {
            FocusTimerView()
        }
        // 완성된 컨테이너를 앱 전체에 꽂아줍니다!
        .modelContainer(sharedModelContainer)
    }
}
```

### 🚨 현업 꿀팁: 데이터 구조가 완전히 바뀌었다면? (V1 ➔ V2)

만약 이사를 하면서 "기존에는 '점수(score)'만 있었는데, 이번 업데이트부터는 '눈 깜빡임 횟수(blinkCount)'라는 데이터도 추가하고 싶어!" 라고 구조를 바꿔버리면 앱이 실행되자마자 크래시가 나며 튕겨버립니다. 옛날 서류와 새 서류 양식이 맞지 않기 때문입니다.

이때는 **`SchemaMigrationPlan`**이라는 기능을 써서 "버전 1에서 버전 2로 넘어갈 때, 예전 기록들의 눈 깜빡임 횟수는 전부 0으로 채워줘!"라고 시스템에 마이그레이션 규칙을 아주 상세하게 적어줘야 합니다. (이 부분은 실무에서 데이터를 다룰 때 가장 심혈을 기울이는 부분입니다.)
