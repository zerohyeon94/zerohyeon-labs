**🗣️ 가볍게 시작하기 (Simple): "돼지코 플러그와 멀티탭"**

- **의존성 주입(DI) 없는 코드:** TV를 샀는데 전원선이 벽 속 전선에 아예 용접(Hardcoding)되어 있는 상태입니다. 이사 갈 때 TV를 떼어갈 수가 없죠. (테스트 불가, 재사용 불가)
    
- **의존성 주입(DI):** TV에 '돼지코 플러그(Protocol)'를 달아두는 겁니다. 220V 콘센트든, 발전기든, 가짜 배터리(Mock)든 구멍만 맞으면 밖에서 꽂아줄(Injection) 수 있습니다.
    
- **DI 컨테이너:** 집에 가전제품이 100개가 넘어가면 플러그 꽂는 것도 일입니다. 그래서 **"TV는 거실 콘센트에, 냉장고는 주방 콘센트에 꽂아줘!"라고 규칙을 적어둔 거대한 '스마트 멀티탭'**이 바로 DI 컨테이너입니다.

---

**🤿 딥하게 들어가기 (Deep: 왜 컨테이너까지 필요할까?)**

`FocusSense` 앱을 만들 때, `TimerViewModel`이 `FocusMLService`를 필요로 한다고 해보겠습니다.

**❌ 컨테이너가 없을 때의 고통 (의존성 릴레이)** 앱의 시작점(`App` 또는 `SceneDelegate`)에서 뷰모델을 만들 때, 서비스 객체를 직접 만들어서 손수 꽂아줘야 합니다.

```Swift
// 앱 시작점
let mlService = FocusMLService()
let networkService = QuoteNetworkManager()
// 필요한 걸 다 넣어서 조립... (파라미터가 10개면 끔찍해집니다)
let timerVM = TimerViewModel(mlService: mlService, networkService: networkService)
let timerView = TimerView(viewModel: timerVM)
```

화면이 깊어질수록, A 화면에서 B 화면으로 넘어갈 때 B가 필요한 객체를 A가 받아서 전달해 줘야 하는 '택배 기사' 노릇을 하게 됩니다.

**✅ DI 컨테이너 도입 후 (스마트 멀티탭)** 앱이 켜질 때, 이 거대한 '멀티탭'에 필요한 객체들을 미리 다 등록(Register)해 둡니다. 그리고 화면(ViewModel)을 만들 때는 그냥 **"멀티탭아, 나 ML 서비스 필요하니까 하나만 꺼내줘(Resolve)!"** 라고 외치기만 하면 됩니다.

---

**💻 실전 코드: 초간단 DI Container 직접 만들어보기**

현업에서는 `Swinject`나 `Needle` 같은 거대한 외부 라이브러리를 쓰기도 하지만, 딕셔너리(`Dictionary`)를 이용해 30줄이면 아주 훌륭한 나만의 컨테이너를 만들 수 있습니다.

```Swift
import Foundation

// 1. 스마트 멀티탭 (DI Container)
class DIContainer {
    static let shared = DIContainer() // 앱 전체에서 하나만 존재 (싱글톤)
    
    // 객체들을 담아둘 딕셔너리 (금고)
    private var dependencies: [String: Any] = [:]
    
    // 2. 금고에 넣기 (Register)
    func register<T>(type: T.Type, component: Any) {
        let key = String(describing: type)
        dependencies[key] = component
    }
    
    // 3. 금고에서 꺼내기 (Resolve)
    func resolve<T>(type: T.Type) -> T {
        let key = String(describing: type)
        return dependencies[key] as! T
    }
}
```

**🚀 실제 앱에서는 이렇게 씁니다!**

```Swift
// 1. 앱이 켜질 때 세팅 (보통 AppDelegate나 @main 안에서)
DIContainer.shared.register(type: QuoteServiceProtocol.self, component: RealQuoteManager())
DIContainer.shared.register(type: FocusMLProtocol.self, component: FocusMLManager())

// 2. 뷰모델에서 꺼내 쓰기
class TimerViewModel: ObservableObject {
    private let quoteService: QuoteServiceProtocol
    private let mlService: FocusMLProtocol
    
    // 파라미터로 안 받고, 컨테이너에서 직접 꺼내옵니다!
    init() {
        self.quoteService = DIContainer.shared.resolve(type: QuoteServiceProtocol.self)
        self.mlService = DIContainer.shared.resolve(type: FocusMLProtocol.self)
    }
}
```

### 🚨 모듈화 아키텍처와 DI 컨테이너의 환상적인 궁합

이전 수다에서 앱을 `TimerFeature` (타이머 화면 모듈)와 `FocusMLKit` (로직 모듈)으로 쪼갰었죠?

만약 DI 컨테이너가 없다면, `TimerFeature`는 반드시 `FocusMLKit`을 `import`해서 직접 코드를 결합해야 합니다. 하지만 DI 컨테이너를 쓰면, `TimerFeature`는 **"나는 `FocusMLProtocol`이라는 규격(돼지코)만 있으면 돼!"**라고 선언하고, 진짜 `FocusMLKit` 객체는 가장 껍데기인 `App` 모듈에서 컨테이너를 통해 슬쩍 꽂아주게 됩니다.

즉, 모듈 간의 결합도가 완전히 끊어지면서 진짜 '독립적인 레고 블록'이 완성되는 것입니다!

**💡 오늘의 요약 처방**

> **DI 컨테이너**는 수많은 서비스와 뷰모델들을 딕셔너리 같은 곳에 모아두고, **필요한 곳에서 언제든 꺼내 쓸 수 있게(Resolve) 해주는 전역 객체 관리소**입니다.

이 컨테이너의 개념까지 장착하셨다면, iOS 앱 설계(Architecture)의 가장 깊은 심연까지 완벽하게 내려갔다 오신 겁니다.
