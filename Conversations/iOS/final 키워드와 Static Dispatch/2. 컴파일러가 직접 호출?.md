### **[Daily iOS Chat #7 - Deep Dive]** **🗣️ 딥한 수다: 컴파일러가 '직접 호출' 한다는 게 무슨 뜻일까?**

사용자님, 펌웨어(C언어)를 하셨으니 이 개념이 아주 반가우실 겁니다. C언어의 가장 기본적인 함수 호출 방식이 바로 **직접 호출(Direct Dispatch)**이거든요!

이걸 이해하려면 **'컴파일 시점'**과 **'런타임(실행) 시점'**을 나눠서 생각하면 아주 명확해집니다.

---

### 1. 직접 호출 (Direct Dispatch / Static Dispatch)

> **"야, 0x1004번지로 점프해서 코드 실행해!"**

- **상황:** C언어에서 일반적인 함수(`void func()`)를 만들고 호출할 때의 상황입니다.
    
- **컴파일러의 생각:** "이 함수는 오버라이딩될 리가 없어. 코드가 메모리 어디에 박혀있는지 내가 정확히 알고 있지."
    
- **동작:** 컴파일러는 함수가 위치한 **메모리 주소값을 코드에 그대로 박아버립니다(Hard-coding).**
    
- **실행:** 앱이 실행되다가 함수 호출 코드를 만나면, 딴짓 안 하고 **바로 그 주소로 점프(Jump)**합니다. 가장 빠릅니다.

### 2. 간접 호출 (Dynamic Dispatch / Table Dispatch)

> **"음... 0x1004번지로 가려고 했는데... 잠깐만, 이 객체가 자식 클래스일 수도 있잖아? 함수 목록(V-Table) 좀 확인해보고 올게."**

- **상황:** Swift의 일반적인 `class` 메서드를 호출할 때입니다. (상속 가능성 있음)
    
- **컴파일러의 생각:** "사용자가 `Animal` 클래스의 `cry()`를 호출했는데, 실제 런타임에 들어오는 게 `Dog`인지 `Cat`인지 지금은 확신할 수 없어."
    
- **동작:** 주소로 바로 점프하지 않고, **"이 객체의 함수 테이블(Virtual Table)을 보고, 맞는 함수 주소를 찾아서 실행해라"**라는 명령어를 넣어둡니다.
    
- **실행:**
    
    1. 객체의 메타데이터로 이동
        
    2. `witness_table` (함수 주소록) 조회
        
    3. `cry()` 함수가 진짜 어디 있는지 주소 확인
        
    4. 그제서야 점프
    
    - → **과정이 복잡해서 약간 느립니다.**

---

### 💡 펌웨어 개발자 맞춤 요약

C언어 하실 때를 떠올려보세요.

1. **직접 호출:** 그냥 `main()`에서 `led_on()` 함수 부르는 것. (주소로 바로 `CALL`)
    
2. **간접 호출:** **함수 포인터**를 사용하는 것. (포인터 변수에 담긴 주소값을 읽어온 뒤에 그곳으로 `CALL`)

Swift에서 `final`을 붙인다는 건, **"이 클래스는 상속 안 할 거니까, 복잡하게 함수 포인터(Table) 뒤지지 말고 그냥 C언어처럼 주소로 바로 쏴버려!"** 라고 컴파일러에게 확신을 주는 행위입니다.
