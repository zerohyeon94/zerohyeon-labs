### `formWeakReference`와 Side Table의 비밀

**🗣️ 가볍게 시작하기 (Simple)**

우선 저 길고 복잡한 에러 메시지를 사람이 읽을 수 있게 번역해 볼까요?

- `malloc`: "메모리(Heap)를 새로 할당했다."
    
- `swift::RefCounts`: "Swift의 참조 카운트(Reference Count)를 관리하는 곳에서"
    
- `formWeakReference()`: "**`weak` 참조를 만들면서!**"

즉, **"개발자가 코드 어딘가에서 `weak var`나 `[weak self]`를 사용해서, 그걸 시스템이 처리하느라 메모리를 조금 썼다"**는 뜻입니다. 지극히 정상적인 동작이니 일단 안심하셔도 됩니다!

---

**🤿 딥하게 들어가기 (Deep: 왜 `weak`를 쓰는데 `malloc`이 발생할까?)**

여기서 정말 재밌는 iOS의 딥한 원리가 등장합니다. 바로 **사이드 테이블(Side Table)**이라는 개념입니다.

**1. 평상시의 객체 (Inline Ref Count)** Swift의 클래스 객체(예: `TimerManager`)가 메모리에 올라가면, 객체 내부에 숨겨진 변수 2개가 자동으로 생깁니다.

- **Strong Count** (나를 강하게 참조하는 횟수)
    
- **Unowned Count** (나를 미소유 참조하는 횟수) 이 카운트들은 객체 몸통 안에 딱 붙어있습니다(Inline). 빠르고 효율적이죠.

**2. `weak`가 등장하는 순간!** 그런데 우리가 코드에 `weak var delegate`를 선언하는 순간, 시스템은 고민에 빠집니다.

- `weak`의 특징이 뭔가요? 객체가 죽으면 자동으로 `nil`이 되어야 하죠?
    
- 객체가 메모리에서 날아갔는데, 그 객체를 가리키던 `weak` 변수들을 찾아서 전부 `nil`로 바꿔주려면 **별도의 '연락처 목록'**이 필요해집니다.

**3. Side Table 생성 (이때 `malloc` 발생!)** 그래서 Swift는 객체 몸통 안에 있던 카운트 정보를 밖으로 빼내서 **'사이드 테이블(Side Table)'**이라는 새로운 데이터 공간을 메모리(Heap)에 몰래 만듭니다. (`malloc` 호출!)

- 이제 객체는 사이드 테이블의 주소만 들고 있고,
    
- 사이드 테이블 안에 Strong, Unowned, 그리고 **Weak Count**가 모두 저장됩니다.

---

### 🕵️‍♂️ Memory Graph에서 이 녀석을 마주친 이유

Memory Graph Debugger에서 저 `malloc` 로그가 보였다는 것은, 시스템이 사이드 테이블을 만든 그 찰나의 흔적(메모리 블록)을 보여주는 것입니다.

**🤔 그럼 이게 누수(Leak)인가요?**

- **아닐 확률이 높습니다:** 객체가 살아있어서 `weak` 참조를 유지하기 위해 사이드 테이블이 존재하는 것은 당연합니다.
    
- **누수일 수도 있는 경우:** 만약 메인 객체(예: 화면)는 분명히 죽었는데, 저 `formWeakReference`로 만들어진 메모리 블록만 덩그러니 좀비처럼 남아있다면? 이건 누군가가 클로저 안에서 `[weak self]`를 써놓고, 그 클로저 자체를 영원히 놔주지 않아서 발생하는 미세한 찌꺼기일 수 있습니다.

3. 그 클래스가 정상적으로 메모리에서 해제되고 있다면, 완벽하게 코드를 짜신 겁니다!
