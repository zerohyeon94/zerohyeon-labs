```swift
headers["Authorization"] = "Bearer \(token)"
headers["Accept"] = "application/json"
```

같은 것도 “요청 헤더(request headers)”가 맞습니다. 다만 캐시/유효성 검사에서 말하는 “요청 헤더”는 보통 아래 조건부·캐시 제어용 헤더들을 가리켜요. 즉, 용도가 다른 여러 종류의 요청 헤더가 있고(인증, 포맷 협상, 조건부 요청 등), 지금은 그중 “조건부 요청”을 강조한 거였어요.

## 요청 헤더는 크게 이렇게 나뉩니다

1. **인증/권한**
	- `Authorization: Bearer <token>`
    
	- 목적: 서버가 **누구인지** 확인
    
	- 캐시 관점: 보통 인증 응답은 서버가 `Cache-Control: no-store` 등으로 **캐시 금지**를 명시하는 경우가 많음(민감 데이터 보호).
    
2. **콘텐츠 협상**
    
	- `Accept: application/json`
    
	- 목적: 응답 **형식 지정**(JSON, 이미지 등)
    

3. **조건부 요청(유효성 검사 / 캐시 절약) ← 캐시에서 말하는 “요청 헤더”**
    
	- `If-None-Match: "<ETag값>"` → 서버가 “변화 없음”이면 **304 Not Modified**로 바디 생략
    
	- `If-Modified-Since: <date>` → 마지막 변경 시간 기준으로 변화 없으면 **304**
    
	- 목적: **바뀌지 않은 리소스는 다시 안 내려받기** (네트워크 절약)


> 이 조건부 헤더를 **클라이언트가 붙이는 기준**은, 이전 응답이 갖고 있던 **응답 헤더**(예: `ETag`, `Last-Modified`, `Cache-Control`)예요. 즉,
> 
> - 첫 요청(200 OK, ETag: "abc") 저장
>     
> - 다음 요청에서 `If-None-Match: "abc"`를 보내 검증
>     
> - 안 바뀌었으면 304 → 캐시 재사용, 바뀌었으면 200 새 바디

## Swift에서 조건부 요청 붙이는 예시

### URLSession(직접)

```swift
var req = URLRequest(url: url)
req.httpMethod = "GET"
req.addValue("application/json", forHTTPHeaderField: "Accept")
if let token = TokenManager.shared.currentAccessToken() {
    req.addValue("Bearer \(token)", forHTTPHeaderField: "Authorization")
}

// 이전 응답에서 저장해 둔 ETag를 꺼내 조건부 요청으로 전송
if let etag = etagStore.value(for: url) {
    req.addValue(etag, forHTTPHeaderField: "If-None-Match")
}

let task = URLSession.shared.dataTask(with: req) { data, resp, err in
    guard let http = resp as? HTTPURLResponse else { return }
    if http.statusCode == 304 {
        // 바디 없음 → 캐시에 둔 데이터 재사용
        let cached = cacheStore.loadBody(for: url)
        // ... 사용
    } else if http.statusCode == 200 {
        // 새 바디 저장 + 새 ETag 저장
        if let newETag = http.allHeaderFields["ETag"] as? String {
            etagStore.save(newETag, for: url)
        }
        cacheStore.saveBody(data, for: url)
    }
}
task.resume()
```

### Alamofire(흐름만)

Alamofire도 결국 `URLRequest`를 감싸므로 **헤더 추가/응답 헤더 읽기**는 동일합니다.

```swift
var headers: HTTPHeaders = ["Accept": "application/json"]
if let token = TokenManager.shared.currentAccessToken() {
    headers.add(name: "Authorization", value: "Bearer \(token)")
}
if let etag = etagStore.value(for: url) {
    headers.add(name: "If-None-Match", value: etag)
}

AF.request(url, method: .get, headers: headers)
  .responseData { resp in
    if resp.response?.statusCode == 304 {
        // 캐시 재사용
    } else if resp.response?.statusCode == 200 {
        if let newETag = resp.response?.value(forHTTPHeaderField: "ETag") {
            etagStore.save(newETag, for: url)
        }
        // 바디 저장
    }
  }
```

## 정리 포인트(면접/실무)

- **Authorization/Accept**도 요청 헤더 맞음.
    
- **캐시와 관련된 “요청 헤더”**는 주로 **조건부 요청 헤더**(If-None-Match / If-Modified-Since)를 뜻함.
    
- 조건부 요청은 **서버가 이전에 준 응답 헤더(ETag/Last-Modified)** 를 기반으로 동작.
    
- 인증이 달려 있는 API는 서버가 보통 `no-store`로 캐시를 막지만, **변경 빈도가 낮은 리소스**(예: 이미지/정적 설정)는 인증이 있어도 **ETag 재검증**을 허용하기도 함(서버 정책에 따라).
    
- 클라이언트에서 강제로 캐시 무시: `URLRequest.cachePolicy = .reloadIgnoringLocalCacheData`
    
- 오프라인에선 `.returnCacheDataDontLoad` 같은 정책으로 **stale 캐시 우선 표시** → 연결 복구 후 재검증
