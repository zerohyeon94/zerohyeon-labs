# **Q1. URLCache & Cache-Control**  

`URLCache`와 HTTP 헤더(**Cache-Control**, **ETag/If-None-Match**)가 어떻게 상호작용하나요?  
다음 상황별로 요청/응답 흐름을 설명하세요:

- “신선하면 캐시 쓰고, 아니면 유효성 검사”
- “무조건 네트워크 강제”
- “오프라인: 일단 오래된 캐시를 보여주고, 이후 갱신”

### **내 답변 요지**

- “URLCache는 응답을 메모리에 저장. 토큰이 유효하면 보관. 무조건 네트워크 강제 시 에러 가능. 오프라인이면 공백 데이터.”

### **피드백**

- 장점: “캐시가 있다”는 감각은 OK.
    
- 보완: 캐시 동작은 **토큰 유효성**이 아니라 **HTTP 캐시 규칙**(헤더)로 결정됩니다. 핵심은:
    
    - 서버 응답 헤더: **Cache-Control**, **Expires**, **ETag/Last-Modified**
        
    - 요청 헤더/정책: **If-None-Match**, **If-Modified-Since**, `URLRequest.cachePolicy`
        
    - `URLCache`는 위 헤더를 기준으로 **저장/조회/검증**을 자동 처리.
        
- 상황별 정답 흐름:
    
    - “신선하면 캐시, 아니면 유효성 검사”:
        
        1. 캐시 항목이 **fresh** → 즉시 캐시 반환
            
        2. **stale** → `If-None-Match`로 **검증 요청** → 304이면 바디 없이 캐시 재사용, 200이면 새로 저장
            
    - “무조건 네트워크 강제”: `cachePolicy = .reloadIgnoringLocalCacheData`
        
    - “오프라인, 일단 오래된 캐시 보여주고 나중에 갱신”:
        
        - 앱 레벨에서 **오프라인 감지** 후 **stale 캐시 허용**(정책은 `.returnCacheDataDontLoad` 등을 조합) → 네트워크 가능해지면 **백그라운드로 재검증/갱신**
            

## **업그레이드 답변 예시**

> “`URLCache`는 **HTTP 캐시 헤더**(Cache-Control/ETag)로 동작합니다. 기본은 ‘fresh면 캐시, stale면 서버에 **If-None-Match**로 검증(304면 캐시 재사용)’입니다. 강제 네트워크는 `.reloadIgnoringLocalCacheData`, 오프라인은 `.returnCacheDataDontLoad` 등으로 ‘stale 우선 표시 후 나중 갱신’ 패턴을 씁니다.”

### **Ideal Answer Points**

- 서버: `Cache-Control: max-age=..., must-revalidate`, `ETag: "abc"`
    
- 클라이언트: `If-None-Match: "abc"` → 304
    
- `URLRequest.cachePolicy` 주요 값: `.useProtocolCachePolicy`(기본), `.reloadIgnoringLocalCacheData`, `.returnCacheDataElseLoad`, `.returnCacheDataDontLoad`

# **Q2. 재시도 전략(Exponential Backoff + Jitter)**  

아이들포턴트 GET/HEAD에 한정해 재시도 매트릭스를 설계하세요.
- 어떤 오류/상태코드를 재시도할지
- 최대 시도 횟수, **지수 백오프 + Jitter** 공식
- `async/await` 취소 연동과 앱 포그라운드/백그라운드 전환 시 고려사항

### **내 답변 요지**

- “아이들 포턴트(=idempotent) 모른다.”

### **피드백**

- 핵심 개념:
    
    - **Idempotent(멱등)**: 같은 요청을 여러 번 보내도 **서버 상태가 변하지 않는** 메서드 (GET/HEAD/PUT/DELETE는 보통 멱등, POST는 보통 비멱등).
        
- 재시도 설계(요점):
    
    - **대상**: 네트워크 오류(타임아웃/연결 끊김), 5xx, 429
        
    - **제외**: 4xx(특히 400/401/403/404), 비멱등 요청(POST 등)은 기본 미재시도
        
    - **Backoff**: `delay = base * 2^attempt + jitter` (jitter는 0~base 범위 난수)
        
    - **최대 시도**: 2~4회 수준 + **전체 time budget**
        
    - **취소/전환**: `Task` 취소 반영, `waitsForConnectivity = true` + `NWPathMonitor`로 “연결 회복 시도”와 “즉시 실패”의 경계 설정

## **업그레이드 답변 예시**

> “멱등 요청(GET 등)만 재시도합니다. 5xx/네트워크 에러/429에 한해 **지수 백오프 + 지터**로 3회 내 재시도, 각 시도는 `Task` 취소를 존중합니다. `waitsForConnectivity`와 `NWPathMonitor`로 오프라인에선 곧바로 실패/큐잉을 선택하고, 포그라운드 복귀 시 재시작합니다.”

### **Ideal Answer Points**

- 멱등성 정의 & 메서드 예시
    
- 재시도 매트릭스(재시도/미재시도)
    
- `waitsForConnectivity`, `NWPathMonitor`
    
- backoff+jitter 공식과 상한

# **Q3. Background Session의 생명주기**  

`URLSessionConfiguration.background`을 사용할 때 앱이 종료/재실행되는 동안 **delegate 이벤트 인계**를 어떻게 보장하나요?  
필수 콜백/식별자/파일 처리 위치를 짚어주세요.

### **내 답변 요지**

- “잘 모르겠다.”

### **피드백**

- 핵심 흐름:
    
    1. **고정 식별자**로 `URLSessionConfiguration.background("com.app.bg")` 생성
        
    2. 앱 종료/재시작 후에도 **시스템이 작업 지속**
        
    3. 재실행 시 AppDelegate:  
        `application(_:handleEventsForBackgroundURLSession:completionHandler:)`  
        → 동일 식별자로 세션 재생성 → **delegate 연결 복구**
        
    4. 다운로드 완료 콜백: `didFinishDownloadingTo`에서 **파일 이동/처리**
        
    5. 모든 이벤트 처리 완료 시, 저장해둔 `completionHandler()` 호출
        
    6. 정책/제약: `discretionary`, `allowsExpensiveNetworkAccess`, `allowsConstrainedNetworkAccess`, 배터리/데이터 절약 모드 반영

## **업그레이드 답변 예시**

> “백그라운드 세션은 **식별자 기반**이라 앱이 죽어도 작업이 계속됩니다. 재실행 시 AppDelegate의 `handleEventsForBackgroundURLSession`에서 **동일 식별자로 세션을 복원**하고 delegate를 붙인 뒤, `didFinishDownloadingTo`에서 파일을 이동/저장합니다. 모든 delegate 이벤트 처리가 끝나면 시스템이 준 `completionHandler`를 호출해야 알림이 정상 표시됩니다.”

**Ideal Answer Points**

- background 식별자
    
- AppDelegate `handleEventsForBackgroundURLSession`
    
- delegate 복구 & 파일 처리 위치
    
- `discretionary`/비용/제약 네트워크 플래그

# **Q4. 안전한 이미지 로딩 파이프라인**  

이미지 로더(Nuke/Kingfisher 또는 직접 구현)를 쓴다고 가정.
- ATS/HTTPS 전제에서, **캐싱(메모리/디스크)**, **프리페치**, **네트워크 절약(ETag/HTTP 304)**, **저품질 네트워크 적응**(리사이즈/압축)까지 포함한 아키텍처를 개략 설계하세요.
- 두 가지 주요 함정을 들고, 완화책을 제시하세요.

### **내 답변 요지**

- Nuke 사용: URL 변환 실패 시 기본 이미지, 성공 시 로드, 실패 시 기본 이미지.

### **피드백**

- 장점: “기본 이미지 폴백”은 UX 좋음.
    
- 보완(안정성/네트워크 절약/품질):
    
    - **ATS/HTTPS** 전제
        
    - **메모리/디스크 캐시** 레이어 구분(용량/만료 정책 설정)
        
    - **ETag/If-None-Match(304)** 지원으로 **네트워크 절약**
        
    - **프리페치**(화면에 나타나기 전 Warm-up)
        
    - **저품질 네트워크 대응**: 서버 리사이즈 API나 `URLSessionTaskMetrics` 기반 적응(저해상도/Progressive JPEG/WebP/HEIF)
        
    - **Pitfalls**:
        
        1. 대형 이미지로 인한 **메모리 압박** → 다운샘플링 / `preferredTransformer`
            
        2. 스크롤 빠른 화면에서 **중복 요청** → deduplication/priority/cancel on reuse

## **업그레이드 답변 예시**

> “이미지 로딩은 Nuke의 **메모리+디스크 캐시**를 활용하고, 서버가 **ETag**를 주면 재검증으로 304를 받아 트래픽을 줄입니다. 스크롤 목록은 **프리페치+dedupe+cancel on reuse**로 낭비를 줄이고, 저대역폭에선 다운샘플/리사이즈된 변형을 요청합니다.”

### **Ideal Answer Points**

- 캐시(메모리/디스크)·프리페치·304 재검증
    
- 다운샘플링/중복요청 방지/우선순위/취소
    
- ATS 전제와 실패 폴백
