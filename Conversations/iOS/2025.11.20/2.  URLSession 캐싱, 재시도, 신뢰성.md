## Q1. 캐시(fresh/stale) & 정책

**당신의 요지(요약):** fresh면 캐시, 네트워크 강제는 policy, 오프라인은 오래된 캐시 표시 후 갱신.  
**교정 & 보강:** ✅ 방향 맞습니다. 단, **“fresh/stale 판단은 HTTP 캐시 헤더**(Cache-Control/ETag/Last-Modified)**로 결정**됩니다. 토큰 유무가 기준은 아니며, 인증 API는 보통 서버가 `no-store`로 캐시를 끕니다.

**업그레이드 한 줄 답**

> “`URLCache`는 **HTTP 캐시 헤더**로 동작한다. fresh면 바로 캐시, stale면 **If-None-Match** 등으로 **304** 검증, 강제 네트워크는 `cachePolicy = .reloadIgnoringLocalCacheData`, 오프라인은 `.returnCacheDataDontLoad`로 **stale 우선 후 재검증**.”

**스니펫**

```swift
// 전역 캐시 용량 설정(앱 시작 시 1회)
URLCache.shared = URLCache(memoryCapacity: 50*1024*1024,
                           diskCapacity: 200*1024*1024,
                           diskPath: "urlcache")

// 오프라인일 때는 stale 우선
var req = URLRequest(url: url)
req.cachePolicy = .returnCacheDataDontLoad  // 오프라인 분기에서만
```

## Q2. 멱등(idempotent) & 재시도

**당신의 요지(요약):** 서버 데이터가 변하지 않을 때만 재시도, GET만 재시도, 오프라인은 실패 보여주고 포그라운드에 재시도.  
**교정 & 보강:** 핵심 기준은 **“요청의 멱등성”**입니다. “서버 데이터가 변하느냐”가 아니라 **요청을 여러 번 보내도 같은 결과인가**가 기준(보통 GET/HEAD/PUT/DELETE 멱등, POST는 비멱등). 재시도는 **네트워크 오류/5xx/429** 중심 + **지수 백오프+지터**.

**업그레이드 한 줄 답**

> “**멱등 요청만** 재시도(5xx/429/네트워크 오류). `delay = base * 2^n + jitter`. `Task` 취소 전파, `waitsForConnectivity`/`NWPathMonitor`로 오프라인 시도 억제.”

**스니펫(개념)**

```swift
func shouldRetry(_ resp: HTTPURLResponse?, _ err: URLError?) -> Bool {
    if let e = err { return [.timedOut, .cannotConnectToHost, .networkConnectionLost].contains(e.code) }
    guard let s = resp?.statusCode else { return false }
    return s == 429 || (500...599).contains(s)
}
```

## Q3. Background `URLSession` 생명주기

**당신의 요지(요약):** 앱이 죽어도 진행, AppDelegate에서 완료 핸들러로 알림.  
**교정 & 보강:** ✅ 큰 방향 OK. 구체 포인트를 붙이세요.

- **식별자 기반**: `URLSessionConfiguration.background("com.app.bg")`
    
- 재실행 시 **AppDelegate**:  
    `application(_:handleEventsForBackgroundURLSession:completionHandler:)` 에서  
    동일 식별자로 세션 **복원 → delegate 연결**
    
- 파일은 `didFinishDownloadingTo`에서 **이동/저장**
    
- 모든 delegate 이벤트 처리가 끝난 뒤 **저장한 completionHandler() 호출**

**스니펫(핵심 흐름)**

```swift
// AppDelegate
var bgCompletion: (() -> Void)?

func application(_ app: UIApplication,
  handleEventsForBackgroundURLSession identifier: String,
  completionHandler: @escaping () -> Void) {
  bgCompletion = completionHandler
  backgroundSession = URLSession(configuration: .background(withIdentifier: identifier),
                                 delegate: self, delegateQueue: nil)
}

// URLSessionDelegate 마지막 이벤트에서
bgCompletion?(); bgCompletion = nil
```

## Q4. Nuke 이미지 파이프라인

**당신의 요지(요약):** URL 인코딩/HTTPS 업그레이드, 리사이즈 프로세서, placeholder/실패 처리.  
**교정 & 보강:** 아주 좋아요. 여기에 **캐시/재검증/프리페치/다운샘플링/중복요청 방지**를 붙이면 완성.

- **공유 파이프라인**: 메모리+디스크(DataCache) 용량 지정
    
- **HTTP 재검증(304)**: 기본 `URLSession` + `URLCache`가 헤더 기반으로 자동 활용(서버가 ETag 제공해야 함)
    
- **프리페치/중복요청 병합**: Nuke 기본 coalescing + `ImagePrefetcher`
    
- **다운샘플링**: 지금처럼 `Resize`는 OK. 아주 큰 이미지엔 `ImageProcessors.Downsample`도 고려
    
- **오프라인**: 네트워크 불가 시 캐시 우선(상황에 따라 `returnCacheDataDontLoad` 세션 사용)

**스니펫(초기 구성)**

```swift
var conf = ImagePipeline.Configuration()
conf.isTaskCoalescingEnabled = true
conf.dataCache = try? DataCache(name: "img-cache")
conf.dataCache?.sizeLimit = 200 * 1024 * 1024 // 200MB
let pipeline = ImagePipeline(configuration: conf)
ImagePipeline.shared = pipeline

// 프리페쳐
let prefetcher = ImagePrefetcher(pipeline: .shared)
prefetcher.startPrefetching(with: urls) // 화면 진입 전에 미리
```
