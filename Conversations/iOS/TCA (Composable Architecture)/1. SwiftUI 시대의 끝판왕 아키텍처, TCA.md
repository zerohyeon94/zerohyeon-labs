**🗣️ 가볍게 시작하기 (Simple): "단방향 컨베이어 벨트"**

우리가 앞서 배웠던 MVVM에서는 View가 ViewModel의 함수를 찌르고, ViewModel은 `@Published` 값을 바꿔서 View를 다시 그렸습니다. 조금 복잡해지면 화살표가 이리저리 엉키는 '거미줄'이 되기 쉽죠.

반면 TCA는 아주 엄격한 **'단방향 순환 컨베이어 벨트(Unidirectional Data Flow)'**입니다.

1. 유저가 버튼을 누른다 (**Action**)
    
2. 컨베이어 벨트 관리자가 규칙에 따라 데이터를 바꾼다 (**Reducer**)
    
3. 바뀐 데이터가 화면에 반영된다 (**State**)

---

**🤿 딥하게 들어가기 (Deep: TCA를 지탱하는 4가지 기둥)**

TCA는 어떤 기능(Feature)을 만들든 무조건 아래 4가지 요소를 만들도록 강제합니다. 집중 타이머 앱을 예로 들어볼까요?

**1. State (상태: 화면에 보여질 모든 데이터)** 화면에 띄워야 할 모든 변수를 하나의 구조체(`struct`)로 모아둡니다.

```Swift
struct State: Equatable {
    var score: Int = 0
    var isTimerRunning: Bool = false
}
```

**2. Action (행동: 앱에서 일어나는 모든 사건)** 유저가 누르는 버튼, 혹은 서버에서 데이터가 도착하는 사건 등을 `enum`으로 모두 나열합니다.

```Swift
enum Action {
    case startButtonTapped
    case stopButtonTapped
    case scoreUpdate(Int) // ML 모델이 점수를 계산해서 던져주는 사건
}
```

**3. Dependency (의존성: 외부 세상과의 소통)** 우리가 앞서 만들었던 **DI 컨테이너** 기억나시나요? TCA는 이 DI 시스템을 프레임워크 자체에 아예 내장하고 있습니다! 네트워크 통신이나 CoreML 같은 외부 작업들을 이곳에 등록합니다. (과거에는 Environment라고 불렀습니다.)

**4. Reducer (리듀서: 핵심 뇌 🧠)** MVVM의 ViewModel 역할을 하지만, 훨씬 엄격합니다. **"현재 State에서, 어떤 Action이 들어오면, 다음 State는 어떻게 변해야 해!"**라는 규칙을 적어두는 순수한 수학 함수 같은 곳입니다.

```Swift
// 리듀서의 대략적인 형태
func reduce(into state: inout State, action: Action) -> Effect<Action> {
    switch action {
    case .startButtonTapped:
        state.isTimerRunning = true
        // ML 서비스(Dependency)를 가동시키는 외부 작업(Effect)을 리턴함
        return .run { send in ... } 
        
    case .stopButtonTapped:
        state.isTimerRunning = false
        return .none // 상태만 바꾸고 끝!
    }
}
```

### 🚨 왜 다들 TCA에 열광할까요? (장점 3대장)

**1. 이름부터 'Composable (조립 가능한)'** 우리가 이전 수다에서 **모듈화 아키텍처**를 배웠죠? TCA는 애초에 이 모듈화를 위해 태어났습니다. 타이머 화면(TimerFeature)과 기록 화면(HistoryFeature)을 각각 독립적으로 잘게 만든 다음, 레고 블록처럼 딸깍! 하고 하나로 합치는 기능이 프레임워크 차원에서 기가 막히게 지원됩니다.

**2. 소름 돋는 테스트(Unit Test) 경험** 이전에 뷰모델 테스트할 때 XCTest로 `setUp` 하고 이것저것 세팅했던 것 기억하시나요? TCA는 테스트를 할 때 **"내가 `startButtonTapped` 액션을 보낼 건데, 그러면 `isTimerRunning`이 `true`로 바뀔 거야"**라고 선언하면, 내부적으로 상태가 100% 똑같이 변했는지 프레임워크가 알아서 소름 돋을 정도로 엄격하게 검사해 줍니다. 테스트 짜는 맛이 최고입니다.

**3. 부작용(Side Effect)의 완벽한 통제** 네트워크 통신이나 타이머처럼 언제 끝날지 모르는 작업들을 `Effect`라는 캡슐에 가둬서 철저하게 통제합니다. 앱이 내 예상과 다르게 튀는 버그를 원천 차단합니다.

---

**🤔 그럼 TCA의 단점은요?**

- **극악의 러닝 커브:** 처음 배울 때 알아야 할 개념이 너무 많고, Swift의 최신 문법(매크로 등)을 한계까지 끌어다 씁니다.
    
- **거대한 의존성:** Point-Free라는 회사에서 만든 강력한 외부 라이브러리(`swift-composable-architecture`)에 내 앱의 운명을 100% 맡겨야 합니다. 애플이 SwiftUI를 대대적으로 업데이트할 때마다 TCA 라이브러리가 대응해 줄 때까지 기다려야 하죠.

---

**💡 오늘의 핵심 요약**

> **TCA**는 상태(State), 행동(Action), 로직(Reducer)을 완벽하게 분리하여, 데이터가 무조건 **한 방향으로만 흐르도록 강제**하는 아키텍처입니다. 대규모 SwiftUI 앱을 **레고처럼 조립**하고 **테스트**하기에 현존하는 가장 강력한 도구입니다.

어떠신가요? MVVM, 코디네이터, 모듈화, 테스트, DI까지 우리가 여태껏 쌓아온 지식들이 **TCA라는 하나의 프레임워크 안에 모조리 녹아들어가 있다는 것**이 느껴지시나요?
