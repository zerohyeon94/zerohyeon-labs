### 노란색 경고창(⚠️)과 RxSwift의 반란

**🗣️ 가볍게 시작하기: 지금 무슨 일이 벌어진 건가요?**

스크린샷 우측 끝을 보시면 모든 항목에 **노란색 느낌표 경고창(⚠️)**이 떠 있는 것이 보이실 겁니다. 이 마크는 Xcode가 강력하게 **"여기 메모리 누수(Leak) 났어요! 화면을 닫았는데도 객체들이 안 죽고 살아있어요!"**라고 소리치고 있는 것입니다.

즉, 앱의 어딘가에서 객체들이 서로 강하게 붙잡고(Retain Cycle) 메모리에 둥둥 떠다니는 '좀비' 상태가 된 것이죠.

---

**🤿 딥하게 들어가기: 범인의 몽타주 (RxSwift)**

왼쪽 리스트의 이름들을 쭉 살펴보면 범인의 정체가 확실하게 드러납니다.

1. **`RxSwift` 관련 객체들의 파티:**
    
    - `PublishSubject`, `MainScheduler`, `ObserveOnSerialDispatchQueue`, `SinkDisposer`... 전부 **RxSwift** 프레임워크 내부에서 사용되는 객체들입니다.
        
2. **결정적 단서 `Closure context` (맨 아래):**
    
    - `Closure context (unknown layout)`이 5개나 둥둥 떠 있습니다. 이건 우리가 작성한 클로저(콜백 블록)가 메모리에서 해제되지 않았다는 뜻입니다.
        
3. **조합해 보면?**
    
    - 사용자님의 `HaruUp` 앱(맨 위 모듈 이름) 어딘가에서 **RxSwift로 스트림(`Observable`)을 구독(`subscribe`)했는데, 그 구독이 제대로 해제(Dispose)되지 않은 상황**입니다.
        

---

### 💊 해결을 위한 액션 플랜 (이렇게 잡아보세요!)

이런 상황을 마주했을 때 현업 개발자들이 확인하는 **체크리스트 2가지**입니다. 해당 화면(또는 뷰모델)의 코드를 열고 아래 내용을 확인해 보세요.

**1. Subscribe 내부의 `[weak self]` 누락 확인 (가장 유력함 ⭐️)** 가장 흔한 실수입니다. `subscribe(onNext:)` 블록 안에서 `self`의 프로퍼티나 메서드에 접근할 때 `[weak self]`를 빼먹지 않았는지 확인해 보세요.

Swift

```
// ❌ 누수 발생 (클로저가 self를 꽉 잡음)
viewModel.someSubject
    .subscribe(onNext: { text in
        self.someLabel.text = text 
    })
    .disposed(by: disposeBag)

// ✅ 해결 (약한 참조 사용)
viewModel.someSubject
    .subscribe(onNext: { [weak self] text in
        self?.someLabel.text = text
    })
    .disposed(by: disposeBag)
```

**2. `DisposeBag`의 생명주기 확인** 구독의 끝에 `.disposed(by: disposeBag)`을 잘 붙여주셨더라도, **그 `disposeBag`을 들고 있는 객체(예: ViewController) 자체가 죽지 못하고 있다면** 내부의 스트림도 죽지 않습니다. (이 경우에는 1번 이유로 ViewController 자체가 누수 중일 확률이 높습니다.)

**💡 다음 스텝 제안**

범인을 더 정확하게 색출하려면, 지금 올려주신 왼쪽 리스트에서 **`Closure context` 중 하나를 클릭하신 후, 화면 가운데에 뜨는 거미줄 같은 그래프(참조 관계)를 확인**해야 합니다. 화살표(실선)를 쭉 따라가다 보면 내가 만든 어떤 클래스가 클로저를 꽉 잡고 있는지 이름이 딱! 뜹니다.

### 🕵️‍♂️ 그래프 해독: 완벽한 원(Circle)의 의미

그래프의 화살표 방향을 쭉 따라가 보면, 시작점과 끝점이 없이 **계속 꼬리를 물고 도는 완벽한 사이클(Loop)**이 형성되어 있습니다. 이것이 바로 메모리 누수의 실체입니다.

그래프에 적힌 이름들을 통해 **HaruUp** 앱의 코드 생김새를 어느 정도 유추해 볼 수 있습니다.

1. **`PublishSubject<String>`:** 문자열(String) 이벤트를 전달하는 서브젝트가 쓰였습니다.
    
2. **`ObserveOn...`:** `.observe(on: MainScheduler.instance)` 같은 스케줄러 변경 코드가 들어갔네요. (UI 업데이트를 위해 메인 스레드로 넘긴 것 같습니다)
    
3. **`ShareWhileConnected`:** 스트림을 공유하기 위해 `.share()` 오퍼레이터를 사용하셨군요!
    
4. **`Closure context`:** 이 모든 스트림의 끝에서 `.subscribe(onNext: { ... })` 안에 작성하신 코드 블록(클로저)입니다.

### 💡 범인은 이 안에 있다! (해결 단서)

그래프가 이렇게 둥글게 말렸다는 건, **맨 끝에 있는 `Closure context`(구독 클로저) 안에서 무언가를 강하게(Strong) 붙잡았고, 그 무언가가 다시 이 RxSwift 스트림(혹은 DisposeBag)을 살려두고 있다는 뜻**입니다.

지금 바로 프로젝트 코드를 열고 아래 특징을 가진 부분을 찾아보세요!

**🔍 수색 타겟:**

- `String` 타입을 방출하는 `PublishSubject` (또는 `Observable<String>`)
    
- `.share()`와 `.observe(on:)`가 함께 체이닝된 곳
    
- **그 아래에 붙은 `.subscribe(onNext: { ... })` 클로저 내부!**

**🚨 유력한 용의자 (수정 전):** 아마 해당 코드의 `subscribe` 클로저 내부가 이렇게 생겼을 확률이 매우 높습니다.

```swift
// 문제의 코드 예상
viewModel.someStringSubject
    .observe(on: MainScheduler.instance)
    .share()
    .subscribe(onNext: { text in
        // ⚠️ 여기서 [weak self] 없이 self를 꽉 잡고 있을 겁니다!
        self.titleLabel.text = text 
        self.doSomething()
    })
    .disposed(by: disposeBag)
```

**✅ 해결 처방:** 클로저 시작 부분에 `[weak self]`를 넣고, 내부 코드를 안전하게 바꿔주시면 저 견고한 원형 그래프가 탁! 하고 끊어지면서 메모리 누수가 싹 사라질 겁니다.

```swift
// 해결된 코드
viewModel.someStringSubject
    .observe(on: MainScheduler.instance)
    .share()
    .subscribe(onNext: { [weak self] text in
        guard let self = self else { return } // self 방어
        self.titleLabel.text = text 
        self.doSomething()
    })
    .disposed(by: disposeBag)
```
