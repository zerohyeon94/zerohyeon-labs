### **🗣️ 가볍게 시작하기 (Simple)**

우리가 아무리 `weak`와 `[weak self]`를 숨 쉬듯이 타이핑한다고 해도, 사람은 반드시 실수를 합니다. 특히 화면을 닫았는데도 백그라운드에서 계속 타이머가 돌아가거나, 센서가 데이터를 수집하고 있다면 100% 메모리 누수(Memory Leak)를 의심해 봐야 합니다.

이럴 때 "코드를 눈으로만 보면서" 범인을 찾는 건 모래사장에서 바늘 찾기입니다. 이때 꺼내 들어야 하는 Xcode의 강력한 무기가 바로 **Memory Graph Debugger**입니다.

### **🤿 딥하게 들어가기 (Deep: 누수 잡는 4단계 루틴)**

현업 iOS 개발자들이 앱을 출시하기 전, 혹은 기능 하나를 완성할 때마다 습관적으로 하는 **메모리 점검 루틴**을 소개해 드릴게요.

**1단계: 시나리오 실행 (들어가기 -> 나오기 반복)** 누수를 확인하고 싶은 화면(예: 집중도 측정 화면)에 들어갔다가, 다시 메인 화면으로 빠져나옵니다. 이 행동을 3~4번 정도 반복합니다. 정상적이라면 화면을 빠져나왔을 때 해당 화면의 메모리는 깔끔하게 해제되어야 합니다.

**2단계: 메모리 스냅샷 찍기** 앱이 실행 중인 상태에서 Xcode 하단 디버그 바에 있는 **노드 3개가 연결된 아이콘(Debug Memory Graph)**을 클릭합니다. 그러면 앱이 일시 정지되면서, 현재 메모리(Heap)에 올라가 있는 모든 객체의 사진(스냅샷)을 찰칵! 하고 찍습니다.

**3단계: 범인 색출하기 (왼쪽 패널)** 이제 왼쪽 네비게이터 패널을 유심히 봅니다.

- **보라색 느낌표(!):** Xcode가 1차적으로 "이거 순환 참조 같은데?" 하고 찾아낸 아주 명백한 에러들입니다.
    
- **클래스 검색:** 느낌표가 없더라도 방심하면 안 됩니다. 하단 검색창에 방금 닫았던 화면이나 뷰모델(예: `TimerManager` 등)을 검색해 봅니다. 분명히 화면을 닫았는데 옆에 **(4)** 처럼 숫자가 떠 있다면? 그 객체가 메모리에 4개나 좀비처럼 살아있다는 뜻입니다. (누수 당첨! 🎯)

**4단계: 멱살 잡고 있는 녀석 찾기 (가운데 패널)** 좀비 객체를 클릭하면 가운데 화면에 거미줄 같은 그래프가 그려집니다.

- **굵은 실선:** 강한 참조 (Strong Reference)
    
- **연한 회색 선:** 약한 참조 / 알 수 없음 (Weak / Unowned 등)

화살표를 역추적해서 **"누가 이 객체를 굵은 실선으로 꽉 잡고 안 놔주고 있는지"** 따라가 보면, 십중팔구 우리가 깜빡하고 `[weak self]`를 안 적은 Closure나 Delegate가 범인으로 앉아 있습니다.
