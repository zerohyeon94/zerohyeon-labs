### 1. 왜 계속 쓸까요? (대체 불가한 매력)

- **UIKit의 근본, Delegate:** `UITableView`, `UICollectionView`, `UITextField` 등 Apple이 만든 거의 모든 기본 UI 요소들이 Delegate 패턴으로 동작합니다. 우리가 애플의 규칙을 따라야 하니 안 쓸 수가 없죠.
    
- **유연성의 끝판왕, Closure:** 코드를 변수처럼 휙휙 던질 수 있고, 비동기 작업(네트워크 통신이 끝난 후 화면 업데이트 등)을 처리할 때 이만큼 직관적이고 편한 문법이 없습니다.

### 2. 어떻게 방어할까요? (근육 기억 만들기)

현업 iOS 개발자들은 이걸 고민하면서 쓰지 않습니다. 그냥 **"숨 쉬듯이 자동으로"** 방어 코드를 타이핑합니다. (근육이 기억하는 수준이 됩니다. 💪)

- Delegate 프로토콜을 만들 때는 무조건 `class` 전용으로 만들고, 변수 앞에는 묻지도 따지지도 않고 **`weak`**를 붙입니다.
    
- 네트워크 요청이나 타이머처럼 나중에 실행되는 Closure를 열 때는, 일단 **`[weak self]`**부터 적고 봅니다.

```swift
// 현업 개발자의 무지성(?) 습관
network.fetchData { [weak self] data in
    guard let self = self else { return } // self가 살아있는지 체크하고 안전하게 씁니다.
    self.updateUI(with: data)
}
```

### 🚀 [Deep Dive] 그럼에도 불구하고, '진짜 대체제'가 등장했다?

수많은 iOS 개발자들이 수년 동안 `[weak self]` 지옥(Callback Hell)에 빠져 고통받았습니다. 그래서 Apple이 2021년에 **"이럴 거면 차라리 Closure(콜백)를 안 쓰게 해줄게!"** 하고 내놓은 궁극의 무기가 바로 **`async/await`** (Swift Concurrency)입니다.

- **과거 (Closure 방식):** 데이터를 요청해놓고, 나중에 Closure가 불리면 `[weak self]`로 뷰모델이나 뷰를 업데이트함. (메모리 누수 위험 존재)
    
- **현재 (async/await 방식):** 코드가 그냥 위에서 아래로 직관적으로 흐릅니다. Closure 괄호 열고 닫고 할 필요가 없으니, `self`를 캡처할 일도 획기적으로 줄어듭니다.

그래서 최근 iOS 트렌드는 **"오래된 API나 커스텀 UI는 여전히 Delegate/Closure(+ weak)를 쓰지만, 비동기 데이터 통신 등은 최대한 `async/await`로 교체해 나간다"** 입니다.
