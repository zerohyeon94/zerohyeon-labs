**🗣️ 가볍게 시작하기 (Simple): "최고급 레스토랑의 완벽한 분업"**

앱을 하나의 레스토랑이라고 생각해 볼까요? 코드를 한 곳에 다 때려 넣는 건 홀서빙 직원이 요리도 하고 계산도 하고 재료 발주도 하는 '난장판'입니다. 앱이 조금만 커져도 무너지죠.

그래서 우리는 역할을 3개로 완벽하게 쪼갭니다.

- **View (홀 직원 & 인테리어):** 오직 유저에게 예쁘게 보여주고(UI), 유저의 터치(주문)를 받는 일만 합니다. 뇌를 비우고 스스로 판단하지 않습니다.
    
- **ViewModel (홀 매니저):** 홀 직원이 넘겨준 주문을 받아 주방(Model)에 지시하고, 요리가 나오면 손님상에 올리기 좋게 예쁘게 플레이팅(데이터 가공)해서 홀 직원에게 넘겨줍니다. 앱의 **'핵심 뇌'** 역할입니다.
    
- **Model (주방장 & 식재료 창고):** 서버와 통신하거나(네트워크), DB에서 데이터를 꺼내오고(CoreData/SwiftData), CoreML로 집중도 점수를 계산하는 등 앱의 진짜 **'비즈니스 로직과 데이터'**를 담당합니다.

---

**🤿 딥하게 들어가기 (Deep: FocusSense 앱 실전 적용)**

지금 만들고 계신 집중 타이머 앱을 MVVM으로 깔끔하게 쪼개보겠습니다.

**1. Model (주방장)** 이전 수다들에서 열심히 만들었던 녀석들입니다. UI가 어떻게 생겼는지 전혀 관심이 없습니다.

- `FocusSession` (SwiftData 모델 구조체)
    
- `FocusMLService` (CoreML 모델로 눈 깜빡임 분석)
    
- `QuoteNetworkManager` (명언 서버 통신 API)

**2. ViewModel (홀 매니저 - 뇌)** 여기가 핵심입니다! `ObservableObject`를 써서 상태 변화를 방송할 준비를 합니다.

```Swift
import Foundation
import Combine // 또는 RxSwift

class FocusTimerViewModel: ObservableObject {
    // 1. View가 쳐다보고(Observe) 있을 플레이팅된 데이터들 (@Published)
    @Published var timeRemainingString: String = "25:00"
    @Published var currentFocusScore: Int = 100
    @Published var todayQuote: String = "명언을 불러오는 중..."
    
    // 주방장들(Model)을 고용합니다.
    private let mlService = FocusMLService()
    private let networkService = QuoteNetworkManager()
    
    // 2. View가 유저의 터치를 받으면 호출할 '명령(Action)'들
    func startTimerButtonTapped() {
        // 타이머 시작 로직, ML 서비스 가동 등...
    }
    
    // 네트워크 통신을 요청하고, 결과를 받아서 화면용 데이터로 예쁘게 가공합니다.
    func loadQuote() {
        Task {
            let quote = try? await networkService.fetchQuote()
            // 메인 스레드에서 UI 업데이트용 변수에 값을 넣어줍니다.
            DispatchQueue.main.async {
                self.todayQuote = quote?.text ?? "집중합시다!"
            }
        }
    }
}
```

**3. View (홀 직원 - 껍데기)** View는 이제 아무런 복잡한 계산을 하지 않습니다. 오직 **ViewModel을 쳐다보고 있다가, 값이 바뀌면 화면을 다시 그릴 뿐입니다. (이것이 데이터 바인딩! ⭐️)**

```Swift
import SwiftUI

struct FocusTimerView: View {
    // 1. 홀 매니저(ViewModel)를 고용합니다!
    @StateObject private var viewModel = FocusTimerViewModel()
    
    var body: some View {
        VStack {
            // 2. ViewModel이 가공해준 데이터를 그대로 갖다 씁니다. (스스로 계산 안 함)
            Text(viewModel.timeRemainingString)
                .font(.largeTitle)
            
            Text("현재 집중도: \(viewModel.currentFocusScore)점")
            
            Text(viewModel.todayQuote)
            
            // 3. 버튼을 누르면 ViewModel에게 "버튼 눌렸어!" 하고 보고만 합니다.
            Button("타이머 시작") {
                viewModel.startTimerButtonTapped()
            }
        }
        .task {
            // 화면이 켜지면 매니저에게 명언을 가져오라고 지시합니다.
            viewModel.loadQuote()
        }
    }
}
```

### 🚨 왜 이렇게 귀찮게 쪼개는 걸까요? (핵심 장점)

처음엔 파일을 여러 개 만들어야 해서 귀찮게 느껴지지만, 엄청난 마법이 일어납니다.

1. **테스트(Testability)가 미치도록 쉬워집니다.**
    
    - 화면(UI)을 켜보지 않고도, 코드로 `let vm = FocusTimerViewModel()`을 만든 뒤 `vm.startTimerButtonTapped()`를 호출해 보고 시간이 잘 가는지 '뇌'만 따로 떼서 검사(Unit Test)할 수 있습니다.
        
2. **UI를 통째로 갈아엎기 편합니다.**
    
    - 나중에 아이폰용 `FocusTimerView` 말고, 애플 워치용 화면을 새로 만든다고 상상해 보세요. `FocusTimerViewModel`은 100% 그대로 재사용하고 껍데기(View)만 새로 그리면 끝납니다! ([Daily iOS Chat #14]에서 이야기했던 멀티 플랫폼 타겟 확장이 바로 이렇게 완성됩니다.)
