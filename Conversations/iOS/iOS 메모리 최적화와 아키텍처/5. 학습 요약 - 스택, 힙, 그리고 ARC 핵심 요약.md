### 1. Stack과 Heap의 차이 (값 타입 vs 참조 타입)

- **스택(Stack):** LIFO 방식의 빠르고 효율적인 메모리 공간입니다. 시간 복잡도 O(1)로 할당 및 해제가 이루어지며, 구조체와 같은 값 타입 데이터가 이곳에 저장됩니다.
    
- **힙(Heap):** 동적으로 할당되는 거대한 메모리 창고입니다. 멀티 스레드 환경에서 동기화 오버헤드가 발생하며, 클래스와 같은 참조 타입의 '실제 본체'가 이곳에 저장됩니다.
    
- **실무 관점:** 고빈도로 변경되는 데이터 모델은 구조체로 설계하여 힙 할당 비용과 CPU 캐시 미스(Cache Miss)를 줄이는 것이 성능 최적화의 첫걸음입니다.

### 2. 메모리 주소(Pointer)와 얕은 복사

- **포인터의 역할:** 클래스 인스턴스는 힙에 거대한 본체를 두고, 스택에는 그 위치를 가리키는 작은 메모리 주소(포인터)만 저장하여 동작합니다.
    
- **얕은 복사의 위험성:** 구조체 내부에 클래스를 프로퍼티로 가질 경우, 구조체를 복사할 때 힙의 데이터가 아닌 스택의 포인터만 복사됩니다. 이로 인해 여러 구조체가 하나의 힙 데이터를 공유하게 되어 의도치 않은 상태 오염(Side Effect)이 발생할 수 있습니다.
    
- **실무 관점:** 값 타입의 완벽한 독립성을 지키기 위해, 실무에서는 값을 수정하는 순간에만 힙 데이터를 새로 복제하는 Copy-on-Write 패턴을 활용해 방어 로직을 구축합니다.

### 3. ARC와 순환 참조(Retain Cycle)

- **ARC의 해제 메커니즘:** 힙에 있는 데이터가 스택으로부터 얼마나 참조되고 있는지(Retain Count)를 컴파일 타임에 추적합니다. 카운트가 0이 되는 순간 메모리에서 즉시 해제(O(1))하여 가비지 컬렉터와 같은 런타임 멈춤 현상을 방지합니다.
    
- **순환 참조의 발생:** 두 객체가 서로를 강하게 참조하면 카운트가 영원히 0이 되지 않아, 화면이 닫혀도 힙에 데이터가 남는 메모리 누수(Memory Leak)가 발생합니다. 앱 크래시의 주된 원인입니다.
    
- **실무 관점:** 이를 방지하기 위해 상호 참조가 필요한 객체 설계에서는 한쪽의 참조 횟수를 올리지 않는 약한 참조(weak) 키워드를 반드시 사용하여 연결 고리를 끊어주어야 합니다.
