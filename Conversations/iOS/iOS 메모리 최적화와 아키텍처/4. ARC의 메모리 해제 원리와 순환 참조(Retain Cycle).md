힙에 있는 데이터는 스스로 삭제될 수 없으며, **"아예 사용하지 않는 경우(어디에서도 이 주소를 가리키지 않는 경우)"에만 메모리에서 삭제**됩니다.

그렇다면 iOS(운영체제)는 이 데이터가 '사용 중인지 아닌지'를 어떻게 알 수 있을까요? 바로 애플의 핵심 메모리 관리 기술인 **ARC(Automatic Reference Counting)** 덕분입니다.

### 1. 핵심 CS 지식 연결: 참조 카운팅(Reference Counting)과 ARC

컴퓨터 과학에서 동적 할당된 메모리(Heap)를 해제하는 방식은 크게 두 가지가 있습니다. 자바(Java)나 안드로이드에서 사용하는 '가비지 컬렉터(Garbage Collector)'와 iOS가 사용하는 '참조 카운팅(Reference Counting)'입니다.

- **동작 원리:** 클래스 인스턴스가 힙에 생성되면, 시스템은 이 인스턴스를 가리키는 스택의 포인터(바로가기) 개수를 셉니다. 이 숫자를 Retain Count(참조 횟수)라고 합니다.
    
- 누군가 주소를 가리키면 Count가 +1 증가(Retain)하고, 함수가 종료되어 스택에서 포인터가 사라지면 Count가 -1 감소(Release)합니다.
    
- **해제 순간:** 이 참조 횟수가 **0**이 되는 순간, 즉 "아무도 나를 바라보지 않는 순간" 시스템은 즉시 힙에서 해당 데이터를 삭제(Deallocate)합니다.
    
- iOS의 ARC는 개발자가 직접 +1, -1 코드를 작성할 필요 없이, 컴파일 타임(앱을 빌드하는 순간)에 컴파일러가 알아서 메모리 해제 코드를 적절한 위치에 삽입해 주는 혁신적인 기술입니다.

### 2. 복잡도 분석 (Big-O Notation)

- **시간 복잡도 ($O(1)$):** 가비지 컬렉터는 런타임에 주기적으로 앱 전체의 힙 메모리를 순회하며($O(N)$) 안 쓰는 데이터를 찾기 때문에 앱에 순간적인 버벅임(Pause)을 유발할 수 있습니다. 반면, ARC는 변수가 할당되거나 해제될 때 단순히 숫자를 +1, -1 하는 연산만 수행하므로 시간 복잡도가 $O(1)$로 매우 빠르며 실시간성이 보장됩니다.
    
- **공간 복잡도 ($O(1)$ 추가 공간):** 힙에 생성된 객체마다 자신의 참조 횟수를 기록할 아주 작은 추가 공간(일반적으로 객체 헤더의 몇 바이트)만 필요하므로 공간 복잡도 역시 $O(1)$입니다.

### 3. 실무자 입장에서 반드시 알아야 하는 이유 (트러블슈팅)

ARC는 완벽해 보이지만, 실무에서 앱을 강제 종료(Crash) 시키는 가장 큰 원인 중 하나인 메모리 누수(Memory Leak)를 발생시키는 치명적인 약점이 있습니다. 바로 **순환 참조(Retain Cycle)** 현상입니다.

만약 '객체 A'가 '객체 B'를 프로퍼티로 가지고 있고, 동시에 '객체 B'도 '객체 A'를 프로퍼티로 가지고 있다면 어떻게 될까요?

서로가 서로의 주소를 가리키고 있기 때문에 두 객체의 참조 횟수는 영원히 0이 되지 않습니다. 결국 화면이 닫혀서 스택의 데이터가 모두 날아가도, 힙에는 두 객체가 삭제되지 못한 채 영원히 남아 앱의 메모리를 파먹게 됩니다. 이를 해결하기 위해 실무에서는 참조 횟수를 올리지 않고 주소만 바라보는 **`weak` (약한 참조)** 키워드를 반드시 활용해야 합니다.

위 다이어그램의 왼쪽(NORMAL ARC)은 함수가 종료되어 스택의 참조가 사라지면 힙 객체의 카운트가 0이 되어 정상 해제되는 구조입니다. 
반면, 오른쪽(RETAIN CYCLE)은 두 객체가 서로를 강하게 참조하여 카운트가 영원히 1 이상으로 남아 메모리 누수가 발생하는 구조를 보여줍니다.

### 5. Swift 코드 구현 예시

실제 실무에서 순환 참조가 어떻게 발생하고, 어떻게 해결하는지 코드로 살펴보겠습니다.

```Swift
import Foundation

// 순환 참조가 발생하는 잘못된 설계
class User {
    var name: String
    var profile: UserProfile?
    
    init(name: String) { self.name = name }
    deinit { print("\(name) 메모리에서 삭제됨!") }
}

class UserProfile {
    // 🚨 문제점: UserProfile이 User를 강하게 참조(strong)하고 있음
    var owner: User?
    
    init() {}
    deinit { print("UserProfile 메모리에서 삭제됨!") }
}

func testMemoryLeak() {
    let myUser = User(name: "Zerohyeon")
    let myProfile = UserProfile()
    
    // 서로가 서로를 가리킴 (Retain Count가 둘 다 2가 됨)
    myUser.profile = myProfile
    myProfile.owner = myUser
} // 함수가 종료되어도 myUser와 myProfile의 참조 횟수가 1씩 남아있어 절대 deinit이 호출되지 않음!

// --- 해결책 ---
class SafeUserProfile {
    // ✅ 해결: weak 키워드를 사용하여 참조 횟수(+1)를 증가시키지 않음
    weak var owner: User?
}
```
