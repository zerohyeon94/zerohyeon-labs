## 오늘 이 주제를 보는 이유

전날 Go 멘토링에서 `focus_sessions` 생성·조회 API의 요청, 응답, 오류와 계층 책임을 정했다. 오늘은 새 기능을 늘리지 않고 조회 API 하나를 React 화면이 소비하는 최소 흐름으로 바꾼다.

첫 배포 범위는 대시보드 전체가 아니라 `최근 집중 세션 목록 1개`다. 이 화면에서 `loading / success / empty / error` 네 상태를 구분할 수 있으면 Go API 계약과 React UI 경계가 실제로 연결된다.

## 학습 목표

- Go 목록 API 응답을 React 화면 상태로 변환하는 흐름을 설명한다.
- 첫 화면에 필요한 컴포넌트와 데이터 요청 책임을 과도하지 않게 나눈다.
- 첫 배포에서 포함할 것과 제외할 것을 명시한다.
- 구현 전에 성공·빈 목록·오류의 완료 기준을 정한다.

## 핵심 개념

### 1. 첫 vertical slice

```text
GET /focus-sessions?limit=20
  → fetch
  → JSON 응답 검증
  → 화면 상태 결정
  → 최근 집중 세션 목록 표시
```

오늘은 세션 생성 폼, 통계 차트, 인증, 실시간 갱신을 함께 열지 않는다.

### 2. 화면 상태

| 상태 | 조건 | 최소 표시 |
| --- | --- | --- |
| `loading` | 요청 진행 중 | 로딩 문구 또는 skeleton |
| `success` | `items.length > 0` | 최근 세션 목록 |
| `empty` | 요청 성공, `items.length === 0` | 첫 집중 세션이 없다는 안내 |
| `error` | 네트워크 실패, non-2xx, 응답 해석 실패 | 오류 문구와 다시 시도 버튼 |

HTTP 요청 성공과 데이터 존재 여부를 하나의 boolean으로 합치지 않는다. `200 + 빈 배열`은 오류가 아니라 정상적인 `empty` 상태다.

### 3. 최소 컴포넌트 경계

| 단위 | 책임 |
| --- | --- |
| `FocusSessionsPage` | 요청 시작, 상태 보관, 재시도 |
| `FocusSessionList` | `items` 목록 렌더링 |
| `FocusSessionItem` | 세션 한 건의 시간·목표·집중 시간 표시 |
| `StatusMessage` | loading, empty, error 안내 |

첫 화면에서는 전역 상태 관리 도구를 추가하지 않는다. 한 페이지에서만 쓰는 서버 응답은 페이지 내부 state로 시작하고, 재사용·캐시 요구가 확인될 때 데이터 fetching 도구를 검토한다.

### 4. API 계약에서 확인할 값

전날 Go 답변 기준 목록 응답은 아래 형태로 고정한다.

```json
{
  "items": [
    {
      "id": "session-id",
      "started_at": "2026-06-23T00:00:00Z",
      "ended_at": "2026-06-23T00:25:00Z",
      "goal_minutes": 25,
      "duration_minutes": 25,
      "created_at": "2026-06-23T00:25:00Z"
    }
  ]
}
```

React에서는 API snake_case를 그대로 사용할지, 경계에서 camelCase로 바꿀지 한 가지 기준을 고른다. 첫 범위에서는 응답 변환 함수를 API 모듈에 두어 UI가 서버 필드명 변경에 직접 묶이지 않게 하는 편이 명확하다.

## 실무 연결

### 오늘 채울 범위 표

| 질문 | 오늘 결정 |
| --- | --- |
| 첫 화면 | 최근 집중 세션 목록 |
| 호출 API | `GET /focus-sessions?limit=20` |
| 필수 상태 | `loading / success / empty / error` |
| 첫 배포 포함 | 목록 조회, 상태 표시, 다시 시도 |
| 첫 배포 제외 | 생성 폼, 인증, 차트, 필터, pagination, 실시간 갱신 |

### 완료 기준

- [ ] API 응답 한 건을 화면에서 어떤 정보로 표시할지 정했다.
- [ ] 빈 배열을 정상 empty 상태로 표시한다.
- [ ] non-2xx 응답과 네트워크 실패를 error 상태로 표시한다.
- [ ] 다시 시도 시 같은 목록 요청을 재실행하는 흐름을 설명할 수 있다.
- [ ] 첫 배포에서 제외할 기능을 다섯 개 이하로 고정했다.
