## 오늘 이 주제를 보는 이유

2026-05-24의 핵심 고민은 기술 공부량이 부족해서가 아니라, 선택지가 서로 다른 방향으로 당기는 것이다.

- AIOps 면접 제안은 AI/인프라/운영 자동화 쪽으로 열리는 문이다.
- AI헬스케어 조교 직무는 현재 하고 있는 일과 Python 학습, 교육 경험을 강화한다.
- iOS 네이티브 개발자는 오래 가져가고 싶은 본래의 꿈이다.
- Swyp 같은 앱서비스 지원은 실제 제품 개발 경험과 iOS 커리어를 다시 붙잡을 수 있는 선택지일 수 있다.

따라서 오늘은 무엇이 더 좋아 보이는지보다, 어떤 선택이 장기 방향을 덜 흐리게 하는지 비교한다.

## 학습 목표

- AIOps, 조교, iOS 네이티브, Swyp 지원을 같은 기준으로 비교한다.
- 현재 조교 직무를 `최종 정체성`이 아니라 `브릿지 경험`으로 볼 수 있는지 판단한다.
- Swyp 지원 여부를 감정이 아니라 조건으로 판단한다.
- 다음 주에 버릴 공부와 남길 공부를 분리한다.

## 핵심 개념

### 1. 선택지는 직무명이 아니라 쌓이는 증거로 본다

커리어 선택은 직무명보다 `다음 이력서에 어떤 증거가 남는가`로 판단하는 편이 좋다.

```text
남는 증거:
- 실제 앱 화면 또는 기능
- 배포/운영 경험
- AI/헬스케어 도메인 설명력
- 수강생 피드백/강의자료 같은 교육 산출물
- 면접에서 말할 수 있는 문제 해결 사례
```

지금 가장 중요한 기준은 `iOS 네이티브 개발자로 다시 설명 가능한 증거가 남는가`다.

### 2. AI헬스케어 조교 직무는 브릿지로 보면 강점이다

조교 직무 때문에 Python을 해야 하는 것은 부담이다. 하지만 이것을 장기 정체성으로 고정하지 않고, 아래처럼 쓰면 강점이 된다.

```text
AI헬스케어 조교 경험:
- Python/AI 기초를 설명하는 힘
- 헬스케어 도메인 문제를 이해하는 힘
- 수강생의 막힘을 구조화하는 커뮤니케이션
- 기술을 남에게 전달하는 콘텐츠화 능력
```

즉, 조교 직무는 `iOS를 포기하는 선택`이 아니라 `헬스케어 AI를 설명할 수 있는 모바일 개발자`로 가는 브릿지가 될 수 있다.

### 3. Swyp 지원은 선택지를 여는 행동이다

지원은 곧 입사가 아니다. 지원은 정보를 얻고, 면접 기회를 만들고, 내 시장 반응을 확인하는 행동이다.

Swyp 지원은 아래 3개 중 2개 이상에 해당하면 진행해볼 만하다.

```text
1. iOS 네이티브 또는 모바일 앱 개발 역할이 분명하다.
2. 실제 사용자-facing 앱서비스를 만들 수 있다.
3. 제품 개선, 화면 구현, API 연동, 배포 경험이 남는다.
```

반대로 아래에 가까우면 우선순위를 낮춘다.

```text
- iOS가 아니라 운영/기획/교육 보조 중심이다.
- Python 학습 부담만 늘고 앱 개발 산출물이 남지 않는다.
- 현재 조교 직무와 병행했을 때 건강과 루틴이 무너진다.
```

### 4. AIOps는 면접을 보되, 중심축으로 확정하지 않는다

AIOps는 AI와 운영 자동화라는 성장성이 있지만, 현재의 iOS 네이티브 꿈과는 거리가 생길 수 있다.

따라서 면접을 본다면 목적은 합격 자체보다 아래 확인에 둔다.

```text
- 실제 업무가 개발인지, 운영/모니터링인지
- Python/백엔드/인프라 역량을 어느 정도 요구하는지
- 모바일/헬스케어 경험이 강점으로 작동하는지
- 1년 뒤 iOS 커리어로 돌아올 설명이 가능한지
```

## 실무 연결

오늘 남길 최소 산출물은 아래 표 하나면 충분하다.

| 선택지       | 얻는 것                      | 잃는 것/부담       | iOS 꿈과의 거리 | 다음 행동             |
| --------- | ------------------------- | ------------- | ---------- | ----------------- |
| AIOps 면접  | AI/운영 자동화 면접 경험           | iOS와 멀어질 수 있음 | 중간~멀다      | 직무 내용 확인          |
| AI헬스케어 조교 | 헬스케어 AI 설명력, Python 교육 경험 | Python 준비 부담  | 중간         | 브릿지 경험으로 정리       |
| iOS 네이티브  | 본래 꿈, 앱 개발 정체성            | 당장 포지션 경쟁     | 가장 가깝다     | 주 3회 Swift/iOS 유지 |
| Swyp 지원   | 앱서비스 경험 가능성               | 병행 부담         | 가까울 수 있음   | 역할 확인 후 지원        |

## 2026-05-24 추가 진로 메모

현재 고민의 핵심은 `무엇을 더 공부해야 하는가`가 아니라 `어떤 직무 경험을 내 장기 서사로 연결할 수 있는가`다.

정리하면 세 가지 축이 있다.

```text
1. iOS 네이티브 개발자라는 원래 꿈
2. AI 헬스케어 조교 직무에서 얻는 Python/AI/교육 경험
3. AIOps 면접 제안처럼 새롭게 열리는 AI/운영 자동화 선택지
```

현재 조교 직무에서 Python 문제를 만들고, 직접 학습해서 설명하고, 수강생에게 전달하는 과정에서 뿌듯함을 느꼈다는 점은 중요하다. 이것은 단순히 "개발에서 멀어졌다"가 아니라, 기술을 구조화하고 설명하는 역량이 실제로 작동한다는 신호다.

다만 장기 기준선은 여전히 iOS 네이티브 개발자다. 따라서 다음 선택은 아래 질문으로 판단한다.

```text
이 선택이 1년 뒤 나를
"AI도 이해하고, 헬스케어 도메인도 알고, iOS 앱으로 구현할 수 있는 개발자"
로 설명하게 해주는가?
```

AIOps는 시장성은 있지만, 실제 업무가 운영/모니터링/장애 대응 중심이면 iOS 커리어와 거리가 생길 수 있다. 반대로 AI 기반 운영 자동화 제품을 개발하거나, Python/백엔드/데이터 파이프라인을 직접 다루는 역할이라면 기술 폭을 넓히는 경험이 될 수 있다.

따라서 AIOps 면접은 `가도 되는가`를 바로 결정하기보다, 아래를 확인하는 자리로 둔다.

```text
- 실제 업무가 개발인지 운영인지
- Python, ML, 백엔드, 인프라 중 무엇을 주로 다루는지
- 내가 만든 결과물이 제품/서비스로 남는지
- 6개월~1년 뒤 iOS/모바일 커리어로 설명 가능한지
```

현재 임시 결론은 이렇다.

```text
1. 넥스트러너스 조교 직무는 당장 현실 기반으로 유지한다.
2. Python 학습은 "AI헬스케어 모바일 개발자가 되기 위한 보조 역량"으로 제한한다.
3. iOS 네이티브는 포기하지 않고 주 3회 최소 유지한다.
4. Swyp는 iOS 앱서비스 경험이 남는다면 지원한다.
5. AIOps는 면접을 통해 실제 업무를 확인한 뒤 결정한다.
```

## SWYP를 한다면 iOS 네이티브인가, 크로스플랫폼인가

현재 기준 추천은 `iOS 네이티브 우선`이다.

이유는 SWYP 자체의 성공 가능성보다, 이 경험이 앞으로의 커리어 서사에 어떻게 남는지가 더 중요하기 때문이다. 지금 장기 기준선은 여전히 iOS 네이티브 개발자이고, AI 헬스케어 조교 경험을 모바일 개발과 연결하려면 Swift/SwiftUI/Core ML/Vision/Apple Intelligence 같은 Apple 생태계 산출물이 가장 선명하다.

```text
iOS 네이티브를 선택하면 좋은 경우:
- 내 포트폴리오 중심축을 iOS로 다시 세우고 싶다.
- Swift/SwiftUI 감각을 회복하고 싶다.
- Core ML, Vision, Foundation Models 같은 Apple AI 기능으로 확장하고 싶다.
- App Store 배포, 권한, 푸시, 카메라, 헬스케어 UX 등 iOS 실무 경험을 남기고 싶다.
```

크로스플랫폼은 나쁜 선택이 아니다. 다만 지금 시점에서는 `내가 시장에 어떤 개발자로 보일 것인가`를 흐릴 수 있다. Flutter나 React Native를 선택하면 Android까지 넓힐 수 있지만, 이미 조교 업무로 Python 부담이 있는 상태에서 새 프레임워크까지 깊게 가져가면 핵심축이 흔들릴 수 있다.

```text
크로스플랫폼을 선택해도 되는 경우:
- 팀 목표가 iOS/Android 동시 출시다.
- 디자이너/백엔드와 빠르게 MVP를 검증해야 한다.
- 내가 모바일 전반 확장성을 보여주고 싶다.
- iOS 네이티브 포트폴리오보다 서비스 출시 자체가 더 중요하다.
```

추천 타협안은 이렇다.

```text
1. SWYP 첫 버전은 iOS 네이티브로 만든다.
2. 기술 스택은 SwiftUI + async/await + API 연동 + 간단한 AI 기능 연결 정도로 잡는다.
3. 앱이 검증되거나 Android 요구가 명확해지면 그때 크로스플랫폼 또는 Android 확장을 검토한다.
```

즉, 지금은 `크로스플랫폼도 할 줄 아는 사람`보다 `AI 헬스케어 경험을 iOS 앱으로 구현할 수 있는 사람`으로 포지셔닝하는 편이 더 선명하다.

## 넥스트러너스 정규직 전환과 에듀테크 미래성

에듀테크 자체는 앞으로의 산업성 측면에서 가능성이 있다. AI 때문에 많은 직무의 스킬셋이 바뀌고 있고, 기업과 개인 모두 재교육/업스킬링을 필요로 한다. 따라서 `AI를 배우는 사람을 가르치는 일`, `학습자료를 만들고 개선하는 일`, `수강생이 실제 역량을 갖추도록 돕는 일`은 단기 유행이라기보다 계속 수요가 생길 가능성이 높다.

하지만 돈을 벌 수 있는지는 `에듀테크 회사에 다니는가`보다 `내 역할이 어디까지 확장되는가`에 달려 있다.

```text
돈이 되기 어려운 형태:
- 반복적인 과제 채점
- 단순 질문 응답
- 이미 있는 자료를 조금 고치는 업무
- Python 기초만 계속 설명하는 역할
```

```text
돈이 될 가능성이 커지는 형태:
- AI/헬스케어 교육과정 설계
- 수강생 과제/피드백 시스템화
- 자동 채점, AI 튜터, 학습 분석 도구 기획 또는 개발
- B2B/B2G 교육 프로그램 운영
- 포트폴리오/취업 성과로 이어지는 커리큘럼 제작
- 교육 콘텐츠를 블로그, 강의, 자료집, SaaS 기능으로 자산화
```

넥스트러너스에서 정규직 전환을 노린다면 목표는 `조교 정규직`이 아니라 아래에 가까워야 한다.

```text
AI 헬스케어 교육 콘텐츠를 만들고,
수강생 성과를 개선하는 시스템을 설계하며,
필요하면 직접 도구까지 만들 수 있는 테크니컬 에듀케이터
```

이 방향이라면 iOS 개발자 꿈과 충돌하지 않는다. 오히려 `AI헬스케어를 설명할 수 있고, 모바일 앱으로도 구현할 수 있는 사람`이라는 희소한 포지션이 된다.

6개월 계약직 동안 확인할 기준은 아래 5개다.

```text
1. 내가 만든 자료가 회사 안에서 재사용되는가?
2. 수강생 피드백/성과가 실제로 개선되는가?
3. Python 학습이 내 AI헬스케어 이해를 강화하는가?
4. 교육 운영을 자동화하거나 도구화할 기회가 있는가?
5. 정규직 전환 시 역할명이 조교를 넘어 커리큘럼/콘텐츠/제품 쪽으로 확장되는가?
```

위 5개 중 3개 이상이 보이면 정규직 전환은 충분히 검토할 가치가 있다. 2개 이하라면 조교 경험은 포트폴리오 소재로 정리하되, iOS/모바일 개발자 루트로 다시 무게를 옮기는 편이 좋다.
