### 1. 실무에서의 Struct vs Class 활용 기준 (첫 번째 질문)

말씀하신 대로 실무에서는 데이터의 특성에 따라 두 타입을 명확히 나누어 사용합니다.

- **구조체(Struct) - 데이터 모델링:** 앱 내부의 상태값, 서버에서 내려받는 JSON 데이터 모델 등은 `struct`로 구현합니다. 값 타입이므로 이리저리 전달하며 복사해도 원본이 오염될 걱정이 없고, 메모리 할당/해제가 빨라 성능상 매우 유리합니다.
    
- **클래스(Class) - 화면 구성 및 주요 기능 캡슐화:** `UIViewController`, `UIView` 같은 UI 컴포넌트나, API 통신을 전담하는 `NetworkManager`처럼 **앱 전역에서 유일한 상태를 유지하거나(공유), 생명주기(Lifecycle) 관리가 필요한 객체**들은 `class`로 보관합니다.

### 2. Stack과 Heap의 진짜 역할: "바로가기"는 어디에 있을까? (두 번째 질문)

질문하신 내용 중 *"힙에 바로가기 같은 정보가 저장된다"*는 부분은 사실 **스택(Stack)의 역할**입니다. 완전히 반대입니다.

- **스택(Stack):** 크기가 작고 빠릅니다. 여기에 '바로가기(포인터, 메모리 주소)'가 저장됩니다.
    
- **힙(Heap):** 크기가 크고 유연합니다. 바로가기가 가리키는 '진짜 원본 데이터(무거운 본체)'가 저장되는 거대한 창고입니다.

클래스(참조 타입) 인스턴스를 생성하면, 거대하고 복잡한 실제 데이터 덩어리는 '힙'에 자리를 잡습니다. 그리고 시스템은 "이 데이터는 힙의 0x1234 주소에 있어!"라는 아주 작은 메모리 주소표(바로가기)만 '스택'에 올려둡니다.

### 3. 핵심 CS 지식 연결: 포인터(Pointer)와 역참조(Dereference)

클래스 내부의 데이터를 읽거나 쓰려고 할 때, 컴퓨터는 스택에 있는 '메모리 주소'를 먼저 읽고, 그 주소를 따라가 힙에 있는 '실제 데이터'에 접근합니다. 이 과정을 CS 용어로 역참조(Dereferencing)라고 합니다.

- **공간 복잡도 측면:** 스택은 일반적으로 고정된 작은 크기(포인터 1개당 8바이트)만 차지하지만, 힙은 클래스가 가진 프로퍼티의 개수와 크기에 따라 동적으로 거대한 $O(N)$의 공간을 차지하게 됩니다.
    
- **실무 최적화 관점 (Cache Miss):** CPU는 데이터를 처리할 때 가까운 메모리(스택)의 데이터를 캐시(Cache)에 올려두고 빠르게 처리합니다. 하지만 힙에 있는 데이터에 접근하려면 스택을 읽고 ➡️ 힙의 임의의 먼 주소로 점프해야 합니다. 이 과정에서 CPU 캐시에 원하는 데이터가 없는 현상인 캐시 미스(Cache Miss)가 발생할 확률이 높아지며, 이는 미세한 성능 저하의 원인이 됩니다. 따라서 고빈도 연산에는 스택만 사용하는 구조체가 유리한 것입니다.
