**구조체 내부에 클래스 인스턴스를 프로퍼티로 가질 수 있습니다.** 하지만 복사를 할 때 완전히 새로운 클래스 인스턴스가 생성되는 것은 **아닙니다.**

이 현상은 실무에서 예상치 못한 버그를 만들어내는 주범 중 하나입니다. 왜 이런 일이 발생하는지 CS 핵심 원리와 함께 파헤쳐 보겠습니다.

### 1. 핵심 CS 지식 연결: 포인터(Pointer)와 얕은 복사(Shallow Copy)

이 현상을 이해하려면 컴퓨터 과학의 **포인터(Pointer)** 개념과 얕은 복사(Shallow Copy)의 원리를 알아야 합니다.

- **포인터의 저장:** 구조체(값 타입) 데이터는 스택(Stack) 메모리에 저장됩니다. 하지만 구조체가 품고 있는 클래스(참조 타입) 프로퍼티의 경우, 힙(Heap) 메모리에 존재하는 실제 데이터 대신 그 데이터가 위치한 '메모리 주소(포인터)'만을 스택에 저장하게 됩니다.
    
- **얕은 복사(Shallow Copy):** 구조체를 다른 변수에 할당하여 복사하면, OS는 스택 영역에 있는 값들을 그대로 비트 단위로 복사(Bitwise Copy)합니다. 이때 힙에 있는 클래스 인스턴스 자체가 복사되는 것이 아니라, 스택에 적혀 있던 **'클래스를 가리키는 메모리 주소값'만 복사**됩니다.
    
- 결과적으로 두 개의 서로 다른 구조체가 힙 영역의 동일한 단일 클래스 인스턴스를 공유(참조)하게 됩니다.

### 2. 복잡도 분석 (Big-O Notation)

- **시간 복잡도 ($O(1)$):** 구조체를 복사할 때 힙 영역에 새로운 메모리를 탐색하여 할당하고 데이터를 깊은 복사(Deep Copy)하는 과정이 없습니다. 단순히 스택에 있는 메모리 주소값(일반적으로 8바이트)만 복사하므로 복사 연산의 시간 복잡도는 $O(1)$로 매우 빠릅니다.
    
- **공간 복잡도 ($O(1)$ 추가 공간):** 스택에는 새로운 구조체 크기만큼의 공간이 추가로 필요하지만, 힙 영역에는 새로운 클래스 인스턴스가 생성되지 않으므로 힙 공간 복잡도는 $O(1)$입니다. (단, 동일한 인스턴스를 가리키는 포인터가 늘어났으므로 내부적으로 ARC의 Retain Count가 +1 증가하는 연산 오버헤드는 발생합니다.)

### 3. 실무자 입장에서 반드시 알아야 하는 이유 (트러블슈팅 및 최적화)

실무에서 이 개념을 간과하면 **'값 타입의 독립성(Value Semantics)'이 깨지는 심각한 버그**를 겪게 됩니다.

데이터를 모델링할 때 구조체를 사용했다면 당연히 복사본을 수정해도 원본에 영향이 없을 것이라고 기대하게 됩니다. 하지만 내부에 참조 타입이 있다면, 복사본의 내부 참조 데이터를 수정했을 때 원본 데이터까지 동시에 바뀌어버리는 의도치 않은 변형(Unintended Mutation)이 발생합니다. 이는 상태 관리를 예측 불가능하게 만듭니다.

**실무 해결책:**

이러한 문제를 방지하면서도 구조체의 성능 이점을 살리기 위해, 실무에서는 구조체 내부의 참조 값을 수정하려고 하는 순간에만 명시적으로 힙 데이터를 새로 복사해 내는 **Copy-on-Write (CoW)** 패턴을 직접 구현하여 방어 로직을 세웁니다.

### 4. Swift 코드 구현 예시

실제 타이머 로직이나 세션 관리를 할 때 이 버그가 어떻게 발생하는지 코드로 확인해 보겠습니다.

```Swift
import Foundation

// Heap에 할당되는 참조 타입
class SessionTimer {
    var duration: TimeInterval
    init(duration: TimeInterval) { self.duration = duration }
}

// Stack에 할당되지만, 내부에 참조 타입을 가진 구조체
struct StudyRecord {
    var id: Int
    var timer: SessionTimer
}

// 1. 초기 생성
let recordA = StudyRecord(id: 1, timer: SessionTimer(duration: 3600))

// 2. 구조체 복사 발생 (Stack 복사 O, Heap 복사 X - 얕은 복사)
var recordB = recordA 
recordB.id = 2

// 3. recordB의 타이머(클래스 내부 값)를 변경
recordB.timer.duration = 7200

// 4. 결과 확인 (Side Effect 발생)
print("Record A 타이머 시간: \(recordA.timer.duration)") // 출력: 7200.0 (A도 함께 변경됨!)
print("Record B 타이머 시간: \(recordB.timer.duration)") // 출력: 7200.0
```

스택 상에서 `recordA`와 `recordB`는 서로 다른 고유한 `id`를 가진 별개의 구조체이지만, 내부에 품고 있는 `timer` 포인터는 결국 같은 힙 주소를 가리키고 있기 때문에 발생하는 현상입니다.
