## Q1 — ARC RC 규칙 (업데이트 답변에 대한 피드백)

### 나의 답변
- 대입/소유는 strong val, let에 대입할 때 +1이고 다른 변수에서 참조가 해제될 때 -1이 됩니다. 
- 옵셔널 바인딩, 반환은 대입할 때는 +1이고 함수 리턴 값을 강한 변수에 할당하면 또 +1이 됩니다. 
- 컬렉션은 내부에 넣게되면 strong값을 보유하기 때문에 +1이고 제거를 하거나 컬렉션을 해제했을 때 -1이 됩니다. 
- 클로저 캡처의 경우 escaping인 경우에만 self를 캡처하면 +1이 됩니다. (여기서 캡처라는게 어떤 뜻인지는 잘 모르겠습니다.) 
- 값타입 내부에서 존재하는 class가 서로를 참조하게 되면 +1이 되고 내부에서 해제할 때 -1이 됩니다. (여기에서 프로퍼티로 가진다는 뜻은 잘 모르겠습니다.) 
- deinit은 자동으로 참조 카운트가 0이되면 호출됩니다.

**Strengths**

- 대입/해제, 컬렉션 보유, `@escaping`에서의 보유 증가 등 핵심 축을 잘 잡았습니다.
    
- `deinit` 조건도 정확해요.

**Gaps & 교정**

- **“캡처”의 뜻**: 클로저가 바깥 스코프의 값을 **자신의 스토리지로 복사(참조)해 보유**하는 것.
    
    - non-escaping: 호출 범위 안에서만 쓰여서 보통 RC를 늘려 오래 보유하지 않음.
        
    - **@escaping**: 클로저가 어딘가에 **저장**되어 수명 연장 → 그때 **`self`를 강하게 잡으면 +1**.
        
- **“프로퍼티로 가진다”의 뜻**: 어떤 타입(예: `struct`)이 `class` 인스턴스를 **프로퍼티로 보관**하면, 그 프로퍼티에 값을 **대입하는 순간 RC +1**입니다. (값 타입/참조 타입 여부와 무관하게 “보관하면 +1, 놓으면 -1” 원칙은 동일)
    
- **옵셔널 바인딩/반환**: “바인딩=대입”이라 RC 규칙 그대로 적용됨(필요 시 +1). 함수에서 반환만 했다고 RC가 오르는 건 아니고, **반환값을 강한 변수에 ‘할당’할 때** +1.

**Ideal Points — 한눈 정리**

- 강한 저장 프로퍼티/변수에 **대입 시 +1 / 해제 시 -1**
    
- 컬렉션에 넣으면 해당 컬렉션이 **strong 보유(+1)**, 제거/컬렉션 파기 시 -1
    
- **@escaping** 클로저가 `self`를 **강하게 캡처하면 +1** → `[weak self]` / `[unowned self]` 필요
    
- 값 타입이더라도 내부에 **참조 타입 프로퍼티**가 있으면 그 프로퍼티 **대입 시 +1**
    
- `deinit`은 Reference Count가 0일 때만, 순환 참조/전역 보유가 남아 있으면 호출 안 됨.

## Q2 — Retain Cycle 4종 & 한 줄 해법

### 나의 답변
- viewModel을 보관한 클로저가 viewController에서 self로 강하게 캡처해서 발생합니다. 그래서 만약에 사용을 한다면 viewcontroller에서 weak 로 약한참조로 접근하는게 좋아보입니다. 
- delegate에서는 child에서 owner에 강한 참조로 가지고 있으면서 child와 owner사이에서 순환참조가 발생하게 됩니다. 이때 delegate를 child에 선언할 때 옵셔널로 선언합니다.
- Timer가 self를 강하게 참조하게되면 타이머는 self에 따라 행동하기 때문에 timer가 끝나기 전까지 해제되지 않는다. 결과적으로 self를 참조하기 위해선 weak를 사용해야한다. 
- 토큰을 가져와서 deinit시 마다 해당 notification을 dispose나 remove해줘야한다.

**Strengths**

- VM 콜백/Timer/Notification에서 `weak` 필요성을 잘 언급했습니다.

**Gaps & 교정**

- **delegate 패턴**은 “옵셔널”보다 **`weak`** 가 핵심입니다. (옵셔널은 선택사양, **순환 차단은 weak가 담당**)

**정답 패턴(한 줄)**

- (a) VC ↔ VM 콜백

```swift
viewModel.onUpdate = { [weak self] in self?.render() }
```
    
- (b) delegate

```swift
protocol ChildDelegate: AnyObject { ... }
final class Child { weak var delegate: ChildDelegate? }
```
    
- (c) Timer / DisplayLink

```swift
timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] _ in self?.tick() }
deinit { timer?.invalidate() }
```
    
- (d) Notification / Combine

```swift
// NotificationCenter
let token = NotificationCenter.default.addObserver(forName: ..., object: nil, queue: .main) { [weak self] _ in self?.handle() }
deinit { NotificationCenter.default.removeObserver(token) }

// Combine
publisher.sink { [weak self] _ in self?.handle() }.store(in: &bag)

```

## Q3 — Concurrency × ARC

### 나의 답변
- Task에 대해서 정확하게 파악하지는 못하였지만 문제라고 할 수 있는 부분은 @escaping 클로저네서는 self를 강하게 참조하기 때문에 순환참조가 발생할 수 있어 이때 weak로 구현을 해야합니다. 
- 또한 deinit일 때 timer를 invalidate로 해제시켜줘야 timer에 대한 순환 참조를 방지할 수 있습니다.

**Strengths**

- `@escaping`에서 강한 캡처 위험, `timer.invalidate()` 필요성 언급 Good.

**Gaps & 교정(핵심 포인트)**

- **문제의 본질**: 비동기 경계(`@escaping` 클로저, `Task{}`)는 클로저가 **오래 살아** `self`를 붙잡기 쉬움. **타이머가 동기/비동기냐**는 누수와 직접 상관 없음.
    
- **취소 ≠ 해제**: `Task`를 cancel해도 캡처된 `self`가 다른 소유 경로로 남아 있으면 해제되지 않을 수 있음.
    
- **MainActor**: UI 업데이트는 메인 액터에서. `Task.detached`는 메인 액터가 아니라서 **추가 주의**.

**정답 패턴**

```swift
@MainActor
func load() {
  Task { [weak self] in
    guard let self else { return }
    let user = try await api.fetchUser()
    self.title = user.name
    self.startTimer()
  }
}

private var timer: Timer?
func startTimer() {
  timer?.invalidate()
  timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] _ in
    self?.view.alpha.toggle()
  }
}
deinit { timer?.invalidate() }
```

## Q4 — 누수 진단 루틴

### 나의 답변
- 디버깅을 통해 메모리 그래프를 보면서 해당 페이지에서 메모리가 얼마나 사용되는지 확인하는 방법이 있습니다.

**Strengths**

- Memory Graph 사용 의지 Good.

**보강 루틴(실전용 6단계)**

1. **재현 시나리오 축소**: A 화면 Push → Pop을 10회 반복 같은 최소 케이스.
    
2. **Memory Graph 스냅샷**: 예상과 달리 남아 있는 객체를 선택 → **Owner Path** 따라가 **강한 엣지**를 확인.
    
3. **의도된 보유 vs 누수** 구분: 싱글톤/캐시/글로벌 큐가 잡고 있나? 의도면 OK, 아니면 후보.
    
4. **핫스팟 패턴 점검**: Timer 미무효화, `NotificationCenter` 미해제, delegate strong, `@escaping` strong self.
    
5. **Instruments(Leaks/Allocations)**: 타임라인에서 인스턴스가 **누적 증가**하는지/어디서 할당됐는지 스택 추적.
    
6. **수정→재검증**: `[weak/unowned]` 추가, 설계 정리, 해제 타이밍 개선 후 다시 1)~5) 반복.
