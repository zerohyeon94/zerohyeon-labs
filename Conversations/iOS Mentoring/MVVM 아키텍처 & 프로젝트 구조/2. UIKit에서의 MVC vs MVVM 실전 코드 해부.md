우선, 두 패턴 모두에서 공통으로 사용할 **순수한 데이터(Model)**를 하나 만들고 시작하겠습니다. UI나 로직이 전혀 없는 순수 구조체입니다.

```Swift
// 공통 Model (식재료)
struct FocusSession {
    var score: Int
    var isFocusing: Bool
}
```

---

### 1. UIKit의 전통적인 MVC 패턴 (Massive View Controller)

MVC에서는 `UIViewController`가 **Controller** 역할을 하며, 뷰(화면)와 모델(데이터) 사이에서 모든 중재를 다 합니다.

**🚨 특징:** `UIViewController`가 데이터를 직접 들고 있고, 로직도 계산하며, UI도 직접 업데이트합니다. 혼자 북 치고 장구 치는 형태입니다.

```Swift
import UIKit

// Controller (겸 View 역할까지 해버림)
class FocusViewController: UIViewController {
    
    // 1. Model을 직접 소유함
    var session = FocusSession(score: 0, isFocusing: false)
    
    // UI 컴포넌트들
    let scoreLabel = UILabel()
    let focusButton = UIButton()
    
    override func viewDidLoad() {
        super.viewDidLoad()
        updateUI() // 초기 화면 세팅
    }
    
    // 2. 유저 액션(버튼 클릭) 처리
    @objc func focusButtonTapped() {
        // 비즈니스 로직 연산을 컨트롤러가 직접 수행!
        session.isFocusing.toggle()
        if session.isFocusing {
            session.score += 10
        }
        
        // 3. UI 업데이트도 직접 지시
        updateUI()
    }
    
    // UI 갱신 로직
    func updateUI() {
        scoreLabel.text = "점수: \(session.score)"
        let title = session.isFocusing ? "집중 중지" : "집중 시작"
        focusButton.setTitle(title, for: .normal)
    }
}
```

- **문제점:** 네트워크 통신이나 CoreML 연산까지 여기에 들어가면 파일 하나가 2,000줄이 넘어가는 지옥이 펼쳐집니다. 테스트하기도 불가능에 가깝습니다.

---

### 2. UIKit에 적용한 MVVM 패턴 (역할 분담의 마법)

MVVM에서는 `UIViewController`를 과감하게 **단순한 View(껍데기)**로 취급해 버립니다. 비즈니스 로직과 데이터는 모두 `ViewModel`이라는 새로운 클래스로 쫓아냅니다.

**✅ Step 1: ViewModel (홀 매니저 & 뇌)** UI 프레임워크(`UIKit`)를 절대 `import` 하지 않는 것이 핵심입니다! 오직 로직만 담당합니다.

```Swift
import Foundation // UIKit 없음!

class FocusViewModel {
    // 1. Model 소유
    private var session = FocusSession(score: 0, isFocusing: false)
    
    // 2. View에게 데이터가 변했다고 알려줄 클로저 (바인딩의 핵심)
    var onSessionUpdated: ((_ scoreText: String, _ buttonTitle: String) -> Void)?
    
    // 3. View에서 호출할 명령 (유저 액션)
    func toggleFocus() {
        // 비즈니스 로직 연산
        session.isFocusing.toggle()
        if session.isFocusing {
            session.score += 10
        }
        
        // 데이터가 변했으니 View를 위해 데이터를 예쁘게 가공해서 클로저로 쏴줌!
        let scoreText = "점수: \(session.score)"
        let buttonTitle = session.isFocusing ? "집중 중지" : "집중 시작"
        
        onSessionUpdated?(scoreText, buttonTitle)
    }
}
```

**✅ Step 2: View (기능이 축소된 UIViewController)** 이제 컨트롤러는 스스로 판단하지 않습니다. ViewModel이 시키는 대로 화면만 그리고, 유저의 터치만 ViewModel에게 전달합니다.

```Swift
import UIKit

// 이제 이 녀석은 Controller가 아니라 단순한 View입니다.
class FocusViewController: UIViewController {
    
    // 1. 뇌(ViewModel)를 가져옴
    private var viewModel = FocusViewModel()
    
    let scoreLabel = UILabel()
    let focusButton = UIButton()
    
    override func viewDidLoad() {
        super.viewDidLoad()
        bindViewModel() // ⭐️ 데이터 바인딩!
    }
    
    // 2. 데이터 바인딩: "ViewModel아, 데이터 바뀌면 이 클로저 실행해줘!"
    private func bindViewModel() {
        viewModel.onSessionUpdated = { [weak self] scoreText, buttonTitle in
            // UI 업데이트만 수행 (계산은 1도 안 함)
            self?.scoreLabel.text = scoreText
            self?.focusButton.setTitle(buttonTitle, for: .normal)
        }
    }
    
    // 3. 유저 액션: "ViewModel아, 버튼 눌렸어. 네가 알아서 계산해!"
    @objc func focusButtonTapped() {
        viewModel.toggleFocus()
    }
}
```

### 💡 핵심 요약: 무엇이 달라졌는가?

1. **데이터의 소유권:** MVC에서는 뷰 컨트롤러가 `Model`을 들고 있었지만, MVVM에서는 `ViewModel`이 들고 있습니다.
    
2. **`import UIKit`의 위치:** 뷰모델 파일에는 `UIKit`이 없습니다. 즉, 아이폰 화면이 없어도 뷰모델의 로직만 따로 떼어내서 아주 쉽게 검증(Unit Test)을 할 수 있게 됩니다.
    
3. **데이터 바인딩(Binding):** UIKit 자체에는 상태 변화를 감지하는 기능이 없어서 위 예시처럼 **클로저(Closure)**를 사용해 수동으로 연결했습니다.

**🔥 여기서 앞서 배운 지식이 연결됩니다!** 위 코드에서 쓴 투박한 클로저 바인딩(`onSessionUpdated`) 대신, **RxSwift**나 **Combine**을 사용하면 이 바인딩 코드가 예술적으로 짧고 우아해집니다. 현업에서 "RxSwift + MVVM"을 세트 메뉴로 묶어서 쓰는 진짜 이유가 바로 이 '데이터 바인딩'을 기가 막히게 잘해주기 때문입니다.
