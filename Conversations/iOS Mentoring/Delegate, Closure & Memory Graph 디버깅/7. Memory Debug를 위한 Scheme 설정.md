### `Malloc Scribble`과 `Malloc Stack Logging`의 마법

**🗣️ 가볍게 시작하기 (Simple)**

우리가 Xcode에서 앱을 실행할 때, 보이지 않는 곳에서 메모리를 할당(`malloc`)하고 해제(`free`)하는 작업이 수만 번씩 일어납니다. Scheme의 Diagnostics(진단) 탭에 있는 이 옵션들은, 시스템에게 **"메모리를 줬다 뺏을 때, 그냥 두지 말고 특별한 흔적을 남겨라!"**라고 지시하는 아주 강력한 CCTV입니다.

---

**🤿 딥하게 들어가기 (Deep: 메모리 추적기)**

**1. Malloc Scribble (낙서하기)**

- **상황:** 메모리를 해제(Free)하면, 그 공간은 비워지지만 '데이터 찌꺼기'는 남아있을 수 있습니다. 만약 실수로 해제된 메모리에 다시 접근하는 버그(Use-After-Free)가 발생했을 때, 우연히 옛날 데이터가 남아있어서 앱이 안 죽고 정상처럼 작동하는 무서운 일이 벌어집니다. (나중에 유저 폰에서 뜬금없이 터지죠.)
    
- **기능:** 이 옵션을 켜면, 시스템이 메모리를 해제할 때 그 공간에 **`0x55`라는 쓰레기 값(낙서)을 강제로 덮어씌워 버립니다.** * **효과:** 해제된 메모리에 실수로 접근하는 순간 앱이 **확실하고 즉각적으로 크래시(Crash)**를 내뱉게 만들어, 숨은 버그를 개발 단계에서 멱살 잡고 끌어냅니다.

**2. Malloc Stack Logging (스택 로깅) ⭐️⭐️⭐️⭐️⭐️**

- **상황:** Memory Graph에서 누수된 객체를 찾았습니다. (예: `Closure context`). 그런데 "도대체 내 코드 수만 줄 중에 **어느 파일, 몇 번째 줄**에서 이 객체를 만든 거야?"를 알 수가 없습니다.
    
- **기능:** 메모리가 할당(`malloc`)되는 그 찰나의 순간에, **콜스택(Call Stack, 함수 호출 경로)을 전부 기록해 둡니다.**
    
- **효과:** FocusSense 앱처럼 눈 깜빡임이나 머리 움직임 같은 생체 데이터를 추적하고 SwiftData에 저장하는 복잡한 로직에서 누수가 났을 때, Memory Graph 우측 인스펙터의 **Backtrace(백트레이스)** 패널을 볼 수 있게 해줍니다.
    
    > _"아! `TimerManager.swift` 45번째 줄에서 만든 클로저가 문제였네!"_ 하고 정확한 코드 라인을 찍어주는 기적을 보여줍니다.

>[!WARNING] 주의할 점: 
>이 두 옵션은 시스템의 리소스를 엄청나게 잡아먹습니다. 버그를 잡을 때만 잠깐 켜시고, 평상시 개발이나 성능(속도) 테스트를 하실 때는 **반드시 끄셔야 합니다!**
