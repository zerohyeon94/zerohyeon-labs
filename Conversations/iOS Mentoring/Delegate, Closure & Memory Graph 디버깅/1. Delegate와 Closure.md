### Delegate와 Closure, 순환 참조(Retain Cycle) 피하기 대작전

**🗣️ 가볍게 시작하기 (Simple)**

iOS 앱을 만들다 보면 화면 간에, 혹은 객체 간에 데이터를 주고받아야 할 일이 참 많죠. 이때 가장 많이 쓰는 두 가지 무기가 바로 **Delegate 패턴**과 **Closure(콜백 함수)**입니다.

그런데 이 두 녀석은 아무 생각 없이 쓰다 보면 앱의 메모리를 야금야금 갉아먹는 주범이 됩니다. 화면을 닫았는데도 객체가 메모리에서 사라지지 않는 **메모리 누수(Memory Leak)**가 발생하거든요.

---

**🤿 딥하게 들어가기 (Deep: ARC와 강한 참조 순환)**

iOS는 **ARC(Automatic Reference Counting)**라는 시스템으로 메모리를 관리합니다. 쉽게 말해 "나를 참조(바라보고)하는 애가 1명이라도 있으면, 난 메모리에서 절대 죽지 않아!"라는 규칙이죠.

문제는 **서로가 서로를 강하게 바라볼 때** 생깁니다. 이걸 **순환 참조(Retain Cycle)**라고 부릅니다.

**1. Delegate의 함정**

- **상황:** `TimerManager` 클래스가 타이머 화면(`TimerViewController`)에 이벤트를 전달하려고 합니다. 화면이 매니저를 생성해서 소유합니다. (화면 ➔ 매니저)
    
- **문제:** 매니저에서 이벤트가 발생했을 때 화면에 알려주려고 `manager.delegate = self(화면)`를 해버립니다. (매니저 ➔ 화면)
    
- **결과:** 두 객체가 서로 멱살을 잡고 놓지 않습니다. 사용자가 타이머 화면을 닫고 뒤로 가기를 눌러도, 둘 다 메모리에서 죽지 못하고 계속 살아있게 됩니다.
    
- **해결:** 그래서 Delegate를 선언할 때는 무조건 한쪽의 힘을 빼야 합니다.
    
    ```swift
    // "나는 널 바라보긴 할 건데, 생명줄을 꽉 잡진 않을게(weak)"
    weak var delegate: TimerDelegateProtocol?
    ```

**2. Closure의 함정**

- **상황:** 화면 안에서 네트워크 통신이나 타이머 비동기 작업을 위해 Closure를 사용합니다.
    
- **문제:** Closure 안에서 화면의 UI를 업데이트하려고 `self.timeLabel.text = "완료"` 라고 쓰는 순간! Closure가 `self`(화면)를 강하게 꽉 붙잡아 버립니다.
    
- **결과:** 화면은 Closure를 소유하고, Closure는 `self`를 소유하는 무한 굴레에 빠집니다.
    
- **해결:** Closure가 `self`를 약하게 참조하도록 **캡처 리스트(Capture List)**를 사용해야 합니다.
    
    ```swift
    network.request { [weak self] result in
        // self가 이미 메모리에서 날아갔을 수도 있으니 옵셔널(?) 처리
        self?.timeLabel.text = result
    }
    ```
