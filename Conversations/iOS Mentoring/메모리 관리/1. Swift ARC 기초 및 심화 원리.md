## 📋 1. 핵심 CS 지식: Stack과 Heap의 이중주

프로그램이 메모리를 사용하는 방식은 크게 두 가지로 나뉩니다.

*   **Stack (정적 할당)**:
    *   **대상**: 구조체(Struct), 열거형(Enum), 기본 자료형.
    *   **동작**: 함수 호출 시 스택 포인터가 이동하며 즉시 할당/해제됩니다 ($O(1)$).
    *   **특징**: 매우 빠르며, 참조 카운팅이 필요 없습니다.
*   **Heap (동적 할당)**:
    *   **대상**: 클래스(Class) 인스턴스, 클로저(Closure).
    *   **동작**: 빈 메모리 공간을 찾아 할당하며, 관리가 없으면 메모리 누수가 발생합니다.
    *   **특징**: 런타임에 크기가 결정되며, **ARC가 관리하는 주 무대**입니다.

## 🔍 2. ARC의 저수준 메커니즘: SIL 단계의 자동 주입

ARC는 가비지 컬렉터(GC)와 달리 **컴파일 타임**에 동작합니다. Swift 컴파일러는 코드 분석을 통해 객체의 생명주기를 파악하고, **SIL(Swift Intermediate Language)** 단계에서 참조 횟수를 조절하는 명령어를 삽입합니다.

```sil
// 컴파일러가 생성하는 중간 코드 예시
%1 = alloc_ref $MyClass  // 객체 생성 (Count: 1)
strong_retain %1        // 참조 추가 시 삽입 (Count: 2)
strong_release %1       // 사용 완료 시 삽입 (Count: 1)
```

## 🧠 3. 참조 카운트의 저장소: Inline RC vs Side Table

Swift 객체의 참조 횟수는 단순히 숫자 하나로 관리되지 않습니다.

1.  **Inline Reference Count**: 객체 본체의 헤더에 직접 저장됩니다. 가장 빠르지만, `weak` 참조가 발생하면 한계가 옵니다.
2.  **Side Table**: `weak` 참조가 발생하거나 카운트가 오버플로우될 때 생성되는 별도 메모리 블록입니다.
    *   **왜 필요한가?**: 객체가 `deinit`되어 사라진 후에도, `weak` 변수들은 여전히 그 자리를 가리키고 있습니다. 이때 Side Table이 남아있어 "그 객체는 이미 사라졌으니 `nil`을 반환해"라고 알려주는 역할을 합니다.

## ⚠️ 4. 실무적 병목: Atomic Operation과 멀티코어 성능

참조 카운팅은 멀티스레드 환경에서 안전해야 하므로 **Atomic(원자적) 연산**을 수행합니다. 
*   이는 CPU 캐시 간의 데이터 동기화를 강제하므로, 초당 수천 번의 객체 복사가 일어나는 고성능 앱(`ZipJoong` 등)에서는 예상치 못한 CPU 부하를 일으킬 수 있습니다.
*   따라서 **"값 타입을 선호하라"**는 조언은 단순히 메모리 구조 때문이 아니라, 이 원자적 연산 비용을 줄이기 위함이기도 합니다.

## 🛠️ 5. 실무 패턴: Strong, Weak, Unowned의 선택

| 종류 | 참조 횟수 영향 | 특징 | 용도 |
| :--- | :--- | :--- | :--- |
| **Strong** | +1 | 객체 생존 보장 | 일반적인 소유 관계 |
| **Weak** | 0 | 자동 `nil` 변환 (Side Table 사용) | 순환 참조 방지 (Delegate 등) |
| **Unowned** | 0 | 해제 후 접근 시 Crash (직접 참조) | 생명주기가 확실히 같을 때 |
