# 1-Summary. Swift ARC 기초 및 심화 마스터 요약

> **주제**: Swift ARC (Automatic Reference Counting)
> **학습 완료일**: 2026-05-08
> **세션 구성**: 기초 원리(1) → 저장소 및 Atomic 연산(1-1) → Side Table 생명주기(1-2) → 코드 검증(1-3)

---

## 🚀 1. 핵심 기술 요약 (Key Takeaways)

1.  **ARC의 본질**: 가비지 컬렉션(GC)과 달리 **컴파일 타임**에 `retain`/`release` 명령어를 주입하여 힙(Heap) 메모리를 관리함.
2.  **참조 카운트 저장소**:
    *   **Inline RC**: 객체 헤더에 직접 저장. 빠르지만 객체 해제 시 사라짐.
    *   **Side Table**: `weak` 참조 시 생성되는 별도 공간. 객체 본체 해제 후에도 `weak` 포인터의 안전성(nil 반환)을 보장함.
3.  **객체 생명주기 5단계**: Alive → Deiniting → Deinitialized → Freed → Dead. Side Table은 마지막 `weak` 참조가 사라질 때(Dead) 비로소 해제됨.
4.  **성능 최적화**: ARC는 멀티코어 안전성을 위해 **Atomic 연산**을 수행함. 이로 인한 CPU 캐시 동기화 비용을 줄이기 위해 빈번한 작업에는 **값 타입(Struct)** 사용이 권장됨.
5.  **참조 순환 방지**: `weak`와 `unowned`를 적절히 사용하며, 클로저에서는 `[weak self]`와 `guard let self = self` 패턴을 통해 논리적 일관성을 확보함.

## 🎙️ 2. 면접 대비 예상 질문 (Interview Q&A)

### [Level 1. 기초 개념]
**Q: ARC와 가비지 컬렉션(GC)의 차이점은 무엇인가요?**
> **A**: ARC는 **컴파일 타임**에 참조 카운팅 코드를 주입하여 런타임 오버헤드가 적고 해제 시점이 예측 가능합니다. 반면, GC는 **런타임**에 별도의 감시 프로세스가 돌며 메모리를 해제하므로 예측이 어렵고 자원 소모가 더 큽니다.

**Q: Strong, Weak, Unowned 참조의 차이를 설명해 주세요.**
> **A**: **Strong**은 참조 횟수를 올리고 인스턴스를 소유합니다. **Weak**는 횟수를 올리지 않고 대상이 해제되면 자동으로 `nil`이 됩니다. **Unowned** 역시 횟수를 올리지 않지만, 대상이 해제된 후 접근하면 런타임 에러를 발생시키므로 대상의 수명이 더 길 때 사용합니다.

### [Level 2. 심화 원리]
**Q: Side Table이란 무엇이며 왜 필요한가요?**
> **A**: Side Table은 `weak` 참조나 카운트 오버플로우 시 생성되는 별도의 메모리 블록입니다. 객체 본체가 해제되어도 Side Table은 마지막 약한 참조가 끝날 때까지 남아있어, `weak` 변수가 안전하게 `nil`을 반환할 수 있도록 '좀비 객체' 접근을 방어하는 역할을 합니다.

**Q: ARC 동작이 앱 성능에 어떤 영향을 줄 수 있나요?**
> **A**: 참조 카운팅은 멀티스레드 안전을 위해 **Atomic(원자적) 연산**을 수행합니다. 이는 CPU 코어 간의 캐시 동기화 비용을 발생시키므로, 고빈도로 생성/복사되는 객체가 많을 경우 CPU 부하가 증가하고 성능 병목이 생길 수 있습니다.

### [Level 3. 실무 및 트러블슈팅]
**Q: 클로저에서 [weak self]를 사용한 후 guard let self = self를 하는 이유는 무엇인가요?**
> **A**: `weak self`만 사용하면 클로저 실행 도중에 `self`가 메모리에서 사라져 로직이 중단되거나 불일치가 발생할 수 있습니다. `guard let`을 통해 클로저가 실행되는 동안만 일시적으로 강한 참조(Strong Promotion)를 만들어 로직의 **완결성과 데이터 일관성**을 보장하기 위함입니다.

**Q: 메모리 누수(Memory Leak)를 발견하는 본인만의 노하우가 있나요?**
> **A**: Xcode의 **Memory Graph Debugger**를 활용하여 순환 참조를 시각적으로 확인합니다. 또한 실무에서는 `deinit`에 로그를 찍어 객체가 예상 시점에 해제되는지 체크하고, Instruments의 **Leaks** 도구를 사용하여 지속적으로 메모리가 증가하는 구간을 프로파일링합니다.
