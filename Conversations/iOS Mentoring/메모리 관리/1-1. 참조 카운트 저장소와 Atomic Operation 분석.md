## 🔍 1. 참조 카운트의 두 가지 거처: Inline vs Side Table

Swift 객체는 힙(Heap)에 할당될 때 자신의 생존 여부를 결정할 '참조 횟수'를 저장해야 합니다. 이 저장 방식은 객체의 상태에 따라 동적으로 변합니다.

### A. Inline Reference Count (기본 모드)
*   **물리적 위치**: 객체의 메모리 레이아웃 중 가장 앞부분인 **헤더(Header)** 공간에 직접 포함됩니다.
*   **동작**: 객체가 생성되면 헤더에 64비트 정수형으로 참조 카운트가 기록됩니다.
*   **장점**: 메모리 접근 속도가 매우 빠릅니다 ($O(1)$). 객체 데이터를 읽으러 갈 때 헤더도 같이 읽히기 때문입니다.
*   **한계**: 객체가 메모리에서 해제(`deinit`)되면 헤더도 같이 사라집니다. 만약 이때 누군가 이 객체를 `weak`로 가리키고 있다면, 그 `weak` 포인터는 갈 곳을 잃고 쓰레기 값(Dangling Pointer)을 가리키게 됩니다.

### B. Side Table (확장 모드)
*   **물리적 위치**: 객체 외부의 별도 메모리 공간입니다.
*   **생성 조건**: 
    1. 해당 객체에 대해 **`weak` 참조**가 처음 발생할 때.
    2. 참조 횟수가 너무 많아 헤더의 64비트 공간을 넘어설 때(거의 드묾).
*   **동작**: 객체 헤더는 이제 숫자를 직접 저장하는 대신 **Side Table의 주소값**을 가집니다.
*   **왜 필요한가? (핵심)**: 객체가 해제되어 힙에서 본체 데이터가 사라져도, **Side Table은 즉시 사라지지 않고 남겨집니다.** 
    - `weak` 포인터들은 이 Side Table을 바라보고 있습니다.
    - 객체 본체가 사라진 후에도 Side Table이 "객체는 죽었어"라는 정보를 유지해주기 때문에, `weak` 포인터가 안전하게 `nil`을 반환할 수 있는 것입니다.

## ⚡ 2. Atomic Operation (원자적 연산)이란?

컴퓨터 과학에서 **Atomic(원자적)**이라는 말은 "더 이상 쪼개질 수 없는 하나의 단위"를 의미합니다.

### A. ARC에서의 의미
여러 개의 CPU 코어가 동시에 하나의 객체를 참조하려고 할 때를 상상해 보세요.
1. 스레드 A가 참조 카운트를 1 증가시키려 함 (현재 10 -> 11).
2. 동시에 스레드 B도 1 증가시키려 함 (현재 10 -> 11).
3. 원자적이지 않다면 결과가 12가 아닌 11이 되어, 객체가 제때 해제되지 않거나 너무 빨리 해제되는 대참사가 벌어집니다.

### B. 비용과 오버헤드
ARC는 이 문제를 막기 위해 **Atomic 연산**을 수행합니다. 이는 하드웨어 수준에서 "내가 이 메모리 값을 바꾸는 동안 다른 놈은 절대 건드리지 마!"라고 잠금(Lock)을 거는 것과 같습니다.
*   **성능 저하 원인**: 이 잠금 과정은 CPU의 캐시 동기화(Cache Coherency)를 강제합니다. 코어끼리 대화하며 "내가 바꿨으니까 너네 캐시 다 무효화해!"라고 신호를 보내는 과정에서 시간이 소모됩니다.
*   **실무적 교훈**: 클래스(참조 타입)를 수천 번 복사하는 루프가 느린 이유는 단순히 메모리 할당 때문이 아니라, 이 **원자적 연산으로 인한 CPU 동기화 비용**이 크기 때문입니다.
