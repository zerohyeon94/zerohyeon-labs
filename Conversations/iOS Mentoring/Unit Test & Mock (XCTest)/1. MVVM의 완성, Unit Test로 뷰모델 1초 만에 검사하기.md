**🗣️ 가볍게 시작하기 (Simple): "알파고 테스터 고용하기"**

앱을 만들다 보면 "버튼을 누르면 점수가 10점 오르는지" 확인하기 위해 매번 앱을 빌드(Cmd+R)하고, 해당 화면까지 찾아 들어가서 직접 버튼을 눌러봅니다. 만약 코드를 고칠 때마다 이걸 수십 번 반복해야 한다면 엄청난 시간 낭비겠죠?

Unit Test는 애플이 제공하는 **`XCTest`**라는 프레임워크를 사용해서, 나 대신 코드를 실행해 주고 결과가 맞는지 확인해 주는 '자동화 로봇'을 고용하는 일입니다.

---

**🤿 딥하게 들어가기 (Deep: 테스트의 황금 룰, Given-When-Then)**

모든 단위 테스트 코드는 전 세계 개발자들이 암묵적으로 합의한 **GWT 패턴**으로 작성합니다.

1. **Given (준비):** 테스트할 환경을 세팅합니다. (예: 뷰모델을 하나 만든다, 초기 점수는 0점이다)
    
2. **When (실행):** 실제 검사할 액션을 발생시킵니다. (예: `toggleFocus()` 함수를 호출한다)
    
3. **Then (검증):** 예상한 결과가 나왔는지 확인합니다. (예: 점수가 10점인지 검사한다)

---

**💻 실전 코드 (FocusViewModel 테스트하기)**

이전에 만들었던 집중 타이머의 뷰모델을 테스트해 보겠습니다. Xcode 프로젝트에 'Unit Testing Bundle' 타겟을 추가하면 테스트 코드를 짤 수 있는 파일이 생깁니다.

```Swift
import XCTest
// ⭐️ 실제 앱의 코드(FocusSense)를 테스트 파일에서 쓸 수 있게 가져옵니다.
@testable import FocusSense 

final class FocusViewModelTests: XCTestCase {
    
    // SUT (System Under Test): 우리가 지금 테스트할 '주인공'이라는 뜻의 관용어입니다.
    var sut: FocusViewModel! 

    // 1. 테스트 시작 직전에 매번 호출됨 (초기화)
    override func setUpWithError() throws {
        try super.setUpWithError()
        sut = FocusViewModel() // Given: 뷰모델을 깨끗한 상태로 생성
    }

    // 2. 테스트가 끝나면 호출됨 (메모리 정리)
    override func tearDownWithError() throws {
        sut = nil
        try super.tearDownWithError()
    }

    // 3. ⭐️ 실제 테스트 함수 (이름은 무조건 'test'로 시작해야 합니다!)
    func test_toggleFocus_whenCalled_scoreIncreasesBy10() {
        
        // Given (준비)
        // 처음에 집중 상태는 false, 점수는 0점이어야 정상입니다.
        XCTAssertFalse(sut.session.isFocusing)
        XCTAssertEqual(sut.session.score, 0)
        
        // When (실행)
        // 유저가 화면에서 집중 버튼을 눌렀다고 가정하고, 뷰모델의 함수를 직접 찌릅니다!
        sut.toggleFocus()
        
        // Then (검증)
        // 점수가 10점 올랐는지, 집중 상태가 true로 바뀌었는지 로봇에게 검사시킵니다.
        // 만약 10점이 아니면 여기서 빨간색 에러(X)가 뜹니다!
        XCTAssertEqual(sut.session.score, 10, "집중 버튼을 누르면 점수가 10점 올라야 합니다.")
        XCTAssertTrue(sut.session.isFocusing, "상태가 집중 모드(true)로 바뀌어야 합니다.")
    }
}
```

이제 `Cmd + U` (테스트 실행 단축키)를 누르면, 아이폰 화면이 뜨지도 않은 채 0.1초 만에 초록색 체크 마크(✅)가 뜨면서 테스트가 통과됩니다!

### 🚨 여기서 드러나는 MVVM의 진짜 위력

만약 과거의 **MVC 패턴**이었다면 어땠을까요? 뷰 컨트롤러 안에 로직이 섞여 있어서 테스트 코드가 이렇게 끔찍해집니다.

```Swift
// ❌ MVC 시절의 끔찍한 UI 테스트
sut.loadViewIfNeeded() // 억지로 화면을 메모리에 올리고
sut.focusButton.sendActions(for: .touchUpInside) // 버튼이 눌린 '척' 이벤트를 쏘고
XCTAssertEqual(sut.scoreLabel.text, "점수: 10") // 라벨에 적힌 글씨를 파싱해서 검사해야 함...
```

하지만 **MVVM 패턴**으로 뇌(ViewModel)를 순수하게 분리해 냈기 때문에, 화면 렌더링을 기다릴 필요 없이 데이터 구조체의 값(`session.score`)만 딱 깔끔하게 비교하면 끝나는 것입니다!

**💡 오늘의 요약**

> Unit Test는 **`XCTest`**를 활용하여 **Given-When-Then** 흐름에 따라 뷰모델의 로직이 의도대로 돌아가는지 코드로 자동 검증하는 강력한 방패입니다.

이 방패가 있으면, 나중에 앱에 새로운 기능을 마구 추가하거나 코드를 크게 수정(리팩토링)할 때도 "내가 기존 기능을 망가뜨리진 않았을까?" 하는 두려움 없이 당당하게 코딩할 수 있게 됩니다.
