## 핵심 답변

`CameraService`는 "카메라를 켠다"는 기능 파일이라기보다, **카메라 하드웨어에서 들어오는 프레임을 앱이 처리할 수 있는 데이터 흐름으로 바꿔주는 파이프라인 관리자**로 보면 된다.

흐름은 아래처럼 읽으면 가장 단순하다.

```text
Camera Device
→ AVCaptureDeviceInput
→ AVCaptureSession
→ AVCaptureVideoDataOutput
→ CMSampleBuffer
→ CVPixelBuffer
→ Vision/Core ML
→ UI 업데이트
```

이 순서가 중요한 이유는 `CameraService.swift`를 위에서 아래로 읽는 것보다, **데이터가 실제로 어디서 들어와서 어디로 나가는지**를 따라가는 편이 구조 파악이 빠르기 때문이다.

## 1. AVCaptureSession은 카메라 파이프라인의 실행 엔진

`AVCaptureSession`은 입력과 출력을 연결하고 실행하는 객체다. 즉, `CameraService` 안에서 세션은 중심축이다.

```swift
let session = AVCaptureSession()
session.beginConfiguration()
// input/output 추가
session.commitConfiguration()
session.startRunning()
```

분석할 때는 아래를 먼저 찾는다.

- `session`이 어디에서 생성되는가
- `beginConfiguration()` / `commitConfiguration()`이 있는가
- `startRunning()` / `stopRunning()`이 어디에서 호출되는가
- 이 호출이 메인 스레드에서 일어나지는 않는가

중요한 포인트는 `startRunning()`이 비용이 있는 동기 작업이라는 점이다. 이걸 메인 스레드에서 호출하면 화면 진입 시 UI가 순간적으로 멈출 수 있다.

## 2. AVCaptureDeviceInput은 카메라 하드웨어를 세션에 넣는 입구

카메라 프레임은 그냥 앱에 들어오지 않는다. 먼저 실제 카메라 장치를 찾고, 그 장치를 `AVCaptureDeviceInput`으로 감싸서 세션에 넣는다.

```swift
let device = AVCaptureDevice.default(.builtInWideAngleCamera, for: .video, position: .back)
let input = try AVCaptureDeviceInput(device: device)
session.addInput(input)
```

여기서 볼 것은 "어떤 카메라를 쓰는가"와 "실패했을 때 어떻게 처리하는가"다.

- 전면 카메라인가, 후면 카메라인가
- 권한이 없으면 어떤 상태가 되는가
- `canAddInput` 체크가 있는가
- 에러를 조용히 무시하고 있지는 않은가

## 3. AVCaptureVideoDataOutput은 프레임을 코드로 받는 출구

실시간 비전 처리 앱에서 핵심은 `AVCaptureVideoDataOutput`이다. 사진을 찍는 앱이라면 `PhotoOutput`이 중요하지만, 자세 분석이나 실시간 AI 앱은 매 프레임이 필요하므로 `VideoDataOutput`을 본다.

```swift
let output = AVCaptureVideoDataOutput()
output.setSampleBufferDelegate(self, queue: videoQueue)
session.addOutput(output)
```

여기서 가장 먼저 확인할 것은 delegate queue다.

```swift
output.setSampleBufferDelegate(self, queue: DispatchQueue(label: "camera.frame.queue"))
```

이 queue는 "카메라 프레임을 받는 작업대"라고 보면 된다. 이 작업대에서 바로 무거운 모델 추론까지 하면 프레임이 밀릴 수 있다. 그래서 실무에서는 아래 두 가지를 구분한다.

- 프레임 수신 queue
- Vision/Core ML 분석 queue

처음 분석할 때는 둘이 분리되어 있는지만 봐도 충분하다.

## 4. CMSampleBuffer는 카메라 한 프레임의 포장지

`captureOutput(_:didOutput:from:)`에 들어오는 `sampleBuffer`는 카메라에서 들어온 한 프레임이다.

```swift
func captureOutput(
    _ output: AVCaptureOutput,
    didOutput sampleBuffer: CMSampleBuffer,
    from connection: AVCaptureConnection
) {
    guard let pixelBuffer = CMSampleBufferGetImageBuffer(sampleBuffer) else {
        return
    }
}
```

여기서 `CMSampleBuffer`를 바로 모델에 넣는다기보다, 내부의 `CVPixelBuffer`를 꺼내서 Vision/Core ML이 처리하게 만든다.

```text
CMSampleBuffer = 시간 정보 + 포맷 정보 + 이미지 버퍼
CVPixelBuffer = 실제 픽셀 데이터
```

그래서 코드 분석에서는 `CMSampleBufferGetImageBuffer`를 찾으면 된다. 이 지점이 "카메라 프레임이 AI 입력 데이터로 바뀌는 순간"이다.

## 5. CameraService에서 성능 병목을 찾는 순서

성능 병목은 보통 아래 순서로 의심한다.

### 1. 프레임을 너무 많이 처리하는가

카메라는 초당 30프레임 이상 들어올 수 있다. 그런데 모델 추론이 프레임마다 50ms 이상 걸리면 바로 밀린다.

확인할 것:

```swift
output.alwaysDiscardsLateVideoFrames = true
```

이 설정이 있으면 처리 못 한 오래된 프레임을 버릴 수 있다. 실시간 앱에서는 "모든 프레임 처리"보다 "최신 프레임 유지"가 더 중요하다.

### 2. 메인 스레드에서 무거운 일을 하는가

아래 같은 흐름이면 위험하다.

```text
captureOutput
→ Vision/Core ML 추론
→ UI 업데이트
→ 전부 main queue
```

UI 업데이트만 메인에서 해야 하고, 프레임 분석은 백그라운드 queue에서 해야 한다.

### 3. PixelBuffer를 불필요하게 복사하는가

`CVPixelBuffer`를 이미지로 바꾸고, 다시 모델 입력으로 바꾸는 식의 변환이 많으면 비용이 커진다.

분석할 때는 이런 변환을 찾는다.

- `UIImage`
- `CIImage`
- `CGImage`
- `CVPixelBuffer`

변환이 많을수록 실시간 처리에는 불리하다.

## CameraService 읽기 요령

실제 코드에서는 아래 5줄 메모만 채우면 된다.

```text
1. 세션 생성 위치:
2. 카메라 입력 추가 위치:
3. VideoDataOutput delegate queue:
4. SampleBuffer에서 PixelBuffer 꺼내는 위치:
5. Vision/Core ML 또는 UI로 넘기는 위치:
```

이 5개를 찾으면 `CameraService`의 큰 흐름은 잡힌다. 그 다음에야 권한 처리, 화면 생명주기, 프레임 드랍 정책, 모델 추론 최적화를 보면 된다.

## 한 문장 정리

`CameraService`는 `AVCaptureSession`으로 카메라 입력과 프레임 출력을 연결하고, `CMSampleBuffer`로 들어온 한 프레임을 `CVPixelBuffer`로 꺼내 Vision/Core ML 파이프라인에 넘기는 역할을 한다.

오늘은 이 한 문장만 말할 수 있어도 충분하다.
