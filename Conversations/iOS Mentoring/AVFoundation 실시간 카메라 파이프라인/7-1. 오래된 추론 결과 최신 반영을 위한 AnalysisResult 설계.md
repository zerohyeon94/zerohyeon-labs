# 7-1. 오래된 추론 결과 최신 반영을 위한 AnalysisResult 설계

> Daily: [[Daily/2026-06-07]]
> 분야: iOS Mentoring
> 대주제: AVFoundation 실시간 카메라 파이프라인
> 원 질문: "오래된 추론 결과가 늦게 도착할 때 최신 결과만 UI에 반영하려면 AnalysisResult를 어떻게 설계해야 하나요?"
> 관련 학습 파일: [[7. AnalysisResult 설계와 최신 결과만 반영하는 기준]]

## 짧은 답

`AnalysisResult`는 "모델이 계산한 모든 값"이 아니라 "UI가 최신성 판단을 할 수 있는 작은 결과 스냅샷"으로 설계한다.

최소 필드는 아래 정도가 좋다.

```swift
struct AnalysisResult: Sendable, Equatable {
    let frameID: Int64
    let capturedAt: Date
    let finishedAt: Date

    let focusScore: Double
    let confidence: Double
    let shouldShowWarning: Bool
    let summaryText: String?
}
```

여기서 가장 중요한 값은 `frameID`다. `frameID`는 프레임을 받을 때 단조 증가시키는 값이고, ViewModel은 마지막으로 반영한 결과보다 `frameID`가 큰 결과만 UI에 반영한다.

## 왜 frameID가 필요한가

실시간 추론에서는 먼저 들어온 프레임이 항상 먼저 끝나지 않는다.

```text
frame 101 -> 추론 오래 걸림 -> 늦게 도착
frame 102 -> 추론 빨리 끝남 -> 먼저 도착
```

이때 UI가 단순히 "도착한 순서"대로 반영하면 102번 결과를 보여준 뒤 101번 결과가 화면을 덮어쓸 수 있다. 그래서 도착 순서가 아니라 "원본 프레임 순서"를 비교해야 한다.

```swift
@MainActor
final class CameraViewModel: ObservableObject {
    @Published private(set) var currentResult: AnalysisResult?

    private var lastRenderedFrameID: Int64 = -1
    private var lastRenderedAt: Date = .distantPast

    func apply(_ result: AnalysisResult) {
        guard result.frameID > lastRenderedFrameID else {
            return
        }

        guard shouldRender(result) else {
            return
        }

        currentResult = result
        lastRenderedFrameID = result.frameID
        lastRenderedAt = Date()
    }
}
```

## UI 반영 조건

최신 프레임이라고 해서 항상 화면을 갱신할 필요는 없다. 최소한 아래 3가지를 같이 본다.

```swift
private func shouldRender(_ result: AnalysisResult) -> Bool {
    let minimumInterval: TimeInterval = 0.15
    let scoreThreshold = 0.05

    let enoughTimePassed = Date().timeIntervalSince(lastRenderedAt) >= minimumInterval

    guard let currentResult else {
        return true
    }

    let warningChanged = result.shouldShowWarning != currentResult.shouldShowWarning
    let scoreChanged = abs(result.focusScore - currentResult.focusScore) >= scoreThreshold

    return warningChanged || scoreChanged || enoughTimePassed
}
```

이 기준이면 UI는 아래처럼 동작한다.

- 더 오래된 `frameID` 결과는 무시한다.
- 경고 상태가 바뀌면 바로 반영한다.
- 점수 변화가 충분히 크면 반영한다.
- 변화가 작아도 일정 시간이 지나면 갱신한다.

## 넣지 않는 것이 좋은 값

`AnalysisResult`에는 아래 값을 넣지 않는 편이 좋다.

- `CMSampleBuffer`
- `CVPixelBuffer`
- `VNRequest`
- 모델 원시 출력 전체 배열
- 디버깅용 이미지 데이터
- View에서 직접 쓰지 않는 중간 계산값

이 값들은 무겁고, 스레드 경계를 넘기기 어렵고, UI에 필요한 정보보다 훨씬 크다. `AnalysisResult`는 MainActor로 넘어갈 수 있는 작고 안정적인 값이어야 한다.

## 역할 분리

권장 구조는 아래와 같다.

```text
CameraService
- frameID 생성
- sampleBuffer / pixelBuffer 처리
- 추론 실행
- AnalysisResult 생성

ViewModel
- frameID 비교
- UI 반영 간격 판단
- @Published 상태 갱신
```

핵심은 CameraService가 "결과를 만들고", ViewModel이 "화면에 반영할지 판단한다"는 점이다. 이렇게 나누면 추론 로직과 UI 정책이 섞이지 않는다.

## 오늘 기억할 문장

```text
AnalysisResult는 추론 결과 그 자체가 아니라,
UI가 최신성, 의미 변화, 반영 간격을 판단할 수 있게 만든
작은 결과 스냅샷이다.
```
