## 전체 흐름

FocusSense의 현재 카메라 분석 경로는 아래처럼 읽을 수 있다.

```text
TimerViewModel.init()
-> CameraService 생성
-> CameraService.checkPermission()
-> setupSession() / configureSession()
-> AVCaptureVideoDataOutputSampleBufferDelegate.captureOutput(...)
-> frameInterval 기준으로 프레임 스로틀링
-> CameraServiceDelegate로 CMSampleBuffer 전달
-> TimerViewModel.cameraService(_:didOutput:)
-> SimpleFocusDetectionService.processFrame(sampleBuffer)
-> CMSampleBufferGetImageBuffer
-> Vision 얼굴/랜드마크 분석
-> CoreML 모델이 로드되어 있으면 CoreML 졸음 추론
-> @Published 상태 업데이트
```

현재 실제로 연결된 분석 서비스는 `TimerViewModel.init()` 기준 `SimpleFocusDetectionService`다.

## 1. 세션 소유권 확인

### 세션은 singleton처럼 오래 살아 있는가?

아니다. 현재 구조는 singleton이 아니라 `CameraService` 인스턴스가 `AVCaptureSession`을 하나 소유하는 구조다.

```swift
private let captureSession = AVCaptureSession()

var session: AVCaptureSession {
    return captureSession
}
```

`session` computed property는 접근할 때마다 새 세션을 만드는 것이 아니라, 이미 생성된 `captureSession` 참조를 반환한다. 따라서 더 정확한 표현은 아래와 같다.

```text
singleton은 아니다.
CameraService 인스턴스당 하나의 AVCaptureSession이 생성된다.
CameraService 인스턴스가 살아있는 동안 같은 captureSession을 계속 사용한다.
```

### 화면 진입/이탈과 함께 만들어지고 해제되는가?

`TimerViewModel.init()`에서 `CameraService()`가 생성된다.

```swift
init() {
    self.cameraService = CameraService()
    self.focusService = SimpleFocusDetectionService()
    ...
    cameraService.delegate = self
}
```

따라서 `CameraService`와 `captureSession`의 실제 생명주기는 `TimerViewModel`을 누가 소유하고 얼마나 오래 유지하는지에 따라 결정된다. `CameraService` 내부만 보면 앱 전체 singleton이라고 볼 근거는 없다.

### startRunning()과 stopRunning()은 어디에서 호출되는가?

`CameraService.startSession()`과 `CameraService.stopSession()`에서 호출된다.

```swift
func startSession() {
    guard permissionGranted else { return }
    
    sessionQueue.async { [weak self] in
        guard let self = self else { return }
        if !self.captureSession.isRunning {
            self.captureSession.startRunning()
            DispatchQueue.main.async {
                self.isRunning = true
            }
        }
    }
}

func stopSession() {
    sessionQueue.async { [weak self] in
        guard let self = self else { return }
        if self.captureSession.isRunning {
            self.captureSession.stopRunning()
            DispatchQueue.main.async {
                self.isRunning = false
            }
        }
    }
}
```

핵심은 `startSession()` / `stopSession()`이 세션 객체를 생성하거나 해제하는 함수가 아니라, 이미 존재하는 `AVCaptureSession`의 캡처 실행 상태를 제어하는 함수라는 점이다.

## 2. 입력 장치 설정 확인

### 전면/후면 카메라 선택 기준이 있는가?

있다. 사용자의 얼굴 인식 여부를 확인하는 앱이므로 전면 카메라를 선택한다.

```swift
let frontCamera = AVCaptureDevice.default(
    .builtInWideAngleCamera,
    for: .video,
    position: .front
)
```

전면 카메라 미리보기 특성상 출력 connection에서 mirroring도 켠다.

```swift
if connection.isVideoMirroringSupported {
    connection.isVideoMirrored = true
}
```

### 권한이 없을 때 흐름이 끊기지 않는가?

권한 흐름은 비교적 명확하다.

- `.authorized`: `permissionGranted = true`로 설정하고 `setupSession()` 호출
- `.notDetermined`: `requestAccess` 후 허용된 경우에만 `setupSession()` 호출
- `.denied`, `.restricted`: `permissionGranted = false`로 두고 세션 설정을 진행하지 않음

또한 `startSession()`에서도 `permissionGranted`가 false이면 바로 반환한다.

```swift
guard permissionGranted else {
    print("⚠️ Camera permission not granted")
    return
}
```

따라서 권한이 없을 때 카메라 세션 실행까지 무리하게 진행되지는 않는다.

### 입력 추가 실패 시 에러 처리가 있는가?

일부는 있다. `AVCaptureDeviceInput` 생성 실패나 `lockForConfiguration()` 실패는 `catch`에서 처리하고, delegate로 에러를 전달한다.

```swift
catch {
    print("❌ Camera input error: \(error)")
    delegate?.cameraService(self, didFailWithError: error)
    captureSession.commitConfiguration()
    return
}
```

다만 아래 두 경우는 현재 print 또는 무처리에 가깝다.

```text
전면 카메라를 찾지 못한 경우:
print 후 return, delegate 에러 전달 없음

captureSession.canAddInput(input) == false인 경우:
명시적 else 처리 없음
```

출력 추가도 마찬가지로 `canAddOutput(videoOutput)`이 false일 때 명시적 처리 없이 넘어간다.

따라서 체크리스트 답변은 이렇게 쓰는 것이 정확하다.

```text
입력 생성/설정 중 throw되는 에러는 처리한다.
하지만 frontCamera nil, canAddInput false, canAddOutput false는 로그 또는 무처리라 보강 여지가 있다.
```

## 3. 출력과 delegate queue 확인

### delegate queue가 메인 큐가 아닌가?

메인 큐가 아니다. `CameraService` 안에 별도 `outputQueue`가 선언되어 있고, 이 큐가 sample buffer delegate queue로 지정되어 있다.

```swift
private let outputQueue = DispatchQueue(label: "com.focussense.camera.output")

videoOutput.setSampleBufferDelegate(self, queue: outputQueue)
```

따라서 `captureOutput(_:didOutput:from:)`는 메인 큐가 아니라 `com.focussense.camera.output` 큐에서 호출된다.

### 프레임이 밀릴 때 오래된 프레임을 버리는 정책이 있는가?

있다.

```swift
videoOutput.alwaysDiscardsLateVideoFrames = true
```

이 설정은 처리 지연이 생겼을 때 오래된 프레임을 계속 쌓아두지 않고 버리는 정책이다. 실시간 집중도 분석 앱에서는 최신 프레임이 더 중요하므로 적절한 방향이다.

추가로 앱 자체에서도 `frameInterval`로 한 번 더 throttle한다.

```swift
guard currentTime - lastFrameTime >= frameInterval else {
    return
}
```

기본값은 1초에 1번이고, 캘리브레이션 모드에서는 약 6fps로 올라간다.

### 픽셀 포맷 설정이 Vision/Core ML 처리에 맞는가?

`kCVPixelFormatType_32BGRA`로 설정되어 있다.

```swift
videoOutput.videoSettings = [
    kCVPixelBufferPixelFormatTypeKey as String: kCVPixelFormatType_32BGRA
]
```

이 포맷은 `CVPixelBuffer`로 Vision에 넘기기 일반적으로 적합하다. 이후 `SimpleFocusDetectionService`에서 `VNImageRequestHandler(cvPixelBuffer:orientation:options:)`로 Vision 분석을 수행한다.

Core ML 쪽은 `VNCoreMLRequest`를 사용하고, 얼굴 영역만 `regionOfInterest`로 지정한 뒤 `imageCropAndScaleOption = .scaleFill`을 사용한다. 따라서 모델 입력 크기/스케일 조정은 Vision/Core ML 요청 쪽에 일부 위임되어 있다.

## 4. 프레임 처리 경로 확인

### CMSampleBuffer에서 CVPixelBuffer를 꺼내는가?

`CameraService`에서는 꺼내지 않는다. `CameraService`는 `CMSampleBuffer`를 throttle한 뒤 delegate로 전달한다.

```swift
delegate?.cameraService(self, didOutput: sampleBuffer)
```

실제 `CVPixelBuffer` 추출은 현재 메인 분석 서비스인 `SimpleFocusDetectionService.processFrame(_:)`에서 한다.

```swift
func processFrame(_ sampleBuffer: CMSampleBuffer) {
    guard let pixelBuffer = CMSampleBufferGetImageBuffer(sampleBuffer) else {
        updateDebug(step: "❌ 픽셀 버퍼 추출 실패")
        return
    }
    
    analyze(pixelBuffer: pixelBuffer)
}
```

따라서 정리하면:

```text
CameraService: CMSampleBuffer 수신 + throttle + delegate 전달
SimpleFocusDetectionService: CMSampleBuffer -> CVPixelBuffer 변환 + Vision/CoreML 분석
```

### 매 프레임마다 모델 추론을 하는가?

카메라 원본 프레임 기준으로는 아니다. `captureOutput`은 입력 프레임마다 호출되지만, `frameInterval` 기준으로 많은 프레임을 건너뛴다.

```swift
guard currentTime - lastFrameTime >= frameInterval else {
    return
}
```

기본 상태에서는 1초에 1번만 delegate로 전달된다. 다만 delegate로 통과한 프레임에 대해서는 `SimpleFocusDetectionService`가 Vision 분석을 수행하고, Core ML 모델이 로드되어 있으면 Core ML 추론도 수행한다.

```swift
if isMLModelLoaded, let model = mlModel {
    coreMLDrowsyProb = runCoreMLInference(
        pixelBuffer: pixelBuffer,
        faceRect: face.boundingBox,
        model: model
    )
}
```

따라서 더 정확한 답변은 아래와 같다.

```text
30fps 원본 프레임마다 모델 추론하지는 않는다.
CameraService에서 frameInterval 기준으로 프레임을 줄인다.
다만 throttle을 통과한 프레임마다 Vision 분석은 수행하고,
CoreML 모델이 로드되어 있으면 해당 프레임마다 CoreML 추론도 수행한다.
```

### UI 업데이트는 메인 스레드로 안전하게 넘기는가?

대부분 메인 스레드로 안전하게 넘긴다.

`CameraService`의 `permissionGranted`, `isRunning` 업데이트는 `DispatchQueue.main.async`로 처리된다.

`TimerViewModel`은 `@MainActor` 클래스이고, delegate 메서드는 `nonisolated`로 받은 뒤 내부에서 `Task { @MainActor in ... }`로 다시 메인 액터에 진입한다.

```swift
nonisolated func cameraService(_ service: CameraService, didOutput sampleBuffer: CMSampleBuffer) {
    Task { @MainActor in
        if calibrationService.isCalibrating {
            collectCalibrationSample()
        }

        focusService.processFrame(sampleBuffer)
    }
}
```

`SimpleFocusDetectionService`도 최종 상태 업데이트와 debug 업데이트를 `DispatchQueue.main.async`로 수행한다.

```swift
DispatchQueue.main.async { [weak self] in
    self.isAnalyzing = false
    self.currentState = FocusState(...)
    self.faceAnalysisData.focusLevel = focusLevel
}
```

다만 성능 관점에서 중요한 주의점이 있다.

현재 `focusService.processFrame(sampleBuffer)` 호출 자체가 `Task { @MainActor in }` 내부에 있다. `processFrame` 안에서 `VNImageRequestHandler.perform`과 Core ML 추론이 동기적으로 실행되므로, 현재 구조에서는 Vision/Core ML 분석이 MainActor에서 실행될 가능성이 높다.

즉, UI 상태 접근은 안전하지만, 무거운 분석 작업이 MainActor에서 실행될 수 있어 UI 버벅임의 병목 후보가 된다.

## 실무 연결 5줄 메모

```text
1. 세션 생성 위치:
   CameraService 내부 `private let captureSession = AVCaptureSession()`.
   TimerViewModel.init()에서 CameraService를 생성하므로, 세션 생명주기는 TimerViewModel 소유 구조를 따라간다.

2. 입력 설정 위치:
   CameraService.configureSession().
   전면 카메라(.front)를 선택하고 AVCaptureDeviceInput으로 추가한다.
   권한 허용 시에만 setupSession()을 호출한다.

3. 출력 delegate queue:
   `outputQueue = DispatchQueue(label: "com.focussense.camera.output")`.
   `videoOutput.setSampleBufferDelegate(self, queue: outputQueue)`이므로 메인 큐가 아니다.

4. 프레임 처리 함수:
   CameraService.captureOutput(_:didOutput:from:)에서 frameInterval throttle 후 delegate로 CMSampleBuffer 전달.
   SimpleFocusDetectionService.processFrame(_:)에서 CMSampleBufferGetImageBuffer로 CVPixelBuffer 추출.

5. 병목 의심 지점:
   CameraService는 outputQueue와 throttle로 잘 줄이고 있다.
   다만 TimerViewModel에서 MainActor로 들어간 뒤 focusService.processFrame을 호출하므로,
   Vision/CoreML 분석이 MainActor에서 동기 실행될 수 있다.
```
