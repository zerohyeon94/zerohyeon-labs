# 8. 추론 완료 순서가 뒤바뀌는 이유와 frameID 추적 기준

> Daily: [[Daily/2026-06-08]]
> 분야: iOS Mentoring
> 대주제: AVFoundation 실시간 카메라 파이프라인
> 멘토링 시작 문장: "iOS 멘토링을 시작하겠습니다. 질문: 같은 프레임 스트림인데 추론 완료 순서가 뒤바뀔 때 어떤 지점에서 순서가 바뀌고, 이를 어떻게 추적해야 하나요?"

## 오늘 이 주제를 보는 이유

어제는 `최신 결과만 반영하는 기준`을 `frameID`와 `updatedAt` 중심으로 정리했다.

오늘은 그보다 한 단계 앞에서, 왜 같은 카메라 스트림인데도 완료 순서가 바뀌는지 본다. 원인을 이해해야 stale result를 버리는 기준도 설명이 선명해진다.

## 학습 목표

- `캡처 시점`과 `추론 완료 시점`이 왜 다를 수 있는지 설명한다.
- 어떤 단계에서 순서 역전이 생기는지 3개 이하로 나눈다.
- `frameID`와 시간 필드를 어디에 남기면 추적이 쉬운지 정리한다.
- 오늘 메모로 `순서가 갈리는 지점 1개 + 비교 기준 1개`를 남긴다.

## 핵심 개념

### 1. 입력 순서와 완료 순서는 다를 수 있다

카메라는 보통 프레임을 순서대로 넘긴다.

하지만 아래 흐름을 지나면 완료 순서는 쉽게 달라진다.

```text
captureOutput
-> pixelBuffer 변환/전처리
-> Vision/Core ML 추론
-> 후처리
-> UI 반영
```

프레임마다 처리 시간이 조금만 달라도, 먼저 들어온 프레임이 나중에 끝날 수 있다.

### 2. 순서가 바뀌는 대표 지점은 3개면 충분하다

오늘은 아래 세 지점만 먼저 보면 된다.

```text
1. 전처리 시간 차이
2. 추론 실행 시간 차이
3. 메인 스레드 반영 대기
```

- 전처리: 크롭, 리사이즈, 포맷 변환 비용이 프레임별로 다를 수 있다.
- 추론: 모델 내부 계산량이나 시스템 부하에 따라 완료 시점이 달라질 수 있다.
- UI 반영: 이미 메인 스레드가 바쁘면 최신 결과도 늦게 보일 수 있다.

### 3. `frameID`는 입력 순서를, 시간 필드는 지연 구간을 보여준다

최소 추적값은 이 정도면 시작할 수 있다.

```text
frameID
capturedAt
startedAt
finishedAt
```

- `frameID`: 어느 프레임이 더 최신 입력인지 비교
- `capturedAt`: 카메라에서 받은 시점
- `startedAt`: 실제 추론을 시작한 시점
- `finishedAt`: 결과가 나온 시점

이 네 값이 있으면 "늦게 도착한 결과"가 전처리에서 밀렸는지, 추론에서 밀렸는지, UI 반영에서 밀렸는지 구분하기 쉽다.

### 4. 비교 기준은 `finishedAt`보다 `frameID`가 먼저다

실시간 UI에서는 아래 순서로 생각하는 편이 안전하다.

```text
1. 마지막 반영 frameID보다 최신인가?
2. finishedAt 차이가 큰 이유는 어디인가?
3. 이 결과를 UI에 보여줄 의미 변화가 있는가?
```

즉, `finishedAt`이 더 늦다고 해서 더 최신 결과는 아니다. 최신성 판단은 입력 순서 기준이 먼저고, 시간 값은 원인 분석용이다.

### 5. 오늘 질문을 남기는 방식

메모는 길게 적지 말고 아래 한 줄이면 충분하다.

```text
frameID는 최신성 비교용, capturedAt/startedAt/finishedAt은 지연 원인 분리용
```

## 실무 연결

포트폴리오 문장으로는 아래 정도가 좋다.

```text
실시간 카메라 추론에서 입력 순서와 완료 순서가 달라질 수 있다는 점을 전제로,
frameID로 최신성을 판단하고 capturedAt/startedAt/finishedAt으로 지연 구간을 분리해
오래된 결과가 UI를 덮어쓰지 않도록 설계했다.
```
