# 6. 실시간 분석 결과 업데이트 주기와 드랍 정책 정리

> Daily: [[Daily/2026-06-06]]
> 분야: iOS Mentoring
> 대주제: AVFoundation 실시간 카메라 파이프라인
> 멘토링 시작 문장: "iOS 멘토링을 시작하겠습니다. 질문: 실시간 카메라 분석에서 결과 업데이트 주기와 프레임 드랍 정책은 어떤 기준으로 정해야 하나요?"

## 오늘 이 주제를 보는 이유

어제는 `분석 결과를 어떤 스레드에서 UI 상태로 반영할지`를 정리했다.

오늘은 그 다음 단계인 `얼마나 자주 결과를 반영할지`, `늦은 프레임을 어디서 버릴지`를 본다. 이 기준이 없으면 실시간 앱은 계산은 맞아도 사용자 경험이 흔들린다.

## 학습 목표

- 결과 업데이트 주기와 프레임 수신 주기를 같은 것으로 보지 않는다.
- `alwaysDiscardsLateVideoFrames`와 앱 레벨 드랍 정책의 역할을 구분한다.
- UI에 반영할 최소 빈도와 최신성 기준을 4줄 메모로 남긴다.
- 다음 주 WWDC26 전에 이어갈 질문 1개를 고른다.

## 핵심 개념

### 1. 모든 프레임을 모두 반영할 필요는 없다

실시간 카메라 분석에서 중요한 것은 `모든 프레임 처리`보다 `사용자에게 최신 상태를 끊기지 않게 보여주기`다.

예를 들면 아래처럼 나눌 수 있다.

```text
카메라 입력: 30fps
모델 추론: 8~12fps
UI 반영: 4~8fps 또는 상태 변화가 있을 때만
```

즉, 입력 주기와 반영 주기는 의도적으로 다를 수 있다.

### 2. 드랍 정책은 두 층으로 본다

첫 번째는 AVFoundation 레벨이다.

```text
alwaysDiscardsLateVideoFrames = true
```

이 설정은 늦은 프레임을 기본적으로 버려서 입력 큐가 밀리는 것을 줄인다.

두 번째는 앱 레벨이다.

```text
이미 추론 중이면 다음 프레임은 건너뛴다.
결과가 너무 자주 오면 UI 갱신은 간격을 둔다.
같은 경고 상태가 반복되면 화면 갱신을 생략한다.
```

실무에서는 이 두 층을 같이 봐야 한다.

### 3. UI 반영은 최신성 기준이 있어야 한다

MainActor로 넘어가는 값은 "계산이 끝난 모든 결과"가 아니라 "지금 화면에 의미 있는 최신 결과"여야 한다.

오늘 기준으로는 아래 질문만 답하면 충분하다.

- 마지막 UI 업데이트 후 몇 ms가 지났는가?
- 경고 여부나 점수가 의미 있게 바뀌었는가?
- 이미 더 최신 프레임 결과가 준비 중인가?

### 4. 결과 struct에 타임스탬프를 같이 둔다

어제 정리한 요약값에 `updatedAt` 또는 `frameID`를 같이 두면 반영 순서를 판단하기 쉽다.

```text
AnalysisResult
- focusScore
- shouldShowWarning
- updatedAt
- frameID
```

이렇게 해두면 오래된 결과가 뒤늦게 도착했을 때 무조건 덮어쓰지 않도록 설계를 생각해 볼 수 있다.

### 5. 오늘은 4줄만 남기면 충분하다

```text
1. 결과는 몇 fps 또는 어떤 상태 변화에서만 UI에 반영할 것인가
2. 이미 추론 중일 때 다음 프레임을 버릴 것인가
3. MainActor에는 어떤 요약 데이터만 넘길 것인가
4. 다음 주에 더 확인할 질문은 무엇인가
```

## 실무 연결

카메라 + 온디바이스 AI 앱 설명은 아래 정도로 말할 수 있어야 한다.

```text
프레임 입력은 계속 받되,
추론과 UI 반영 주기를 분리하고,
늦은 프레임과 중복 상태 업데이트를 버려
실시간성을 유지했다.
```

이 설명은 단순한 성능 팁이 아니라 `AI를 이해하는 iOS 개발자`라는 포트폴리오 문장으로 이어진다.
