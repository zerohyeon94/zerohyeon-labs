# 7. AnalysisResult 설계와 최신 결과만 반영하는 기준

> Daily: [[Daily/2026-06-07]]
> 분야: iOS Mentoring
> 대주제: AVFoundation 실시간 카메라 파이프라인
> 멘토링 시작 문장: "iOS 멘토링을 시작하겠습니다. 질문: 오래된 추론 결과가 늦게 도착할 때 최신 결과만 UI에 반영하려면 AnalysisResult를 어떻게 설계해야 하나요?"

## 오늘 이 주제를 보는 이유

어제는 `결과 업데이트 주기`와 `프레임 드랍 정책`을 정리했다.

오늘은 그 기준을 실제 데이터 구조로 좁힌다. `무엇을 결과로 들고 갈지`가 흐리면 MainActor 경계와 UI 갱신 기준도 계속 흔들린다.

## 학습 목표

- `분석 원본 데이터`와 `UI 반영용 요약 결과`를 구분한다.
- 오래된 결과를 걸러내기 위한 최소 필드를 정한다.
- ViewModel로 넘길 값이 왜 작고 단순해야 하는지 설명할 수 있다.
- 오늘 메모로 `최신 결과만 반영하는 기준` 3개를 남긴다.

## 핵심 개념

### 1. UI에는 원본이 아니라 요약 결과를 넘긴다

카메라 파이프라인 안에서는 `CMSampleBuffer`, `CVPixelBuffer`, 원시 점수, 중간 후처리 값이 많다.

하지만 UI가 필요한 것은 아래처럼 더 작다.

```text
AnalysisResult
- frameID
- updatedAt
- focusScore
- shouldShowWarning
- summaryText
```

즉, 화면에는 "지금 보여줄 판단"만 넘기고 원본 프레임 정보는 서비스 내부에 남기는 편이 단순하다.

### 2. 오래된 결과를 걸러내는 기준은 2개면 시작할 수 있다

오늘은 아래 두 필드만 있어도 충분하다.

```text
frameID
updatedAt
```

- `frameID`: 더 최신 프레임인지 비교
- `updatedAt`: UI 반영 간격이나 stale 여부 비교

둘 다 있으면 `늦게 도착한 오래된 결과`를 그대로 화면에 덮어쓰지 않도록 판단하기 쉽다.

### 3. 반영 조건은 "정확도"보다 "최신성 + 의미 변화"에 가깝다

UI 갱신은 모든 결과마다 할 필요가 없다.

오늘 기준으로는 아래 세 질문이면 된다.

- 이 결과가 마지막 반영 결과보다 최신인가?
- 경고 여부나 점수가 의미 있게 바뀌었는가?
- 마지막 UI 갱신 후 최소 간격이 지났는가?

이 세 조건 중 일부만 만족할 때 반영하도록 두면 실시간성이 흔들리지 않는다.

### 4. ViewModel은 비교 규칙을 갖고, CameraService는 원본 처리를 맡는다

역할은 이렇게 나누는 편이 좋다.

```text
CameraService
- 프레임 수신
- 추론 실행
- 원본 결과 생성

ViewModel
- 최신 결과 비교
- UI 반영 여부 결정
- @Published 상태 갱신
```

이렇게 두면 MainActor 안으로 무거운 값이 들어오는 일을 줄이기 쉽다.

### 5. 오늘 남길 메모는 3줄이면 된다

```text
1. 최신 결과 비교 기준(frameID / updatedAt)
2. UI 반영 조건(상태 변화 / 최소 간격)
3. MainActor에 넘길 요약 필드
```

## 실무 연결

면접이나 포트폴리오 문장으로는 아래 정도가 좋다.

```text
실시간 추론 결과를 그대로 화면에 밀어 넣지 않고,
frameID와 반영 간격 기준으로 최신 결과만 선택해
UI 업데이트 빈도와 메인 스레드 부담을 함께 줄였다.
```
