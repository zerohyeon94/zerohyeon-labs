### 나의 생각

객체지향이라는 것은 데이터와 어떠한 동작을 하는 함수가 있는데, 이를 하나로 묶어서 객체라고 하는데, 이 객체가 스스로 동작하게 하는 주체가되는 것

---

사용자님이 말씀하신 정의가 바로 객체 지향의 **가장 중요한 두 가지 개념**을 모두 포함하고 있습니다.

1. **"데이터와 동작을 하나로 묶는다"** 👉 **캡슐화 (Encapsulation)**
    
2. **"객체가 스스로 동작하는 주체가 된다"** 👉 **주체성 (Agency/Responsibility)**

---
### 객체지향 프로그래밍의 3가지 핵심 요소

- Encapsulation : 캡슐화
- Polymorphism : 다형성
- Inheritance : 상속

---
## 👨‍💻 개발자 언어로 다시 정리

### **1. "데이터와 함수를 묶음" = 상태(State)와 행위(Behavior)**

- **데이터 (변수)** → 객체 지향에서는 **'상태(State/Property)'**라고 부릅니다.
    
    - 예: `UISwitch`의 `isOn` (켜져 있나? 꺼져 있나?)
        
- **함수 (동작)** → 객체 지향에서는 **'행위(Behavior/Method)'**라고 부릅니다.
    
    - 예: `UISwitch`의 `setOn(true, animated: true)` (스위치를 켜라!)

### **2. "객체가 스스로 동작하는 주체" = 책임(Responsibility)**

이 부분이 진짜 핵심입니다. 예전(절차 지향)에는 개발자가 **'전지전능한 신'**처럼 모든 데이터를 주무르고 함수에 집어넣었습니다. 하지만 객체 지향에서는 개발자가 **'관리자(Manager)'**가 됩니다.

- **관리자(개발자):** "야, 스위치! 너 켜져." (`mySwitch.setOn(true)`)
    
- **스위치(객체):** "네, 알겠습니다. (내부적으로 내 `isOn` 변수를 `true`로 바꾸고, 내 배경색을 녹색으로 칠하고, 동그라미를 오른쪽으로 애니메이션 시키는 건 **내가 알아서 할게요.**)"

---

### 💡 iOS 개발에서의 실제 예시: `UISwitch`

만약 `UISwitch`를 **절차 지향(C 스타일)**으로 만들었다면 우리는 스위치를 켤 때마다 이렇게 코딩해야 했을 겁니다.

```c
// 절차 지향: 개발자가 모든 걸 다 알아야 함 (피곤함)
struct SwitchData sw;
sw.isOn = true;
drawBackground(sw, GREEN); // 배경색 내가 칠해줘야 함
moveKnob(sw, RIGHT);       // 손잡이도 내가 옮겨줘야 함
startAnimation(sw);        // 애니메이션도 내가 시켜야 함
```

하지만 **객체 지향(iOS 스타일)**은 이렇게 끝납니다.

```swift
// 객체 지향: 객체에게 명령만 내림 (편함)
mySwitch.setOn(true, animated: true)
// 끝! 나머지는 mySwitch 객체가 '알아서' 수행함.
```

**결론:** 사용자님의 정의대로, **"나(개발자)는 명령만 내리고, 구체적인 동작 방식은 객체가 스스로 책임지는 것"**이 바로 객체 지향 프로그래밍의 정수입니다.
