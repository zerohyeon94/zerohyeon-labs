## 🗣️ 가볍게 시작하기 (Simple): "건축 도면과 자재를 트럭에 싣고 강남 빌딩으로 이사 가기"

- **로컬 개발:** 내 방에서 레고 블록(코드)을 조립하고 설계도(Compose 파일)를 그려서 테스트를 마친 상태입니다.
    
- **배포 과정:** 이 설계도와 블록들을 트럭(GitHub)에 싣고, 24시간 전기가 끊기지 않는 강남의 빌딩(AWS EC2)으로 가져가서, 도면대로 한 번에 건물을 뚝딱 지어 올리는 과정입니다.

---

## 🤿 딥하게 들어가기 (Deep: AWS 배포 실전 4단계)

AWS EC2(우분투) 터미널에 `ssh`로 무사히 접속한 상태라고 가정하고, 건물을 올리는 순서를 밟아보겠습니다.

### Step 1: 내 코드와 설계도를 EC2로 가져오기 (Git Clone)

내 컴퓨터에 있는 파이썬 파일들과 `Dockerfile`, `docker-compose.yml`을 일일이 복사해서 EC2로 옮기는 것은 너무 힘든 일입니다. 개발자들은 보통 **GitHub**를 트럭으로 사용합니다.

1. 내 컴퓨터에서 프로젝트를 GitHub에 업로드(Push)합니다.
    
2. EC2 터미널에서 아래 명령어를 쳐서 코드를 통째로 다운로드(Clone)합니다.

```Bash
git clone [본인의 깃허브 프로젝트 주소]
cd [프로젝트 폴더명]
```

### Step 2: EC2에 오케스트라 지휘자 모셔오기

우리가 예전 수다에서 EC2에 도커(Docker) 자체는 설치했지만, 여러 방을 동시에 묶어주는 지휘자인 **Docker Compose**는 별도로 모셔와야 합니다.

```Bash
# 우분투 패키지 매니저로 간편하게 지휘자를 설치합니다.
sudo apt update
sudo apt install docker-compose -y
```

### Step 3: 대망의 마법 주문 외우기 (건물 짓기)

자, 이제 프로젝트 폴더 안으로 들어왔고 지휘자도 준비되었습니다. 심호흡을 한 번 하시고, 로컬에서 쳤던 그 마법의 주문을 똑같이 입력합니다!

```Bash
docker-compose up -d --build
```

- `--build`: "EC2 환경에 맞춰서 Dockerfile을 읽고 이미지를 새로 구워서 띄워라!"라는 뜻입니다.
    
- 이 엔터 한 번에, EC2 안에서 FastAPI 카운터, Redis 메모장, Celery 주방장이 차례대로 쫙 켜지며 내부 복도까지 완벽하게 연결됩니다.

### Step 4: AWS 방화벽(보안 그룹) 점검

서버는 완벽하게 켜졌지만, AWS는 기본적으로 모든 외부 접속을 철저히 막아둡니다. AWS 홈페이지의 EC2 관리 콘솔로 가서, 해당 인스턴스의 **'보안 그룹(Security Group)'** 인바운드 규칙에 **8000번 포트(또는 80번 포트)**를 열어주어야 영현님의 아이폰에서 이 서버로 무사히 들어올 수 있습니다.

## 🚨 핵심 포인트: 비밀번호는 절대 GitHub 트럭에 싣지 마세요! (.env 파일)

이 과정에서 겪는 가장 아찔한 보안 사고가 있습니다. DB 비밀번호나 구글 로그인 시크릿 키가 적힌 `.env` 파일을 통째로 GitHub에 올려서 EC2로 가져가는 경우입니다!

`.env` 파일은 절대 GitHub에 올리지 말고(gitignore 처리), **EC2 터미널에 접속한 뒤 직접 빈 파일을 만들어서 비밀번호를 타이핑**해 주어야 합니다.

```Bash
# EC2 터미널에서 프로젝트 폴더 안에 .env 파일 만들기
nano .env

# (안에다가 DB_PASSWORD=1234 등을 적고 저장한 뒤 빠져나옵니다)
```

이렇게 해야 도커 컴포즈가 건물을 올릴 때 EC2에만 있는 이 비밀문서를 읽고 안전하게 서버를 띄울 수 있습니다.

## 💡 오늘의 요약 처방

- **GitHub 이사:** `git clone`을 통해 로컬의 코드를 EC2 서버로 아주 쉽게 가져옵니다.
    
- **설치 및 실행:** EC2에 `docker-compose`를 설치하고 `up -d --build` 명령어로 3개의 컨테이너를 동시에 구워서 띄웁니다.
    
- **보안 철칙:** 민감한 정보가 담긴 `.env` 파일은 절대 Git으로 옮기지 말고, EC2에서 직접 생성합니다.
