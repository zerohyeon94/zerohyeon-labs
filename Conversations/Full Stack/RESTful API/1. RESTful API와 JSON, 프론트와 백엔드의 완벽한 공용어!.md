앞서 도커(Docker)라는 튼튼한 방 안에서 서버와 데이터베이스를 성공적으로 띄우셨습니다! 이제 이 서버가 밖에서 기다리는 프론트엔드 앱과 어떻게 소통하는지 알아볼 차례입니다.

## 🗣️ 가볍게 시작하기 (Simple): 식당 메뉴판과 세계 공용어

- **프론트엔드 (iOS/React):** 식당에 앉아서 메뉴를 고르고 음식을 기다리는 **'손님'**입니다.
    
- **RESTful API:** 식당의 **'메뉴판이자 주문 규칙'**입니다. "지출 내역 추가(POST)해 주세요", "집중 기록 보여(GET)주세요"처럼 서로 약속된 주소와 행동 양식입니다.
    
- **JSON:** 언어가 다른 손님과 요리사가 완벽하게 소통하기 위해 사용하는 아주 직관적인 **'세계 공용어(주문서)'**입니다.

## 🤿 딥하게 들어가기 (Deep: 프론트와 백엔드는 어떻게 대화할까?)

과거에는 모바일 앱(SwiftUI, UIKit)과 백엔드(FastAPI)가 데이터를 주고받을 때 서로의 데이터 구조가 달라서 호환성 문제가 자주 발생했습니다. 이 문제를 완벽하게 해결해 준 두 가지 핵심 규칙을 살펴보겠습니다.

### 1. 통신 규칙의 표준: RESTful API

인터넷 세상에서 식당 메뉴판처럼 미리 정해둔 주소(URL)와 행동(Method)의 규칙입니다. 어떤 작업을 할지는 4가지 핵심 동사(HTTP Method)로 결정합니다.

- **GET (조회):** "오늘 내 지출 내역 다 가져와 줘!"
    
- **POST (생성):** "오늘 커피값 4500원 새로 기록해 줘!"
    
- **PUT/PATCH (수정):** "아까 적은 커피값 4500원이 아니라 5000원이야, 수정해 줘!"
    
- **DELETE (삭제):** "실수로 잘못 적은 내역 지워 줘!"

예를 들어, 가계부 앱에서 지출을 기록한다면 프론트엔드에서는 `POST /expenses`라는 주소로 백엔드 서버에 데이터를 밀어 넣게 됩니다.

### 2. 데이터의 포장지: JSON (JavaScript Object Notation)

데이터를 요청할 때나 응답받을 때 사용하는 아주 가볍고 직관적인 텍스트 형식입니다. Swift의 딕셔너리(Dictionary) 구조와 거의 똑같이 생겼습니다.

백엔드에서 유저의 집중 시간 데이터를 꺼내서 모바일로 보내준다면, 서버는 아래와 같은 JSON 텍스트 형태로 예쁘게 포장해서 던져줍니다.

```JSON
{
  "userId": 1,
  "focusTime": 45,
  "status": "completed"
}
```

### 3. 모바일에서 데이터 받기

iOS 앱에서는 이 통신을 위해 주로 `Alamofire`나 기본 `URLSession`을 사용하게 됩니다. 백엔드가 던져준 저 JSON 텍스트를 받아서, Swift의 강력한 무기인 `Codable` 프로토콜을 이용해 모바일에서 사용하는 구조체(Struct)로 쏙 변환하면 데이터 통신이 완벽하게 끝납니다.

---

## 🚨 핵심 포인트: 서로의 '내부 언어'를 몰라도 됩니다!

RESTful API와 JSON의 가장 위대한 점은 **'완벽한 독립성'**입니다. 백엔드를 파이썬으로 만들든, 자바로 만들든 프론트엔드 개발자는 전혀 신경 쓸 필요가 없습니다. 모바일 앱이든, React 기반의 웹이든 그저 "정해진 API 주소로, JSON 데이터를 주고받자"라는 약속만 지키면 완벽하게 소통할 수 있기 때문입니다.

---

## 💡 오늘의 요약 처방

- **RESTful API**는 프론트엔드와 백엔드가 소통하기 위해 정해둔 메뉴판 같은 **'주소와 행동(GET/POST 등)의 규칙'**입니다.
    
- **JSON**은 양쪽이 데이터를 안전하고 쉽게 주고받기 위해 사용하는 표준화된 **'텍스트 포장지'**입니다.
