## 🗣️ 가볍게 시작하기 (Simple): "정적인 종이 지도" vs "실시간 내비게이션 경로"

- **일반 REST API (종이 지도):** 백엔드가 "여기 네가 요청한 데이터야" 하고 데이터만 툭 던져줍니다. 프론트엔드는 그 데이터를 받고 나서, **'그다음 어떤 주소로 요청을 보내야 할지'** 스스로 지도(코드)를 보고 판단해야 합니다.
    
- **HATEOAS (내비게이션):** 백엔드가 데이터와 함께 **"네가 다음에 누를 수 있는 버튼(행동)들의 주소는 이거야"**라고 링크(경로)까지 친절하게 싹 다 묶어서 보내줍니다. 프론트엔드는 생각할 필요 없이 백엔드가 준 주소로 이동만 하면 됩니다.

---

## 🤿 딥하게 들어가기 (Deep: 코드로 보면 어떻게 다를까?)

이해하기 쉽게, 유저의 은행 계좌 정보를 조회하는 상황을 가정해 보겠습니다.

### 1. 일반적인 REST API (HATEOAS 미적용)

서버에 계좌 조회를 요청하면 백엔드는 이렇게 응답합니다.

```JSON
{
  "accountId": 12345,
  "balance": 50000
}
```

데이터는 잘 받았습니다! 하지만 모바일 앱에서 '출금하기' 버튼이나 '입금하기' 버튼을 누르려면, 앱 개발자가 스위프트 코드 내부에 `https://api.com/withdraw/12345` 라는 주소를 **미리 하드코딩(고정)**해 두어야 합니다.

### 2. HATEOAS를 적용한 궁극의 API

이번엔 HATEOAS 원칙을 지킨 백엔드의 응답입니다.

```JSON
{
  "accountId": 12345,
  "balance": 50000,
  "links": [
    { "rel": "deposit", "href": "/api/accounts/12345/deposit" },
    { "rel": "withdraw", "href": "/api/accounts/12345/withdraw" },
    { "rel": "transfer", "href": "/api/accounts/12345/transfer" }
  ]
}
```

보이시나요? 데이터뿐만 아니라, **"현재 잔액 상태에서 네가 할 수 있는 다음 행동(rel)과 그 API 주소(href)"**들을 `links`라는 배열에 담아 함께 보내주었습니다! (만약 잔액이 0원이라면, 서버가 알아서 `withdraw` 링크를 빼고 보내줄 수도 있습니다.)

## 🚨 핵심 포인트: 왜 그토록 완벽한데 실무에서는 잘 안 쓸까?

HATEOAS가 적용되면 모바일 개발자에게는 엄청난 마법이 일어납니다. 만약 서버 개발자가 API 주소를 `/api/withdraw`에서 `/api/v2/withdraw`로 바꾸더라도, 모바일 앱은 그냥 JSON에 적힌 `href` 주소를 그대로 읽어서 실행하기 때문에 **앱 스토어에 심사를 올려서 앱을 강제로 업데이트할 필요가 전혀 없어집니다!**

하지만 이렇게 완벽해 보이는데도 현업에서 100% 지키기 어려운 이유가 있습니다.

1. **백엔드의 엄청난 피로도:** 백엔드 개발자가 데이터를 보낼 때마다 유저의 권한과 상태를 일일이 계산해서 '다음 행동 링크'를 꼬박꼬박 만들어서 붙여줘야 합니다. 개발 시간과 서버 리소스가 크게 낭비됩니다.
    
2. **모바일/웹 프론트엔드의 구조적 한계:** 이미 모바일 앱 화면 구조(UI)는 고정되어 있는 경우가 많습니다. 버튼 위치나 화면 이동 흐름이 이미 기획 단계에서 픽스되어 있기 때문에, 굳이 백엔드에서 링크를 받아와서 동적으로 화면을 구성할 필요성이 떨어집니다.

## 💡 오늘의 요약 처방

- **HATEOAS**는 API 응답 데이터에 **'다음 상태로 넘어갈 수 있는 관련된 링크(URL)'**를 함께 포함시켜 보내는 REST의 최종 단계 원칙입니다.
    
- 프론트엔드가 백엔드의 API 주소 변경에 영향을 받지 않게 되는 장점이 있습니다.
    
- 하지만 설계가 너무 복잡해지고 프론트엔드 환경과 안 맞는 경우가 많아, 현업에서는 "알아두면 좋은 이상향" 정도로 취급하고 과감히 생략하는 경우가 대부분입니다.
