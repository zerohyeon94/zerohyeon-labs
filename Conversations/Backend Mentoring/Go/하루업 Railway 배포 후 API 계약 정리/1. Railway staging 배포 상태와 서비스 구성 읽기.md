## 오늘 이 주제를 보는 이유

6월 10일에 하루업 백엔드가 Railway staging 환경에 올라갔다. 이제는 로컬 코드만 보는 단계보다, 배포된 서비스가 어떤 구성으로 살아 있고 앱이 어떤 API를 믿고 붙을 수 있는지 읽는 단계가 더 중요하다.

6월 11일부터 6월 14일까지는 기존 iOS/Swift 흐름을 잠시 멈추고 `하루업 Railway 배포 후 API 계약 정리` 큰 주제로 리셋한다. 오늘은 그 첫날이라, 구현 세부보다 서비스 구성과 계약 surface를 먼저 좁히는 데 집중한다.

## 학습 목표

- Railway staging 기준으로 `앱 -> 백엔드 -> DB/Redis -> 외부 의존성` 흐름을 한 번에 설명할 수 있다.
- API를 볼 때 `Method / URL / 인증 / 사용 화면 / 다음 조치` 다섯 칸으로 정리하는 기준을 잡는다.
- `정상 동작`, `앞으로 사용`, `보호 필요`, `보류`를 구분하는 이유를 이해한다.

## 핵심 개념

### 서비스 구성부터 읽기

배포된 백엔드를 읽을 때는 Controller 목록부터 바로 들어가기보다 아래 순서가 낫다.

1. Railway에 어떤 앱이 올라가 있는가
2. 외부에서 접근하는 base URL과 `/health`가 살아 있는가
3. PostgreSQL, Redis, JWT secret, 외부 API key 같은 필수 의존성이 붙어 있는가
4. 앱이 실제로 호출할 API가 무엇인가

이 순서를 잡아 두면 "코드는 있는데 지금 호출 불가능한 API"와 "실제로 팀이 써도 되는 API"를 빨리 분리할 수 있다.

### API 계약은 화면 기준으로 본다

오늘은 [[Projects/Haru Up/하루업 API 점검 및 문서화 계획]]에 아래 칸만 먼저 채우면 충분하다.

```text
API:
Method / URL:
상태: 정상 동작 / 앞으로 사용 / 보호 필요 / 보류
사용 화면:
다음 조치:
```

핵심은 "이 API가 존재하는가"보다 "어느 화면이 언제 어떤 인증으로 부르는가"를 붙이는 것이다.

### 보호 필요 API를 따로 본다

수동 배치 실행, 테스트 푸시, 통계/엑셀 다운로드, Actuator 같은 항목은 "안 쓰는 API"가 아니라 "외부 공개를 줄여야 하는 API"일 수 있다. 삭제 여부보다 먼저 보호 필요 여부를 따로 표시해야 한다.

## 실무 연결

오늘 바로 남길 메모는 세 줄이면 된다.

- Railway staging에서 외부에 보이는 주소 1개
- 앱이 바로 붙을 API 후보 1~3개
- 보호 필요로 의심되는 API 1개

이 세 줄이 잡히면 금요일에는 실제 사용 API 계약으로, 토요일에는 미사용/보호 API 분류로 자연스럽게 이어진다.
