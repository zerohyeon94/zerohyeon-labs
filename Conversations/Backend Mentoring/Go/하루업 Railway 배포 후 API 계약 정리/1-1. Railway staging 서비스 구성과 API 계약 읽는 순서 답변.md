# 1-1. Railway staging 서비스 구성과 API 계약 읽는 순서 답변

> Daily: [[Daily/2026-06-11]]
> 분야: Backend Go
> 대주제: 하루업 Railway 배포 후 API 계약 정리
> 관련 학습 파일: [[Conversations/Backend Mentoring/Go/하루업 Railway 배포 후 API 계약 정리/1. Railway staging 배포 상태와 서비스 구성 읽기]]
> 질문: Railway staging에 올라간 하루업 백엔드를 볼 때 서비스 구성과 API 계약을 어떤 순서로 읽어야 하나요?

## 답변 요약

순서는 `배포가 실제로 살아 있는지 확인 -> 외부 의존성과 인증 경계 확인 -> 앱이 붙을 API만 추리기 -> 각 API의 request/response 계약 확인 -> 보호해야 할 내부 API 분리`가 가장 안전하다.

짧게 적으면 아래 순서다.

```text
1. Runtime 확인
2. 의존성/환경 확인
3. 공개 surface 확인
4. 화면 기준 API 묶기
5. 계약 상세 확인
6. 보호/보류 API 분리
```

## 1. Runtime부터 본다

코드보다 먼저 "지금 이 서비스가 실제로 호출 가능한가"를 확인한다.

여기서 먼저 볼 것은 아래다.

```text
- Railway public URL
- /health 응답
- 배포 로그에 즉시 실패 흔적이 없는지
- SERVER_PORT=${PORT} 같은 런타임 설정이 맞는지
```

이 단계가 안 되면 뒤의 API 문서는 전부 의미가 약해진다. 지금 필요한 건 설계 리뷰가 아니라 "호출 가능한 staging surface"다.

## 2. 외부 의존성과 인증 경계를 본다

그다음은 이 서버가 무엇에 기대고 사는지 확인한다.

```text
- PostgreSQL
- Redis
- JWT secret
- Firebase
- Clova
- CORS
- Cloud storage / file upload 경로
```

이유는 간단하다. API가 존재해도 의존성이 빠져 있으면 `정상 동작 API`가 아니라 `조건부 API`다. 그래서 API 표를 채울 때도 "외부 의존성 필요 여부"를 먼저 알고 들어가야 한다.

같이 봐야 할 것은 인증 경계다.

```text
- 인증 없이 호출 가능한 API
- JWT가 필요한 API
- 운영자/내부 전용이어야 하는 API
```

이 경계를 초기에 나눠야 React나 iOS 쪽에서 붙을 수 있는 범위를 바로 줄일 수 있다.

## 3. 공개 surface를 그룹으로 나눈다

이제 Controller를 하나씩 읽되, 파일 순서가 아니라 앱 기능 순서로 읽는다. 현재 문서 기준으로는 아래 묶음이 적절하다.

1. 인증/회원
2. 프로필/설정
3. 미션
4. 관심사/추천/AI
5. 랭킹/알림/기타

이 단계에서 목표는 모든 API를 외우는 게 아니다. 각 그룹마다 아래 한 줄만 남기면 된다.

```text
이 그룹에서 앱이 지금 바로 붙을 API는 무엇인가?
```

## 4. 화면 기준으로 API를 다시 묶는다

여기서부터가 계약 읽기의 핵심이다. API를 endpoint 기준으로 보지 말고, 화면이나 사용자 흐름 기준으로 다시 묶는다.

예를 들면 아래처럼 본다.

```text
로그인 화면 -> 인증 API
홈/오늘 화면 -> 미션/추천 API
프로필 화면 -> 프로필/설정 API
알림 화면 -> 알림 API
```

이렇게 묶어야 "이 API가 왜 필요한가"가 분명해진다. 그리고 실제 팀 문서도 endpoint 나열보다 화면 기준으로 만들수록 프론트엔드와 모바일이 쓰기 쉽다.

## 5. 각 API는 다섯 칸만 먼저 채운다

처음부터 상세 Swagger 문서를 만들려고 하면 너무 느리다. 먼저 아래 다섯 칸만 채운다.

```text
Method / URL
인증 필요 여부
사용 화면
응답에서 화면이 실제로 쓰는 핵심 필드
다음 조치
```

`다음 조치`에는 보통 아래 네 종류 중 하나가 들어간다.

```text
- 바로 앱 연동 가능
- seed/외부 API 준비 후 재확인
- 응답 형식 수정 필요
- 보호 필요 또는 비공개 처리 필요
```

핵심은 request/response 전체를 다 적는 게 아니라 "앱이 지금 붙기 위해 모자란 정보가 무엇인가"를 남기는 것이다.

## 6. 보호/보류 API를 마지막에 따로 뺀다

수동 배치, 테스트 푸시, 통계 다운로드, Actuator, 내부 운영용 endpoint는 일반 API 목록과 섞어 두면 안 된다.

이 항목들은 아래 셋 중 하나로 바로 태깅하는 편이 좋다.

```text
보호 필요
Swagger 비공개 후보
삭제/보류 후보
```

여기서 중요한 점은 `사용하지 않음 = 삭제`가 아니라는 것이다. staging에서는 안 써도 운영에서 필요한 API가 있다. 그래서 마지막 단계에서 한 번 더 분리해야 한다.

## 오늘 바로 적용할 읽기 순서

오늘 기준으로는 아래처럼 30~40분 안에 끝내는 흐름이 좋다.

1. [[Projects/Haru Up/하루업 API 점검 및 문서화 계획]]에서 `/health`, 환경변수, 의존성 체크 항목 확인
2. 인증/회원 API부터 보고 앱이 지금 붙을 API 1~2개만 고르기
3. 미션 또는 추천 API에서 화면에 바로 연결될 API 1~2개 더 고르기
4. 랭킹/배치/푸시 쪽에서 보호 필요 API 1개 표시
5. 표에 `Method / URL / 인증 / 사용 화면 / 다음 조치` 다섯 칸만 채우기

이렇게 하면 오늘은 "서비스 구성 이해", 금요일은 "실사용 API 계약", 토요일은 "보호/보류 정리"로 자연스럽게 이어진다.

## 실무 기준 한 줄

배포된 백엔드는 "코드 구조를 다 이해하는 순서"보다 "앱이 오늘 붙을 수 있는 계약 surface를 줄여 가는 순서"로 읽는 편이 훨씬 실무적이다.

## 오늘 기억할 3줄

- 먼저 `/health`, 환경변수, DB/Redis, JWT 같은 런타임 경계를 확인한다.
- API는 Controller 순서가 아니라 화면 기준으로 묶어서 읽는다.
- 일반 API와 보호 필요 API를 마지막에 반드시 분리한다.
