# 3. 실제 사용 API 계약 5개 이하로 정리하기

> Daily: [[Daily/2026-06-12]]
> 분야: Backend Go
> 대주제: 하루업 Railway 배포 후 API 계약 정리
> 멘토링 시작 문장: "Go 멘토링을 시작하겠습니다. 질문: 지금 앱 화면 기준으로 먼저 고정해야 할 실제 사용 API 5개를 어떤 형식으로 정리하면 좋을까요?"

## 오늘 이 주제를 보는 이유

어제는 Railway staging에 올라간 서비스 구성과 계약 surface를 읽는 순서를 잡았다. 오늘은 그다음 단계로, 앱이 실제로 붙을 가능성이 높은 API만 5개 이하로 줄여서 `Method / URL / 인증 / 응답 핵심 필드 / 다음 조치` 형식으로 정리해야 한다.

특별 리셋 기간의 금요일 목표는 "모든 endpoint 파악"이 아니라 "이번 주 안에 팀과 화면 기준으로 합의할 API 계약 묶음"을 만드는 것이다. 수를 줄여야 토요일의 보호 필요 API 분리와 일요일 요약도 깔끔해진다.

## 학습 목표

- 앱 화면 기준으로 우선순위가 높은 API를 5개 이하로 고를 수 있다.
- 각 API를 `Method / URL / 인증 / 응답 핵심 필드 / 다음 조치` 다섯 칸으로 적을 수 있다.
- `정상 동작`, `응답 확인 필요`, `보호 필요`를 섞지 않고 구분할 수 있다.

## 핵심 개념

### API는 화면 기준으로 줄인다

우선 아래처럼 화면이나 사용자 흐름에서 바로 필요한 것만 고른다.

1. 로그인 또는 인증 상태 확인
2. 홈 또는 오늘의 추천 미션
3. 나의 기록 또는 완료한 미션 조회
4. 프로필/설정에서 꼭 필요한 조회
5. 알림 또는 기타 핵심 기능 1개

핵심은 Controller별 균등 분배가 아니라 "앱이 지금 보여줄 화면"을 기준으로 줄이는 것이다.

### 계약 표는 다섯 칸이면 먼저 충분하다

처음 문서화는 아래 형식이면 된다.

```text
API:
Method / URL:
인증:
응답 핵심 필드:
다음 조치:
```

`응답 핵심 필드`에는 앱 화면이 실제로 쓰는 값만 남긴다. 전체 JSON을 다 옮기기보다 `id`, `title`, `status`, `completedAt`처럼 UI와 직접 연결되는 필드가 우선이다.

### 다음 조치는 세 가지로만 남긴다

너무 많은 상태를 만들지 말고 우선 아래 셋으로 정리한다.

- 바로 연동 가능
- 응답 형식 또는 인증 확인 필요
- 보호 필요 또는 공개 범위 재검토

이렇게 해야 토요일에 일반 API와 보호 필요 API를 깔끔하게 분리할 수 있다.

## 실무 연결

오늘 저녁에는 [[Projects/Haru Up/하루업 API 점검 및 문서화 계획]]에 아래 결과만 남기면 충분하다.

- 화면 기준 핵심 API 5개 이하
- 각 API의 `Method / URL / 인증 / 응답 핵심 필드 / 다음 조치`
- 팀에 바로 물어봐야 할 애매한 계약 1~2개

이 산출물이 있으면 다음 주 Go/React 적응에서도 `어떤 API가 실제 업무 우선순위인가`를 훨씬 빠르게 읽을 수 있다.
