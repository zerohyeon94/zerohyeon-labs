> 멘토링 질문: "ZipJoong에서 처음 배포해볼 Go API는 어떤 데이터와 엔드포인트부터 잡으면 좋을까요?"

## 결론

첫 배포 범위는 `완료된 집중 세션` 하나를 PostgreSQL에 저장하고 최근 목록으로 다시 읽는 흐름이 좋다.

1. `POST /focus-sessions` — 완료된 집중 세션 생성
2. `GET /focus-sessions?limit=20` — 최근 집중 세션 목록 조회
3. `GET /healthz` — 배포 환경에서 Go 서버가 실행 중인지 확인

첫 단계에서는 세션 시작·종료를 별도 API로 나누지 않는다. 인증, 사용자·장치 등록, 실시간 이벤트, 통계, AI 분석도 제외한다.

## 후보 비교

| 접근 | 장점 | 단점 | 판단 |
| --- | --- | --- | --- |
| DB 연결 확인 API만 만들기 | 가장 빨리 배포 가능 | 실제 서비스 흐름과 React 연결을 배우기 어렵다 | `healthz`로만 보조 |
| 완료된 `focus_sessions` 생성·조회 | HTTP → Go → PostgreSQL → React 흐름을 작게 완주 가능 | 진행 중 세션은 아직 표현하지 못한다 | **첫 범위로 추천** |
| `focus_events`부터 저장 | 이벤트 로그와 관제 형태에 가깝다 | 세션·장치 식별과 이벤트 종류 설계가 먼저 필요하다 | 두 번째 단계로 보류 |

## 첫 데이터 모델

첫 테이블은 `focus_sessions` 하나만 둔다.

| 컬럼 | 타입 후보 | 제약 |
| --- | --- | --- |
| `id` | `uuid` | primary key, 서버 생성 |
| `started_at` | `timestamptz` | not null |
| `ended_at` | `timestamptz` | not null, `ended_at > started_at` |
| `goal_minutes` | `smallint` | nullable, 입력 시 `1..480` |
| `created_at` | `timestamptz` | not null, 기본값 `now()` |

`duration_minutes`는 `ended_at - started_at`으로 계산할 수 있으므로 처음에는 중복 저장하지 않는다. `status`도 첫 범위를 완료된 세션으로 한정하면 아직 필요하지 않다.

## API 계약 초안

### 세션 생성

```http
POST /focus-sessions
Content-Type: application/json
```

```json
{
  "started_at": "2026-06-22T19:00:00+09:00",
  "ended_at": "2026-06-22T19:25:00+09:00",
  "goal_minutes": 25
}
```

성공 시 `201 Created`:

```json
{
  "id": "b9a2cb14-cd9e-45de-8b0f-fc5d08fe9845",
  "started_at": "2026-06-22T10:00:00Z",
  "ended_at": "2026-06-22T10:25:00Z",
  "goal_minutes": 25,
  "duration_minutes": 25,
  "created_at": "2026-06-22T10:25:01Z"
}
```

서버 내부 저장과 응답은 UTC로 통일하고, 클라이언트가 화면에서 로컬 시간대로 변환한다.

### 최근 세션 조회

```http
GET /focus-sessions?limit=20
```

성공 시 `200 OK`:

```json
{
  "items": [
    {
      "id": "b9a2cb14-cd9e-45de-8b0f-fc5d08fe9845",
      "started_at": "2026-06-22T10:00:00Z",
      "ended_at": "2026-06-22T10:25:00Z",
      "goal_minutes": 25,
      "duration_minutes": 25,
      "created_at": "2026-06-22T10:25:01Z"
    }
  ]
}
```

- 기본 `limit`: 20
- 최대 `limit`: 100
- 정렬: `started_at DESC, id DESC`
- 데이터가 없을 때: `{"items":[]}`
- 첫 배포에서는 cursor pagination을 넣지 않는다.

## 실패 응답

첫 범위에서는 세 가지만 일관되게 처리한다.

| 상황 | 상태 코드 | 오류 코드 |
| --- | --- | --- |
| JSON 형식 또는 필수 필드 오류 | `400 Bad Request` | `invalid_request` |
| 종료 시각이 시작 시각보다 빠르거나 같은 경우 | `422 Unprocessable Entity` | `invalid_time_range` |
| PostgreSQL 연결·저장 실패 | `500 Internal Server Error` | `internal_error` |

```json
{
  "error": {
    "code": "invalid_time_range",
    "message": "ended_at must be later than started_at"
  }
}
```

DB 오류의 실제 내용은 로그에만 남기고 API 응답에는 노출하지 않는다.

## 첫 배포의 완료 기준

- `POST /focus-sessions` 호출 결과가 PostgreSQL에 1행 저장된다.
- 서버를 재시작한 뒤에도 `GET /focus-sessions`에서 같은 데이터가 조회된다.
- 빈 DB에서는 `items: []`가 반환된다.
- 잘못된 시간 범위 요청은 저장되지 않고 정해진 오류 JSON을 반환한다.
- React에서는 목록의 `loading / success / empty / error` 네 상태를 구분할 수 있다.
