# 2-1. focus_sessions 계층 책임과 오류 흐름 답변

> Daily: [[Daily/2026-06-22]]
> 분야: Backend Go
> 대주제: ZipJoong Go 서버 첫 실행 범위
> 질문 파일: [[Conversations/Backend Mentoring/Go/ZipJoong Go 서버 첫 실행 범위/2. focus_sessions API 계약과 계층 책임 정하기]]
> 멘토링 질문: "ZipJoong의 focus_sessions 생성·조회 API를 처음 구현할 때 요청 검증, 비즈니스 규칙, PostgreSQL 저장, HTTP 응답 책임을 handler/service/repository에 어떻게 나누면 좋을까요?"

## 결론

각 계층이 아는 대상을 하나로 제한한다.

- `handler`: HTTP와 JSON을 안다.
- `service`: 집중 세션의 규칙과 사용 사례를 안다.
- `repository`: PostgreSQL과 SQL을 안다.

의존 방향은 `handler → service → repository`이며, 반대 방향으로 HTTP 타입이나 PostgreSQL 타입이 새어 나오지 않게 한다.

## 전체 흐름

### 생성

```text
POST /focus-sessions
  → handler: JSON 해석과 HTTP 형식 검증
  → service: 시간·목표 시간 규칙 검증
  → repository: INSERT ... RETURNING
  → service: 도메인 결과 반환
  → handler: 201 JSON 응답으로 변환
```

### 목록 조회

```text
GET /focus-sessions?limit=20
  → handler: limit 문자열 해석
  → service: 기본값·최댓값 적용
  → repository: 최근 세션 SELECT
  → handler: 200 {"items": [...]} 응답
```

## Handler 책임

Handler는 요청을 Go 값으로 바꾸고 결과를 HTTP 응답으로 바꾸는 어댑터다.

### 해야 할 일

- JSON body decode
- 알 수 없는 JSON 필드 거부
- 필수 필드 누락과 잘못된 RFC 3339 시간 형식 확인
- query parameter인 `limit`를 정수로 변환
- service 호출
- service 오류를 HTTP status와 오류 JSON으로 매핑
- domain 값을 response DTO로 변환

### 하지 않을 일

- `ended_at > started_at` 같은 집중 세션 규칙 판단
- 집중 시간 계산
- SQL 작성
- `pgx.ErrNoRows` 같은 DB 라이브러리 오류 직접 처리

```go
type CreateFocusSessionRequest struct {
    StartedAt   time.Time `json:"started_at"`
    EndedAt     time.Time `json:"ended_at"`
    GoalMinutes *int      `json:"goal_minutes"`
}

type FocusSessionResponse struct {
    ID              string    `json:"id"`
    StartedAt       time.Time `json:"started_at"`
    EndedAt         time.Time `json:"ended_at"`
    GoalMinutes     *int      `json:"goal_minutes"`
    DurationMinutes int       `json:"duration_minutes"`
    CreatedAt       time.Time `json:"created_at"`
}
```

HTTP DTO는 handler 패키지에 두고 service나 repository가 이 타입을 받지 않게 한다.

## Service 책임

Service는 하나의 사용 사례를 완성하며 HTTP와 PostgreSQL 사이의 규칙 경계가 된다.

### 생성 규칙

- 시작·종료 시각이 비어 있지 않아야 한다.
- `ended_at`은 `started_at`보다 늦어야 한다.
- `goal_minutes`가 있다면 `1...480` 범위여야 한다.
- `duration_minutes`는 서버가 두 시각의 차이로 계산한다.
- 유효한 값만 repository에 전달한다.

### 조회 규칙

- `limit` 미입력 시 20
- 최댓값 100
- 정렬 기준은 최근 시작 시각순
- repository가 빈 slice를 반환하면 그대로 빈 목록으로 처리

```go
type CreateFocusSessionCommand struct {
    StartedAt   time.Time
    EndedAt     time.Time
    GoalMinutes *int
}

type FocusSessionService struct {
    repository FocusSessionRepository
}

func (s *FocusSessionService) Create(
    ctx context.Context,
    command CreateFocusSessionCommand,
) (FocusSession, error)

func (s *FocusSessionService) ListRecent(
    ctx context.Context,
    limit int,
) ([]FocusSession, error)
```

Service는 `http.StatusUnprocessableEntity`나 `pgxpool.Pool`을 알지 않아야 한다.

## Repository 책임

Repository는 domain 입력을 PostgreSQL 쿼리로 바꾸고 DB row를 domain 값으로 되돌린다.

### 해야 할 일

- `INSERT ... RETURNING`으로 세션 저장
- `SELECT ... ORDER BY started_at DESC, id DESC LIMIT $1` 실행
- `timestamptz`, UUID, nullable 값을 Go 값으로 scan
- context 취소와 DB 오류를 보존하여 상위 계층에 반환
- 필요하면 DB 고유 오류를 repository 수준 오류로 변환

### 하지 않을 일

- JSON 태그가 붙은 response DTO 반환
- `400`, `422`, `500` 같은 HTTP 상태 결정
- 목표 시간이 적절한지 판단
- React 화면에서 필요한 문구 결정

```go
type FocusSessionRepository interface {
    Create(
        ctx context.Context,
        input NewFocusSession,
    ) (FocusSession, error)

    ListRecent(
        ctx context.Context,
        limit int,
    ) ([]FocusSession, error)
}
```

Service가 구체적인 PostgreSQL 구현체가 아니라 이 interface에 의존하면 repository를 fake로 교체해 규칙을 빠르게 테스트할 수 있다.

## Domain 값

domain 값은 HTTP 요청이나 DB row에 종속되지 않는 프로젝트 내부 표현이다.

```go
type NewFocusSession struct {
    StartedAt   time.Time
    EndedAt     time.Time
    GoalMinutes *int
}

type FocusSession struct {
    ID              string
    StartedAt       time.Time
    EndedAt         time.Time
    GoalMinutes     *int
    DurationMinutes int
    CreatedAt       time.Time
}
```

첫 범위에서는 `duration_minutes`를 DB에 중복 저장하지 않고 service 또는 조회 과정에서 계산한다. API 응답에는 클라이언트 편의를 위해 포함한다.

## 오류 흐름

Service는 HTTP 상태가 아닌 의미 있는 오류를 반환한다.

```go
var (
    ErrInvalidTimeRange = errors.New("invalid time range")
    ErrInvalidGoal      = errors.New("invalid goal minutes")
)
```

Handler가 `errors.Is`로 HTTP 응답에 매핑한다.

| 발생 위치 | 내부 오류 | Handler 응답 |
| --- | --- | --- |
| handler | JSON decode 실패, 잘못된 `limit` | `400 invalid_request` |
| service | `ErrInvalidTimeRange` | `422 invalid_time_range` |
| service | `ErrInvalidGoal` | `422 invalid_goal_minutes` |
| repository | 연결·query·scan 실패 | 로그 기록 후 `500 internal_error` |

Repository 오류에는 `%w`로 원인을 감싸 디버깅 정보를 유지하되, handler는 실제 SQL이나 접속 정보를 사용자에게 노출하지 않는다.

## Transaction 판단

첫 생성 API가 `focus_sessions` 한 행만 INSERT한다면 명시적인 transaction은 필요 없다. SQL 한 문장 자체가 원자적이다.

다음처럼 여러 저장 작업이 모두 성공하거나 모두 실패해야 할 때 service가 transaction 경계를 요청한다.

- 세션 생성과 첫 이벤트 생성을 함께 저장
- 세션 종료와 일별 통계를 동시에 갱신
- 여러 repository 호출을 하나의 사용 사례로 묶음

Transaction 필요성을 미리 넣지 말고, 두 개 이상의 DB 변경이 하나의 원자적 작업이 되는 시점에 추가한다.

## 최소 테스트

### Handler

- 잘못된 JSON은 service를 호출하지 않고 `400`을 반환한다.
- service의 `ErrInvalidTimeRange`를 `422` 오류 JSON으로 변환한다.
- 정상 생성 결과를 `201` response DTO로 변환한다.
- 빈 목록을 `{"items":[]}`로 반환한다.

### Service

- 종료 시각이 시작 시각보다 이르면 repository를 호출하지 않는다.
- 목표 시간이 범위를 벗어나면 repository를 호출하지 않는다.
- 유효한 command를 repository 입력으로 변환한다.
- repository 오류를 삼키지 않고 상위로 전달한다.

### Repository

- INSERT 후 반환된 row를 domain 값으로 올바르게 변환한다.
- 목록이 `started_at DESC, id DESC` 순서로 반환된다.
- DB에 데이터가 없으면 오류가 아니라 빈 slice를 반환한다.

## 오늘 확정할 표

### API 계약

| 항목 | 결정 |
| --- | --- |
| 생성 요청 필수 필드 | `started_at`, `ended_at`; `goal_minutes`는 선택 |
| 생성 성공 응답 | `id`, 두 시각, 목표 시간, 계산된 `duration_minutes`, `created_at` |
| 목록 기본 정렬 | `started_at DESC, id DESC` |
| 빈 목록 응답 | `{"items":[]}` |
| 시간 표준과 형식 | API는 RFC 3339, DB는 `timestamptz`, 응답은 UTC |

### 계층 책임

| 질문 | 결정 |
| --- | --- |
| handler에서 끝낼 검증 | JSON·시간 형식·필수 필드·query 문자열 변환 |
| service에서 처리할 규칙 | 시간 순서, 목표 시간 범위, 기본·최대 limit, 집중 시간 계산 |
| repository가 반환할 값과 오류 | domain 값 또는 원인을 감싼 저장소 오류 |
| transaction이 필요한 지점 | 첫 단일 INSERT에는 불필요; 다중 DB 변경부터 service 사용 사례 경계에서 적용 |
