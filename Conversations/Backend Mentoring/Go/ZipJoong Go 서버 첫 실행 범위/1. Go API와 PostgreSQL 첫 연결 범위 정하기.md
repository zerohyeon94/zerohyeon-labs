# 1. Go API와 PostgreSQL 첫 연결 범위 정하기

> Daily: [[Daily/2026-06-15]]
> 분야: Backend Go
> 대주제: ZipJoong Go 서버 첫 실행 범위
> 멘토링 시작 문장: "Go 멘토링을 시작하겠습니다. 질문: ZipJoong에서 처음 배포해볼 Go API는 어떤 데이터와 엔드포인트부터 잡으면 좋을까요?"

## 오늘 이 주제를 보는 이유

Go 멘토링은 하루업이 아니라 ZipJoong의 서버 배포와 실제 동작 확인에 연결한다.
ZipJoong은 아직 프로젝트가 시작되지 않았으므로, 오늘은 구현을 크게 열지 않고 Go API와 PostgreSQL의 첫 연결 범위를 작게 정한다.

## 학습 목표

- ZipJoong의 첫 Go API 범위를 1-2개 리소스로 줄인다.
- PostgreSQL에 저장할 최소 테이블 후보를 정한다.
- React 대시보드가 읽을 수 있는 응답 형태를 미리 생각한다.

## 핵심 개념

- 첫 Go 서버는 인증, 실시간 처리, AI 분석보다 `생성/조회`가 되는 작은 API부터 시작한다.
- ZipJoong의 첫 리소스 후보는 `focus_sessions` 또는 `focus_events`가 적합하다.
- PostgreSQL 테이블과 API 응답을 함께 생각하면 React 대시보드 범위를 작게 잡기 쉽다.

## 실무 연결

- Go: REST API, handler/service/repository 구조, DB 연결
- PostgreSQL: 집중 세션 또는 이벤트 로그 테이블
- React: 세션 목록 또는 이벤트 로그 화면
- 배포: Docker Compose로 Go API와 PostgreSQL을 함께 실행하는 흐름
