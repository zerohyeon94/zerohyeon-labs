## 1. 오늘의 주제: FastAPI와 REST API의 이해

백엔드 서버는 클라이언트(iOS 앱)와 데이터베이스, 또는 AI 모델 사이에서 요청(Request)을 받고 응답(Response)을 돌려주는 중앙 제어소 역할을 합니다. 그중에서도 **FastAPI**는 파이썬(Python) 기반의 현대적인 웹 프레임워크로, 이름처럼 처리 속도가 매우 빠르고 타입 힌트(Type Hint)를 적극적으로 지원하여 최근 AI 모델 서빙용으로 가장 널리 쓰이고 있습니다.

오늘은 클라이언트가 서버에 데이터를 요청하는 표준 방식인 **REST API**의 개념과, FastAPI가 수많은 클라이언트의 요청을 어떻게 빠르고 효율적으로 처리하는지 그 구조를 학습하겠습니다.

## 2. 전체 아키텍처에서의 역할 및 협업 포인트

전체 시스템에서 FastAPI 서버는 클라이언트의 요청을 검증하고, 필요한 비즈니스 로직(예: 헬스케어 데이터 가공)이나 AI 추론을 실행한 뒤 그 결과를 다시 클라이언트에게 전달하는 다리 역할을 합니다.

- **프론트엔드/클라이언트(iOS)와의 협업 포인트:** 서버로 보낼 데이터와 받을 데이터의 형태(JSON 구조)를 사전에 명확히 약속해야 합니다. Swift의 `Codable`을 이용해 안전하게 디코딩할 수 있도록, 백엔드와 API 명세서를 논의하고 데이터 타입을 맞추는 과정이 필수적입니다.
    
- **AI 직군과의 협업 포인트:** AI 모델러가 만든 모델(PyTorch 등)을 서버 메모리에 로드하고 실행해야 합니다. 모델의 추론 함수가 어떤 형태의 텐서(Tensor) 입력값을 필요로 하고 실행에 얼마나 시간이 걸리는지 파악하여, 서버 병목 현상이 일어나지 않도록 API 엔드포인트를 설계하는 역할을 백엔드가 담당합니다.

## 3. 핵심 CS 지식: 비동기 I/O와 라우팅(Routing)

FastAPI가 압도적인 성능을 내는 이유는 내부적으로 비동기 I/O (Asynchronous I/O)와 이벤트 루프(Event Loop)라는 CS 원리를 적극 활용하기 때문입니다.

- **비동기 I/O와 이벤트 루프:**
    
    기존의 동기식 서버는 DB 조회나 AI 모델 추론처럼 오래 걸리는 작업이 시작되면, 그 작업이 끝날 때까지 해당 스레드(Thread)가 멈춰서 다른 사용자의 요청을 받지 못합니다(Blocking). 하지만 FastAPI의 이벤트 루프는 **큐(Queue)** 자료구조를 활용해 대기 시간이 발생하는 동안 제어권을 넘겨 다른 사용자의 요청을 먼저 처리합니다(Non-blocking).
    
- **복잡도 분석 (라우팅 룩업):**
    
    클라이언트가 요청한 URL(예: `/predict/risk`)을 찾아 알맞은 함수를 실행하는 과정을 '라우팅'이라고 합니다. 프레임워크 내부적으로는 이 경로들을 해시 테이블(Hash Table)이나 **트라이(Trie)** 자료구조로 관리합니다. 해시 테이블을 기준으로 URL을 검색해 매핑하는 작업은 시간 복잡도 $O(1)$에 가깝게 처리되어 검색 속도를 극대화합니다. 반면, 동시 요청을 큐에 적재하는 공간 복잡도는 접속자 수 $N$에 비례하여 $O(N)$의 메모리를 차지합니다.
    

## 4. 실무 최적화 관점 (트러블슈팅)

만약 딥러닝 모델의 이미지 분류나 위험도 예측처럼 CPU 연산량이 매우 큰 무거운 작업을 `async def` 함수 안에서 단순히 실행해버리면 어떻게 될까요? CPU 자체를 계속 점유하게 되어 이벤트 루프 전체가 블로킹되고, 결국 서버가 멈추는 대형 장애가 발생합니다.

실무에서는 이러한 CPU 바운드(CPU-bound) 작업은 `ThreadPoolExecutor`와 같은 스레드 풀(Thread Pool)로 분리(Offloading)하여 메인 이벤트 루프가 멈추지 않도록 설계하는 것이 가장 중요한 최적화 포인트입니다.

## 5. 코드 예시: FastAPI 비동기 라우팅

아래는 iOS 클라이언트와 통신하며 AI 예측 결과를 반환하는 가장 기초적인 FastAPI 서버 코드입니다.

```Python
from fastapi import FastAPI
import asyncio

# 1. FastAPI 애플리케이션 인스턴스 생성
app = FastAPI()

# 2. 일반적인 동기식 엔드포인트 (가벼운 작업)
@app.get("/health")
def check_health():
    return {"status": "Server is running"}

# 3. 비동기식(Async) 엔드포인트 - AI 추론 등 대기가 필요한 작업 모사
@app.post("/predict/risk")
async def predict_health_risk():
    # 실제로는 이곳에 PyTorch 모델 등의 추론 코드가 들어갑니다.
    # asyncio.sleep(2)는 모델 연산에 걸리는 대기 시간을 모사합니다.
    await asyncio.sleep(2) 
    
    return {
        "message": "Prediction complete", 
        "risk_score": 85
    }
```

생성된 다이어그램은 iOS 클라이언트에서 요청이 들어왔을 때 FastAPI 서버가 비동기적으로 어떻게 처리하는지 보여줍니다.

1. **CLIENT_IOS_APP**: 클라이언트 앱입니다.
    
2. **FASTAPI_SERVER (점선 영역)**: 서버 내부 환경입니다.
    
3. **EVENT_LOOP**: 서버의 심장부로, 모든 요청을 비동기로 수신하고 제어합니다.
    
4. **TASK_QUEUE**: 비동기 I/O 작업이나 스레드 풀로 넘겨진 CPU 바운드 작업의 대기 상태를 관리하는 **큐(Queue)** 자료구조입니다.
    
5. **DATABASE_MYSQL**: 데이터베이스와 같은 외부 시스템과의 통신은 비동기 I/O로 처리되어 이벤트 루프를 멈추지 않습니다.
    
6. **THREAD_POOL_CPU_BOUND**: **핵심 트러블슈팅 포인트**입니다. 무거운 **AI 모델 추론** 작업은 메인 이벤트 루프를 멈추지 않게 **별도의 스레드 풀**로 위임(Offload)하여 처리합니다.
