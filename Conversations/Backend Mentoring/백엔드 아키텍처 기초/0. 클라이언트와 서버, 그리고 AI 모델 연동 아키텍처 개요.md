## 클라이언트, 백엔드, 그리고 AI의 역할 분담

단일 기기(On-Device)에서 모든 것을 처리하던 환경을 벗어나 서버와 통신하는 시스템을 구축할 때, 가장 먼저 이해해야 할 것은 관심사의 분리(Separation of Concerns)입니다.

헬스케어 서비스처럼 사용자의 생체 데이터를 수집하고 AI로 분석하는 애플리케이션을 상상해 봅시다. iOS 앱은 사용자와 직접 맞닿아 데이터를 수집하고 결과를 시각화하는 데 집중합니다. 반면 FastAPI로 구축된 백엔드 서버는 클라이언트로부터 받은 대량의 요청을 안정적으로 처리하고, 무거운 연산이 필요한 AI 모델(PyTorch 등)로 데이터를 넘겨 추론 결과를 받아오는 '중앙 교통 통제소' 역할을 수행합니다.

프론트엔드와 백엔드가 협업할 때는 '어떤 포맷(JSON)으로, 어떤 경로(Endpoint)를 통해 데이터를 주고받을 것인가(API 명세)'를 명확히 정의하는 것이 핵심입니다.

## 핵심 CS 원리: HTTP 통신과 프로세스/스레드

클라이언트와 서버 간의 대화는 주로 HTTP(HyperText Transfer Protocol)라는 약속된 규칙을 통해 이루어집니다.

- **비연결성(Stateless):** HTTP 통신의 가장 큰 특징은 서버가 클라이언트의 이전 상태를 기억하지 않는다는 점입니다. iOS에서 서버로 데이터를 보낼 때마다, 이 요청이 누구로부터 왔는지(인증 토큰 등)를 매번 함께 보내야 합니다.
    
- **네트워크 계층:** 이 통신은 근본적으로 TCP/IP 계층 위에서 동작하며, 데이터의 무손실과 순서를 보장하는 3-Way Handshake 과정을 거쳐 연결을 수립한 뒤 데이터를 전송합니다.

서버는 동시에 수많은 사용자의 요청을 처리해야 합니다. 운영체제(OS) 관점에서 서버 애플리케이션은 하나의 프로세스(Process)로 실행되며, 내부적으로 여러 개의 스레드(Thread)를 생성하여 요청을 병렬로 처리합니다. FastAPI는 여기서 한 발 더 나아가 비동기 이벤트 루프(Asynchronous Event Loop)를 활용하여, 적은 수의 스레드로도 수많은 I/O 작업(데이터베이스 조회, AI 모델 추론 대기 등)을 효율적으로 처리할 수 있는 강력한 장점을 가집니다.

## 복잡도 및 최적화: 동기화와 병목 현상

AI 모델 추론은 필연적으로 높은 컴퓨팅 자원과 긴 시간을 요구합니다. 만약 백엔드 서버가 순차적으로(동기적으로) 사용자 요청을 처리한다면, 한 사용자의 AI 분석이 끝날 때까지 다른 모든 사용자는 대기해야 합니다.

사용자가 $N$명일 때, 모든 요청을 동기적으로 처리하는 아키텍처의 대기 시간은 $O(N)$으로 선형 증가하게 됩니다. 이는 서비스 장애로 직결됩니다.

**실무 최적화 포인트:**

이를 해결하기 위해 실무에서는 FastAPI의 `async/await` 구조를 활용하여 비동기(Non-blocking) 처리를 구현합니다. I/O 대기 시간 동안 서버 자원이 멈춰있지 않고 다른 클라이언트의 요청을 받아 처리하도록 함으로써, 실질적인 체감 성능을 $O(1)$에 가깝게 유지하고 서버의 처리량(Throughput)을 극대화합니다. 더 심화된 구조에서는 Redis나 메시지 큐(Message Queue)를 도입하여 AI 작업을 백그라운드 워커(Worker)로 완전히 분리하기도 합니다.
