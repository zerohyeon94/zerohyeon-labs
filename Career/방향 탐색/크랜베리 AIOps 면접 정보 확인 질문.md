## 1. 공개 정보 기준으로 보이는 회사 방향

면접 전 확인한 공개 정보 기준으로 크랜베리는 단순 소프트웨어 회사라기보다, AI와 IoT를 결합한 관제/안전 솔루션 회사에 가깝다.

공식 홈페이지에서는 음원 분석 기반 솔루션, 시설 관리, 스마트 케어, B2B/공공 솔루션 시장을 강조한다. 특히 음향 특성 분석, 실시간 이벤트 분석, 예방/예측, 데이터 중심, 프라이빗 공간 안전 같은 키워드가 보인다.

잡코리아 기업정보 기준으로는 2022년 설립, 중소기업, 임직원 약 20명 규모이며, AI와 IoT 기술을 활용한 사물지능융합관제 토탈 시스템 개발/제조를 주요 사업으로 설명한다.

보도 기사에서는 음원 분석 기반 객체 인식 지능형 융합 관제 시스템이 조달청 우수제품으로 선정되었다고 소개되며, 음원과 영상을 결합한 멀티모달 AI 안전 솔루션 방향이 보인다.

---

## 2. AIOps라는 말의 의미부터 확인해야 한다

AIOps는 일반적으로 AI를 활용해 IT 운영을 자동화하고 최적화하는 영역이다.

대표 키워드:

```text
- 로그/메트릭/이벤트 수집
- 이상 징후 탐지
- 근본 원인 분석
- 장애 예측
- 알림 노이즈 감소
- 자동 대응
- 클라우드/인프라/서비스 모니터링
```

다만 회사마다 AIOps라는 말을 다르게 쓸 수 있다.

크랜베리의 경우 공개 정보상 `AI 음원 분석`, `IoT 관제`, `공공/안전 솔루션`, `예방/예측`에 가까워 보이므로, 면접에서는 아래를 반드시 분리해서 확인해야 한다.

```text
1. IT 인프라 AIOps인가?
2. IoT/관제 시스템의 이상 탐지인가?
3. AI 모델 개발인가?
4. 솔루션 운영/기술지원인가?
5. 데이터 파이프라인/백엔드 개발인가?
```

## 3. 면접에서 반드시 확인할 질문

### 1. 실제 업무 범위

```text
이 포지션에서 하루 업무는 개발, 운영, 기술지원, 데이터 분석 중 어디에 가장 가깝나요?
```

확인 포인트:

- 코드 작성 비중
- 고객사 대응 비중
- 관제/운영 모니터링 비중
- 연구개발과 유지보수 비중

### 2. AIOps의 내부 정의

```text
회사에서 말하는 AIOps는 IT 인프라 운영 자동화인가요, 아니면 IoT/관제 데이터 기반 이상 탐지인가요?
```

이 질문은 꼭 해야 한다. 이름은 AIOps여도 실제로는 관제 솔루션 운영, 데이터 라벨링, 고객사 기술지원일 수 있다.

### 3. 다루는 데이터

```text
주로 다루는 데이터는 로그/메트릭인가요, 음원/영상/센서 데이터인가요?
```

크랜베리 공개 정보상 음원/영상/센서 데이터 가능성이 있어 보인다.

내 커리어와 연결되는 데이터:

```text
좋음:
- 음원/영상/센서 데이터
- 이상 탐지
- 시계열 데이터
- 실시간 이벤트 처리
- 엣지/IoT 데이터

주의:
- 단순 운영 로그 확인
- 고객사 장애 접수 처리
- 반복 리포트 작성
```

### 4. 개발 스택

```text
개발 스택은 Python, 백엔드, 클라우드, 임베디드/IoT, 프론트엔드 중 어디에 가깝나요?
```

추가 질문:

```text
- Python을 어느 정도 사용하나요?
- ML 모델을 직접 학습/평가하나요?
- API 서버나 데이터 파이프라인을 개발하나요?
- 클라우드는 AWS/GCP/Azure 중 무엇을 쓰나요?
- 온프레미스/공공기관 구축도 있나요?
```

### 5. AI 모델 관련성

```text
AI 모델은 직접 개발하나요, 기존 모델을 운영/적용하나요?
```

좋은 신호:

```text
- 이상 탐지 모델 성능 개선
- 음원/영상 모델 평가
- 데이터 전처리 파이프라인 구축
- 모델 배포/모니터링
- 실시간 추론 시스템 최적화
```

주의 신호:

```text
- 모델은 외부/기존 것을 쓰고 나는 설정만 함
- 장애 알림을 확인하고 고객사에 전달하는 업무 중심
- 데이터 라벨링/엑셀 정리 중심
```

### 6. 제품과 고객

```text
제가 맡게 될 기능이 실제 제품에 들어가나요, 내부 운영/고객 지원에 더 가까운가요?
```

확인 포인트:

- 제품 로드맵
- 공공 조달/고객사 구축 여부
- B2B 솔루션 납품 방식
- 유지보수 계약 구조

### 7. 커리어 전환 가능성

```text
이 역할을 6개월~1년 수행하면 어떤 역량이 쌓였다고 말할 수 있을까요?
```

원하는 답변:

```text
- AI 기반 이상 탐지 시스템 경험
- IoT/센서 데이터 처리 경험
- 실시간 관제 솔루션 개발 경험
- 클라우드/백엔드/데이터 파이프라인 경험
- 멀티모달 AI 제품화 경험
```

애매한 답변:

```text
- 고객 응대 경험
- 다양한 이슈 처리 경험
- 운영 안정화 경험
```

운영 경험도 의미는 있지만, 현재 목표인 `AI 헬스케어 + 모바일 개발자`와 연결하려면 기술 산출물이 분명해야 한다.

## 4. 내 배경과 연결해서 물어볼 질문

의료기기 펌웨어, 헬스케어 앱, AI 헬스케어 부트캠프, Python 조교 경험을 연결해 아래처럼 질문할 수 있다.

```text
제가 의료기기 펌웨어와 헬스케어 앱 개발 경험이 있고,
최근에는 AI 헬스케어 교육 과정에서 Python/AI 실습을 다루고 있습니다.
이 배경이 해당 AIOps 포지션에서 강점으로 작동할 수 있는 지점이 있을까요?
```

이 질문의 목적은 회사가 나를 어떻게 쓰려는지 보는 것이다.

좋은 답변:

```text
- IoT/센서 데이터 이해에 도움이 된다.
- 실시간 시스템과 현장 장비 이해가 중요하다.
- AI 모델 결과를 제품/사용자 경험으로 연결할 수 있다.
- 헬스케어/안전 도메인 이해가 솔루션 설계에 도움이 된다.
```

위 답변이 나오면 면접을 진지하게 볼 만하다.

## 5. 합류를 긍정적으로 볼 조건

아래 중 3개 이상이면 크랜베리 AIOps는 검토 가치가 있다.

```text
1. 실제 개발 비중이 높다.
2. Python/ML/데이터 파이프라인을 다룬다.
3. 음원/영상/센서 기반 이상 탐지 또는 예측 모델을 다룬다.
4. 제품 기능으로 남는 산출물을 만든다.
5. 클라우드/백엔드/실시간 시스템 경험이 쌓인다.
6. 내 의료기기/헬스케어/모바일 경험을 강점으로 본다.
```

이 경우 커리어 설명은 아래처럼 가능하다.

```text
의료기기/헬스케어 앱 경험을 바탕으로,
AI 기반 관제/이상 탐지 시스템을 경험했고,
이후 모바일/헬스케어 AI 서비스로 확장했다.
```

---

## 6. 합류를 조심해야 할 조건

아래 중 3개 이상이면 신중해야 한다.

```text
1. 개발보다 고객사 운영/기술지원이 중심이다.
2. AI 모델은 직접 다루지 않는다.
3. 데이터는 보지만 분석/모델링/파이프라인 권한이 없다.
4. 야간/장애 대응 등 운영 부담이 크다.
5. 내 커리어 목표와 연결되는 산출물이 불분명하다.
6. 6개월 뒤 무엇을 배웠다고 말할지 답이 흐리다.
```

이 경우 AIOps라는 이름은 좋아 보여도 실제 커리어 전환에는 도움이 적을 수 있다.

## 7. 면접 전 준비할 내 답변

### 자기소개 방향

```text
저는 의료기기 펌웨어와 헬스케어 앱 개발 경험이 있고,
AI 헬스케어 교육 과정을 통해 Python/AI 실습을 다루고 있습니다.
현장 장비와 사용자 앱, 그리고 AI 학습 흐름을 함께 이해하는 점이 강점입니다.
이번 포지션에서는 AI 기반 이상 탐지나 관제 시스템이 실제 제품으로 동작하는 과정을 경험하고 싶습니다.
```

### 지원 동기 방향

```text
단순히 AI 모델을 학습하는 것보다,
실제 현장 데이터에서 이상을 탐지하고 사용자/운영자가 바로 활용할 수 있는 제품으로 연결하는 흐름에 관심이 있습니다.
특히 음원, 영상, 센서 데이터가 결합된 관제 시스템은 의료기기/헬스케어 도메인에서 쌓은 현장성 있는 경험과도 연결될 수 있다고 생각했습니다.
```

## 8. 결론

크랜베리 AIOps 면접은 `합격해야 하는 자리`가 아니라 `정보를 확인해야 하는 자리`다.

핵심 질문은 하나다.

```text
이 포지션이 나를 1년 뒤
"AI 기반 실시간 데이터/이상 탐지 시스템을 경험한 개발자"
로 만들어주는가?
```

그렇다면 검토할 가치가 있다.

반대로 실제 업무가 운영, 고객사 대응, 단순 모니터링에 가깝다면 지금의 넥스트러너스/OZ 브릿지 경험과 iOS 네이티브 포트폴리오를 유지하는 편이 더 낫다.
