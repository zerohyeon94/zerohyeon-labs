# 크랜베리 React Go 플랫폼 개발 직무 판단

> 작성일: 2026-05-27
> 관련 문서: [[Career/방향 탐색/크랜베리 회사 정보]], [[Career/방향 탐색/크랜베리 AIOps 면접 정보 확인 질문]], [[Career/방향 탐색/크랜베리와 넥스트러너스 도메인 위치와 경쟁력 비교]]
> 목적: 크랜베리에서 들은 실제 업무가 React와 Go 기반 플랫폼 개발일 때, 해당 직무의 성격과 커리어 적합성을 판단한다.

---

## 1. 현재 들은 업무 내용

```text
- React로 웹사이트/플랫폼 화면 개발
- Go 언어로 백엔드 서버 또는 API 개발
- 백엔드 서버로부터 데이터를 받아와 화면에 표시
- 관제/플랫폼 형태의 서비스 개발 가능성
```

이 설명만 보면 `AI 모델 개발자`나 전통적인 `AIOps 엔지니어`라기보다는 `React + Go 기반 플랫폼 개발자` 또는 `AIoT 관제 플랫폼 풀스택 개발자`에 가깝다.

---

## 2. 직무 성격 해석

이 업무는 크게 세 가지 층으로 나눠 볼 수 있다.

```text
1. 프론트엔드
   - React 기반 관제 화면
   - 이벤트/알림/지도/카메라/센서 상태 표시
   - 관리자 페이지, 대시보드, 통계 화면

2. 백엔드
   - Go 기반 API 서버
   - 장비/센서/이벤트 데이터 조회
   - 사용자/권한/기관/설치 장소 관리
   - 이벤트 이력, 알림, 로그, 통계 처리

3. 도메인
   - 음원 분석 결과
   - CCTV/카메라 연동 결과
   - 공공기관/지자체 관제 업무 흐름
   - AI 탐지 이벤트를 사람이 확인하고 조치하는 UX
```

핵심은 `웹 화면을 만드는 일` 자체보다, 어떤 데이터를 다루느냐다.

```text
단순 관리자 페이지라면: 일반 웹/백엔드 개발 경험
실시간 이벤트/관제/센서 데이터를 다룬다면: AIoT 플랫폼 개발 경험
AI 탐지 결과와 오탐/미탐 피드백을 다룬다면: AI 제품화 경험
```

---

## 3. 좋은 신호

```text
1. React + Go 조합은 실무 백엔드/플랫폼 개발 경험으로 가치가 있다.
2. Go는 서버, API, 동시성, 네트워크, 실시간 이벤트 처리에 강점이 있어 관제 플랫폼과 잘 맞는다.
3. 음원/영상/센서 데이터가 실제 제품 화면으로 연결되면 AIoT 제품 개발 경험이 된다.
4. 공공기관/지자체가 쓰는 관제 플랫폼이라면 요구사항, 운영 안정성, 권한 관리, 로그 관리 경험을 쌓을 수 있다.
5. 단순 앱 개발보다 시스템 전체 흐름을 이해하는 데 도움이 된다.
```

---

## 4. 주의할 신호

```text
1. AI 모델 개발이나 데이터 분석을 직접 하는 포지션은 아닐 가능성이 높다.
2. iOS 네이티브 개발과는 거리가 생길 수 있다.
3. React 화면 개발만 반복하면 커리어가 웹 프론트엔드 쪽으로 이동할 수 있다.
4. Go 백엔드가 깊지 않고 단순 API CRUD 수준이면 기술적 성장 폭이 제한될 수 있다.
5. 고객사 요청에 맞춘 SI성 커스터마이징이 많으면 제품 개발 경험보다 납품 대응 경험이 커질 수 있다.
6. 데이터가 이미 가공된 결과만 내려오고, AI/센서/실시간 처리에는 접근하지 못하면 AIoT 경험으로 말하기 약해진다.
```

---

## 5. 커리어 관점 판단

### iOS 네이티브 개발자 관점

```text
직접적인 iOS 경력으로는 약하다.
React와 Go는 모바일 네이티브 스택이 아니므로, iOS 개발자로 바로 연결되는 경험은 아니다.
다만 API, 데이터 흐름, 관제 플랫폼, 서버 구조를 이해한 모바일 개발자로 확장할 수는 있다.
```

### AI 경험이 있는 모바일 개발자 관점

```text
조건부로 도움이 된다.
AI 모델을 직접 학습하거나 Core ML을 다루는 경험은 아니지만,
AI 탐지 결과를 실제 제품/플랫폼으로 연결하는 경험은 쌓을 수 있다.
나중에 모바일 앱에서 AI 이벤트, 센서 데이터, 관제 알림, 실시간 모니터링을 다룰 때 설명 가능한 경험이 된다.
```

### 백엔드/플랫폼 개발자 관점

```text
꽤 도움이 된다.
Go 기반 서버, API, 이벤트 처리, 관제 데이터, 공공/기관용 플랫폼 경험은 백엔드 커리어로 전환할 때 의미가 있다.
특히 실시간 데이터, WebSocket, 메시지 큐, 장비 상태 관리, 알림 시스템까지 다룬다면 강점이 된다.
```

---

## 6. 이 직무가 좋은 선택이 되는 조건

아래 중 3개 이상 해당하면 긍정적으로 볼 수 있다.

```text
1. Go 백엔드 코드를 실제로 작성한다.
2. React 화면만이 아니라 API 설계나 데이터 모델링에도 참여한다.
3. 실시간 이벤트, 알림, 장비 상태, 관제 이력 같은 데이터를 다룬다.
4. 음원 분석 결과나 AI 탐지 이벤트를 제품 화면에 연결한다.
5. 기존 CCTV/VMS/장비와 연동하는 구조를 배운다.
6. 단순 SI 커스터마이징이 아니라 자체 플랫폼 제품을 만든다.
7. 모바일 앱 또는 외부 클라이언트 API로 확장할 가능성이 있다.
```

---

## 7. 조심해야 하는 조건

아래가 많으면 신중해야 한다.

```text
1. React 화면 유지보수만 한다.
2. Go 백엔드는 이미 만들어져 있고 단순 조회 API만 붙인다.
3. AI/음원/센서 데이터에는 접근하지 못한다.
4. 고객사별 화면 수정과 납품 대응이 대부분이다.
5. 개발보다 설치, 운영, 기술지원, 장애 대응 비중이 크다.
6. 제품 로드맵 없이 수주 건마다 임시 개발을 반복한다.
```

## 8. 면접에서 꼭 확인할 질문

```text
1. React와 Go 중 실제 업무 비중은 어느 쪽이 더 큰가요?
2. Go 백엔드에서는 API 개발 외에 실시간 이벤트 처리, 알림, 메시지 큐, DB 설계도 하게 되나요?
3. 프론트엔드에서는 단순 관리자 페이지인지, 실시간 관제 대시보드인지 궁금합니다.
4. 음원 분석 결과나 AI 탐지 이벤트 데이터 구조를 직접 다루게 되나요?
5. 백엔드 서버가 장비/센서에서 직접 데이터를 받나요, 아니면 별도 AI 서버가 가공한 결과만 받나요?
6. WebSocket, SSE, MQTT, Kafka, Redis Pub/Sub 같은 실시간 처리 기술을 사용하나요?
7. 기존 CCTV/VMS/PTZ 카메라와 연동하는 API나 프로토콜을 다루나요?
8. 이 플랫폼은 회사의 자체 제품인가요, 아니면 고객사별 SI 프로젝트인가요?
9. 모바일 앱이나 외부 클라이언트 API로 확장할 계획이 있나요?
10. 입사 후 3개월 동안 맡게 될 첫 번째 기능은 무엇인가요?
```

## 9. 최종 판단

```text
React + Go 플랫폼 개발이라면, 크랜베리 포지션은 AIOps 연구직보다 AIoT 관제 플랫폼 개발직에 가깝다.
이 직무는 iOS 네이티브 경력으로 바로 쌓이진 않지만, AI/센서/관제 데이터를 제품으로 연결하는 경험이 될 수 있다.
Go 백엔드와 실시간 관제 데이터까지 다룬다면 커리어 가치가 꽤 있다.
반대로 React 화면 수정과 고객사별 납품 대응만 많다면, 장기적으로 원하는 모바일/AI 방향과 멀어질 수 있다.
```

따라서 판단 기준은 아래 한 문장으로 정리할 수 있다.

```text
이 일이 “웹 화면 개발”인지, 아니면 “AIoT 데이터를 다루는 제품 플랫폼 개발”인지 확인해야 한다.
```
