## 1. 현실 기준

현재 가장 중요한 문제는 기술 스택 취향이 아니라 `다음 1년을 어떤 직무명과 어떤 산출물로 남길 것인가`다.

현재 경력 현실:

```text
1. 녹십자MS 3개월은 개발보다 다른 업무 비중이 컸다.
2. 이후 1년 공백이 있었다.
3. 실제 iOS 개발경력은 약 1년 6개월이다.
4. 이후 약 1년 동안 Python/AI 학습을 했다.
5. 현재 넥스트러너스는 6개월 계약직 조교/운영 업무다.
6. 크랜베리는 정규직 플랫폼 개발 가능성이 있다.
```

이 상황에서는 `가장 이상적인 스택`보다 `정규직 개발자로 다시 경력을 붙이는 것`의 가치가 커진다.

## 2. 수정된 핵심 결론

```text
React + Go 자체를 커리어 목표로 삼는 것은 추천하지 않는다.
하지만 크랜베리가 정규직 소프트웨어 개발자 포지션이고,
AIoT/AIOps 플랫폼을 실제로 개발하는 역할이라면,
넥스트러너스 계약직 조교보다 현실적으로 더 안정적인 경력 복구 선택일 수 있다.
```

즉, 판단 기준은 아래다.

```text
나쁜 선택:
React/Go 웹 풀스택이 되려고 크랜베리에 간다.

좋은 선택:
정규직 플랫폼 개발자로 경력을 복구하고,
AIoT/AIOps 제품 개발 경험을 쌓으며,
iOS + 헬스케어 AI 방향을 병행 유지한다.
```

## 3. 바이브코딩 기반 플랫폼 개발의 의미

바이브코딩은 장점과 위험이 모두 있다.

좋은 경우:

```text
1. AI 도구를 사용해 개발 속도를 높인다.
2. 사람이 요구사항, 구조, 데이터 흐름, 테스트, 리뷰를 책임진다.
3. React/Go 코드를 이해하고 수정/검증한다.
4. 실제 제품 기능을 배포하고 운영한다.
```

위험한 경우:

```text
1. AI가 만든 코드를 붙여넣기만 한다.
2. 테스트, 코드리뷰, 설계, 보안 검토가 없다.
3. 개발자가 시스템을 이해하지 못한 채 기능만 찍어낸다.
4. 이력서에 남는 것이 "AI 도구로 화면 만든 사람"이 된다.
```

따라서 크랜베리에서 확인해야 할 질문은 아래다.

```text
바이브코딩을 쓰더라도 코드리뷰, 테스트, 배포, 운영 책임은 개발자가 지나요?
제가 맡는 기능은 실제 제품에 배포되나요?
AI가 만든 코드를 이해하고 개선하는 엔지니어링 프로세스가 있나요?
```

## 4. 크랜베리를 선택해야 하는 조건

아래 조건이 맞으면 크랜베리는 현실적으로 선택 가치가 있다.

```text
1. 정규직 소프트웨어 개발자 포지션이다.
2. 개발 비중이 70% 이상이다.
3. React만이 아니라 Go 백엔드/API/DB/실시간 이벤트를 실제로 다룬다.
4. AIOps/AIoT 플랫폼의 데이터 구조, 알림, 장비 상태, 로그, 관제 이벤트를 다룬다.
5. 설치/기술지원/고객사 대응보다 제품 개발 비중이 높다.
6. 바이브코딩은 보조 도구이고, 개발자가 구조와 품질을 책임진다.
7. 긴 출퇴근을 보완할 방법이 있다.
```

이 조건이 맞으면, 크랜베리는 아래 경력 문장을 만들 수 있다.

```text
AIoT/AIOps 관제 플랫폼에서 React와 Go를 활용해 센서/음원/이벤트 데이터를 제품 기능으로 구현한 개발자
```

이 문장은 현재 계약직 조교보다 개발자 시장에서 더 강하다.

## 5. 넥스트러너스를 선택해야 하는 조건

넥스트러너스는 아래 조건이 명확할 때만 장기 선택지다.

```text
1. 정규직 전환 가능성과 기준이 구체적이다.
2. 전환 후 운영만이 아니라 콘텐츠/커리큘럼 제작 권한이 있다.
3. 과제 풀이, FAQ, 학습자료, 커리큘럼 개선안이 회사 자산으로 남는다.
4. 1년 안에 보충 세션이나 일부 강의 기회가 있다.
5. 교육/코칭을 장기 직업 정체성으로 받아들일 수 있다.
```

이 조건이 없다면 넥스트러너스는 안정적으로 보이지만 실제로는 `계약직 운영 업무`에 머물 가능성이 있다.

## 6. 출퇴근 문제

크랜베리의 출퇴근은 리스크가 맞다. 하지만 정규직 개발자 경력 복구가 확실하다면, 출퇴근은 개선 전략으로 관리해야 한다.

현재 확인된 조건:

```text
재택근무: 없음
야근: 거의 없음
시차출퇴근: 있음
```

이 조건이면 출퇴근 리스크는 조금 완화된다.

```text
재택이 없는 것은 아쉽다.
하지만 야근이 거의 없고 시차출퇴근이 가능하다면,
하루 종료 시간이 예측 가능하므로 출퇴근 루틴을 설계할 수 있다.
```

가능한 개선책:

```text
1. 시차출근 가능 여부 확인
2. 재택은 기대하지 않고, 시차출퇴근으로 피로도를 줄이는 전략을 세운다.
3. 출퇴근 중 지하철 구간을 iOS 학습/영어/문서 읽기 시간으로 고정
4. 야근 빈도 확인
5. 3개월 수습 후 근처 거점/스터디카페/운동 동선 재설계
6. 6개월 후 실제 피로도가 너무 크면 이직 전략 재평가
```

출퇴근이 길더라도 `정규직 개발 경력`이 확실하면 감수할 가치가 생긴다.
반대로 업무가 애매하면 긴 출퇴근은 손실만 커진다.

### 추천 출퇴근 전략

시차출퇴근이 가능하다면 무조건 러시아워를 피하는 방향이 좋다.

추천안:

```text
1안: 늦은 출근형
출근: 10:30 ~ 11:00
퇴근: 19:30 ~ 20:00
장점: 아침 러시아워 회피, 출근 전 40분 iOS 학습 가능
단점: 저녁 시간이 늦어짐

2안: 이른 출근형
출근: 08:00 ~ 08:30
퇴근: 17:00 ~ 17:30
장점: 저녁 시간을 확보하기 좋음
단점: 아침 기상 부담, 출근 피크와 겹칠 수 있음

3안: 중간형
출근: 09:30 ~ 10:00
퇴근: 18:30 ~ 19:00
장점: 생활 리듬이 무난함
단점: 출근/퇴근 모두 일부 혼잡과 겹칠 수 있음
```

현재 우선 추천은 `늦은 출근형`이다.

이유:

```text
1. 수지에서 가산디지털단지역 방향은 출근 피크 부담이 크다.
2. 오전에 40분만 확보해도 iOS/영어/기술문서 루틴을 유지할 수 있다.
3. 야근이 거의 없다면 퇴근 시간이 늦어져도 예측 가능하다.
4. 지하철에서 서서 가는 시간이 줄면 피로도가 크게 낮아진다.
```

다만 실제로는 2주간 테스트해야 한다.

```text
1주차: 늦은 출근형 테스트
2주차: 이른 출근형 테스트
비교 기준: 수면, 피로도, 운동 가능성, iOS 학습 가능성, 지각 리스크
```

출퇴근 시간을 학습 시간으로 쓸 때는 코딩이 아니라 읽기/정리만 한다.

```text
출근 지하철:
- Apple 문서/WWDC 요약 읽기
- Swift/iOS 개념 복습
- 영어회화 표현 3개 저장

퇴근 지하철:
- 오늘 배운 업무 기술 3줄 정리
- 내 iOS 포트폴리오에 연결할 아이디어 1개 기록
- 피곤하면 오디오 학습만 수행
```

```text
넥스트러너스는 정규직 전환과 콘텐츠/커리큘럼 역할이 확실하지 않다면
장기 선택지로는 불안정하다.
```
