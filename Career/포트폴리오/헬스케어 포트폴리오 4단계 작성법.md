# 헬스케어 포트폴리오 4단계 작성법

> **원칙**: 포트폴리오 1개를 4단으로 쓰되, 각 단마다 '어떤 시야가 들어 있는가'를 명시한다.
> 학생 90%가 1~3단에서 멈춘다. 4단을 쓸 수 있으면 그 자체로 채용 신호다.

---

## 4단계 구조

### 01 임상적 질문 정의 `[외부고객 시야]`

> "이 프로젝트가 풀려는 문제는 무엇인가?"

- 의사/환자/병원이 실제로 겪는 불편을 한 문장으로 정의
- 숫자나 기술 이전에: **누가, 왜 이 문제로 손해를 보는가**
- 나쁜 예: "건강 데이터를 분석하는 앱을 만들었다"
- 좋은 예: "퇴원 후 환자의 복약 이탈률이 40%에 달해 재입원 비용이 발생하는 문제를 타겟했다"

### 02 데이터 출처와 한계 `[기술의 원리 시야]`

> "왜 이 데이터가 이 문제에 맞는가?"

- 단순히 "HealthKit 데이터 사용"이 아니라 **왜 이 데이터여야 하는가**
- 수집 방식, 편향 가능성, 일반화 한계를 명시
- 나쁜 예: "HealthKit으로 심박수 데이터를 수집했다"
- 좋은 예: "HealthKit 심박수는 수동 측정이 아닌 passively collected — 측정 누락률이 높아 결측 처리 전략이 모델 성능에 직접 영향을 미쳤다"

### 03 모델·검증 결과 `[기술 시야]`

> "어떤 조건에서 잘 작동하고, 어떤 조건에서 실패하는가"

- AUROC 하나만 쓰면 탈락
- 포함해야 하는 것: subgroup 성능(성별·연령·기저질환), 임계값 선택 근거, 실패 케이스
- 나쁜 예: "AI 모델 정확도 87%"
- 좋은 예: "전체 AUROC 0.87이나, 65세 이상 subgroup에서 0.71로 하락. 만성질환자 데이터 부족이 원인 — 실제 배포 시 해당 그룹 별도 검증 필요"

### 04 실제 적용 시 한계 `[내부고객 + 외부고객 시야]`

> "이걸 실제로 병원에서 쓰려면 무엇이 문제인가?"

- 의사가 왜 안 쓸까? / 병원이 왜 도입을 꺼릴까?
- 수가 구조, 인허가(SaMD, IEC 62304), 워크플로우 마찰, 책임 소재
- 나쁜 예: (대부분 여기서 멈춤)
- 좋은 예: "SaMD Class II 해당 — FDA 510(k) 또는 식약처 인허가 없이 임상 판단 보조 불가. 현재 버전은 '참고용'으로 범위를 제한하고, 추후 인허가 경로를 명시했다"

---

## 마이헬스버디에 적용

> 발표 이후 포트폴리오 작성 시 이 구조로 작성할 것

### 01 임상적 질문 정의
```
[작성 예시 방향]
"개인 건강 데이터를 가진 사용자가 증상과 데이터의 연관성을 
스스로 해석하지 못하는 문제 — 의료진 방문 없이 의미 있는 
건강 패턴을 발견할 수 있는 도구가 필요했다"
```

### 02 데이터 출처와 한계
```
[작성 예시 방향]
"사용자 입력 기반 데이터 — 의료 기록이 아니기 때문에 
임상적 진단이 아닌 '패턴 탐색'으로 범위를 제한.
편향 가능성: 건강에 관심 있는 사용자 집단 편향"
```

### 03 모델·검증 결과
```
[발표 후 실제 수치로 채울 것]
- 사용한 AI 기능의 성능 지표
- 어떤 케이스에서 잘 작동하고 어떤 케이스에서 실패했는가
- 임계값이나 분류 기준의 근거
```

### 04 실제 적용 시 한계
```
[이것이 90%와 차별화되는 지점]
"현재 구현은 SaMD 인허가 범위 밖 — 의료 진단이 아닌 
생활 습관 제안으로 서비스 포지셔닝. 임상 판단 보조 기능 
추가 시 식약처 디지털치료기기 가이드라인 검토 필요"
```

## 다른 프로젝트에 적용 시 체크리스트

- [ ] 01: 의사/환자/병원 중 **누가** 이 문제로 손해를 보는가 명시했는가
- [ ] 02: 데이터를 선택한 **이유**와 그 **한계**를 함께 썼는가
- [ ] 03: 성능 지표 + **실패 조건**을 함께 썼는가
- [ ] 04: 실제 병원 도입 시 **현실적 장벽**을 언급했는가

> 4번이 있으면 면접관에게 "이 사람은 임상 현장을 안다"는 신호를 준다.
