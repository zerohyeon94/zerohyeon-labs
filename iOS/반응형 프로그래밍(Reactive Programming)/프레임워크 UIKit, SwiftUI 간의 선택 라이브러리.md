### 1. UIKit + RxSwift : "명령형 UI의 한계를 극복하기 위한 커뮤니티의 선택"

- **등장 배경:** UIKit은 2008년부터 존재했던 Apple의 전통적인 '명령형' UI 프레임워크입니다. 오랫동안 Apple은 공식적인 반응형 프레임워크를 제공하지 않았습니다. 비동기 처리를 위해 Delegate 패턴이나 클로저(콜백)를 사용해야 했고, 코드가 복잡해지는 문제가 있었습니다.
    
- **매칭의 이유 (RxCocoa의 존재):** 이 불편함을 해소하기 위해 전 세계 개발자들이 도입한 것이 바로 서드파티 라이브러리인 **RxSwift**입니다. 특히 RxSwift는 UIKit과 결합하기 위해 **`RxCocoa`**라는 강력한 확장 라이브러리를 함께 제공합니다.
    
- **동작 방식:** `RxCocoa`를 사용하면 UIKit의 UI 요소들(예: 버튼, 텍스트 필드)을 반응형 데이터 스트림으로 쉽게 바꿀 수 있습니다. 예를 들어, 버튼 클릭을 `button.rx.tap`이라는 하나의 데이터 흐름으로 만들어 비동기 로직과 매끄럽게 연결할 수 있었기 때문에, UIKit 시절에는 RxSwift가 사실상 표준(De facto standard)으로 자리 잡았습니다.

### 2. SwiftUI + Combine : "처음부터 함께 설계된 Apple의 네이티브 반응형 생태계"

- **등장 배경:** 2019년 WWDC에서 Apple은 **SwiftUI**와 **Combine**을 동시에 발표했습니다. 이 둘은 태생부터가 서로를 위해 설계된 '영혼의 단짝'입니다.
    
- **매칭의 이유 (네이티브 통합):** 앞서 말씀드린 대로 SwiftUI는 `View = f(State)`라는 반응형 UI 공식을 따릅니다. 이 상태(State)에 들어갈 데이터의 흐름을 만들어주는 엔진 역할을 Combine이 완벽하게 수행합니다.
    
- **동작 방식:** SwiftUI에서 뷰의 상태를 관찰할 때 사용하는 `@Published`, `ObservableObject` 같은 핵심 속성 래퍼(Property Wrapper)들이 내부적으로 모두 Combine 프레임워크를 기반으로 동작합니다. 별도의 변환 과정이나 서드파티 라이브러리 없이, 데이터의 흐름(Combine)과 화면의 업데이트(SwiftUI)가 물 흐르듯 네이티브로 연결됩니다.

### 💡 그럼 반드시 이 조합으로만 써야 할까요? (크로스오버)

이 부분이 실무 앱 개발이나 포트폴리오를 구성할 때 가장 중요한 포인트입니다. **절대적인 공식은 아닙니다.**

1. **Combine + UIKit:** iOS 13 이상을 지원하는 프로젝트라면, UIKit으로 UI를 그리면서도 비동기 데이터 처리는 Combine을 사용하는 경우가 점점 많아지고 있습니다. 무거운 RxSwift 의존성을 덜어내고 Apple의 네이티브 기능을 활용하기 위함입니다.
    
2. **RxSwift + SwiftUI:** 반대로 SwiftUI를 사용하면서도 기존에 구축해둔 방대한 RxSwift 로직을 그대로 활용해야 하는 경우도 있습니다. 이럴 때는 RxSwift의 데이터를 Combine의 `Publisher`처럼 변환해서 사용하기도 합니다.

실제 프로젝트를 진행하다 보면 하나의 앱 안에서 UIKit과 SwiftUI를 혼용해서 사용하는 아키텍처를 구성하게 되는 경우가 아주 흔합니다. 이때 데이터 흐름을 어떻게 일관되게 가져갈 것인지 고민하는 과정에서 이 두 프레임워크의 차이를 아는 것이 큰 무기가 됩니다.
