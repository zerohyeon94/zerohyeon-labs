반응형 프로그래밍(Reactive Programming)의 가장 핵심적인 정의는 **"데이터의 흐름(Stream)과 변화의 전파에 초점을 맞춘 프로그래밍 패러다임"**입니다.

이를 가장 쉽게 이해하기 위해 기존의 '명령형 프로그래밍'과 비교해 보겠습니다.

### 1. 명령형(Imperative) vs 반응형(Reactive)

**명령형 프로그래밍 (Pull 방식)** 우리가 흔히 작성하는 방식입니다.

- `A = B + C`라고 코드를 작성했다고 가정해 보겠습니다.
    
- 이 코드가 실행되는 그 순간의 `B`와 `C` 값을 더해서 `A`에 할당합니다.
    
- 이후에 `B`나 `C`의 값이 바뀌더라도, 개발자가 다시 `A = B + C`를 실행하라고 **명령**하지 않는 이상 `A`의 값은 변하지 않습니다.

**반응형 프로그래밍 (Push 방식)** 엑셀(Excel)의 스프레드시트를 떠올리면 완벽합니다.

- 셀 A에 `= B + C`라는 수식을 걸어둡니다.
    
- 이후 B나 C 셀의 값이 바뀌면, 우리는 아무런 명령을 하지 않았음에도 A 셀의 값이 **자동으로 반응하여 업데이트**됩니다.
    
- 즉, 데이터의 변화가 발생하면 그 변화가 연결된 곳으로 알아서 흘러가는(Push) 구조입니다.

---

### 2. 반응형 프로그래밍의 핵심 개념

반응형 프로그래밍을 구성하는 두 가지 중요한 기둥이 있습니다.

- **모든 것은 데이터 스트림(Stream)이다:** 변수의 값, 사용자의 버튼 클릭, 타이머, 네트워크 요청 등 컴퓨터에서 일어나는 모든 이벤트를 '시간의 흐름에 따라 발생하는 데이터의 연속'으로 바라봅니다.
    
- **구독(Subscribe)과 반응(React):** 개발자는 이 데이터 스트림을 지켜보고(Observe/Subscribe) 있다가, 스트림에서 새로운 데이터가 흘러나올 때마다 미리 정의해둔 동작(UI 업데이트, 계산 등)이 자동으로 실행되도록 설계합니다.

### 3. 왜 반응형 프로그래밍을 사용할까요?

최근의 앱이나 웹 개발(SwiftUI나 React 등의 선언형 UI 프레임워크 포함)에서는 상태(State) 관리가 매우 중요합니다.

사용자가 화면을 스와이프하고, 그와 동시에 네트워크에서 데이터를 받아오고, 데이터가 오면 화면의 리스트를 갱신해야 하는 등 수많은 비동기(Asynchronous) 이벤트가 동시다발적으로 일어납니다. 기존 방식으로는 이런 복잡한 콜백(Callback)이나 상태 변화를 추적하기가 너무 어렵고 코드가 지저분해집니다.

반응형 프로그래밍을 도입하면 **"데이터가 변하면 -> UI는 그에 반응해 자동으로 다시 그려진다"**는 흐름을 일관되고 깔끔하게 구축할 수 있습니다.
