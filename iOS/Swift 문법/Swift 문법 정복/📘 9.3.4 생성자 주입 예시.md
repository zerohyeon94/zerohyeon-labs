`AViewController` 내부에서 `AViewModel()`을 직접 생성하지 않고, 뷰 컨트롤러가 생성되는 시점(`init`)에 외부로부터 필요한 뷰모델을 전달(주입)

특히 MVVM 패턴을 사용할 때 뷰 컨트롤러와 뷰모델을 연결하는 가장 권장되는 방식이기도 합니다. 코드로 보면 다음과 같은 형태가 됩니다.

```Swift
class AViewController: UIViewController {
    // 주입받은 뷰모델을 저장할 프로퍼티 (let으로 선언 가능하여 안전함)
    private let viewModel: AViewModel
    
    // ✅ 생성자 주입
    init(viewModel: AViewModel) {
        self.viewModel = viewModel
        super.init(nibName: nil, bundle: nil) // UIViewController의 지정 이니셜라이저 호출
    }
    
    // 스토리보드 지원을 위한 필수 이니셜라이저 (보통 코드 기반 UI에서는 fatalError 처리)
    required init?(coder: NSCoder) {
        fatalError("init(coder:) has not been implemented")
    }
}

// 외부(예: SceneDelegate나 이전 화면)에서 객체를 조립하여 주입하는 모습
let myViewModel = AViewModel()
let aViewController = AViewController(viewModel: myViewModel)
```

### 💡 면접을 위한 추가 팁 (UIKit 실무 포인트)

이 내용을 면접에서 답변하실 때, 실무적인 경험을 더해 어필할 수 있는 꼬리 질문 대비 포인트가 있습니다.

면접관이 **"UIKit에서 커스텀 생성자(`init`)를 만들면 `required init?(coder:)`를 반드시 구현하라고 에러가 뜨는데, 그 이유는 무엇인가요?"**라고 물어볼 수 있습니다.

> **답변 방향:** "UIViewController는 기본적으로 `NSCoding` 프로토콜을 채택하고 있기 때문입니다. 스토리보드나 XIB 같은 인터페이스 빌더를 통해 뷰 컨트롤러가 생성될 때는 `init(coder:)`를 통해 초기화가 이루어집니다. 따라서 코드 기반으로 UI를 작성하기 위해 커스텀 `init`을 만들더라도, 스토리보드로 생성될 가능성을 대비해 시스템이 강제로 `init(coder:)`를 구현하도록 요구하는 것입니다."
