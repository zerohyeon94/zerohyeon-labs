# ARC(Automatic Reference Counting)란?

- **Swift의 메모리 관리 방식**으로, **클래스(참조 타입)** 인스턴스의 **참조 카운트**를 컴파일러가 자동으로 증가/감소시켜, **카운트가 0이 되면 즉시 메모리 해제**합니다.
- **가비지 컬렉션(GC)이 아님** → 주기적으로 스캔하지 않고, 필요 없어지는 순간(deinit 시점)에 해제됨.
- **구조체/열거형(값 타입)**은 ARC 대상이 아닙니다.

## 기본 규칙 (세 줄 요약)

1. **strong(기본)**: 소유(ownership). 참조 카운트를 +1.
2. **weak**: 비소유, **Optional**이어야 함. 대상이 해제되면 **자동으로 nil**.
3. **unowned**: 비소유, **Non-optional**. 대상이 먼저 해제되면 **크래시** → **수명 보장이 되는 관계에만** 사용.

---

# 왜 중요한가? (버그 패턴)

- **강한 순환 참조(Retain Cycle)**가 생기면 참조 카운트가 0이 되지 않아 **메모리 누수**.
- 대표 사례
    - 양방향 객체 관계(예: Parent ↔ Child)
    - **클로저가 self를 캡처**(네트워크 콜백, Timer, Animation, Combine/AsyncStream 등)
    - **NotificationCenter**, **Timer**, **CADisplayLink** 등 런루프 기반 객체의 강한 참조

---

# 언제 weak, 언제 unowned?

- **weak**: 상대가 **나보다 먼저 해제될 수 있음** / 수명 불확실. (가장 안전, 면접 기본값)  
    예) `delegate` 프로퍼티, 클로저 캡처에서 대부분 `weak self`
- **unowned**: **수명이 반드시 나와 같거나 더 길다**는 **강한 보장**이 있을 때.  
    예) `CreditCard`가 항상 `Customer`와 동일 생명주기를 가정하는 샘플, 또는 `lazy` 클로저에서 `self`가 먼저 해제되지 않음이 확실할 때

# 코드 예제

## 1) 순환 참조 예시 → 해결

```swift
final class Person {
    let name: String
    var apartment: Apartment?   // 기본 strong
    init(name: String) { self.name = name }
    deinit { print("Person deinit") }
}

final class Apartment {
    let unit: String
    var tenant: Person?         // 여기도 strong이면 순환
    init(unit: String) { self.unit = unit }
    deinit { print("Apartment deinit") }
}

// ❌ 순환
var p: Person? = Person(name: "Alex")
var a: Apartment? = Apartment(unit: "101")
p!.apartment = a
a!.tenant = p
p = nil   // deinit 호출 안 됨
a = nil   // 여전히 보관 → 누수
```

해결(역방향을 약참조로 전환):

```swift
final class Apartment {
    let unit: String
    weak var tenant: Person?    // ✅ 약하게 참조 (Optional)
    init(unit: String) { self.unit = unit }
    deinit { print("Apartment deinit") }
}
```

## 2) 클로저 캡처로 인한 순환 → 캡처 리스트

```swift
final class MyVC: UIViewController {
    var work: (() -> Void)?

    func start() {
        // ❌ self를 강하게 캡처 → self 속성이 work를 갖고 있으면 순환
        work = {
            self.view.backgroundColor = .systemBlue
        }
    }
}
```

해결(weak/unowned):

```swift
work = { [weak self] in
    self?.view.backgroundColor = .systemBlue
}
// 또는 lifetime 보장 시
// work = { [unowned self] in self.view.backgroundColor = .systemBlue }
```

> **팁**: 캡처 리스트의 기본은 `strong`. 의식적으로 `[weak self]`를 고려하세요.

## 3) Timer / NotificationCenter

```swift
final class MyVC: UIViewController {
    var timer: Timer?

    override func viewDidLoad() {
        super.viewDidLoad()
        // ❌ target-selector는 보통 강한 참조 경로가 생기기 쉬움
        timer = Timer.scheduledTimer(withTimeInterval: 1.0, repeats: true) { [weak self] _ in
            self?.tick()
        }

        NotificationCenter.default.addObserver(
            forName: .someEvent, object: nil, queue: .main
        ) { [weak self] _ in
            self?.handle()
        }
    }

    deinit {
        timer?.invalidate() // ✅ 수동 해제
        // addObserver 반환 토큰을 저장했다면 여기서 remove
    }
}
```

## 4) deinit 확인

```swift
final class Foo {
    deinit { print("Foo deinit") }
}
var foo: Foo? = Foo()
foo = nil  // 즉시 "Foo deinit"
```

# autoreleasepool은 언제?

- Swift도 **Obj-C 브리징** 때문에 **autorelease pool**이 존재합니다.
- 이미지/대용량 객체를 루프에서 많이 만들 때 **임시 객체를 빨리 비우고 싶을 때** 사용:

```swift
for i in 0..<10_000 {
    autoreleasepool {
        // 큰 데이터 처리
    }
}
```

일반 앱 코드 대부분은 신경 쓸 필요 없음(특수 상황 최적화 용도).

# 실무 체크리스트 (바로 적용)

- **delegate는 항상 `weak`** (protocol은 `AnyObject`로 제한)
    
- **클로저 콜백은 기본적으로 `[weak self]`**, 내부에서 `guard let self = self else { return }` 패턴
    
- **타이머/디스플레이링크/노티**: 캡처는 `weak`, 수명 끝에 **invalidate/remove**
    
- **메모리 그래프 디버거**로 누수 확인(Xcode ▶ Debug Memory Graph) Instruments(Leaks/Allocations)도 활용
    
- **테스트로 누수 감지**:  `weak var sut = makeSUT(); ...; sut = nil; XCTAssertNil(sut)` 패턴

# 면접에서 자주 나오는 꼬리 질문 & 한 줄 답변

1. **ARC vs GC 차이?**  
    ARC는 **컴파일 타임**에 retain/release 삽입, 참조 0이면 **즉시 해제**. GC는 주기적 스캔.
    
2. **weak vs unowned 선택 기준?**  
    수명 불확실/먼저 없어질 수 있음 → **weak(안전)**. 수명 보장 → **unowned(위험하지만 비용↓)**.
    
3. **클로저 캡처 기본은? 왜 이슈?**  
    기본 **strong**이므로 self를 쉽게 붙잡아 **순환 참조** 유발.
    
4. **값 타입은 ARC 대상인가?**  
    **아니오.** 클래스/클로저 같은 참조 타입만.
    
5. **deinit 언제 호출?**  
    참조 카운트 0이 되는 **즉시**(단, autorelease 상황 등으로 지연될 수 있음).
    
6. **대표적 누수 사례?**  
    양방향 strong, 콜백 클로저 strong self, Timer/Notification 미정리.
