## ✅ ViewController와 ViewModel 간 접근 제어 기본 원칙

|위치|일반 접근 제어|이유|
|---|---|---|
|`ViewController`의 UI 요소|`private` 또는 `private(set)`|외부에서 조작 불필요 (캡슐화)|
|`ViewModel`의 input (호출용 API)|`public` 또는 `internal`|VC에서 호출해야 하므로 노출 필요|
|`ViewModel`의 내부 상태|`private` 또는 `fileprivate`|외부에서 조작 금지|
|`ViewModel`의 output (Observable, 바인딩 데이터)|`public` 읽기 + 내부 쓰기 (`private(set)` or 바인딩 패턴)|View가 참조하되 직접 수정은 못하게|

---

## 🔍 질문에 대한 핵심 답변

> “ViewController에서 API 통신에 필요한 값을 넘겨 ViewModel이 API를 호출하도록 할 때, ViewModel의 API 메서드를 public으로 만들면 API 통신에 필요한 정보가 모두 외부에 노출되는 것 아닌가요?”

➡️ **정확히 말하면 "통신 메서드"만 외부에 노출되는 것이고, 통신에 사용되는 데이터 내부 구조는 노출되지 않도록 설계할 수 있습니다.**

### 예시:

```swift
// ✅ ViewModel
final class UserViewModel {
    
    // 내부 저장소: 외부에서 접근 못함
    private var authToken: String = "secret123"

    // 외부에서 호출할 수 있는 메서드만 public/internal
    func fetchUserProfile(userID: String) {
        let endpoint = "https://api.domain.com/user/\(userID)"
        APIService.shared.get(endpoint, headers: ["Authorization": "Bearer \(authToken)"]) { result in
            ...
        }
    }
} 
```

👉 이 구조에서는 외부에서:

- `authToken`은 **절대 볼 수 없음**
    
- `fetchUserProfile(userID:)` 만 **ViewController에서 호출 가능**

즉, “API 호출 메서드는 공개되더라도 내부 민감 정보는 숨길 수 있습니다.”

---

## 🔒 보안과 설계 균형을 맞추는 방법

|기술|설명|
|---|---|
|`private(set)`|외부는 읽기만 가능하게 (값은 노출, 수정은 불가)|
|의존성 주입 (DI)|ViewModel이 외부에서 민감 정보를 받아 사용하되, 저장은 하지 않음|
|input/output 분리|Input 프로토콜로 API 요청만 받고, Output 프로토콜로 UI 반영만 전달|

---

## 🧠 정리하면

> ✅ **“ViewController에서 접근해야 할 만큼만 public/internal로 열고, 나머지는 철저히 숨긴다”**는 것이 Swift 접근 제어의 핵심입니다.

**API 통신 메서드 자체는 공개하되, 내부 상태와 민감한 로직은 숨기면 정보가 모두에게 공유되는 문제는 발생하지 않습니다.**
