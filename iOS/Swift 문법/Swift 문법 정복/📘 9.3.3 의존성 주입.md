### 1. DI를 적용하지 않은 경우 (강한 결합)

객체 내부에서 의존할 객체를 직접 생성하는 방식입니다. `InterviewViewModel`이 `RealNetworkManager`라는 구체적인 클래스에 강하게 결합되어 있습니다.

```Swift
class RealNetworkManager {
    func fetchCards() -> [String] {
        // 실제 네트워크 통신을 통해 데이터를 가져오는 로직
        return ["iOS 생명주기는?", "클래스와 구조체의 차이는?"]
    }
}

class InterviewViewModel {
    // ❌ 뷰모델 내부에서 직접 네트워크 매니저를 생성 (강한 결합)
    let networkManager = RealNetworkManager()
    
    func loadData() {
        let cards = networkManager.fetchCards()
        print(cards)
    }
}
```

**문제점:** 나중에 네트워크 매니저를 다른 클래스로 교체하거나, 테스트를 위해 가짜 데이터를 넣고 싶을 때 `InterviewViewModel`의 코드를 직접 수정해야 합니다.

### 2. DI를 적용하는 3가지 방법

먼저 구체적인 클래스 대신, 역할을 추상화한 **프로토콜(Protocol)**을 정의합니다.

```Swift
protocol CardFetchable {
    func fetchCards() -> [String]
}

// 실제 통신을 하는 객체 (프로토콜 채택)
class RealNetworkManager: CardFetchable {
    func fetchCards() -> [String] { return ["실제 데이터 1", "실제 데이터 2"] }
}
```

이제 이 프로토콜을 기반으로 외부에서 의존성을 주입받습니다.

#### 방법 A: 생성자 주입 (Initializer Injection) - ⭐️ 가장 권장

객체가 생성될 때 이니셜라이저를 통해 의존성을 전달받습니다. 필수적인 의존성을 누락 없이 주입할 수 있어 가장 안전합니다.

```Swift
class InterviewViewModel {
    let networkManager: CardFetchable // 프로토콜 타입으로 선언
    
    // ✅ 외부에서 생성할 때 주입받음
    init(networkManager: CardFetchable) {
        self.networkManager = networkManager
    }
}

// 사용 시점 (외부에서 주입)
let realManager = RealNetworkManager()
let viewModel = InterviewViewModel(networkManager: realManager)
```

#### 방법 B: 프로퍼티 주입 (Property Injection)

객체를 먼저 생성한 뒤, 프로퍼티에 접근하여 의존성을 할당합니다. Storyboard를 사용할 때처럼 이니셜라이저를 직접 제어하기 어려울 때 종종 사용됩니다.

```Swift
class InterviewViewModel {
    // ✅ 밖에서 변경할 수 있도록 var로 선언 (Optional이거나 암시적 추출 사용)
    var networkManager: CardFetchable?
}

// 사용 시점
let viewModel = InterviewViewModel()
viewModel.networkManager = RealNetworkManager() // 외부에서 주입
```

#### 방법 C: 메서드 주입 (Method Injection)

의존성이 객체 전체의 상태를 유지할 필요 없이, 특정 메서드를 실행할 때만 일회성으로 필요할 때 파라미터로 전달받습니다.

```Swift
class InterviewViewModel {
    // ✅ 메서드 호출 시 파라미터로 주입받음
    func loadData(using manager: CardFetchable) {
        let cards = manager.fetchCards()
    }
}

// 사용 시점
let viewModel = InterviewViewModel()
viewModel.loadData(using: RealNetworkManager())
```

### 3. DI로 얻는 가장 큰 이점: 유닛 테스트(Unit Test)

DI의 진가는 **테스트 코드**를 작성할 때 나타납니다. 실제 네트워크 통신은 느리고 서버 상태에 따라 실패할 수 있어 테스트용으로 부적합합니다. DI를 사용하면 실제 객체 대신 **가짜 객체(Mock)**를 주입하여 뷰모델의 로직만 순수하게 테스트할 수 있습니다.

```Swift
// 테스트를 위한 가짜 객체 생성
class MockNetworkManager: CardFetchable {
    func fetchCards() -> [String] {
        // 네트워크 통신 없이 바로 가짜 데이터 반환
        return ["테스트용 카드 1", "테스트용 카드 2"]
    }
}

// 테스트 코드 환경
func testInterviewViewModel() {
    // ✅ 가짜 매니저를 주입! (기존 뷰모델 코드는 1줄도 수정하지 않음)
    let mockManager = MockNetworkManager()
    let viewModel = InterviewViewModel(networkManager: mockManager)
    
    // 뷰모델이 잘 동작하는지 빠르고 안정적으로 테스트 가능
    viewModel.loadData() 
}
```

이렇게 의존성을 외부로 분리해 두면(제어의 역전, IoC), 코드를 수정하지 않고도 부품을 갈아 끼우듯 유연하게 객체를 교체할 수 있습니다.
