## 요약

- `weak`는 **클래스 인스턴스(참조 타입)** 에만 가능 → 제약으로는 **`AnyObject`**.
    
- RxCocoa의 `emit(with:)` / `drive(with:)`는 **소유자(owner)를 약한 참조로 캡처**해 **해제 시 이벤트 무시**를 보장.
    
- 값 타입(구조체/열거형)은 `weak` 불가 → `with:`에 넘길 수 없음.

---

## 왜 `AnyObject` 제약인가?

`with:`는 내부에서 넘긴 객체를 **weak로 잡아야** 함 → 스위프트에서 **weak는 클래스만** 가능.  
그래서 제네릭 제약을 `Object: AnyObject`로 둠.

```swift
// RxCocoa (개념적 형태)
public func emit<Object: AnyObject>(
  with object: Object,
  onNext: @escaping (Object, Element) -> Void
) -> Disposable
```

- `Object: AnyObject` → “반드시 클래스” 보장
    
- 내부 구현은 `weak object`로 캡처 → 객체가 해제되면 콜백 자체가 **호출되지 않음**

> 값 타입(Struct/Enum)은 `weak` 불가 → `with:` 사용 불가

---
## `weak` 한눈 정리

- **목적:** 순환 참조(retain cycle) 방지 + 소유자 수명에 따라 **안전하게 무시**
    
- **가능 대상:** 클래스 인스턴스(참조 타입)만
    
- **옵셔널:** `weak var ref: SomeClass?` (항상 Optional)
    
- **해제 후 동작:** 자동으로 `nil`, 접근 시 안전 가드 필요

### `weak` vs `unowned`

- `weak` : 객체가 **먼저 해제될 수 있음** → 안전, 옵셔널, 가드 필요
    
- `unowned` : 객체가 **반드시 더 오래 살아야 함** → 비옵셔널, 해제되면 **크래시**
    
- 규칙: **의심나면 `weak`**. 정말 수명 보장될 때만 `unowned`.

---

## 언제 `weak`를 쓸까?

- **클로저 캡처**: VC/뷰모델/셀 등 자신을 캡처할 때
    
- **델리게이트 패턴**: `delegate`는 관례적으로 `weak`
    
- **Rx 바인딩**: `emit(with:)`, `drive(with:)` or `[weak self]` 패턴

---
## 자주 하는 실수

- 값 타입(Struct)을 `with:`에 넘기려 함 → **컴파일 에러** (AnyObject 제약 불일치)
    
- `unowned self` 남발 → **해제 후 크래시**
    
- Rx 바인딩에서 `weak` 없이 `self` 강캡처 → **retain cycle** (특히 VC↔VM/Service)
