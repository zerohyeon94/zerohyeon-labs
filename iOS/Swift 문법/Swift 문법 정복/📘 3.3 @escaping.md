`Escaping Closure`를 이해하는 핵심은 **'클로저가 실행되는 타이밍'**과 **'함수의 종료 타이밍'**의 관계입니다.

차이를 명확히 확인하기 위해 두 가지 상황을 코드로 비교해 보겠습니다.

---

### 1. 일반 클로저 (Non-Escaping Closure)

기본적으로 스위프트의 클로저는 Non-escaping입니다. 즉, 함수 안으로 전달된 클로저는 **함수가 끝나기 전에 무조건 실행을 완료**해야 합니다.

```Swift
func runSynchronously(closure: () -> Void) {
    print("1. 함수 실행 시작")
    
    // 함수가 종료되기 전에 클로저가 호출됨
    closure() 
    
    print("3. 함수 실행 종료")
}

runSynchronously {
    print("2. 클로저 내부 실행")
}
```

- **실행 순서:** 1 -> 2 -> 3
    
- 함수의 생명주기 내에서 클로저가 소비되므로 별도의 키워드가 필요 없습니다.

---

### 2. 탈출 클로저 (@escaping)

클로저가 함수의 실행 흐름(Scope)을 벗어나서 작동해야 할 때 컴파일러에게 이를 알려주는 키워드입니다. 대표적으로 **비동기 통신**과 **외부 저장** 두 가지 상황에서 쓰입니다.

#### A. 비동기 작업의 콜백 (네트워크 요청 등)

가장 흔하게 접하는 상황입니다. 함수는 이미 종료되었는데, 나중에 서버에서 데이터를 받아왔을 때 클로저를 실행해야 하는 경우입니다.

```Swift
func fetchUserData(completion: @escaping (String) -> Void) {
    print("1. 네트워크 요청 시작")
    
    // 2초 뒤에 서버에서 응답이 왔다고 가정 (비동기 처리)
    DispatchQueue.main.asyncAfter(deadline: .now() + 2.0) {
        // 이 시점에는 이미 fetchUserData 함수가 완전히 종료된 상태입니다.
        // 함수가 죽은 뒤에도 클로저가 살아남아 실행되어야 하므로 @escaping이 필수입니다.
        completion("3. 데이터 로드 완료") 
    }
    
    print("2. 함수 자체는 여기서 바로 종료됨")
}

fetchUserData { result in
    print(result)
}
```

- **실행 순서:** 1 -> 2 -> (2초 대기) -> 3
    
- 만약 여기서 `@escaping`을 빼면, 컴파일러는 "함수가 끝나면 클로저도 메모리에서 사라지는데, 나중에 어떻게 호출하려고 해?"라며 에러를 발생시킵니다.

#### B. 클로저를 외부 변수에 저장할 때

함수의 파라미터로 받은 클로저를 클래스나 구조체의 프로퍼티(변수)에 저장해 두고 나중에 꺼내 쓰고 싶을 때 사용합니다.

```Swift
class NetworkManager {
    // 클로저를 저장할 배열
    var completionHandlers: [() -> Void] = []
    
    // 파라미터로 받은 클로저를 외부 변수에 저장하므로 @escaping 필요
    func addHandler(handler: @escaping () -> Void) {
        completionHandlers.append(handler)
    }
}
```

> **핵심 요약:** 클로저가 함수의 중괄호 `{ }` 가 닫히고 난 **이후**에도 메모리에 살아남아 호출될 가능성이 1%라도 있다면, 반드시 `@escaping` 키워드를 붙여야 합니다.

---

면접에서 `@escaping` 클로저를 설명할 때는 꼬리 질문으로 **"순환 참조(Retain Cycle)와 메모리 누수 문제"**가 아주 높은 확률로 따라옵니다
