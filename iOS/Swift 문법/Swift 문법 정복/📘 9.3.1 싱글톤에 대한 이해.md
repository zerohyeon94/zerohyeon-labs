## 나의 의견

- **싱글톤**: `static let shared = LoginProviderStore()` 한 번만 생성(게으른 초기화 + Swift에서 스레드 안전). 앱 생애 동안 **하나의 인스턴스**를 모두가 공유.
    
- **매번 `init`**: `LoginProviderStore()`를 필요할 때마다 생성하면 **여러 인스턴스**가 생김.

## 추가 설명

- 차이는 **설계 품질(결합도/테스트성/확장성)** 과 **수명/전역상태 관리**에 있어요.

# 어떤 방식을 쓸까? (실무 기준)

## A) 싱글톤로 단순하게

- 장점: 편함, 어디서나 바로 접근
    
- 단점: **전역 가변 상태**가 커짐(결합 ↑), **테스트 어려움**, 나중에 저장소를 교체하기 어려움
	- 전역 가변 상태 (Global Mutable State) : `LoginProviderStore.shared` 같은 전역 인스턴스를 여러 곳에서 직접 참조하면, 코드의 많은 부분이 **같은 전역 객체**에 의존합니다. 이 전역 객체의 내부 구현이 바뀌면, 해당 전역을 사용하는 **모든** 코드가 직간접적으로 영향을 받아 **결합도(coupling)** 가 커져요.
	- 테스트 어려움 : 전역은 **교체/격리**가 어렵습니다. 테스트마다 상태를 초기화해야 하고, 동시에 실행되는 테스트 간 **상태 오염**이 발생하기 쉽습니다.
	- 교체 난이도 : 나중에 저장소를 `UserDefaults`→`Keychain`/`File`/`In‑Memory`등으로 바꾸려면, 전역에 하드코딩되어 있으면 사용처를 **전부 수정**해야 할 수도 있어요.
    
- 적용 예:
```swift
final class LoginProviderStore: LoginProviderStoring {
    static let shared = LoginProviderStore()
    private init(defaults: UserDefaults = .standard) { self.defaults = defaults }
    // ... 나머지 동일
}

// 사용
LoginProviderStore.shared.setCurrentLoginProvider(provider)
```

## B) 의존성 주입(DI)로 유연하게

- 장점: **테스트 용이**(mock/suite 주입), **유연성/교체 용이**, **결합↓**
	- 의존성 주입 의미 : 필요한 의존 객체(예: 저장소)를 **내부에서 생성하지 않고** 바깥(컴포지션 루트)에서 **주입**받는 설계.
		```swift
			init(store: LoginProviderStoring) { self.store = store }
		```
	- 결합↓ : 코드가 **구체 타입**(UserDefaults 기반 구현)이 아니라 **프로토콜**(인터페이스)에 의존 → 구현이 바뀌어도 사용처는 그대로.
	- 테스트 용이 : 테스트에서 **가짜 구현(Mock/Stubs)** 이나 **격리된 UserDefaults(suite)** 를 주입하면, 실제 환경과 분리된 상태로 **안전하게 검증** 가능. 테스트 간 상태 오염도 줄어듭니다.
   
- 단점: 주입 코드가 약간 번거로울 수 있음

- 핵심: “여러 개 만들지 않으려면” **컴포지션 루트**(App 시작 지점)에서 **한 번만 만들고** 필요한 곳에 **같은 인스턴스를 전달**하세요. 이렇게 하면 **싱글톤이 아니어도** “실제로는 하나만 써서” 수명 관리가 됩니다.

- 적용 예:
```swift
// AppDIContainer (컴포지션 루트)
final class AppDIContainer {
    let loginProviderStore: LoginProviderStoring = LoginProviderStore()
}

// ViewModel 주입
final class LoginViewModel {
    private let loginProviderStore: LoginProviderStoring
    init(loginProviderStore: LoginProviderStoring) {
        self.loginProviderStore = loginProviderStore
    }
}
```

# 스레드/동시성 관련 메모

- `UserDefaults`는 내부적으로 스레드 안전하게 설계되어 있으나, **즉시 디스크 동기화**는 보장하지 않습니다(요즘은 `synchronize()` 안 씀).
    
- UI는 메인 스레드에서, 저장 로직은 간단해서 보통 그냥 사용합니다.
    
- 외부에서 변경을 감지하고 싶다면 `UserDefaults.didChangeNotification`을 고려하세요.

# 결론 가이드(의사결정 트리)

- **작은 앱·간단한 상태**: 싱글톤 OK
    
- **테스트가 중요/교체 가능해야 함/멀티 저장소 가능성**: DI 권장
    
- 싱글톤을 쓰더라도 **프로토콜(LoginProviderStoring)** 을 유지해 **추후 교체 가능성** 확보가 베스트
