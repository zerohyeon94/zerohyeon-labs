`[weak self]`와 `[unowned self]`는 **"ARC의 참조 횟수(Reference Count)를 증가시키지 않아 순환 참조를 막는다"**는 핵심 목적은 완전히 똑같습니다. 하지만 **참조하던 객체(`self`)가 메모리에서 먼저 사라졌을 때 어떻게 행동하느냐**에서 결정적인 차이가 발생합니다.

두 개념의 차이를 코드로 비교해 보겠습니다.

---

### 1. [weak self] (약한 참조) : "안전 제일주의"

가장 보편적이고 안전한 방식입니다. `self`가 언제든 메모리에서 해제될 수 있다고 가정하기 때문에 항상 **옵셔널(Optional)** 타입으로 취급됩니다.

- **동작 방식:** 객체가 메모리에서 해제되면, `weak` 참조는 자동으로 **`nil`**로 변환됩니다.
    
- **장점:** 객체가 사라진 후 클로저가 실행되더라도 `nil`인지 확인하고 안전하게 넘어갈 수 있으므로 **앱이 크래시(강제 종료)되지 않습니다.**

```Swift
class SafeClass {
    var name = "안전한 객체"
    var closure: (() -> Void)?
    
    func doSomething() {
        closure = { [weak self] in
            // self는 옵셔널이므로 언래핑이 필수
            guard let self = self else { 
                print("객체가 이미 사라졌습니다.")
                return 
            }
            print(self.name)
        }
    }
}
```

---

### 2. [unowned self] (미소유 참조) : "강한 확신"

`self`가 클로저보다 먼저 메모리에서 해제될 일이 **절대 없다**고 확신할 때 사용합니다. 따라서 **논옵셔널(Non-Optional)** 로 취급됩니다.

- **동작 방식:** 객체가 메모리에서 해제되어도 **`nil`로 변하지 않고, 예전 메모리 주소를 그대로 가리키고 있습니다.** (이를 Dangling Pointer라고 합니다.)
    
- **위험성:** 만약 객체가 이미 해제되었는데 클로저가 실행되어 `self`에 접근하려고 하면, **그 즉시 런타임 에러가 발생하며 앱이 강제 종료(Crash)** 됩니다.

```Swift
class RiskyClass {
    var name = "위험한 객체"
    var closure: (() -> Void)?
    
    func doSomething() {
        closure = { [unowned self] in
            // 옵셔널 바인딩(guard let) 없이 바로 접근 가능
            print(self.name) 
            // ⚠️ 만약 이 시점에 RiskyClass가 메모리에서 해제되었다면 앱 크래시 발생!
        }
    }
}
```

---

### 3. 언제 [unowned self]를 써야 할까?

위험성을 안고서라도 `unowned`를 쓰는 이유는 옵셔널 바인딩 코드를 생략해 코드를 간결하게 만들고, 약간의 오버헤드를 줄이기 위함입니다. 주로 **클래스(self)와 클로저의 생명주기(Lifecycle)가 완전히 동일할 때** 사용합니다.

가장 대표적인 예시는 객체 내부의 `lazy var` 프로퍼티에서 클로저를 사용할 때입니다.

```Swift
class MyViewController {
    var titleText = "메인 화면"
    
    // 이 UI 요소는 MyViewController와 운명 공동체입니다.
    // 뷰 컨트롤러가 살아있는 한 클로저도 살아있고, 뷰 컨트롤러가 죽으면 클로저도 즉시 죽습니다.
    lazy var headerView: UIView = { [unowned self] in
        let view = UIView()
        // unowned를 쓰면 강제 언래핑 없이 바로 self 접근 가능
        view.accessibilityLabel = self.titleText 
        return view
    }()
}
```

다만, 현대의 iOS 개발에서는 앱의 안정성을 최우선으로 고려하여 약간의 귀찮음(guard let)을 감수하더라도 **대부분의 상황에서 무조건 `[weak self]`를 사용하는 것을 권장**하는 추세입니다.

> **면접 답변 요약(Cheat Sheet):** "두 키워드 모두 순환 참조를 방지하지만, 참조하던 객체가 해제되었을 때의 안전성에 차이가 있습니다. `[weak self]`는 객체가 해제되면 `nil`이 되는 옵셔널 타입이라 안전하지만, `[unowned self]`는 해제된 메모리 주소를 계속 가리키는 논옵셔널 타입이라 접근 시 크래시가 발생할 수 있습니다. 따라서 클로저와 객체의 생명주기가 완전히 동일하다고 확신할 때만 `unowned`를 사용하고, 그 외 비동기 네트워크 통신 등에서는 안전한 `weak`를 사용해야 합니다."
