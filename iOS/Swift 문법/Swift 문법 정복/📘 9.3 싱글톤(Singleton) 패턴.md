## ✅ `shared`란?

> 앱 전역에서 하나의 인스턴스를 공유하고 싶을 때 사용하는 **Singleton 객체의 접근자(property)** 입니다.

```swift
final class APIService {
    static let shared = APIService()
    private init() {} // 외부에서 생성 금지
}
```

---

## 📌 왜 `shared`를 쓰는가?

- 인스턴스를 여러 번 만들지 않기 위해 (메모리 절약, 상태 공유)
    
- 공통된 네트워크, 로깅, 설정, 알림 처리 등에 적합
    
- 앱 전역에서 동일한 동작과 상태를 유지

---

## 👇 사용 예시

```swift
APIService.shared.fetchUserProfile(...)
```

- `APIService.shared`: 단일 인스턴스 접근
    
- `fetchUserProfile`: shared 인스턴스의 메서드 호출

---

## 🔍 `shared` vs `shared()` 차이점

|표현|의미|사용 조건|예시|
|---|---|---|---|
|`shared`|**프로퍼티** (보통 `static let`)|값처럼 접근|`APIService.shared.fetch()`|
|`shared()`|**메서드** (보통 `static func`)|실행해서 인스턴스를 반환|`SomeManager.shared().fetch()`|

---

### ✅ 정리

| 구분         | 설명                                                                  | 예시                                    |
| ---------- | ------------------------------------------------------------------- | ------------------------------------- |
| `shared`   | `static let shared = SomeClass()` <br>→ 항상 같은 인스턴스 반환               | `URLSession.shared`                   |
| `shared()` | `static func shared() -> SomeClass` <br>→ 필요 시마다 생성 가능 (보통 싱글톤은 아님) | `DateFormatter.shared()` 처럼 커스터마이즈 가능 |

---

## ✅ 예시 비교

### 1. `shared`를 사용하는 전형적인 싱글톤

```swift
final class Logger {
    static let shared = Logger()
    private init() {}
}
```

- 사용: `Logger.shared.log("Hello")`
    
- 항상 같은 인스턴스

---

### 2. `shared()` 처럼 메서드로 인스턴스 생성

```swift
class FormatterFactory {
    static func shared() -> DateFormatter {
        let formatter = DateFormatter()
        formatter.dateFormat = "yyyy-MM-dd"
        return formatter
    }
}
```

- 사용: `let f = FormatterFactory.shared()`
    
- 호출마다 새 인스턴스 (or 필요 시 캐싱 가능)

---

## 🧠 결론

| 상황                          | 사용 방식                         |
| --------------------------- | ----------------------------- |
| 인스턴스 1개만 유지 (진짜 싱글톤)        | `static let shared = ...`     |
| 조건마다 다르게 생성하고 싶거나, 매번 새로 생성 | `static func shared() -> ...` |

---

> ✅ **Swift 개발자 대부분은 `static let shared` 형태를 선호하며**,  
> 보안이나 설계 이유가 특별히 없다면 `shared()` 메서드 방식은 **일반적이지 않습니다.**

---

## ✅ 비교 요약

| 항목                   | `static let shared`                  | `class func shared()`                  |
| -------------------- | ------------------------------------ | -------------------------------------- |
| ✅ **가장 일반적인 싱글톤 방식** | ✅                                    | ❌                                      |
| 접근 방식                | `ApiMember.shared.registerUser(...)` | `ApiMember.shared().registerUser(...)` |
| 쓰기 편의성               | ✅ 깔끔하고 간결                            | ❌ 괄호 필요로 덜 자연스러움                       |
| 메모리 안정성              | ✅ 확실하게 1회 생성                         | ⚠️ 클로저 잘못 쓰면 다중 생성 위험                  |
| 설계 의도 전달             | ✅ 명확함 (`shared` = singleton)         | ❌ 의도 해석 필요                             |
| 테스트/확장 유연성           | ❌ 단점 있음 (테스트 대체 어려움)                 | ✅ 내부 인스턴스 은닉 가능, 교체 가능                 |
| 보안성                  | ❌ 접근 경로 명확히 드러남                      | ✅ 내부 인스턴스를 숨기고 메서드로만 제어                |
| 실무에서 많이 쓰는가?         | ✅ 널리 쓰임                              | 🔸 특정 상황만                              |

## ✅ 대중적인 방식: `static let shared`

```swift
final class ApiMember {
    static let shared = ApiMember()
    private init() {}
}
```

- 🍎 Apple SDK도 대부분 이 방식 사용:
    
    - `URLSession.shared`
        
    - `NotificationCenter.default`
        
- ✍️ 가장 읽기 쉽고 쓰기 편함: `ApiMember.shared.registerUser()`

👉 **iOS 커뮤니티, 팀 프로젝트, 오픈소스 등 거의 90% 이상 이 방식을 채택합니다.**

---

## 🔐 메서드 방식 (`shared()`): 보안/은닉 목적

```swift
final class ApiMember {
    private static let shared_ = ApiMember()
    class func shared() -> ApiMember {
        return shared_
    }
}
```

- ✅ 내부 인스턴스를 완전히 숨기고 싶을 때
    
- ✅ 테스트 시 내부 구현을 스와핑하거나 mocking 가능하게 만들고 싶을 때
    
- ✅ 외부에서 shared를 사용하더라도 내부 구조를 변경하고 싶을 때

→ 보안 목적이 **매우 명확한 경우에는 좋은 선택**입니다. 하지만 일반적이지는 않습니다.

---

## 🧠 실무 팁 (정리)

|상황|추천 방식|
|---|---|
|일반적인 싱글톤 (API, 로깅, 설정 등)|✅ `static let shared`|
|내부 로직 숨기기, 보안 목적, 확장 고려 (DRM, 보안 API 등)|🔸 `class func shared()` (단 내부 구조 명확히 주석 또는 설계 문서 필요)|
|테스트 대체 필요 (의존성 주입 없이)|🔸 `shared()` + 테스트 인젝션|

---

## 🧾 결론

> **대중적인 Swift 스타일에서는 `static let shared`를 가장 많이 사용합니다.**
> 
> 하지만 `class func shared()` 방식은 **설계상 보안이나 확장성, 테스트 유연성**을 고려할 때 적절한 선택이 될 수 있습니다.
