RxSwift(리액티브 프로그래밍)는 **복잡한 비동기/이벤트 흐름**을 “깔끔하고, 재사용 가능하게, 선언적으로” **‘조립’**할 수 있게 해줍니다.

### 1. **이벤트 흐름이 복잡할 때, 코드가 훨씬 읽기 쉽고 관리가 쉬움**

- 버튼 클릭, 네트워크 응답, UI 입력, 타이머, 알림 등등
    
- 여러 비동기 작업을 **순차, 병렬, 합성, 취소, 필터링** 등 다양하게 조합할 때  
    → Rx 없이 하면 콜백 지옥, GCD 남용, 상태 꼬임, 버그 유발
    
- Rx는 “스트림”처럼 연결해서 한눈에 보이게 해줌

#### 예시

- 버튼 클릭 → 서버 요청 → 결과 오면 리스트 갱신 (에러나면 토스트, 로딩 중이면 인디케이터 등)

---

### 2. **코드의 일관성 & 재사용성 증가**

- Rx로 작성하면 “패턴화”된 코드가 많아서 다양한 이벤트 소스(네트워크, Notification, KVO, UI 이벤트 등)를 **동일한 방식**으로 다룰 수 있음

---

### 3. **상태관리와 비동기 조합이 쉬워짐**

- 여러 개의 비동기 소스(예: 3개 API 요청 결과를 합쳐야 할 때)를 콜백으로 짜면 진짜 복잡
    
- Rx의 `zip`, `combineLatest`, `flatMap` 등 연산자로 “데이터 흐름”을 자연스럽게 만들 수 있음

---

### 4. **취소, 에러처리, 재시도, 스케줄링 등이 쉽다**

- Rx는 디스포저블(disposeBag)로 **자동 취소**
    
- `.retry()`, `.catchError()`로 **에러처리와 재시도**
    
- `.observeOn(.main)` 등으로 **스레드 전환**도 명확하게 컨트롤

---

### 5. **UI 바인딩이 매우 자연스럽다**

- `Observable` → UI까지 “연결”하는 패턴이 익숙해지면 MVC, MVVM 구조에서 코드가 매우 간결해짐
    
- RxCocoa 등과 연계하면 UI의 변화(버튼, 텍스트필드, 테이블 등)도 전부 Rx로 다룰 수 있음

---

## ✅ **언제 RxSwift를 꼭 써야 할까?**

- **이벤트/비동기/데이터 흐름**이 복잡한 앱
    
- 여러 이벤트의 “조합”, “필터링”, “취소”, “재시도” 등이 필요한 앱
    
- MVVM 구조에서 “UI와 데이터”를 분리하고 싶은 앱
    
- 코드의 재사용, 테스트, 일관성이 중요한 팀 프로젝트

---

## ❗️**반대로 RxSwift를 굳이 쓰지 않아도 되는 경우**

- 간단한 앱, 이벤트 흐름이 복잡하지 않으면 **async/await, GCD, 콜백**으로도 충분함
    
- Rx는 러닝커브(학습 곡선)가 높고, 필요 이상으로 도입하면 코드가 오히려 어려워질 수 있음
