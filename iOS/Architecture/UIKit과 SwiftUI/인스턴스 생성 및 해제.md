## 인스턴스 생성/해제 비용이 '비싸다'는 의미

### 1. 힙(Heap) 할당의 복잡성 (Class)
- **탐색(Search):** 메모리의 빈 공간을 찾아 헤매는 과정이 필요함.
- **동기화(Synchronization):** 멀티스레드 환경에서 메모리 충돌을 막기 위한 잠금(Lock) 비용 발생.
- **관리(Bookkeeping):** 메모리가 해제된 후 조각난 공간(Fragmentation)을 관리해야 함.

### 2. ARC 오버헤드 (Class)
- 인스턴스가 생성되고 사라질 때마다 참조 횟수(Reference Count)를 계산(+1, -1)하는 CPU 연산 비용이 듬.

### 3. 반면 스택(Stack)은? (Struct)
- **O(1) 비용:** 스택 포인터(Stack Pointer)만 단순히 증가/감소시키면 되므로 탐색이나 관리가 전혀 필요 없음.
- **즉시성:** CPU 명령어 몇 줄로 처리가 끝나기 때문에 수천, 수만 번 반복해도 부하가 거의 없음.

## 이해하기 쉽게
### 1. 비유로 이해하기 (호텔 vs 캠핑)

**Class (Heap 영역)**는 **'호텔 체크인'**과 같습니다.

1. **할당(체크인):** 빈 방이 어디 있는지 시스템을 조회해야 합니다. (탐색 비용)
    
2. **관리(숙박부):** 누가 묵는지 이름을 적고, 키를 발급받아야 합니다. (참조 카운팅)
    
3. **접근(이동):** 방에 가려면 엘리베이터를 타고 복도를 걸어가야 합니다. (포인터 참조)
    
4. **해제(체크아웃):** 방을 비우고, 청소하고, 시스템에서 빈 방으로 상태를 바꿔야 합니다.

- 👉 **결론:** 절차가 복잡하고 시간이 걸립니다.

**Struct (Stack 영역)**는 **'텐트 치기'**와 같습니다.

1. **할당:** 그냥 내 발밑 빈 땅에 텐트를 툭 던지면 끝입니다. (스택 포인터 이동)
    
2. **관리:** 별도의 장부 작성이 필요 없습니다.
    
3. **접근:** 바로 옆에 있으니 즉시 들어갈 수 있습니다.
    
4. **해제:** 텐트를 걷어서 가방에 넣으면 끝입니다.

- 👉 **결론:** 절차가 단순하고 즉각적입니다.

### 2. 기술적으로 파헤치기 (진짜 이유)

컴퓨터 공학적으로 "비용이 비싸다"는 것은 구체적으로 다음 3가지 작업 때문입니다.

#### ① 메모리 탐색 비용 (Allocation Cost)

- **Class (Heap):** 힙 메모리는 데이터가 뒤죽박죽 섞여 있습니다. 새로운 클래스 인스턴스를 만들려면, 운영체제는 **"이 크기의 데이터를 넣을 만한 빈 공간이 어디 있지?"** 하고 메모리 전체를 훑어서 빈 공간을 찾아야 합니다. 이 과정(Search)이 CPU를 씁니다.
    
- **Struct (Stack):** 스택은 책을 쌓는 것과 같습니다. **"그냥 제일 위에 올려"**라고 하면 끝입니다. 빈 공간을 찾을 필요가 없이, 스택 포인터(위치 표시기)만 한 칸 올리면 됩니다. (CPU 사이클이 거의 0에 수렴)

#### ② 참조 카운팅 (Reference Counting / ARC)

- **Class:** 여러 곳에서 공유될 수 있으므로, **"지금 몇 명이 나를 보고 있지?"**라는 숫자(Reference Count)를 계속 세야 합니다. 변수에 할당할 때마다 숫자를 +1, 다 쓰면 -1을 해야 하는데, 이 덧셈 뺄셈 과정(retain/release)이 생각보다 무겁습니다. 특히 멀티스레드 환경에서는 안전장치(Lock)까지 걸어야 해서 더 느려집니다.
    
- **Struct:** 복사해서 주면 끝입니다. "몇 명이 보는지" 셀 필요가 없습니다.

#### ③ 간접 참조 (Indirection)

- **Class:** 변수에는 '주소(번지수)'만 적혀 있습니다. 데이터를 읽으려면 그 주소를 따라가서(Follow pointer) 실제 데이터를 가져와야 합니다. 한 다리 건너야 하니 느립니다.
    
- **Struct:** 변수 안에 데이터 자체가 들어있습니다. 바로 읽으면 됩니다.
