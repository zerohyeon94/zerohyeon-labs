## 1. 먼저 “라이브러리”가 뭐냐부터

**라이브러리(Library)** : 내가 만든 코드 말고, _다른 사람이 미리 만들어 놓은 함수/클래스 묶음_.

- 예: Alamofire, RxSwift 같은 것들
    
- 우리가 `import Alamofire` 해놓고 쓰는 “재사용 가능한 코드 뭉치”

---

## 2. Static Library (정적 라이브러리)

### 개념

- **빌드할 때**(컴파일 + 링크 단계)에 라이브러리 코드가 **앱 실행파일 안으로 복붙**된다고 생각하면 됩니다.
    
- 결과적으로, **최종 .app 안의 실행파일 하나에 코드가 다 들어감**.

### 특징

1. **앱 크기 (바이너리 크기)가 커진다**
    
    - 라이브러리 코드가 통째로 실행파일에 합쳐지니까.
        
2. **실행 시점에는 추가 로딩이 거의 없음**
    
    - 이미 실행파일 안에 있으니까, 별도 “라이브러리 파일”을 찾거나 로드할 필요가 적음
        
    - 보통 **앱 실행 속도가 빠른 편**(런타임 오버헤드 적음)
        
3. **앱마다 따로 포함**
    
    - A앱, B앱이 같은 라이브러리를 쓰더라도,
        
    - 각 앱 실행파일에 각자 포함되므로, **앱마다 자기 복사본을 가지고 있음**
        
4. iOS에서의 예
    
    - 옛날식 `.a` 파일
        
    - 요즘은 **Static Framework**, **Static XCFramework** 형태로 많이 사용
        
    - Swift Package Manager(SPM)로 받는 라이브러리도 기본이 static인 경우가 많음

## 3. Dynamic Library (동적 라이브러리)

### 개념

- **앱을 실행할 때** 또는 **실행 도중(run-time)** 에 라이브러리를 **따로 로드해서 붙이는 방식**.
    
- 라이브러리가 **별도 파일(.dylib, dynamic framework)** 로 존재하고, 실행 시점에 OS가 그걸 메모리에 올려서 연결해 줍니다.

### 특징

1. **실행파일은 가벼워질 수 있음**
    
    - 앱 실행파일(.exe에 해당하는 것)에는 “라이브러리를 참조한다”는 정보만 있고
        
    - 실제 코드는 외부 `.dylib` / dynamic `.framework` 안에 있음
        
2. **여러 앱이 공유 가능**
    
    - 이론상, A앱과 B앱이 **같은 dynamic library**를 쓰면 메모리에서 코드를 공유할 수 있음 (iOS에선 대부분 시스템 프레임워크가 이런 구조).
        
    - 예: UIKit, Foundation 같은 **시스템 프레임워크**들은 동적 라이브러리 느낌으로 동작
        
3. **앱 실행 시점에 로딩 오버헤드**
    
    - 앱 실행할 때 OS가 “어, 이 앱은 무슨 무슨 dynamic library 필요하네?” 하고
        
    - 그걸 찾아서 메모리에 올리고, 심볼 연결을 해야 함
        
    - 그래서 **dynamic 라이브러리가 많으면 앱 런치가 느려질 수 있음**
        
4. **바꾸기 쉽다**
    
    - 이론적으로는,
        
        - 실행파일은 그대로 두고 dynamic library만 교체하면
            
        - 앱을 다시 빌드하지 않고도 기능을 바꿀 수 있음 (iOS에선 보안/스토어 정책 때문에 마음대로 못하지만, PC OS에선 흔한 패턴)
            
5. iOS에서의 예
    
    - `.framework` 중에서 **“Dynamic Framework”** 로 빌드한 것
        
    - Apple 시스템 프레임워크들 (UIKit, AVFoundation 등)은 전형적인 dynamic library

---

## 4. 그러면 “dynamic” vs “static” 차이를 한 줄로 정리하면?

- **Static**:
	- “빌드할 때 실행파일 안에 다 섞어서 넣는 라이브러리”
	- “컴파일 타임에 꽉 붙어 있는 동료”
    
- **Dynamic**:
	- “빌드할 때는 참조 정보만 넣어두고, 실행할 때(OS가) 따로 불러서 붙이는 라이브러리”
	- “런타임에 불러와 같이 일하는 외주”

## 5. iOS 개발에서 실제로 체감하는 포인트

### (1) 앱 실행 속도 / 런치 타임

- **Static 라이브러리 위주 → 런치 타임 유리**
    
    - iOS에서 SPM이 static을 기본으로 사용하는 이유 중 하나
        
- **Dynamic Framework를 너무 많이 쓰면 → 앱 처음 켤 때 느려짐**

### (2) 빌드 결과 구조

- Static:
    
    - `MyApp` 실행파일 하나가 엄청 뚱뚱해짐
        
- Dynamic:
    
    - `MyApp` + `SomeSDK.framework` + `AnotherSDK.framework` 이런 식으로 여러 파일이 함께 번들 안에 존재

### (3) 유지보수/배포 관점

- iOS는 앱스토어 심사 + 샌드박스 때문에 **PC처럼 dynamic library만 교체하는 식의 업데이트는 사실상 불가능** 
  → 보통 “Static이든 Dynamic이든” 다시 빌드해서 새 버전 앱을 올려야 함

그래서 iOS에서는:

- 퍼포먼스 측면 + 앱 구조 단순화를 위해 **Static 라이브러리를 기본 선택으로 보는 흐름이 꽤 강함**
    
- 그러나 SDK 공급사에서 “Dynamic Framework”로만 제공하는 경우도 있고 플러그인 구조 등에서 dynamic이 필요한 케이스도 있음

---

## 6. 머릿속에 남기기 좋은 요약

- **Dynamic Library**
    
    - 실행할 때(OS가) 따로 로딩하는 라이브러리
        
    - 여러 앱이 공유 가능, 바이너리는 가벼움
        
    - 런치 타임 오버헤드 존재
        
- **Static Library**
    
    - 빌드 시 실행파일 안에 녹여 넣는 라이브러리
        
    - 앱마다 코드가 복사되지만
        
    - 런치 타임이 빠르고 구조 단순
