## 한 줄 정의

> **UIKit 안에서 SwiftUI 뷰를 화면으로 보여주기 위한 UIViewController 서브클래스**

UIKit 기반 프로젝트에 SwiftUI 뷰(예: `Chart`, `NavigationStack`, 커스텀 View 등)를 끌어와서 사용하고 싶을 때 사용하는 브릿지 역할의 컨트롤러이다.

---

## 등장 배경

- **도입 시기**: iOS 13, Xcode 11 (SwiftUI 첫 등장과 함께 공개)
- 목적
  - 기존 UIKit 프로젝트를 전부 갈아엎지 않고도, **일부 화면만 SwiftUI로 작성**할 수 있도록 하기 위함
  - SwiftUI → UIKit 사이를 연결하는 “호스팅 컨트롤러”

---

## 기본 개념

- 클래스 정의: `final class UIHostingController<Content> : UIViewController where Content : View`
- 제네릭 파라미터 `Content`:
  - SwiftUI의 `View` 프로토콜을 채택한 타입
  - `UIHostingController(rootView: Content)` 형태로 전달
- 역할
  - 내부적으로 SwiftUI의 뷰 계층을 관리
  - 외부적으로는 **일반 UIViewController처럼 취급**되기 때문에
    - `navigationController?.pushViewController(_:)`
    - `present(_:animated:)`
    - child view controller로 embed
    등이 모두 가능하다.

---

## 언제 사용하는가?

1. **기존 UIKit 프로젝트에 SwiftUI를 도입**하고 싶을 때
   - 전체 리빌드 대신 일부 화면(예: 그래프, 설정 화면, 프로필 카드 등)만 SwiftUI로 작성
2. **SwiftUI 전용 기능을 쓰고 싶을 때**
   - 예: iOS 16+에서 제공하는 **Swift Charts** 라이브러리를 UIKit 프로젝트에서 쓰고 싶을 때
3. 실험/프로토타입
   - 새로운 UI를 SwiftUI로 빠르게 만들어 보고, 기존 UIKit 앱 안에 꽂아 넣어 테스트할 때
4. 점진적 마이그레이션 전략
   - 앱의 특정 탭, 특정 플로우부터 SwiftUI로 천천히 바꾸고 싶을 때

---

## 관련 개념 비교

- **SwiftUI → UIKit**
  - `UIHostingController`: SwiftUI 뷰를 UIKit 컨트롤러로 감싸서 사용
- **UIKit → SwiftUI**
  - `UIViewRepresentable` / `UIViewControllerRepresentable`
  - UIKit의 `UIView` / `UIViewController`를 SwiftUI에서 재사용할 때 사용

이 둘을 잘 조합하면, UIKit · SwiftUI를 섞어서 점진적으로 구조를 변경할 수 있다.
