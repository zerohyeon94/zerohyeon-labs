## 큰 그림

- **상태(state)의 주인**이 누구냐가 핵심
  - UIKit이 상태를 들고 있고 SwiftUI는 뷰만 그리는 경우
  - SwiftUI가 상태를 들고 있고 UIKit은 그 결과를 구독하는 경우

---

## 1. UIKit → SwiftUI: 초기 값 전달

가장 단순한 방법: SwiftUI 뷰의 이니셜라이저 파라미터 사용

```swift
struct MyChartView: View {
    let title: String
    let values: [Double]

    var body: some View {
        VStack {
            Text(title)
                .font(.headline)
            Chart(values, id: \.self) { v in
                LineMark(
                    x: .value("Index", values.firstIndex(of: v) ?? 0),
                    y: .value("Value", v)
                )
            }
        }
    }
}
```

UIKit에서:

```swift
코드 복사
let swiftUIView = MyChartView(
    title: "오늘의 혼잡도",
    values: [1, 2, 3, 4, 5]
)

let hostingVC = UIHostingController(rootView: swiftUIView)
```

이 방식은 초기 값 전달에는 좋지만, 이후 값 업데이트에는 한계가 있다.
따라서 상태가 변하는 경우에는 ObservableObject를 활용하는 것이 일반적.

## 2. UIKit → SwiftUI: ObservableObject 사용
### 2-1. ViewModel 정의

```swift
import SwiftUI
import Combine

final class ChartViewModel: ObservableObject {
    @Published var title: String = "혼잡도 차트"
    @Published var values: [Double] = []
}
```

### 2-2. SwiftUI 뷰에서 사용

```swift
struct MyChartView: View {
    @ObservedObject var viewModel: ChartViewModel

    var body: some View {
        VStack {
            Text(viewModel.title)
                .font(.headline)

            Chart(viewModel.values, id: \.self) { v in
                LineMark(
                    x: .value("Index", viewModel.values.firstIndex(of: v) ?? 0),
                    y: .value("Value", v)
                )
            }
        }
        .padding()
    }
}
```

### 2-3. UIKit에서 ViewModel을 주입하고 갱신

```swift
final class DashboardViewController: UIViewController {

    private let chartViewModel = ChartViewModel()

    override func viewDidLoad() {
        super.viewDidLoad()

        chartViewModel.title = "오늘의 혼잡도"
        chartViewModel.values = [1, 4, 2, 6, 3]

        let swiftUIView = MyChartView(viewModel: chartViewModel)
        let hostingVC = UIHostingController(rootView: swiftUIView)

        // … child로 embed하는 코드 (생략)
    }

    private func updateChart() {
        // 예: 네트워크 응답 후 값 업데이트
        chartViewModel.values = [2, 3, 5, 1, 4]
    }
}
```

> @Published 프로퍼티가 바뀌면 SwiftUI 뷰가 자동으로 다시 렌더링된다.

## 3. SwiftUI → UIKit: 이벤트 전달 (콜백)
SwiftUI에서 UIKit 쪽으로 “이벤트”만 전달하고 싶을 때는 클로저 콜백 패턴이 가장 단순하다.

```swift
struct MyChartView: View {
    let onPointSelected: (Double) -> Void

    let values: [Double]

    var body: some View {
        Chart(values, id: \.self) { v in
            LineMark(
                x: .value("Index", values.firstIndex(of: v) ?? 0),
                y: .value("Value", v)
            )
            .onTapGesture {
                onPointSelected(v)
            }
        }
    }
}
```

UIKit:

```swift
let swiftUIView = MyChartView(
    onPointSelected: { [weak self] value in
        print("선택된 값: \(value)")
        // 여기서 UIKit 쪽 로직 수행 (예: 상세 화면 push 등)
    },
    values: [1, 2, 3, 4, 5]
)

let hostingVC = UIHostingController(rootView: swiftUIView)
```

## 4. 다른 커뮤니케이션 패턴
1. NotificationCenter
	- SwiftUI에서 Notification을 post → UIKit에서 observer로 받기
	- 프로젝트 전반에 브로드캐스팅할 때 사용
2. Combine의 Publisher/Subject
	- SwiftUI 쪽에서 PassthroughSubject 등에 값을 보내고
	- UIKit 쪽에서 구독하는 방식도 가능
3. 환경 값 (@Environment)
	- SwiftUI 내부 구조에서 공통 의존성 전달에 좋지만
	- UIKit과 직접 연결보다는 SwiftUI 내부용으로 이해하는 것이 편함

### 키 포인트 요약
- 상태(State)는 한 곳에서 관리하고, 나머지는 렌더링 + 이벤트 전달만 한다.
- UIKit이 상태의 주인:
	- ObservableObject + @ObservedObject
- SwiftUI가 이벤트를 UIKit에 알려야 할 때:
	- 클로저 콜백이 가장 단순하고 직관적이다.
