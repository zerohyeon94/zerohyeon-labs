## 1. Auto Layout & Safe Area

- `hostingController.view.translatesAutoresizingMaskIntoConstraints = false` 꼭 설정
- containerView의 네 변에 pin 하는 제약을 확실히 잡아야 레이아웃이 깨지지 않는다.
- safe area 안에 넣고 싶다면 `view.safeAreaLayoutGuide`를 기준으로 제약 설정

```swift
NSLayoutConstraint.activate([
    hostingVC.view.topAnchor.constraint(equalTo: container.topAnchor),
    hostingVC.view.leadingAnchor.constraint(equalTo: container.leadingAnchor),
    hostingVC.view.trailingAnchor.constraint(equalTo: container.trailingAnchor),
    hostingVC.view.bottomAnchor.constraint(equalTo: container.bottomAnchor)
])
```

## 2. 배경색 이슈

SwiftUI 뷰 자체에 .background(Color(UIColor.systemBackground)) 등을 지정하지 않으면

UIKit 뷰의 배경이 그대로 비칠 수 있음

특히 다크 모드 지원 시 SwiftUI 쪽에서도 적절한 background를 설정하는 것이 좋다.

예:

```swift
var body: some View {
    Chart(values, id: \.self) { v in
        // ...
    }
    .padding()
    .background(Color(UIColor.secondarySystemBackground))
    .clipShape(RoundedRectangle(cornerRadius: 12))
}
```

## 3. 네비게이션 구조 혼합 시 주의

- UIKit의 UINavigationController 위에 UIHostingController를 올리는 경우:
	- 제목/네비게이션 바는 UIKit에서 관리
	- SwiftUI의 NavigationStack은 내부 navigation 전용으로 사용하는 것을 권장
- 필요 이상으로 Navigation 구조를 이중으로 만들면 사용자 경험이 꼬일 수 있음

## 4. 성능 관점 메모

- SwiftUI + Charts는 내부적으로 레이아웃/드로잉을 알아서 최적화해주지만,
	- 실시간으로 매우 자주 데이터가 변하는 경우 (예: 오실로스코프 수준 애니메이션)에는
		- 차트 타입, 데이터 개수에 따라 성능을 꼭 체크해야 한다.
- UIHostingController 자체가 느린 건 아니지만:
	- SwiftUI 뷰를 아주 많이 동적으로 생성/파괴하는 구조는 비용이 커질 수 있다.

## 5. 다른 대안과 비교
### 5-1. DGCharts 같은 UIKit 기반 라이브러리 vs Swift Charts + UIHostingController
- DGCharts (or Charts 라이브러리)
	- UIKit/코어그래픽 기반
	- 오래된 프로젝트에도 쉽게 들어감
	- 완전 커스텀한 스타일/제어에 강점
- Swift Charts (iOS 16+) + UIHostingController
	- 선언형 문법으로 차트 정의 → 코드 가독성 높음
	- Apple 공식 제공이라 향후 기능 추가 기대
	- 차트 레이아웃과 상호작용을 SwiftUI 스타일로 쉽게 조합 가능
프로젝트 상황:
- iOS 16 이상 타깃 + UIKit 기반 → SwiftUI + UIHostingController로 차트만 SwiftUI로 만드는 전략이 현실적인 선택.

## 6. iOS 16+에서의 대안: UIHostingConfiguration

리스트/컬렉션 셀 안에 SwiftUI 뷰를 넣고 싶다면
UIHostingController 대신 UIHostingConfiguration이 더 간편할 수 있다. (iOS 16+)

예 (UITableViewCell):

```swift
cell.contentConfiguration = UIHostingConfiguration {
    MyChartView(values: [1, 2, 3])
}
```

- 장점
	- child view controller 관리 코드 불필요
	- 셀 안에 SwiftUI 뷰를 자연스럽게 넣을 수 있음
- 단점
	- iOS 16 이상에서만 사용 가능

## 7. 기억해 두면 좋은 패턴 요약
1. 전체 화면 SwiftUI로 만들기
	- let vc = UIHostingController(rootView: SwiftUIView())
	- navigationController?.pushViewController(vc, animated: true)
2. 부분 뷰로 embed
	- child view controller 패턴 (addChild / didMove(toParent:))
3. 데이터 바인딩
	- UIKit → SwiftUI: ObservableObject + @ObservedObject
	- SwiftUI → UIKit: 콜백 클로저
4. 버전 전략
	- iOS 16+ 타깃이면:
		- 리스트 셀에는 UIHostingConfiguration도 고려
	- 그 외:
		- UIHostingController가 기본 선택지
