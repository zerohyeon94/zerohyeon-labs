>[!Note] "선언형 데이터 바인딩 구조와의 부조화"의 의미
 앱의 데이터 흐름(Rx 등)은 데이터 변경 시 자동으로 반응하는 **선언형 구조**를 지향하고 있으나, DGCharts 라이브러리는 뷰를 업데이트하기 위해 개발자가 직접 데이터를 변환하고 갱신 메서드를 호출해야 하는 **명령형 구조**를 강제함. 이로 인해 데이터 스트림과 UI 업데이트 로직 사이에 불필요한 변환 코드(Boilerplate)가 늘어나고, 유지보수성이 떨어지는 충돌(부조화)이 발생했음.
### 1. 명령형(Imperative) UI vs 선언형(Declarative) UI

이해를 돕기 위해 화면에 차트를 그리는 두 가지 방식을 비교해 보겠습니다.

- **명령형 방식 (DGCharts, 기존 UIKit): "어떻게(How) 그릴 것인가"** 데이터가 변경될 때마다 개발자가 **하나하나 지시**를 내려야 합니다.
    
    1. 기존 차트 데이터를 지워라.
        
    2. 새로운 배열을 `BarChartDataEntry` 객체로 변환해라.
        
    3. 이걸 `DataSet`에 담아라.
        
    4. 차트 뷰에 데이터를 주입해라.
        
    5. `chart.notifyDataSetChanged()`를 호출해서 화면을 새로고침해라.
    
    > **문제점:** 코드가 길어지고, 개발자가 실수로 '새로고침' 호출을 빼먹으면 데이터와 화면이 일치하지 않는 버그가 발생합니다.
    
- **선언형 방식 (SwiftUI Charts): "무엇을(What) 그릴 것인가"**
    
    화면의 상태(State)와 UI를 묶어두고(Binding), **"이 데이터 배열을 바탕으로 바 차트를 그려라"**라고 선언만 해둡니다.
    
    1. 데이터가 변경된다.
        
    2. (끝) 알아서 차트가 새로운 데이터에 맞게 다시 그려짐.

### 2. "부조화"가 발생했던 이유 (문제의 핵심)

프로젝트에서 이미 Rx를 활용해 이벤트 스트림을 구축하고 상태를 관리하고 계십니다. 즉, 데이터의 흐름은 **"데이터가 바뀌면 알아서 흘러가는(반응형/선언형) 구조"**를 띄고 있습니다.

그런데 여기에 DGCharts를 붙이게 되면:

1. Rx를 통해 "데이터가 변경됨!" 이라는 깔끔한 신호가 들어왔는데,
    
2. 이걸 처리하기 위해 다시 `subscribe` 내부에서 DGCharts를 위한 길고 복잡한 **명령형 세팅 코드(데이터 변환, 뷰 갱신 지시 등)**를 주저리주저리 작성해야 합니다.
    
3. 이게 바로 **"부조화(Mismatch)"**입니다. 데이터는 현대적인 스트림으로 흐르는데, UI 업데이트는 과거의 수동 방식으로 억지로 끼워 맞춰야 해서 코드가 지저분해지고 유지보수가 어려워집니다.

### 3. SwiftUI Charts 도입이 가져온 "조화"

- SwiftUI는 태생부터 **선언형 데이터 바인딩**을 위해 만들어졌습니다.
    
- `UIHostingController`를 통해 SwiftUI Charts를 UIKit에 삽입하면, Rx나 상태 변수에서 전달된 데이터 배열만 SwiftUI View에 넘겨주면 됩니다.
    
- 차트를 업데이트하기 위한 복잡한 명령형 코드가 전부 사라지고, "데이터 = 차트 화면" 이라는 공식이 성립되어 **결합도가 낮아지고 코드가 훨씬 간결**해집니다.
