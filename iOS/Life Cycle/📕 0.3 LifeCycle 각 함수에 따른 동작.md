## 1. 코드로 UI(button 등)를 구현하는 위치

**질문:**

> button의 UI 구현은 viewDidLoad에서 하는 게 좋은지, viewWillAppear에서 하는 게 좋은지?

### **정답: 거의 항상 `viewDidLoad`에서 한다!**

- **이유**
    
    - `viewDidLoad`는 **뷰가 메모리에 올라온 직후 1회만 호출**
        
    - UI(버튼, 레이블, 오토레이아웃 등)는 한 번만 만들고, 화면에 보일 때마다 새로 만들 필요가 없음
        
    - `viewWillAppear`는 **화면이 보일 때마다** 여러 번 호출 → UI가 중복 생성될 위험

**예시**

```swift
override func viewDidLoad() {
    super.viewDidLoad()
    let button = UIButton()
    // button 설정, addSubview, 레이아웃 등
    self.view.addSubview(button)
}
```

**실무 Tip:**

- **UI 요소(뷰, 버튼, 레이블 등) 생성/초기화/레이아웃 → viewDidLoad**
    
- 단, UI “업데이트”나 “상태 변경”은 viewWillAppear, viewDidAppear에서 처리할 수도 있음

## 2. 타이머(start/stop) 시작 위치

**질문:**

> 화면 접근 시 타이머 동작은 viewDidLoad vs viewDidAppear?

### **정답: 일반적으로 `viewDidAppear`에서 시작!**

- **이유**
    
    - `viewDidLoad`: 뷰가 메모리에 올라왔지만, 실제로 화면에 **보이기 전** (ex: 네비게이션 push, 모달, 탭 이동 등 아직 안보임)
        
    - `viewDidAppear`: 뷰가 **화면에 완전히 보인 직후** (유저가 UI를 실제로 보기 시작)
        
    - 타이머가 “화면에 표시된 후” 동작해야 정확 (실제로 보여주는 시간 측정, 애니메이션 동기화 등)

**예시**

```swift
override func viewDidAppear(_ animated: Bool) {
    super.viewDidAppear(animated)
    startTimer()
}
```

**실무 Tip:**

- 타이머는 “실제 사용자 눈에 보이기 시작할 때” 동작해야 UX 오차가 없음
    
- 화면을 벗어날 때는 viewWillDisappear/viewDidDisappear에서 stop/pause

## 3. 타이머 → Label 값 변경 UI

**질문:**

> 타이머마다 label.text로 값 넣으면 바로 UI가 바뀌는가? 아니면 화면을 강제로 다시 그려야 하나?

### **정답: label.text만 변경하면 **자동으로 UI가 바로 갱신**됨!**

- **이유**
    
    - iOS의 UI(UIKit)는 메인스레드에서 label.text를 바꾸면 바로 화면에 반영
        
    - 별도로 화면을 리프레시하거나 재실행할 필요 없음
        

**예시**
```swift
timer = Timer.scheduledTimer(withTimeInterval: 1.0, repeats: true) { [weak self] _ in
    self?.seconds -= 1
    self?.label.text = String(format: "%02d:%02d", self?.seconds ?? 0 / 60, self?.seconds ?? 0 % 60)
}
```

**실무 Tip:**

- 반드시 **메인스레드**에서 label.text 변경!
    
    - (Timer는 기본적으로 메인스레드에서 실행됨, 비동기 처리할 땐 DispatchQueue.main.async 필요)

## 4. 화면 진입 시(예: 인증) API 요청 위치

**질문:**

> 인증 등으로 화면이동하자마자 API 요청은 어느 라이프사이클 함수에서?

### **정답: 일반적으로 `viewDidLoad` 또는 `viewWillAppear`!**

- **API 요청이 즉시/한번만 필요**  
    → **viewDidLoad**  
    (ex: “문자인증 화면 진입과 동시에, 인증번호 발송 API 호출”)
    
- **매번 필요(화면에 진입할 때마다 새로고침/재요청)**  
    → **viewWillAppear**  
    (ex: “화면을 다시 진입할 때마다 매번 인증번호 재전송”)

**실무 Tip:**

- 네트워크 로딩 애니메이션/로딩뷰는 viewDidLoad에서 바로 표시하는 경우 많음
    
- 특정 상황에 따라(특정 조건, 인증 실패 등) viewWillAppear에서 추가 요청 가능

## 5. UIViewController LifeCycle별 실무 활용 포인트

|함수|주로 하는 일|
|---|---|
|**init**|인스턴스 변수 초기화, DI, 주입|
|**loadView**|뷰 직접 코드로 만들 때만 오버라이드(거의 안 씀)|
|**viewDidLoad**||

- UI 생성/초기화
    
- 데이터/상수 초기 설정
    
- 네트워크 최초 요청
    
- RxSwift/Combine 바인딩
    
- delegate/dataSource 연결 |

| **viewWillAppear** |

- 매번 리프레시해야 하는 UI/데이터
    
- 네비게이션 바 스타일 조정
    
- 화면에 진입할 때마다 필요한 액션(상태 클린업 등) |

| **viewDidAppear** |

- 타이머/애니메이션 시작
    
- 사용자 분석/이벤트 트래킹
    
- 모달/트랜지션 등 화면 효과
    
- 오디오/비디오 재생 시작 |

| **viewWillDisappear** |

- 타이머/애니메이션 정지
    
- 입력 종료/저장
    
- 알림/콜백 등 화면 나갈 때 필요한 처리 |

| **viewDidDisappear** |

- 리소스 해제
    
- 화면 완전히 사라졌을 때 처리할 일 |

| **deinit** |

- 강한 참조/메모리릭 클린업
    
- Notification, KVO, RxSwift Dispose 등 안전한 해제 |
