# 1) 최초 진입 시 라이프사이클 흐름

(내비 push 기준, modal도 유사)

1. `init(viewModel:)`
    
2. `loadView()` → `view` 생성 (스토리보드면 nib 로드)
    
3. `viewDidLoad()`
    
    - 여기서 **Rx 바인딩**(버튼 탭 → ViewModel Input, ViewModel Output → UI) 설정
        
4. `viewWillAppear(_:)`
    
5. **레이아웃 사이클**
    
    - 필요 시 `viewWillLayoutSubviews` → `layoutSubviews` → `viewDidLayoutSubviews`
        
6. `viewDidAppear(_:)`

> 당신은 `viewWillAppear`/`viewDidAppear`를 오버라이드하지 않았어도 **시스템은 호출합니다.** 단지 코드가 없을 뿐.  
> **중요:** 여기까지는 “화면 준비” 단계일 뿐, 네트워크는 아직(버튼 안 눌렀다면) 안 갔겠죠.

# 2) 버튼 탭 → 네트워크 → UI 갱신의 타임라인

1. **사용자 탭 이벤트** (항상 메인 스레드)
    
    - `button.rx.tap`이 **메인 스레드에서** 이벤트 방출
        
2. **ViewModel로 Input 전달**
    
    - `flatMapLatest { runLoginFlow() }` 같은 흐름 시작
        
3. **네트워크 시작**
    
    - 권장: `subscribe(on: background)`로 **백그라운드 스레드**에서 시작
        
    - 실제 통신/파싱은 SDK/URLSession이 **비동기**로 수행
        
4. **결과 수신**
    
    - 권장: `.observe(on: MainScheduler.instance)`로 **이후 부수효과/상태 업데이트를 메인**에서 실행
        
    - 예) `loadingRelay.accept(false)`, `errorRelay.accept(...)`, `routeRelay.accept(...)`, `tableViewItems.accept(newItems)` 등
        
5. **UI 갱신**
    
    - Relay/Subject → ViewModel **Output(Driver/Signal)** → ViewController에서 `.drive/.emit`로 UI에 바인딩
        
    - 이때 UI 속성 변경(텍스트, isHidden, reload 등)이 일어나면, 필요 시 **레이아웃/디스플레이 사이클**이 한 번 더 돌 수 있어요.
        
    - (코어 애니메이션은 현재 RunLoop 턴 마지막에 트랜잭션 commit)
        

> **포인트:** **버튼 탭 자체만으로는 VC 라이프사이클 메서드가 다시 호출되지 않습니다.**  
> 단, 결과에 따라 **화면 전환**(push/present)이 일어나면 **그때**:
> 
> - 현재 VC: `viewWillDisappear` → 레이아웃 사이클 → `viewDidDisappear`
>     
> - 다음 VC: `viewWillAppear` → 레이아웃 사이클 → `viewDidAppear`

# 3) “화면 갱신” 방법 레벨별 정리

## (A) 값 바인딩(가장 Rx-스러운 방법)

- ViewModel Output을 **Driver/Signal**로 노출 → VC에서 `.drive/.emit`로 UI에 연결

```swift
// VM
let nickname: Driver<String>
let isLoading: Driver<Bool>
let error: Signal<String>
let items: Driver<[CellVM]>

// VC
output.nickname.drive(nicknameLabel.rx.text)
output.isLoading.drive(indicator.rx.isAnimating)
output.error.emit(onNext: showAlert)
output.items.drive(tableView.rx.items(cellIdentifier:...))
```
    
- 장점: **메인 스레드 보장**, 에러 전파 안 함(Driver/Signal), 중복 이벤트 억제 가능.

## (B) 명령형 갱신(필요 시)

- 특정 시점에 직접 변경:
    
```swift
label.text = "완료"
tableView.reloadData()
collectionView.performBatchUpdates { ... }
```
    
- 레이아웃 갱신이 필요하면:
    
```swift
view.setNeedsLayout()
view.layoutIfNeeded() // 즉시 레이아웃
```
    

## (C) 리스트 최적화

- **Diffable Data Source** 또는 RxDataSources로 **diff 기반** 업데이트 → 깜빡임/스크롤 점프 최소화.

## (D) 애니메이션 동반

- 값 바인딩 + `UIView.animate`:
    
```swift
UIView.animate(withDuration: 0.25) {
  self.view.layoutIfNeeded()
}
```
    
- 코어 애니메이션 변경은 **RunLoop 끝에서 커밋**됨(CATransaction).

## (E) 라우팅

- ViewModel에서 `route: Signal<LoginRoute>` 방출 → Coordinator/VC가 구독해서 `push/present` 수행.  
    이때에만 양 VC의 **appear/disappear** 라이프사이클이 다시 호출.

# 4) 생명주기와 Rx 구독 관리 (화면 이탈/취소 대비)

- **문제:** 네트워크가 끝나기 전에 사용자가 뒤로 가면?  
    → 결과가 도착해도 더 이상 UI를 만지면 안 됨.
    
- 해결책:
    
    1. **수명 기반 해제**
        
        ```swift
        observable
			  .takeUntil(rx.viewWillDisappear) // 화면 이탈 시 자동 종료
			  .observe(on: MainScheduler.instance)
			  .bind(to: ...)
        ```
        
        또는 `takeUntil(self.rx.deallocated)`
        
    2. **바인딩용 DisposeBag 교체**
        
        - `viewWillAppear`에서 `bindingBag = DisposeBag()`
            
        - `viewWillDisappear`에서 `bindingBag = DisposeBag()`

    3. **flatMapLatest**
        
        - 최신 요청만 유지, 이전 요청은 dispose.

# 5) 흔한 오해/FAQ

- **Q. 버튼 눌러서 데이터만 갱신했는데 왜 `viewWillAppear`가 안 불리죠?**  
    A. **화면 전환**이 있어야 불립니다. 같은 VC에서 뷰만 갱신하면 라이프사이클 메서드는 재호출되지 않습니다(대신 레이아웃/디스플레이 사이클은 돌 수 있음).
    
- **Q. 로딩 스피너가 깜빡여요.**  
    A. `flatMapLatest`로 이전 흐름을 끊을 때 `onDispose`에서 `loading=false` → 곧바로 새 흐름에서 `loading=true`가 올라가며 깜빡일 수 있습니다.  
    → `ActivityIndicator`(카운트 방식) 사용 또는 `distinctUntilChanged()` 적용.
    
- **Q. 네트워크 스레드는?**  
    A. `subscribe(on: Concurrent…Scheduler)`로 **시작을 백그라운드**로 강제, UI/상태 변경은 `.observe(on: Main)`.
