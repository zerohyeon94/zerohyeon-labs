`SetupViewModel.swift`는 지금 **화면 상태 및 유효성 검사**를 담당하고 있는데,  
여기서 말씀하신 **`SetupModel.swift`가 필요할지 여부**는 앱의 구조 확장성과 역할 분리에 따라 결정됩니다.

---

## ✅ ViewModel과 Model은 뭐가 다른가요?

| 역할           | ViewModel                               | Model                                                    |
| ------------ | --------------------------------------- | -------------------------------------------------------- |
| 💡 화면 상태와 로직 | View의 입력을 받고, 상태 관리, 유효성 검사             | 실제 저장/비즈니스에 사용할 구조체                                      |
| 💾 실제 데이터 구조 | ❌                                       | ✅ 예: `BudgetConfigModel`, `FixedCostModel` 등             |
| 💬 예시        | `viewModel.salary`, `viewModel.isValid` | `struct BudgetConfigModel { salary: Int, payday: Date }` |

---

## ✅ 결론: 지금 시점에서는 꼭 필요하지는 않지만, **확장성이나 테스트 목적이 있다면 도입을 추천**합니다.

---

## 👇 이런 경우 `SetupModel.swift`를 추가하는 게 좋습니다:

### ✅ 1. ViewModel과 CoreData 사이에서 변환용 데이터 구조가 필요할 때

```swift
struct BudgetConfigModel {
    var salary: Int
    var payday: Date
    var fixedCosts: [FixedCostModel]
}

struct FixedCostModel {
    var title: String
    var amount: Int
}
```

이걸 CoreData 저장 시 변환해서 사용:

```swift
CoreDataManager.shared.save(model: BudgetConfigModel)
```

---

### ✅ 2. ViewModel을 테스트하거나 Preview에 사용할 때

ViewModel에서 이런 식으로도 접근할 수 있습니다:

```swift
class SetupViewModel {
    var model = BudgetConfigModel(salary: 0, payday: Date(), fixedCosts: [])

    var isValid: Bool {
        return model.salary > 0
    }
}
```

|파일|역할|지금 필요한가?|

## ✅ 1. `SetupModel` 구조체 안에 `BudgetConfigModel`이 들어가야 하나요?

> 결론부터 말씀드리면: **필요 없습니다.**

`SetupModel`이라는 구조체를 따로 만들고 그 안에 `BudgetConfigModel`을 포함시키는 방식은 **오히려 구조를 불필요하게 복잡하게 만듭니다.**

---

### 🎯 이유:

- `BudgetConfigModel`은 앱 전체에서 재사용 가능한 “도메인 모델”입니다.
- `SetupModel`이라는 상위 모델 안에 포함시키면,  `Setup` 화면에서만 쓰는 모델처럼 **범위가 불필요하게 좁아집니다.**
- **ViewModel이 필요한 모델들을 조합**해서 쓰는 게 자연스럽습니다.

```swift
// ViewModel 내부에서 이렇게 관리하면 됩니다
var model = BudgetConfigModel(
    salary: 0,
    payday: Date(),
    fixedCosts: []
)
```

---

## ✅ 2. “Model”이라는 단어를 네이밍에 꼭 써야 하나요?

> 결론: **"기능에 따라 선택적으로 쓰는 것이 좋습니다."**

|네이밍|설명|예시|
|---|---|---|
|`BudgetConfigModel`|✅ 명확하고 중복 회피 가능|실체 데이터, CoreData ↔︎ 변환용|
|`BudgetConfig`|❌ CoreData Entity 이름과 중복될 수 있음|충돌 위험 있음|
|`FixedCostModel`|✅ OK (구조체와 CoreData entity 구분됨)|안전한 네이밍|

---

### 💡 실전 네이밍 전략

|상황|추천|
|---|---|
|**CoreData Entity 이름과 충돌할 가능성 있음**|`Model`, `DTO`, `ViewData` 등의 접미어 추가|
|**특정 화면 전용 데이터만 다룰 때**|`SetupModel`, `InputModel` 등의 **기능 기반 네이밍** 사용|
|**공용으로 사용하는 도메인 모델**|`BudgetConfigModel`, `UserModel` 등 **범용적 이름** 사용|

---

## ✅ 최종 정리

|항목|권장 방식|
|---|---|
|`SetupModel { BudgetConfigModel }`처럼 중첩|❌ 하지 마세요 (불필요한 추상화)|
|`BudgetConfigModel`, `FixedCostModel`|✅ 각자 독립 구조체로 사용|
|`Model` 접미어|✅ 상황에 따라 사용 (특히 CoreData 충돌 방지 시 유용)|
