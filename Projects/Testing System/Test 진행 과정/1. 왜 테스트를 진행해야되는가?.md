## **리드 개발자 입장에서 실무 피드백**

### 1) **내가 개발한 것에 대해 테스트를 진행해야 하는가?**

- **필수입니다.**  
    실무에서 테스트는 코드 품질을 보장하는 데 핵심적인 역할을 합니다.
    
    - 데이터 저장/조회/수정/삭제가 실제로 원하는 대로 동작하는지 검증
        
    - 변경(리팩토링, 기능 추가) 시 버그 발생 가능성 줄이기
        
    - 나와 팀원, 미래의 내가 신뢰하고 사용할 수 있는 코드가 됨

#### 실무 조언

- **모든 기능에 대해 100% 테스트를 작성할 필요는 없습니다.**  
    하지만, 데이터 무결성과 비즈니스 로직의 핵심 부분, 버그가 잦은 부분, 외부와 연결되는 부분(네트워크, DB 등)은 꼭 테스트하세요.
    
- **특히 CoreDataManager 같은 데이터 계층 로직**은 테스트를 해두면 리팩토링/추가 개발에 훨씬 유리합니다.

---

### 2) **테스트를 진행한다면 어떤 방식으로 하는가? (실무 흐름)**

#### 1. **유닛 테스트 (Unit Test)**

- **XCTest**로 CoreDataManager의 각 메서드를 독립적으로 테스트합니다.
    
    - 예) createSpendingRecord가 실제로 객체를 추가하는지
        
    - fetch, update, delete 동작을 각각 검증
        
- **CoreData의 인메모리 스토어(in-memory store)**를 사용해 실제 파일 DB를 건드리지 않고 테스트할 수 있습니다.

#### 2. **TDD (Test-Driven Development)**

- 실무에서는 TDD를 “엄격하게” 적용하는 곳은 많지 않지만,
    
    - **중요한 신규 기능**이나 **버그 픽스**에는 먼저 테스트를 만들고(실패), 그다음 구현, 그리고 성공 확인 순서로 진행하는 팀도 있습니다.
        
- **추천**:
    
    - 혼자 개발할 때는 “기존 코드 → 테스트 추가 → 버그 수정 또는 리팩토링” 순으로 연습해보세요.
        
    - 신규 기능 개발 시에는 테스트 케이스부터 한두 개 써보고, 코드를 작성하면서 동작을 검증하세요.

#### 3. **실무 개발자들의 테스트 흐름 예시**

1. **기능 명세**(요구사항) 작성
    
2. **테스트 코드** 먼저 작성 (필수 케이스만)
    
3. **기능 구현**
    
4. **테스트 돌리며** 실패/성공 확인
    
5. **코드 리뷰 시에도 테스트 코드를 같이 제출**
    
6. **CI(Continuous Integration) 환경에서 자동으로 테스트 실행**

#### 4. **참고 예시 (코드)**

```swift
// CoreDataManagerTests.swift
import XCTest
@testable import YourApp

class CoreDataManagerTests: XCTestCase {
    var sut: CoreDataManager!
    override func setUp() {
        super.setUp()
        sut = CoreDataManager(inMemory: true) // in-memory 스토어
    }
    override func tearDown() {
        sut = nil
        super.tearDown()
    }
    
    func testCreateSpendingRecord() {
        let model = SpendingInputModel(title: "Test", amount: 1000, date: Date())
        let result = sut.createSpendingRecord(model)
        XCTAssertTrue(result)
        
        let records = sut.fetchSpendingRecords(on: model.date)
        XCTAssertEqual(records.count, 1)
        XCTAssertEqual(records.first?.title, "Test")
    }
}
```

- 실제로 `inMemory: true`와 같이 인메모리 스토어를 쓰면, 앱 데이터와 분리되어 안전하게 테스트할 수 있습니다.

### **정리 및 주니어 개발자에게 조언**

- **단어 사용** 모두 적절합니다.
    
- **실무에서도 데이터 저장/조회/수정/삭제는 꼭 테스트합니다.**
    
- **최소한의 유닛 테스트부터 시작**하세요.  
    작은 성공을 반복하면 테스트에 대한 부담이 줄고, 자연스럽게 TDD, Mock 등도 접할 수 있습니다.
    
- **테스트는 “내가 만든 코드에 자신감을 주는 안전망”입니다.**
    
- 팀에서는 “테스트 코드 없는 PR은 받지 않는다”라는 문화도 있으니, 습관적으로 연습해두면 성장에 매우 좋습니다.
