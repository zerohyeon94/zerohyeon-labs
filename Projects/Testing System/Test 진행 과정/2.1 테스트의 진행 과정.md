## **리드 개발자 입장에서 단계별 가이드 및 주니어 피드백**

### 1) **Unit Test 작성 흐름 (실무 기준)**

1. **테스트 타겟 추가 및 파일 생성**
    
    - Xcode 프로젝트에서 “File > New > Target...”을 눌러 _Unit Testing Bundle_을 추가 (이미 있다면 생략).
        
    - CoreDataManagerTests.swift처럼 별도의 테스트 파일을 생성.
        
2. **테스트 환경(인메모리) CoreDataManager 준비**
    
    - 실제 앱 DB를 망치지 않기 위해, **테스트 전용 in-memory store**를 사용해야 함.
        
    - 실무에서는 `init(inMemory: Bool)`을 추가해서 인메모리와 실제 파일 분리.
        
3. **테스트 코드 구조화**
    
    - 각 CRUD 메서드별로 테스트 메서드 작성.
        
    - 테스트 데이터(모델 등) 미리 생성.
        
4. **setUp/tearDown 메서드로 초기화 및 정리**
    
    - 각 테스트 시작 전/후, 항상 깨끗한 환경이 되게 관리.
        
5. **실제 메서드 호출 → 기대 결과 검증**
    
    - 예: 생성 → 정상적으로 fetch 됨을 XCTAssert로 확인
        
6. **의미 있는 이름의 테스트 케이스 작성**
    
    - ex) `testCreateBudgetConfig_success()`, `testDeleteFixedCost_removesEntity()`

### 2) **실무에서 사용하는 순서 예시 (코드 포함)**

#### (1) **CoreDataManager를 인메모리로 쓸 수 있게 보완**

- 실무에서 많이 쓰는 패턴:
    
```swift
final class CoreDataManager {
    static let shared = CoreDataManager()
    let persistentContainer: NSPersistentContainer

    // 신규 생성자 추가!
    init(inMemory: Bool = false) {
        persistentContainer = NSPersistentContainer(name: "GagaeSsi")
        if inMemory {
            persistentContainer.persistentStoreDescriptions.first?.url = URL(fileURLWithPath: "/dev/null")
        }
        persistentContainer.loadPersistentStores { (desc, error) in
            if let error = error {
                fatalError("Core Data store failed: \(error)")
            }
        }
    }
    ...
}
```

- 테스트에서만 `CoreDataManager(inMemory: true)`로 생성하면 테스트용 DB가 만들어짐.

---

#### (2) **테스트 클래스 기본 구조**

```swift
import XCTest
@testable import GagaeSsi

final class CoreDataManagerTests: XCTestCase {
    var sut: CoreDataManager! // System Under Test

    override func setUpWithError() throws {
        sut = CoreDataManager(inMemory: true)
        sut.resetAllData() // 매 테스트 전 DB 깨끗하게 초기화
    }

    override func tearDownWithError() throws {
        sut = nil
    }
    
    func testCreateBudgetConfig() {
        let model = BudgetConfigModel(salary: 5000000, payday: 25, fixedCosts: [])
        let result = sut.createBudgetConfig(from: model)
        XCTAssertTrue(result)
        
        let config = sut.fetchBudgetConfig()
        XCTAssertNotNil(config)
        XCTAssertEqual(config?.salary, 5000000)
    }
    
    // 이 방식으로 각 CRUD 메서드별로 테스트 케이스 작성!
}
```

### 3) **실제 테스트 진행 순서 (요약)**

1. **CoreDataManager에 inMemory 옵션 생성**
    
2. **테스트 파일(예: CoreDataManagerTests.swift) 생성**
    
3. **setUp/tearDown에서 sut(CoreDataManager) 인스턴스 생성 및 초기화**
    
4. **각 CRUD 메서드에 대해**
    
    - ① 테스트 모델 준비
        
    - ② 메서드 호출(예: create, update, delete)
        
    - ③ 결과 검증(XCTAssert 계열 함수로)
        
5. **실행: Command+U 또는 Product > Test**
    
6. **결과 확인 및 실패/성공 분석**
    
7. **의미 있는 이름/주석/예상 실패 조건(엣지 케이스)도 테스트**

## **주니어 개발자에게 추가 팁**

- **작게, 자주 커밋**: 테스트 케이스 추가할 때마다 커밋하면 추적이 용이.
    
- **실패하는 테스트도 일부러 만들어보기**: 실수로 잘못 구현한 경우 테스트가 실패하는지 경험.
    
- **엣지 케이스 신경쓰기**: 예를 들어, 없는 id로 삭제 요청, nil/empty 데이터 등.
    
- **모델(예: BudgetConfigModel)도 테스트 대상**: entity/model 변환에서 데이터 유실 없는지 검증.
    
- **테스트 코드도 “동작하는 코드”와 동일하게 리뷰/관리**: 실무에선 테스트 없는 기능은 리뷰가 안되는 경우도 많음.
