## 1. **Error 메시지의 원인과 CoreDataManager.swift에서의 수정법**

### 🔶 **에러 원인**
`
```swift
CoreData: warning: Multiple NSEntityDescriptions claim the NSManagedObject subclass 'BudgetConfig' ...
```


- **중복 선언**:  
    NSPersistentContainer가 “BudgetConfig” 엔티티를 중복해서 로드하고 있습니다.
    
    - **원인 1:** 테스트 타겟과 앱 타겟 양쪽에 `.xcdatamodeld`나 `BudgetConfig.swift` 등 NSManagedObject subclass가 _중복_ 포함됨
        
    - **원인 2:** CoreDataManager의 초기화에서 NSPersistentContainer(name: "GagaeSsi")가 동일 모델을 여러 번 로드하거나,  
        `inMemory` store를 적용하지 않아서 테스트 환경/앱 환경이 충돌

---

### 🔶 **실무에서 자주 쓰는 해결법**

#### 1) **CoreDataManager의 inMemory 지원**

- 테스트와 실제 앱 실행을 분리할 수 있게 생성자 추가

```swift
final class CoreDataManager {
    static let shared = CoreDataManager()
    let persistentContainer: NSPersistentContainer

    // ✅ inMemory 옵션 추가
    init(inMemory: Bool = false) {
        persistentContainer = NSPersistentContainer(name: "GagaeSsi")
        if inMemory {
            persistentContainer.persistentStoreDescriptions.first?.url = URL(fileURLWithPath: "/dev/null")
        }
        persistentContainer.loadPersistentStores { (desc, error) in
            if let error = error {
                fatalError("Core Data store failed: \(error)")
            }
        }
    }

    // ... 이하 동일
}
```

#### 2) **타겟 멤버십 확인**

- Xcode에서 `BudgetConfig.swift` (NSManagedObject subclass)가 **앱 타겟**과 **테스트 타겟**에 _동시에_ 포함되어 있다면
    
    - **테스트 타겟의 멤버십을 해제**
        
    - `xcdatamodeld` 파일도 **테스트 타겟에만 추가**하면 안됨.
        
    - _보통, 앱 타겟만 체크되어 있으면 정상_

#### 3) **테스트 타겟의 Build Phases > Compile Sources 확인**

- `BudgetConfig.swift` 등 CoreData 엔티티 파일이 두 번 추가되어 있지 않은지 확인

---

#### **(요약)**

- **코드 레벨**에서는 inMemory 지원
    
- **프로젝트 세팅**에서는 타겟 멤버십과 빌드 소스 중복 체크

## 2. **BudgetConfig 관련 함수별 테스트 필요성과 실무 피드백**

### **각 함수 테스트가 필요한가?**

**YES.**  
실무에서는 “각 CRUD(생성, 조회, 수정, 삭제)”에 대해 최소 1개씩 테스트가 작성되어야 합니다.

- **createBudgetConfig**: 잘 생성되는지
    
- **fetchBudgetConfigEntity/fetchBudgetConfig**: 저장한 데이터를 올바르게 가져오는지
    
- **updateBudgetConfig**: 업데이트한 값이 실제로 반영되는지
    

#### 실무자 팁

- “중복 생성 시 어떻게 되는가?”, “업데이트 시 없는 데이터를 수정하면?”, “조회 시 데이터가 없으면?” 등 _엣지 케이스_도 테스트해야 신뢰할 수 있음

## 3. **핵심 요약 & 주니어 개발자 피드백**

- **단어 사용, 문제 인식, 질문 방향 모두 적절합니다.**
    
- **에러 메시지는 프로젝트 세팅과 CoreDataManager에 inMemory store 지원, 타겟 멤버십 중복 문제를 반드시 점검해야 함.**
    
- **CRUD 모든 함수에 대해 테스트 코드 작성은 실무에서도 필수이며, 다양한 케이스(성공, 실패, 예외)로 늘리는 습관이 좋음**
    
- **테스트 작성 후, 에러 경고가 뜬다면 무시하지 말고 원인을 바로 해결하는 습관을 갖길 추천합니다.**
