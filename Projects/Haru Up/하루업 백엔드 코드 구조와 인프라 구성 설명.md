## 이 노트의 목적

하루업 백엔드를 처음 보는 사람이 “어떤 코드가 어떤 역할을 하는지”, “Docker Compose에 있는 서비스들이 왜 필요한지”, “AWS 또는 Railway에 배포할 때 무엇을 최소로 올려야 하는지”를 이해하기 위한 정리다.

한 줄로 요약하면 하루업 백엔드는 **Spring Boot API 서버 + PostgreSQL/pgvector + Redis + 외부 AI/Push API + 선택적 모니터링 도구**로 구성되어 있다.

## 코드 계층 구조

하루업 백엔드는 대체로 아래 흐름을 따른다.

```text
Controller
  -> UseCase
    -> Service
      -> Repository / External API Client
        -> Domain Entity / DB / Redis / Clova / Firebase
```

각 계층의 의미:

| 계층 | 역할 | 예시 |
|---|---|---|
| Controller | HTTP 요청을 받는다. URL, request body, 인증 사용자 정보를 받아 UseCase/Service로 넘긴다. | `MemberAuthController`, `MemberMissionController`, `InterestController` |
| UseCase | 사용자 행동 단위의 업무 흐름을 묶는다. 여러 Service를 조합한다. | `MemberAuthUseCase`, `MemberMissionUseCase`, `MemberCurationUseCase` |
| Service | 실제 비즈니스 로직을 처리한다. 검증, 계산, 저장, 외부 API 호출을 담당한다. | `MemberService`, `MissionRecommendService`, `HybridInterestRecommendationService` |
| Repository | DB 조회/저장을 담당한다. JPA Repository 또는 QueryDSL 구현체가 있다. | `MemberRepository`, `MemberMissionRepository`, `PgVectorInterestRepository` |
| Domain Entity | DB 테이블과 연결되는 핵심 데이터 모델이다. | `Member`, `MemberMissionEntity`, `InterestEmbeddingEntity` |
| DTO | API 요청/응답 또는 내부 전달용 데이터 구조다. | `MemberDto`, `InterestRecommendationResponse`, `MemberMissionDto` |
| Config | 외부 시스템이나 Spring Bean 설정을 담당한다. | `SecurityConfig`, `RedisConfig`, `FirebaseConfig`, `ClovaApiConfig` |

이 구조의 장점은 Controller가 얇고, 실제 로직이 UseCase/Service에 모이는 것이다. 단, 현재 일부 Controller는 엔드포인트가 많고 로직 흐름이 길어 보이므로 운영 전에는 핵심 API부터 테스트 케이스를 잡는 것이 좋다.

## 실행 시작점

### `HaruUpApplication.kt`

Spring Boot 앱의 시작점이다.

- `@SpringBootApplication`: Spring Boot 앱으로 실행
- `@EnableScheduling`: `@Scheduled`가 붙은 배치/스케줄러 실행 허용

즉 서버가 켜지면 REST API뿐 아니라 자정 미션 생성, 랭킹 배치, 알림 배치 같은 스케줄 작업도 같이 동작할 수 있다.

### `ServletInitializer.kt`

WAR 배포를 위한 보조 진입점이다. 현재 Gradle에서 `war` 플러그인을 사용하고 있고 Dockerfile도 `bootWar`로 WAR를 만든다.

## 데이터 저장소

### PostgreSQL

PostgreSQL은 서비스의 기본 DB다.

저장하는 데이터 예시:

- 회원 정보
- 프로필/설정
- 목표
- 미션
- 커스텀 미션
- 관심사
- 랭킹
- 알림 기기 토큰
- 파일 메타데이터
- refresh token 일부

현재 `application.yml`은 다음 환경변수를 사용한다.

```text
POSTGRES_HOST
POSTGRES_PORT
POSTGRES_DB
POSTGRES_USER
POSTGRES_PASSWORD
```

중요한 점:

- `ddl-auto: none`이라 Spring이 테이블을 자동 생성하지 않는다.
- 배포 전 schema 생성 SQL 또는 migration 도구가 필요하다.
- pgvector 확장이 필요한 기능이 있으므로 `CREATE EXTENSION IF NOT EXISTS vector;`가 필요하다.

### pgvector

pgvector는 PostgreSQL 안에서 벡터 데이터를 저장하고 유사도 검색을 할 수 있게 해주는 확장이다.

쉽게 말하면:

```text
"운동 습관 만들기" -> [0.12, -0.31, 0.77, ...]
"아침 러닝 시작" -> [0.10, -0.28, 0.80, ...]
```

이런 숫자 배열끼리 얼마나 비슷한지 비교해서 추천에 활용한다.

하루업에서는 관심사 추천, 미션 추천, 임베딩 기반 검색에 연결되는 중요한 기능이다.

### Redis

Redis는 빠른 임시 저장소다.

PostgreSQL과의 차이:

| 구분 | PostgreSQL | Redis |
|---|---|---|
| 목적 | 오래 보관할 데이터 | 빠르게 읽고 만료되는 데이터 |
| 예시 | 회원, 미션, 랭킹 | refresh token, cache, rate limit |
| 저장 방식 | 디스크 중심 DB | 메모리 중심 저장소 |
| 장점 | 안정적 영속 저장 | 빠름, 만료 시간 설정 쉬움 |

현재 코드에서는 인증, 캐시, rate limit 쪽에 Redis가 연결된다. Redis 없이 배포하면 로그인 유지나 일부 제한 기능에서 문제가 날 수 있다.

Clova Studio는 Naver의 AI API다.

하루업에서 하는 일:

- 큐레이션 챗봇 질문/답변 생성
- 관심사 추천
- 미션 추천
- 텍스트 임베딩 생성

필요 환경변수:

CLOVA_API_KEY
```

### Firebase Admin SDK

Firebase는 앱 푸시 알림 전송에 사용된다.

하루업에서 하는 일:

- 앱에서 전달받은 push token 저장
- 특정 회원 또는 테스트 대상에게 push notification 발송
- 미션 리마인더/알림 배치와 연결

관련 파일:

FIREBASE_SERVICE_ACCOUNT_JSON
```

## 보안과 인증

### Spring Security

`SecurityConfig`에서 인증 없이 접근 가능한 API와 인증이 필요한 API를 나눈다.

현재 permitAll에 포함된 대표 경로:

- `/api/member/auth/**`
- `/swagger-ui/**`
- `/v3/api-docs/**`
- `/actuator/prometheus/**`
- `/health`
- `/members/statistics/**`

나머지 요청은 기본적으로 인증이 필요하다.

### JWT

JWT는 로그인 후 서버가 발급하는 토큰이다. 앱은 이후 요청마다 이 토큰을 함께 보내고, 서버는 토큰을 검증해 “누가 요청했는지”를 판단한다.

관련 파일:

- `JwtTokenProvider`
- `JwtAuthenticationFilter`
- `MemberPrincipal`
- `SecurityUtils`

필요 환경변수:

```text
JWT_SECRET_KEY
JWT_ISSUER
```

### CORS

CORS는 “어떤 프론트엔드/앱 origin에서 이 API를 호출할 수 있는지”를 정하는 설정이다.

현재는 localhost 중심으로 열려 있다.

```text
http://127.0.0.1:5500
http://localhost:5500
```

Railway/AWS 배포 후에는 앱 테스트 도메인 또는 실제 클라이언트 origin을 반영해야 한다.

## 배치와 스케줄러

서버가 켜져 있으면 API 요청이 없어도 정해진 시간에 실행되는 작업들이 있다.

| 파일 | 실행 시점 | 역할 |
|---|---|---|
| `DailyMissionGenerationBatch` | 매일 00:00 | 활성 목표를 기반으로 일일 미션 생성 |
| `RankingBatchScheduler` | 기본 매일 00:00 | 전날 기준 랭킹 집계 |
| `NotificationPushScheduler` | 매일 19:00 | 미션 리마인더 푸시 발송 |

주의할 점:

- 서버를 여러 대 띄우면 같은 배치가 중복 실행될 수 있다.
- 운영에서는 배치 서버를 분리하거나, 한 인스턴스에서만 실행되도록 제어해야 한다.
- staging/production에서 배치 활성 여부를 환경변수로 분리하는 것이 좋다.

## Docker Compose 구성 설명

현재 `docker-compose.yml`에는 아래 서비스가 들어 있다.

| 서비스 | 무엇인가 | 쉽게 말하면 |
|---|---|---|
| `app` | Spring Boot 앱 | 실제 하루업 API 서버 |
| `app-qa` | Spring Boot 앱의 QA 복제본 | QA용 API 서버 |
| `postgres` | PostgreSQL + pgvector | 서비스 데이터 저장소 |
| `redis` | Redis | 빠른 임시 저장소 |
| `redis-qa` | QA용 Redis | QA 서버 전용 Redis |
| `prometheus` | 메트릭 수집기 | 서버 지표를 주기적으로 긁어가는 도구 |
| `grafana` | 대시보드 | 지표/로그를 화면으로 보는 도구 |
| `loki` | 로그 저장소 | 애플리케이션 로그를 저장하고 검색하는 도구 |

## Prometheus, Grafana, Loki 쉽게 이해하기

### Prometheus

Prometheus는 **숫자 지표를 주기적으로 수집하는 도구**다.

예시:

- API 요청 수
- 응답 시간
- JVM 메모리 사용량
- HTTP 상태 코드 개수
- DB 커넥션 상태
- batch 실행 관련 지표

하루업 설정:

```text
Prometheus -> /actuator/prometheus -> Spring Boot app
```

`docker/prometheus/prometheus.yml`에서 15초마다 `/actuator/prometheus`를 읽도록 되어 있다.

한 줄 요약:

```text
Prometheus = 서버의 숫자 상태를 계속 수집하는 기록장
```

### Grafana

Grafana는 **수집된 지표와 로그를 대시보드로 보여주는 도구**다.

Prometheus만 있으면 숫자 데이터는 쌓이지만 보기 어렵다. Grafana를 붙이면 그래프, 표, 패널로 볼 수 있다.

예시:

- 오늘 API 요청이 몇 번 왔는지
- 평균 응답 시간이 얼마나 되는지
- 특정 시간대에 에러가 늘었는지
- batch 로그가 언제 발생했는지

하루업 설정:

```text
Grafana
  -> Prometheus 데이터 조회
  -> Loki 로그 조회
```

한 줄 요약:

```text
Grafana = Prometheus/Loki 데이터를 사람이 보기 쉽게 보여주는 화면
```

### Loki

Loki는 **로그를 저장하고 검색하는 도구**다.

로그 예시:

```text
미션 생성 시작
Clova API 호출 실패
JWT token invalid
랭킹 배치 완료
```

하루업의 `logback-spring.xml`은 prod가 아닌 환경에서 Loki appender를 켜도록 되어 있다. 즉 로컬/QA 환경에서는 애플리케이션 로그를 Loki로 보낼 수 있다.

한 줄 요약:

```text
Loki = 서버 로그를 모아서 나중에 검색할 수 있게 해주는 저장소
```

### 세 도구의 관계

```text
Spring Boot app
  ├─ 숫자 지표 -> Actuator -> Prometheus -> Grafana
  └─ 로그     -> Logback  -> Loki       -> Grafana
```

비유하면:

- Prometheus: 체온, 심박수, 걸음 수 같은 숫자를 기록
- Loki: 오늘 어떤 일이 있었는지 문장으로 기록
- Grafana: 숫자와 문장을 화면에서 확인

## Railway 배포 관점

Railway에서는 Docker Compose 전체를 그대로 올리기보다 서비스를 나누는 것이 자연스럽다.

권장:

```text
backend-app
postgres
redis
```

초기에는 Prometheus/Grafana/Loki를 Railway에 같이 올리지 않아도 된다. Railway 자체 로그와 `/health` 확인으로 먼저 배포를 안정화하는 편이 좋다.

Railway에서 주의할 점:

- `docker/app/Dockerfile` 경로 지정 필요
- `SERVER_PORT=${{ PORT }}` 설정 필요
- PostgreSQL에서 pgvector 사용 가능 여부 확인
- Redis 연결 변수 확인
- Firebase/Clova/JWT 환경변수 주입
- 파일 업로드는 Volume 또는 기능 범위 제한 필요

## AWS 배포 관점

AWS에 최소로 올릴 경우 필요한 것은 아래 정도다.

```text
app + postgres + redis + reverse-proxy
```

| 구성 | AWS 최소 배포 포함 여부 | 이유 |
|---|---|---|
| `app` | 포함 | 실제 API 서버 |
| `postgres` | 포함 | 영속 데이터 저장소 |
| `redis` | 포함 | refresh token/cache/rate limit |
| `reverse-proxy` | 포함 권장 | HTTPS와 도메인 처리 |
| `app-qa` | 제외 | 같은 서버에 QA까지 올리면 복잡도/메모리 증가 |
| `redis-qa` | 제외 | app-qa 제외 시 필요 없음 |
| `prometheus` | 제외 가능 | 초반에는 로그와 healthcheck로 충분 |
| `grafana` | 제외 가능 | 운영 안정화 후 추가 |
| `loki` | 제외 가능 | 운영 안정화 후 추가 |

최소 서버 크기:

- Lightsail 2GB: 테스트/초기 운영 최소선
- Lightsail/EC2 4GB: 운영 초반 권장
- 8GB 이상: 모니터링까지 같은 서버에 올릴 때 여유
