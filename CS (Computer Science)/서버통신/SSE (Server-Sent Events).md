## 1. 정의

> SSE는 클라이언트(브라우저나 앱)가 **한 번 서버에 연결을 열어두고**, 서버가 그 연결을 통해 **일방적으로 여러 번 데이터를 흘려보내는 방식**의 통신입니다.  HTTP 위에서 동작하는 **단방향 스트리밍**이라고 보면 됩니다.

- 클라이언트 → 서버 :  “이 URL로 이벤트 생기면 계속 보내줘요” 하고 **한 번 요청**
    
- 서버 → 클라이언트 : 이벤트가 생길 때마다 `data: ...\n\n` 형식으로 **계속 푸시**

응답의 Content-Type

```http
Content-Type: text/event-stream
```

응답 바디

```text
event: step
data: {"currentStep":1}

event: step
data: {"currentStep":2}

event: completed
data: {"currentStep":4}
```

클라이언트는 이 스트림을 계속 읽으면서 이벤트를 처리

---

## 2. SSE의 핵심 특징

1. **단방향(서버 → 클라이언트)**
    
    - 클라이언트는 HTTP 요청 한 번만 보냄
        
    - 이후에는 **서버만** 계속 이벤트를 밀어줌
        
    - 쌍방 통신(WebSocket)과 다름
        
2. **HTTP 기반**
    
    - 별도 프로토콜이 아니라 기존 HTTP 위에서 동작
        
    - 보통 `GET` 요청 + `text/event-stream` 응답
        
3. **연결 유지 (long-lived connection)**
    
    - 서버가 응답을 “끝내지 않고” 상태가 바뀔 때마다 계속 `data:` 덩어리를 추가로 보냄
        
    - 클라이언트는 같은 HTTP 응답을 계속 읽으면서 이벤트를 소비

---

## 3. SSE 쓸 때 장점

### ✅ 1) 이벤트 기반 “푸시”에 가깝다

- 폴링처럼 계속 “지금 어때요?” 안 물어봐도 됨
    
- 상태가 바뀌는 순간 서버에서 바로 이벤트를 밀어줄 수 있음 → “실시간에 가깝게” 상태 반영 가능

### ✅ 2) 네트워크/서버 리소스 효율 (폴링 대비)

- 폴링: 상태가 안 바뀌어도 계속 요청/응답 반복
    
- SSE: 상태가 바뀔 때에만 데이터 전송
    
- 특히 업데이트 빈도가 낮을수록 SSE가 효율적

### ✅ 3) HTTP 위라서 웹 환경에서는 비교적 쉽게 사용

- 브라우저는 `EventSource` 라는 기본 API 제공

```js
const es = new EventSource("/events");
es.onmessage = (event) => {
  console.log("data", event.data);
};
```

- 프록시/로드밸런서/방화벽도 “일반 HTTP 트래픽”으로 볼 확률이 높음

## 4. SSE 쓸 때 고려해야 하는 점 (중요 포인트들)

### 4-1. 클라이언트(iOS 앱) 측 고려사항

1. **iOS에는 기본 EventSource가 없다**
    
    - 브라우저처럼 `EventSource` 객체가 없음
        
    - `URLSession` + `URLSessionDataDelegate` 를 사용해서
        
        - 응답 스트림을 계속 읽고,`\n\n` 단위로 잘라서 `event:` / `data:` 파싱하는 로직을 직접 구현해야 함
            
    - 혹은 서드파티 라이브러리 사용
        
2. **연결 끊김 처리 & 재연결**
    
    - 모바일은
        
        - 네트워크 전환(Wi-Fi ↔ LTE), 앱 백그라운드 진입, OS가 프로세스 kill 등 상황 변수가 많음
            
    - SSE 연결이 끊어지면:
        
        - **자동 재연결 로직** 필요
            
        - 어디까지 이벤트를 받았는지(마지막 event id) 저장해두고 이어받을지 정책 필요
            
3. **백그라운드 동작**
    
    - 앱이 백그라운드로 가면 긴 연결을 유지하기 어렵고, iOS가 네트워크 활동을 제한함
        
    - 분석 진행 화면처럼 **“사용자가 보고 있는 동안만 실시간 업데이트”** 라면 괜찮지만, 앱이 내려가도 계속 받고 싶다면 푸시/백그라운드 작업 등 다른 설계 필요

---

### 4-2. 서버/인프라 측 고려사항

1. **서버 구현**
    
    - 응답 헤더:
        
        `Content-Type: text/event-stream Cache-Control: no-cache Connection: keep-alive`
        
    - 응답을 끝내지 않고:
        
        - 상태 변화 시마다 `data: ...\n\n` 형식으로 쓰기
            
    - 서버 프레임워크가 SSE를 별도로 지원하는지 확인 필요 (예: Spring, Node 등)
        
2. **커넥션 유지 & 타임아웃**
    
    - 한 유저가 SSE 연결 하나를 계속 붙잡고 있음 → 유저 수가 많아지면 동시 연결 수가 크고, 서버 자원 사용량도 커짐
        
    - 프록시/로드밸런서/게이트웨이 레벨에서:
        
        - idle timeout이 너무 짧으면 연결이 자주 끊어짐
            
        - keep-alive 정책, 버퍼링 끄기 등 세팅 필요
            
3. **재연결 프로토콜**
    
    - 서버 입장에서도:
        
        - 유저가 다시 연결했을 때 새로 시작할지 특정 jobId/status부터 이어서 보낼지
                
        - 필요하다면 `id:` 필드 사용해서 “최근 이벤트 id” 기반 재전송 설계 가능

---

### 4-3. 기획/비즈니스 측 고려사항

1. **실시간성이 얼마나 중요한가?**
    
    - 예: 1초 이내로 반영해야 하는 채팅, 주식 가격 → SSE/WebSocket이 유리
        
    - 예: 5~10초 정도 늦어도 괜찮은 진행률 표시 → 폴링도 충분히 현실적인 선택
        
2. **처음부터 SSE를 도입할 만큼의 가치가 있는지**
    
    - 기능 규모가 크지 않고, 단계도 4개 정도이며, 유저 수도 많지 않다면:
        
        - 개발 속도·복잡도까지 고려하면 V1에는 폴링 나중에 실시간 기능 많이 늘어나면 SSE/WebSocket 도입 같은 전략도 합리적

---

## 5. 한 줄로 요약하면

> **SSE(Server-Sent Events)** 는 클라이언트가 한 번 서버에 연결을 맺은 뒤, 서버가 그 연결을 통해 이벤트를 계속 푸시하는 HTTP 기반 단방향 스트리밍 방식입니다.
> 
> 실시간성 & 요청 효율성 면에서 장점이 있지만, 모바일 환경에서는 **연결 끊김·재연결·백그라운드·인프라 설정** 같은 부분을 반드시 함께 고려해야 합니다.
