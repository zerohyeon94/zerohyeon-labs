## ✅ 상황 ①: `Hashable`을 사용해서 `Set`, `Dictionary` 등에 담는 경우

### 🔹 예시

```swift
struct User: Codable, Hashable {
    let id: Int
    let name: String
}
```

이 경우:

- `User`는 서버로부터 JSON으로 받아오는 **모델 구조체**이고
    
- `Hashable`을 채택하면 아래처럼 사용 가능합니다:

```swift
var uniqueUsers = Set<User>()
uniqueUsers.insert(user1)
```

✅ 즉, 서버에서 받은 JSON을 디코딩한 후  
**중복 제거를 위해 Set에 넣거나**,  
**Dictionary의 Key로 사용하려면 반드시 Hashable**이 필요합니다.

---

## ✅ 상황 ②: SwiftUI 또는 DiffableDataSource에서 Hashable 요구

특히 iOS 개발에서 `Hashable`이 필요한 케이스는 다음과 같습니다:

|상황|이유|
|---|---|
|**SwiftUI** `List`|식별을 위해 `Identifiable` 또는 `Hashable` 요구|
|**UICollectionViewDiffableDataSource**|셀 구분을 위해 `Hashable` 타입 필요|
|**Set, Dictionary Key**|중복 제거 또는 빠른 탐색을 위해 필요|

### 예시: DiffableDataSource

```swift
struct Message: Codable, Hashable {
    let id: Int
    let content: String
}
```

> 이때 `Hashable`은 UI 업데이트를 빠르게 하기 위해 필요한 조건입니다.

---

## ✅ 왜 `Codable` + `Hashable`을 함께 쓰는 경우가 많을까요?

|역할|프로토콜|설명|
|---|---|---|
|네트워크로부터 JSON을 받기 위한 기능|`Codable`|JSON ↔ 모델 자동 변환|
|컬렉션(Set, Dictionary 등)에서 사용하거나, SwiftUI에서 ID 추적|`Hashable`|고유 식별값 부여 & 비교 가능|

> ✅ 그래서 요즘은 대부분의 모델이 아래처럼 작성됩니다:

```swift
struct Product: Codable, Hashable {
    let id: Int
    let name: String
    let price: Double
}
```
