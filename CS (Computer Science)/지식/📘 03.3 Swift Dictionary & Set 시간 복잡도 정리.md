배열(Array)이 순서가 있는 목록이라면, **Dictionary(딕셔너리)**와 **Set(집합)**은 **키(Key)**를 기반으로 데이터를 관리하는 해시 테이블 구조이다. 
탐색 속도가 매우 빠르다는 것이 가장 큰 장점이다.

## 1. Dictionary (딕셔너리) 시간 복잡도

키(Key)와 값(Value)의 쌍으로 저장된다. 키를 통해 값을 찾는 속도가 배열과 달리 데이터 크기에 영향을 받지 않는다.

| 연산 종류 | 메서드 예시 | 평균 시간 복잡도 | 설명 |
| :--- | :--- | :--- | :--- |
| **접근/검색** | `dict[key]` | **$O(1)$** | 키를 해싱하여 메모리 주소를 바로 찾음 |
| **추가/수정** | `dict[key] = value`, `updateValue` | **$O(1)$** | 해당 키 위치에 값을 쓰기만 하면 됨 |
| **삭제** | `dict[key] = nil`, `removeValue` | **$O(1)$** | 해당 키 위치의 값을 제거 |
| **키 확인** | `keys.contains(key)` | $O(1)$ | 키 존재 여부 확인 |

> [!NOTE] Worst Case
> 해시 충돌(Hash Collision)이 모두 발생하는 최악의 경우 $O(n)$이 될 수 있지만, 코딩 테스트에서는 보통 **$O(1)$**로 간주하고 푼다.

### 실전 활용: 빈도수 세기 (Frequency Counter)
배열을 순회하며 개수를 셀 때 아주 유용하다.
```swift
let arr = ["A", "B", "A", "C", "B", "B"]
var counts = [String: Int]()

for item in arr {
    // 키가 없으면 0을 기본값으로 하여 1 더함 (O(1))
    counts[item, default: 0] += 1
}
// 결과: ["A": 2, "B": 3, "C": 1]
````

## 2. Set (집합) 시간 복잡도

중복을 허용하지 않고, 순서가 없는 값들의 모음이다. **"특정 값이 존재하는지"** 확인할 때 가장 강력하다.

|**연산 종류**|**메서드 예시**|**평균 시간 복잡도**|**설명**|
|---|---|---|---|
|**삽입**|`insert()`|**$O(1)$**|값을 해싱하여 저장|
|**삭제**|`remove()`|**$O(1)$**|값을 해싱하여 찾아 삭제|
|**검색 (포함 여부)**|`contains()`|**$O(1)$**|**가장 중요!** 배열의 `contains`($O(n)$)보다 월등히 빠름|
|**집합 연산**|`union`, `intersection`|$O(n + m)$|두 집합의 크기에 비례|

### 실전 활용: 중복 제거 및 빠른 검색

```swift
let nums = [1, 2, 3, 1, 2, 4, 5]

// 1. 중복 제거 (O(n))
let uniqueNums = Set(nums) // {1, 2, 3, 4, 5}

// 2. 빠른 검색 (O(1))
if uniqueNums.contains(3) { // 배열이었다면 O(n)
    print("3이 있습니다")
}
```

## 3. 핵심 조건: Hashable 프로토콜

Dictionary의 **Key**와 Set의 **Element**가 되기 위해서는 반드시 `Hashable` 프로토콜을 준수해야 한다.

- Swift의 기본 타입(`Int`, `String`, `Double`, `Bool` 등)은 모두 `Hashable`이다.
    
- 커스텀 구조체나 클래스를 Key로 쓰려면 `Hashable`을 직접 채택해야 한다.

```swift
struct Point: Hashable {
    let x: Int
    let y: Int
}

var points = Set<Point>() // 가능
```

## 4. Array vs Set/Dictionary 선택 기준

|**상황**|**추천 자료구조**|**이유**|
|---|---|---|
|**순서**가 중요할 때|**Array**|인덱스로 관리, 정렬 유지 용이|
|**중복 데이터**가 허용될 때|**Array**|Set은 중복 불가|
|데이터 존재 여부(**검색**)가 빈번할 때|**Set**|`contains` 속도가 $O(1)$|
|**특정 값(ID 등)으로 데이터**를 찾아야 할 때|**Dictionary**|Key로 조회 속도가 $O(1)$|

### 💡 요약

코딩 테스트 문제를 풀 때 시간 초과가 난다면 다음을 의심해 보세요:
1.  이중 반복문($O(n^2)$)을 쓰고 있는가?
2.  반복문 안에서 배열의 `contains`($O(n)$)를 호출하고 있는가?

2번의 경우, 배열을 **Set**이나 **Dictionary**로 바꾸는 것만으로도 $O(n) \times O(n) = O(n^2)$의 로직을 $O(n) \times O(1) = O(n)$으로 최적화할 수 있습니다.
