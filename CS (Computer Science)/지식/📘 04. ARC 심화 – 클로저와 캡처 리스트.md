### ✅ 목표

- 클로저와 ARC의 관계 이해
    
- 클로저 내부에서 발생하는 순환 참조(Capture Cycle)의 원인 파악
    
- `weak`, `unowned`, `[capture list]`의 실전 사용법 익히기

---

## 1. 🧠 클로저는 참조 타입이다

```swift
class SomeClass {
    var message = "Hello"

    func doSomething() {
        let closure = {
            print(self.message)
        }
        closure()
    }
}
```

> 이 코드에서 `self`는 클로저에 의해 **캡처(capture)** 됩니다.  
> 클로저가 `self`를 참조하게 되면 ARC는 **self와 클로저가 서로를 참조하는 상황**이 생겨  
> 메모리 해제가 되지 않는 순환 참조(capture cycle)가 발생할 수 있어요.

---

## 2. 🔁 클로저에서 순환 참조 발생 예시

```swift
class ViewModel {
    var onUpdate: (() -> Void)?

    func setup() {
        onUpdate = {
            print("updating \(self)")
        }
    }

    deinit {
        print("ViewModel deinitialized")
    }
}
```

- `ViewModel`이 클로저를 참조 (`onUpdate`)
    
- 클로저가 `self`를 캡처 (`self.message` 등)
    
- ➡️ 서로 참조 → 순환 참조 발생 → `deinit` 호출 안됨 ❌

---

## 3. ✅ 해결: 캡처 리스트 `[weak self]`, `[unowned self]`

```swift
onUpdate = { [weak self] in
    guard let self = self else { return }
    print("updating \(self)")
}
```

또는

```swift
onUpdate = { [unowned self] in
    print("updating \(self)")
}
```

### 🔍 차이 요약

| 항목     | `weak self`       | `unowned self`       |
| ------ | ----------------- | -------------------- |
| 옵셔널 여부 | `Optional(self?)` | `Non-Optional(self)` |
| 안전성    | 안전, 해제되면 nil      | 위험, 해제 후 접근 시 크래시    |
| 사용 상황  | 해제 가능성 있음         | self가 무조건 존재해야 할 때   |

---

## 4. 🔒 클로저 내부에서 `self`를 쓸 때의 룰

- ✅ ViewController → 클로저에서 `self` 쓰면 **반드시 `[weak self]` 또는 `[unowned self]` 사용**
    
- ✅ 타이머, 네트워크 콜백, 애니메이션 클로저 등 → **캡처 리스트 필수**
    
- ✅ `self`가 더 오래 살아야 하는 경우 → `unowned`, 그 외는 `weak`

---

## 💡 핵심 요약

- 클로저는 참조 타입 → self 캡처 시 순환 참조 가능
    
- [weak self], [unowned self]로 순환 참조 해결
    
- self를 반드시 사용해야 한다면 **명확하게 의도를 표현하는 것**이 중요

---

## ✅ Q1. 클로저에서 `self`를 캡처할 때 순환 참조가 발생하는 이유는?

### 🟢 사용자 답변 요약:

> 클로저가 `self`를 참조하고, `self`가 클로저를 참조해서 서로 강하게 잡고 있는 상태 → 순환 참조 → 메모리 해제 안 됨.

### ✅ 피드백:

- **정확한 개념**을 짚었습니다.
    
- 표현 중 **“메모리가 해제되는 순간을 알 수 없다”**는 문장은 의미는 알겠지만 약간 모호하게 들릴 수 있어요.
    
    - 더 명확하게: “참조 카운트가 0이 되지 않기 때문에 ARC가 해제하지 못한다” 정도로 표현하면 좋습니다.

---

### ✅ 개선된 모범답안:

> 클로저는 참조 타입이기 때문에, 클로저 내부에서 `self`를 사용할 경우 `self`를 **강하게 참조**하게 됩니다.  
> 만약 `self`도 해당 클로저를 속성으로 보유하고 있다면, 두 객체가 **서로를 참조하는 순환 참조(cycle)**가 발생합니다.  
> 이 경우 ARC는 두 객체의 참조 카운트를 0으로 만들 수 없기 때문에, **메모리를 해제하지 못하고 누수(memory leak)**가 발생합니다.

## ✅ Q2. `weak self`와 `unowned self`는 어떤 상황에서 사용하는 것이 적절한가요?

### 🟢 사용자 답변 요약:

> `self`가 반드시 존재할 때는 `unowned`를 사용한다.

### ✅ 피드백:

- 핵심 **의도는 맞지만 설명이 너무 간단**해서 오해의 여지가 있어요.
    
- 단순히 "무조건 있으면 unowned"가 아니라, **“존재가 보장되며, 옵셔널 처리를 하지 않아도 될 때”** unowned를 선택합니다.
    
- 그리고 `weak`는 일반적으로 더 안전하고 널처리할 수 있어야 하는 상황에서 사용하죠.

---

### ✅ 개선된 모범답안:

> `weak self`는 참조 대상이 **해제될 수 있는 가능성이 있는 경우**에 사용하며, 옵셔널로 선언되어야 합니다.  
> 예: 뷰컨트롤러가 클로저를 가지고 있고, 뷰컨트롤러가 먼저 사라질 수 있는 상황에서는 `weak self`로 참조해야 안전하게 `nil` 체크가 가능합니다.
> 
> 반면 `unowned self`는 참조 대상이 **항상 메모리에 존재할 것이 확실한 경우**에 사용합니다.  
> 옵셔널이 아니기 때문에, 해제된 인스턴스에 접근하면 앱이 크래시납니다.  
> 따라서 클로저보다 `self`가 더 오래 살아야 하는 상황에서만 사용하는 것이 적절합니다.
