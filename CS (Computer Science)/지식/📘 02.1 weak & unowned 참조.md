## 🧠 먼저 배경 설명: 왜 메모리 누수가 생기나?

Swift는 자동으로 메모리를 관리해주는 **ARC(Automatic Reference Counting)**를 사용합니다.  
→ 즉, 객체를 누가 참조하고 있는지 확인해서, **더 이상 참조하는 곳이 없으면 메모리를 자동으로 해제**합니다.

하지만!  
두 객체가 서로를 강하게 참조(`strong`)하면 참조 카운트가 0이 되지 않아 **메모리가 해제되지 않는 문제**가 생깁니다.

> [!success] ARC와 메모리 누수
> - ARC는 참조 카운트가 0이 되면 메모리를 자동 해제
> - 순환 참조가 생기면 참조 카운트가 0이 되지 않아 누수 발생

### 🔁 예: 순환 참조(circular reference)

```swift
class A {
    var b: B?
}
class B {
    var a: A?
}

let a = A()
let b = B()
a.b = b
b.a = a // 서로가 서로를 참조 → 메모리 해제되지 않음
```

> 이걸 방지하기 위해 **약한 참조(weak)** 또는 **비소유 참조(unowned)**를 사용합니다.

---

## 🌱 weak 참조

### ✅ 특징

- 참조 카운트를 **증가시키지 않음**
- 참조 대상이 해제되면 자동으로 `nil`이 됨
- **옵셔널 타입만 가능 (`var x: SomeType?`)**

### ✅ 언제 사용하나?

- **순환 참조를 방지하고 싶을 때**
- 참조 대상이 먼저 해제될 수 있는 상황 (예: Delegate, 부모 → 자식)

### ✅ 예시

```swift
class Person {
    var pet: Pet?
}
class Pet {
    weak var owner: Person? // 여기 weak 사용
}
```

---

## 🌾 unowned 참조

### ✅ 특징

- 역시 참조 카운트를 **증가시키지 않음**
- **하지만 nil이 될 수 없음 (비옵셔널)**
- 참조 대상이 해제되었는데 접근하면 **앱이 크래시남**

### ✅ 언제 사용하나?

- 참조 대상이 **나보다 오래 살아있어야 하는 상황**
- 예: 클로저 내부에서 self를 캡처할 때

### ✅ 예시

```swift
class Person {
    var name: String
    var card: CreditCard?

    init(name: String) {
        self.name = name
    }
}

class CreditCard {
    unowned let owner: Person // 여기 unowned 사용

    init(owner: Person) {
        self.owner = owner
    }
}
```

> `CreditCard`는 항상 `Person`과 함께 존재하므로 `owner`는 절대 `nil`이 되지 않아야 합니다.

---

## 🧾 정리 표

|항목|`weak`|`unowned`|
|---|---|---|
|참조 카운트 증가|❌|❌|
|참조 대상 해제 시|`nil`로 자동 설정|크래시 발생|
|옵셔널 여부|`Optional` 필수|`Non-Optional` 필수|
|사용 예시|델리게이트, 부모-자식 관계|상위 객체가 항상 먼저 존재할 때|

---

## 💡 핵심 요약

- `weak`은 안전하지만 옵셔널
- `unowned`는 안전하지 않지만 깔끔함
- **상황에 맞게 선택해서 사용**

### 🧠 오늘의 질문
## 📘 Q1. `weak`과 `unowned`는 왜 ARC에서 필요한가요?

### 🟢 사용자 답변:

> 참조를 사용할 때, 서로다른 객체가 서로를 참조하게되면 메모리를 해제할 수 없고, 이는 ARC에서도 해제할 수 없기 때문에 이러한 순환참조를 해결하기 위해서 사용합니다. 그래서 두경우 모두 참조 카운트를 증가시키지 않아서,초기 참조 카운트가 0으로 되어있어 메모리를 해제할 수 있다. 단 weak는 옵셔널이 가능하지만 unowned는 옵셔널이 불가능하다.

### ✅ 평가:

- 순환 참조 → 메모리 누수 → ARC로도 해제 불가 → `weak` / `unowned`로 해결  
    이 흐름을 잘 잡으셨어요!
- "초기 참조 카운트가 0"이라는 표현은 약간 오해의 소지가 있어요.

### ✅ 개선안:

```diff
- 초기 참조 카운트가 0으로 되어있어
+ weak / unowned는 참조 카운트를 증가시키지 않기 때문에
```

### ✅ 모범답안 예시:

> `weak`과 `unowned`는 **순환 참조로 인해 ARC가 메모리를 해제하지 못하는 상황을 방지하기 위해 사용됩니다.**  
> 두 방식 모두 **참조 카운트를 증가시키지 않기 때문에**, 객체가 다른 곳에서 참조되지 않으면 ARC에 의해 정상적으로 해제됩니다.  
> `weak`은 대상이 사라지면 자동으로 `nil`이 되며, 옵셔널 타입으로만 사용할 수 있습니다. 반면, `unowned`는 비옵셔널로 사용되며, 대상이 해제된 후 접근하면 앱이 크래시납니다.

---

## 📘 Q2. 각각을 어떤 상황에서 사용하는 것이 적절한가요?

### 🟢 사용자 답변:

> weak의 경우 A와 B라는 두 클래스에서 먼저 참조하여 존재하는 A보다 뒤늦게 참조한 B 클래스가 먼저 해제되는 경우에 weak를 사용합니다. 이때 weak는 B class에 추가됩니다. unowned의 경우는 반대로 A 클래스가 먼저 해제되는 경우 B class에 unowned를 추가합니다.

### ✅ 평가:

- **해제 순서에 따른 기준**을 나누신 건 좋았어요!
- 하지만 표현이 조금 모호하거나 방향이 반대인 부분이 있습니다.  
    예를 들어:

```diff
- A보다 뒤늦게 참조한 B 클래스가 먼저 해제되는 경우에 weak를 사용합니다.
+ 참조 대상(B)이 먼저 해제될 수 있는 상황에서는 weak를 사용합니다.
```

- **unowned는 참조 대상이 "절대 먼저 해제되지 않을 때" 사용하는 것이 맞습니다.**  
    → 답변에 반대로 서술된 것처럼 보입니다.

---

### ✅ 개선된 설명:

| 키워드       | 사용 상황              | 예시                                        |
| --------- | ------------------ | ----------------------------------------- |
| `weak`    | 참조 대상이 먼저 해제될 수 있음 | delegate, 부모 → 자식                         |
| `unowned` | 참조 대상이 항상 존재해야 함   | Person → CreditCard 등 하위 객체가 상위 객체를 참조할 때 |

---

### ✅ 모범답안 예시:

> `weak`은 참조하는 대상이 **먼저 메모리에서 해제될 수 있는 경우**에 사용합니다.  
> 예를 들어 delegate 패턴처럼, 자신이 참조하는 객체가 언제든지 사라질 수 있을 때 `weak`로 선언하여 안전하게 `nil`로 만들 수 있습니다.
> 
> `unowned`는 참조 대상이 **반드시 먼저 존재해야 하고, 해제되지 않을 것을 보장할 수 있는 경우**에 사용합니다.  
> 예: `CreditCard`는 항상 `Person` 객체와 함께 존재하므로, `unowned`로 참조해도 안전합니다.

## 🧷 1. **델리게이트 패턴에서 `weak`를 사용하는 이유**

### ✅ 상황 설명

델리게이트(Delegate)는 iOS 앱에서 **객체 간 위임 관계**를 만드는 대표적인 패턴입니다.
보통 이런 구조예요:

```swift
class ChildViewController: UIViewController {
    weak var delegate: SomeDelegate? // 여기에 weak 사용
}
```

### ✅ 왜 `weak`로 선언해야 하나요?

#### ❗ 문제: 순환 참조 발생 가능

- A(ViewController)가 B(ChildVC)를 **소유**
- B는 다시 A를 **delegate로 참조**

```swift
class A {
    var child: B?
}
class B {
    var delegate: A? // ← 이게 strong이면 서로 참조하는 상태
}
```
이 경우 ARC가 **둘 다 참조 카운트가 0이 되지 않아서 메모리 누수**가 발생해요.

### ✅ 해결: `weak` 참조로 순환 고리 끊기

```swift
class B {
    weak var delegate: A?
}
```
→ 이제 `delegate`가 `nil`로 되면서 ARC가 메모리를 정상 해제할 수 있어요.

---

### 📌 핵심 요약

| 항목        | 설명                                    |
| --------- | ------------------------------------- |
| 왜 weak?   | delegate는 참조 대상보다 오래 살지 않으므로 해제될 수 있음 |
| 무슨 문제 예방? | 순환 참조(메모리 누수)                         |
| 반드시 옵셔널?  | 네, `weak`는 옵셔널로만 선언 가능                |

---

## 🧵 2. **클로저에서 `[unowned self]`를 사용하는 이유**

### ✅ 상황 설명

클로저는 참조 타입입니다.  
그래서 클로저 내부에서 `self`를 사용하면 **self를 캡처(capture)**하게 됩니다.

```swift 
class MyViewController {
    func loadData() {
        fetchData { result in
            self.updateUI(result) // ← self를 캡처
        }
    }
}
```
### ❗문제: 순환 참조 발생

- `MyViewController`가 `fetchData` 클로저를 가지고 있음
- 클로저가 `self`를 참조
→ 서로 참조 = **메모리 누수 가능**

---

### ✅ 해결: `[unowned self]` 또는 `[weak self]` 사용

```swift
fetchData { [unowned self] result in
    self.updateUI(result)
}
```
→ self의 참조 카운트는 증가하지 않음.  
→ `MyViewController`가 해제되면 클로저도 함께 해제됨.

---

### ✅ `weak self` vs `unowned self`

|항목|`weak self`|`unowned self`|
|---|---|---|
|옵셔널 여부|옵셔널 (`self?`)|비옵셔널 (`self`)|
|안전성|안전 (nil 체크 가능)|위험 (해제된 후 접근하면 크래시)|
|주 사용처|self가 먼저 해제될 수 있음|self가 항상 존재한다고 확신 가능할 때|

---

### 📌 정리

|상황|해결 방법|이유|
|---|---|---|
|델리게이트|`weak` delegate|참조 대상이 먼저 해제될 수 있으므로|
|클로저 내부 self|`[unowned self]` 또는 `[weak self]`|self를 클로저가 캡처하면 순환 참조 발생 가능|
