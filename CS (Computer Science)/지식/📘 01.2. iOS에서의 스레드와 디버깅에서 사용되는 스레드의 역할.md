# iOS에서 흔히 쓰는 스레드/동시성 도구

- **GCD(DispatchQueue)**: `DispatchQueue.main`, `.global(qos:)`, `.asyncAfter` 등.
    
- **OperationQueue**: 의존성, 취소 등 고수준 제어.
    
- **Swift Concurrency**: `async/await`, `Task`, `@MainActor`, actor 등 (내부적으로 스레드 풀에서 실행).
    
- **RxSwift**: `subscribe(on:)`(생산), `observe(on:)`(소비)로 어느 스케줄러(=큐/스레드)에서 실행할지 분리.

# 디버깅할 때 보이는 “스레드”는 뭔 역할?

Xcode의 **Debug Navigator**(⌘+7)나 LLDB에서 보이는 Thread 1, 2, …가 **현재 앱에서 살아 있는 실행 흐름들**이에요.

대표적으로:

- **Thread 1 (main)**: UI 이벤트, 터치, 레이아웃, 타이머 등. 여기서 오래(block) 걸리면 **UI 멈춤/스크롤 버벅임**.
    
- **백그라운드 스레드들**: 네트워킹 콜백, 이미지 디코딩, DB I/O, BLE/파일 처리 등 **무거운 일 처리**.

디버깅 시 자주 보는 것:

- **Backtrace(스택트레이스)**: 해당 스레드가 “어떤 함수 호출 경로로” 여기까지 왔는지.  
    LLDB: `thread backtrace` 또는 `bt`
    
- **현재 스레드가 메인인지**: `Thread.isMainThread` 로깅 or Xcode 표시
    
- **교착상태(Deadlock) 의심**: 여러 스레드가 서로 락을 기다리며 멈춤 → 모든 스레드가 `semaphore.wait()`/`dispatch_sync` 같은 곳에서 멈춰 있는 패턴
    
- **데이터 레이스**: 여러 스레드가 동일한 가변 상태를 동시에 읽고/쓰기
    

# iOS 개발자가 알아야 할 최소 체크리스트

1. **UI는 무조건 메인 스레드**

```swift
DispatchQueue.main.async {
    self.label.text = "완료"
}
// RxSwift
observable
  .observe(on: MainScheduler.instance)
  .bind(to: label.rx.text)
  .disposed(by: bag)
```
    
2. **무거운 일은 백그라운드**
    
```swift
DispatchQueue.global(qos: .userInitiated).async {
    let image = decodeLargeImage(data)
    DispatchQueue.main.async { self.imageView.image = image }
}
```
    
3. **스레드-세이프한 공유 상태 관리**
    
    - 한 큐(직렬)로 단일화하거나, **락(NSLock)**, **actor**, **Serial DispatchQueue**, **@MainActor** 사용.
        
    - Rx라면 `share(replay:)`, `observeOn`/`subscribeOn` 의도 분리.

4. **도구로 잡기**
    
    - **Main Thread Checker**(스키마 > Diagnostics): 메인 아닌 스레드에서 UI 건드리면 경고.
        
    - **Thread Sanitizer(TSan)**: 데이터 레이스 탐지(강력 추천).
        
    - **Instruments – Time Profiler**: 메인 스레드가 어디에서 오래 걸리는지.
        
    - **Instruments – Thread States/Logging**: 스레드 상태 전환, 락 대기 확인.

5. **QoS 이해(우선순위)**
    
    - `userInteractive`(즉각 반응), `userInitiated`(빠른 완료), `utility`(진행 표시 가능), `background`(눈에 안 띄는 작업).
        
    - 과도한 high QoS는 배터리·발열·스케줄링에 악영향.
    

# 디버깅 실무 팁

- **프리즈/버벅임**: Time Profiler로 **Main Thread flame graph** 확인 → 오래 걸리는 함수 offload.
    
- **크래시**: 크래시 스레드의 **backtrace 최상단**부터 읽기. 다른 스레드에서 공유 자원 접근 중인지도 같이 본다.
    
- **교착 의심**: 모든 스레드 backtrace 찍어서 동일 리소스 락 대기 확인.
    
- **로그에 스레드 찍기(디버그용)**:
    
```swift
func log(_ msg: String) {
    print("[\(Thread.isMainThread ? "main" : "bg")] \(msg)")
}
```

# Swift Concurrency(참고)

- `@MainActor` 붙이면 해당 메서드/타입은 **항상 메인 스레드 컨텍스트**에서 실행.
    
- `Task { ... }` 내부는 **스레드 풀**에서 실행되며 런타임이 적절히 스케줄.
    
- **actor**로 가변 상태를 캡슐화 → 데이터 레이스 방지에 매우 유용.

# RxSwift와 스레드 감각 정리

- `subscribe(on:)` = **생산**(업스트림) 시작 스케줄러. (예: 네트워크 호출 트리거 위치)
    
- `observe(on:)` = **소비/후처리** 스케줄러. (UI 바인딩은 MainScheduler)
    
- 화면 생명주기·셀 재사용에 따라 **DisposeBag** 분리 + `observe(on: MainScheduler.instance)` 일관 적용.

## 한 줄 요약

- **스레드 = 실행 흐름**. 메인 스레드=UI, 백그라운드=무거운 일.
    
- 디버깅에 보이는 스레드는 현재 앱의 실행 흐름 목록이며, **Backtrace로 원인 추적**.
    
- **Main Thread Checker/Thread Sanitizer/Instruments**로 문제를 빠르게 잡고, **메인은 가볍게, 무거운 건 백그라운드로**가 기본 원칙입니다.
