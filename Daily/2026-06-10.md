## 🌙 오늘 하루 피드백 (2026-06-10)

### ✅ 완료한 것

- iOS 멘토링에서 `presentationTime`과 `frameID`의 역할을 분리했다.
- Swift 멘토링에서 actor가 만든 `ResultSnapshot`을 `@MainActor` ViewModel로 넘기는 안전한 경계를 정리했다.
- 하루업 백엔드를 Railway staging 환경에 올리고, 배포된 API를 기준으로 `정상 동작 / 앞으로 사용 / 사용하지 않음`으로 나눴다.
- 멘토링 운영 방식에 대해 리셋 필요성을 인식했고, 6/11~6/14는 하나의 큰 주제로 다시 시작하기로 결정했다.

### 💬 기술 피드백

오늘 가장 큰 진전은 하루업 백엔드를 “코드가 존재하는 서버”에서 “앱 팀이 실제로 호출할 수 있는 배포된 서비스”로 바꾼 점이다. Railway 배포가 됐다는 사실 자체보다, 배포된 상태에서 정상 동작 API와 사용하지 않을 API를 구분한 것이 더 중요하다. 이 구분은 곧 앱-서버 계약을 정리하는 작업이고, 이후 프론트엔드/모바일 팀이 어떤 endpoint를 믿고 붙을 수 있는지 결정하는 기준이 된다.

다음 단계에서는 `사용하지 않음`을 바로 삭제 대상으로 보면 위험하다. `삭제`, `Swagger 비공개`, `운영 보호 필요`, `나중에 사용할 후보`처럼 한 번 더 나누는 것이 좋다. 특히 수동 배치 실행, 테스트 푸시, 통계/엑셀 다운로드, Actuator/Swagger 노출 범위는 “안 쓰는 API”가 아니라 “외부에 공개되면 안 되는 운영 API”일 수 있다.

Railway 관점에서는 `/health`, `SERVER_PORT=${{ PORT }}`, PostgreSQL/Redis 연결, JWT secret, CORS origin, Clova/Firebase 환경변수, 파일 업로드 저장소를 별도 체크리스트로 남겨야 한다. 오늘 분류한 API 목록에 `사용 화면`, `인증 필요 여부`, `배포 환경 테스트 상태`, `다음 조치` 열을 붙이면 바로 팀 공유 문서가 된다.

멘토링은 오늘 기준으로 한 번 끊는 판단이 맞다. iOS/Swift 내용이 많이 누적되어서 더 이어가면 학습 밀도보다 맥락 유지 비용이 커진다. 목~일은 `하루업 Railway 배포 후 API 계약 정리` 하나만 보고, 다음 주 월요일부터 Go/React 중심으로 다시 주차를 여는 편이 지금 업무 전환에 더 맞다.
