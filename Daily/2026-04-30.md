## 📚 학습 내용 (직접 작성)

### Task & @MainActor
- **이해한 것**:
	- Task : 동기로 진행되는 코드 환경에서 해당 Task {} 내에서 비동기로 코드를 실행할 수 있게 작업공간을 만들어준다.
	- @MainActor : 블록안에 있는 코드가 메인 스레드(Main Thread)에서만 실행되도록 강제. 현재 적용된 이유는 화면 상태를 변경하는 작업은 무조건 Main Thread에서 진행되야한다.
- **ZipJoong과 연결되는 부분**: 
	- viewModel에서 존재하는 값들을 수정하고 viewModel에서 UI를 업데이트 하는 부분에서 코드를 사용한다.
- **면접 답변으로 쓸 수 있는 한 줄**:
	- ViewModel과 같이 UI보다 기능이나 설정값을 저장하는 class에서 UI변경을 시도하는 함수의 경우 Task와 @MainActor를 통해 비동기로 메인스레드에서 동작하게 해야합니다.

### Git 병합 전략
- **세 방식 중 내가 고른 것**: Squash Merge
- **이유**: 
	- 다른 것에 비해 main 브랜치에는 커밋 기록이 기능에 대한 내용만 정의되어있어서 좋은 것 같다고 판단했습니다.
	- 실제로 상세 기록들은 PR 내에 커밋메시지가 다 확인된다고 생각됩니다.

### 💬 기술 피드백

**Task & @MainActor 이해 — 맞는데 "왜"가 빠졌다**

이해한 내용은 정확하다. 다만 면접 답변 한 줄을 더 날카롭게 다듬을 수 있다:

> 현재: "ViewModel에서 UI변경을 시도하는 함수의 경우 Task와 @MainActor를 통해 비동기로 메인스레드에서 동작하게 해야합니다."

이건 "어떻게"다. 면접관이 원하는 건 "왜":

> 개선: "SwiftUI는 @Published 변경들을 한 RunLoop cycle에 묶어 처리합니다. 그 사이에 무거운 초기화 작업이 끼어들면 fullScreenCover 같은 화면 전환 애니메이션을 방해합니다. `Task { @MainActor in }`으로 감싸면 현재 동기 흐름이 끝난 뒤 다음 RunLoop에서 실행되어, SwiftUI가 화면 전환을 먼저 완료할 시간을 얻습니다."

이 설명은 "현상 → 원인 → 해결 → 왜 작동하는가" 구조다. 이 순서로 말할 수 있으면 면접에서 경험으로 배운 것임이 명확하게 전달된다.

**ZipJoong — 버그 하나 고치려다 앱 전체를 봤다**

뽀모도로 수정 코드 확인이 목적이었는데, 앱을 실행하면서 개선점 8가지를 직접 발견한 것이 더 가치 있다. 이 중 몇 가지는 구조적인 문제다:

- **팝업 종류 미분류 (자리 이탈 vs 졸음)**: UX 흐름 설계 문제. 각 상황에 맞는 대응이 없으면 사용자는 어떤 팝업이든 그냥 닫는다.
- **타이머 종료 후 로딩 지연**: CameraService 세션 종료가 동기 처리될 가능성이 있다. 내일 금요일 CameraService 분석 일정이 이 문제를 파고드는 것이다 — 준비된 상태로 들어가는 것.
- **일반 타이머 실행 시 뽀모도로 버튼 사라짐**: 상태 관리 문제. `timerState`에 따라 버튼 노출 조건을 따로 설계해야 할 수 있다.

졸음 판정 시 미니게임/게이미피케이션 아이디어는 포트폴리오에 쓸 수 있는 차별점이다. 헬스케어 도메인에서 "집중 유지"에 행동과학을 붙이는 접근으로 설명할 수 있다.

**Git Squash Merge — 선택 타당, 이유도 명확**

"main 브랜치에는 기능 단위 커밋만 남는다"는 이유가 정확하다. 추가로 기억해 두면 좋은 것: Squash Merge 후 브랜치를 삭제하지 않으면 나중에 같은 브랜치에서 다시 PR을 열 때 커밋 히스토리가 꼬일 수 있다. 개인 프로젝트라도 **PR 머지 후 브랜치 삭제**를 루틴으로 잡는 것이 좋다.

**조교 직무 확정 — 구조가 바뀌었다**

이건 오늘 가장 큰 변화다. 학습→취업을 목표로 하던 구조에서, **일하면서 학습하는 구조**로 전환됐다. 조교 직무는 헬스케어 + AI + 교육의 교차점인데, 이 경험을 쌓으면서 동시에 포트폴리오 프로젝트를 개발하는 것이 가능해진 셈이다.

### 🧭 새 iOS 프로젝트 5가지 후보 분석

5가지를 커리어 임팩트 관점에서 줄 세우면:

| 순위 | 프로젝트 | 이유 |
|------|----------|------|
| ★★★ | **1. 자세 교정 헬스 가이드** | ZipJoong 카메라 경험 연장. Vision Body Pose + CoreML. On-device AI 포트폴리오 직결. 헬스케어 + AI + iOS 세 가지 다 담김. |
| ★★★ | **5. 의료 용어 AI 해설 앱** | Vision OCR + AI(온디바이스 or API). 조교 직무에서 "학생들이 의료 용어 몰라서 막히는 상황"을 실제로 접하면서 소재가 생긴다. |
| ★★ | **2. HealthKit AI 대시보드** | HealthKit 경험 자체는 희소. 서버 AI 연동 추가되면 복잡도 올라가고 완성도 관리가 어려워짐. |
| ★ | **3. Context-Aware 루틴 플래너** | CoreLocation + ActivityKit 연습은 되지만 AI 차별화가 낮다. |
| ★ | **4. 복약 알림 앱** | 완성도 높은 유틸리티이지만 포트폴리오 임팩트가 낮다. 부업 프로젝트로는 좋음. |

**추천은 1번 또는 5번**. 조교 직무를 시작하면서 학생들이 어떤 부분에서 막히는지 직접 보게 될 텐데, 그 경험이 5번 앱의 소재가 된다. 1번은 ZipJoong에서 CameraService를 이미 다뤘기 때문에 진입 비용이 낮다. 둘 다 "왜 이걸 만들었나"를 헬스케어 도메인 경험으로 설명할 수 있다.
