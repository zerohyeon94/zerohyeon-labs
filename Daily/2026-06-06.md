## 🌙 저녁 회고 (Codex 자동 작성 영역)

> 피드백 기준일: 2026-06-06

### ✅ 완료한 것

- 하루업 백엔드 테스트 기간 배포 방향을 Railway staging으로 정리하고, 운영 배포는 2026-07-06 전후 AWS로 다시 검토하는 흐름을 잡았다.
- Railway 배포 전에 먼저 봐야 할 API 목록, Swagger/Controller 비교, 인증 필요 여부, 앱 사용 여부를 문서화 대상으로 분리했다.
- Jday를 iOS/iPadOS/macOS에서 정상 동작하는지 확인하고, 스토어 업로드용 스크린샷/이미지 준비 상태를 정리했다.
- 기존 스픽 루틴 대신 듀오링고로 영어 루틴을 테스트했다.

### 💬 기술 피드백

오늘 제일 좋은 판단은 하루업을 바로 배포부터 밀어붙이지 않고, 테스트 기간의 목적을 `팀 API 연동용 staging base URL 확보`로 좁힌 점이다. Railway를 1차 테스트 서버로 두고 AWS는 운영 전환 시점에 다시 보는 결정은 현재 단계의 비용, 속도, 관리 부담을 잘 나눈 선택이다.

다만 Railway 배포 전에는 API 목록 정리만으로는 부족하고, `앱에서 실제 호출하는 API / Swagger에 보이는 API / 인증 없이 열려 있는 API`를 분리해야 한다. 특히 수동 배치, 테스트 푸시, 통계성 API, actuator/Swagger 공개 범위는 staging에서는 편해 보여도 운영 전환 때 보안 리스크가 된다. 내일은 "있는 API"보다 "앱 팀이 지금 쓸 API"를 먼저 표시하는 편이 좋다.

Jday는 스토어 업로드 전 단계로 아주 의미 있는 QA를 했다. iOS/iPadOS/macOS를 모두 확인했다는 건 단순 실행 확인이 아니라 SwiftUI Multiplatform 앱의 출시 리스크를 줄인 것이다. 다음 단계는 기능을 더 고치는 것이 아니라 Archive, 빌드 번호, App Privacy, 개인정보 처리방침 URL, 스크린샷 해상도처럼 App Store Connect에서 막히는 항목을 차례로 닫는 것이다.
