### ✅ 완료한 것

- iOS 멘토링을 완료했다. `AnalysisResult`를 단순 추론 결과가 아니라 `frameID / capturedAt / finishedAt / focusScore / confidence / shouldShowWarning` 중심의 최신성 판단용 스냅샷으로 설계하는 방향을 잡았다.
- Swift 멘토링을 완료했다. `CameraService -> ResultSnapshot -> @MainActor ViewModel -> CameraUIState` 흐름으로 MainActor 범위를 줄이는 기준을 정리했다.
- Jday App Store 제출용 앱 이름, 부제, 설명, 키워드, SKU, 스크린샷 위치, 배포 체크리스트를 실제 프로젝트 문서로 추가했다.
- Jday의 CloudKit 설정을 진전시켰다. entitlements 기준으로 `iCloud.com.zerohyeon.jday`, `CloudKit`, `aps-environment`, `remote-notification`이 반영된 상태까지 확인했다.

### 💬 기술 피드백

오늘 iOS/Swift 멘토링의 핵심은 같은 문제를 두 층에서 정리했다는 점이다. iOS 쪽에서는 "오래된 추론 결과가 늦게 도착했을 때 무엇을 버릴 것인가"를 `frameID` 기준으로 잡았고, Swift 쪽에서는 "무거운 원본을 MainActor로 가져오지 않으려면 어떤 경계 타입을 둘 것인가"를 `ResultSnapshot`으로 정리했다. 이 둘은 별개 주제가 아니라 같은 실시간 파이프라인의 앞뒤다.

`stale result`가 아직 낯설다고 적어둔 것도 좋은 메모다. 여기서 stale은 "늦게 도착한 결과" 전체가 아니라, **이미 UI가 더 최신 frameID를 반영한 뒤 도착한 과거 frameID 결과**를 뜻한다. 예를 들어 102번 프레임 결과를 이미 화면에 보여줬는데 101번 프레임 결과가 뒤늦게 오면, 101번은 stale result다. 그래서 `finishedAt`이 늦은지보다 `frameID`가 마지막 반영값보다 큰지가 먼저다.

Jday는 6월 7일에 방향이 크게 좋아졌다. 단순히 "스토어 올려야지"가 아니라, 앱 이름/부제/설명/키워드 같은 메타데이터와 `Signing & Capabilities`, CloudKit 컨테이너, Push Notifications, Remote notifications까지 실제 제출 차단 항목을 직접 만졌다. 이건 출시 준비에서 가장 지루하지만 가장 중요한 구간이다.

다만 하루업 API 분류는 아직 비어 있다. 6월 7일은 Jday 제출 준비와 멘토링 완료가 중심이었기 때문에 API 분류를 못 한 것 자체는 괜찮다. 대신 다음에는 "전체 API 정리"가 아니라 `앱에서 실제 호출하는 API 5개만 먼저 표시`처럼 범위를 더 작게 잡는 편이 좋다.
