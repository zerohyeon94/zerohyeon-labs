## ✅ Rx로 이벤트 전달

|요소|구성|
|---|---|
|발신자|`SpendViewModel.saveSpending()` → `.onNext(())`|
|이벤트 허브|`AppEventBus.shared.spendingAdded`|
|수신자|`HomeViewModel` → `subscribe { fetchTodayBudget() }`|

---

## ✅ Subject를 정의하는 위치에 따른 차이점

|비교 항목|ViewModel에 각각 정의|Singleton 이벤트 허브 사용|
|---|---|---|
|🔄 중복 발생|✅ 매 화면마다 `PublishSubject` 생성|❌ 한 곳에 모아둠|
|📦 재사용성|❌ 다른 화면에서 접근 어려움|✅ 전역에서 접근 가능|
|🧩 ViewModel 결합도|높음 (서로 알거나 넘겨야 함)|낮음 (관찰만 하면 됨)|
|🧼 코드 정리|중복 많아짐|한 곳에서 이벤트 정의 가능|
|🧪 테스트 용이성|❌ 테스트마다 주입 필요|✅ 주입 없이 구독 가능|

## ✅ 현실적인 예

### 📍 상황:

- `SpendViewModel` → 소비가 추가됨
    
- `FixedExpenseEditViewModel` → 고정비 항목이 추가됨
    
- `HomeViewModel` → 둘 중 어떤 액션이 일어나도 예산 갱신 필요

### ❌ ViewModel마다 subject 정의 시

- `HomeViewModel`은 **둘 다 알고 있어야** 하고, **의존성이 증가**
    
- 재사용이 어려움

### ✅ 싱글톤 이벤트 허브 사용 시


>[!Tip] 앱 전체에서 사용하는 글로벌 이벤트 중심 객체 
> '기능'이나 '모듈'보다는 '공통 자원' 또는 '전역 도구'

#### 📄 AppEventBus.swift

```swift
// App : 앱 전역에서 사용됨 (모든 화면/모듈에 열려 있음)
// Event : 이벤트 중심 (사용자 액션, 데이터 변경)
// Bus : '모두가 타고 내릴 수 있는 통신 버스'개념 (이벤트를 발행하고 구독하는 허브)
final class AppEventBus {
    static let shared = AppEventBus()
    private init() {}

    let spendingAdded = PublishSubject<Void>()
    let fixedExpenseChanged = PublishSubject<Void>()
}
```

#### ✏️ 어디서든 이렇게 사용

```swift
AppEventBus.shared.spendingAdded.onNext(())
AppEventBus.shared.fixedExpenseChanged.onNext(())
```

#### 👂 HomeViewModel에서 구독

```swift
AppEventBus.shared.spendingAdded
    .merge(with: AppEventBus.shared.fixedExpenseChanged)
    .subscribe(onNext: { [weak self] in
        self?.refreshTodayBudget()
    })
    .disposed(by: disposeBag)
```

이 구조는 iOS에서 흔히 다음 이름으로 쓰입니다:

|용도|이름 예시|
|---|---|
|RxSwift용 이벤트 허브|`AppEventBus`, `AppSignal`, `GlobalEventBus`|
|Combine용|`EventStream`, `AppEventSubject` 등|
|알림 중심 구조|`AppNotificationCenter`, `AppMessageBus`|

➡️ 모두 공통적으로 "앱 전체 이벤트 흐름을 하나의 통로로 모은다"는 개념입니다.

## ✅ 결론

> ❗️**ViewModel에 각각 PublishSubject를 두는 방식은 간단하지만 곧 복잡도가 폭발합니다.**  
> ✅ 지금처럼 **다양한 ViewModel에서 발생하는 이벤트를 한 곳으로 묶어**  
> 다른 화면이 **독립적으로 반응**할 수 있도록 만드는 것이 Rx 구조의 진짜 강점입니다.

---
## ✅ 주요 확인 포인트

```swift
final class HomeViewModel {
    private let disposeBag = DisposeBag() // ✅ 반드시 필요

    func bind() {
        AppEventBus.shared.spendingAdded
            .observe(on: MainScheduler.instance)
            .subscribe(onNext: { [weak self] in
                self?.fetchTodayBudget()
            })
            .disposed(by: disposeBag)
    }
}
```

### 🔍 각 줄의 의미

- `AppEventBus.shared.spendingAdded`:  
    소비 추가 이벤트를 발행하는 `PublishSubject<Void>`
    
- `.observe(on: MainScheduler.instance)`:  
    UI 관련 로직(`fetchTodayBudget()`이 UI 갱신을 포함할 가능성 있음)은 메인 쓰레드에서 처리
    
- `.subscribe(onNext:)`:  
    이벤트 수신 후 실제 작업 실행
    
- `.disposed(by: disposeBag)`:  
    메모리 누수 방지를 위한 Rx의 구독 해제 처리

---

## 🧠 권장: `bind()`는 `init()` 또는 ViewController에서 호출

```swift
init() {
    bind()
}
```

또는 ViewController에서:

```swift
override func viewDidLoad() {
    viewModel.bind()
}
```

## ✅ 보완 가능 사항 (옵션)

### 1. 에러 처리 추가 (예: debug 로그)

```swift
.subscribe(
    onNext: { [weak self] in self?.fetchTodayBudget() },
    onError: { print("❌ HomeViewModel - spendingAdded error: \($0)") }
)
```

### 2. 여러 이벤트를 통합 구독 (e.g., 고정비 변경 포함)

```swift
Observable.merge(
    AppEventBus.shared.spendingAdded,
    AppEventBus.shared.fixedExpenseChanged
)
.observe(on: MainScheduler.instance)
.subscribe(onNext: { [weak self] in
    self?.fetchTodayBudget()
})
.disposed(by: disposeBag)
```

## ✅ Rx 구조와 완전히 연결되었는지 체크 (이벤트 → 갱신 흐름)

현재 구조가 아래와 같다면 이상적입니다:

```swift
[SpendViewController] 저장 버튼
      ↓
AppEventBus.shared.spendingAdded.onNext(())
      ↓
HomeViewModel.bind() 안에서 이벤트 감지
      ↓
fetchTodayBudget() 실행
      ↓
BehaviorRelay에 새로운 값 적용
      ↓
ViewController에서 bind → UI 자동 갱신
```
