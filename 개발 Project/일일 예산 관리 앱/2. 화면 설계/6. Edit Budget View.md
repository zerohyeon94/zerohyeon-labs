## ✅ 화면 명세: EditBudgetViewController

|항목|설명|
|---|---|
|목적|월급과 월급일 설정을 수정|
|입력 필드|`월급` (정수) / `월급일` (1~31 정수 선택)|
|저장 버튼|누르면 CoreData/Model 업데이트 및 뒤로 이동|

```swift
Settings/

```

---

## ✅ 파일을 불러오는 구조

> **✅ CoreData에서 데이터를 가져오는 것은 `SettingsViewController`에서 하고,**  
> 그 값을 **`EditBudgetViewController`에 초기값으로 전달**하는 것이 일반적입니다.

이유는 다음과 같습니다.

---

## 🧠 왜 ViewController 진입 전에 CoreData 데이터를 불러오는가?

### 1. **단일 책임 원칙(SRP)**을 지키기 위해

- `EditBudgetViewController`의 역할은 **"사용자가 월급과 월급일을 편집하는 화면"**입니다.
    
- `CoreData` 접근, 데이터 로딩 책임까지 지면 **역할이 모호해지고 테스트/유지보수가 어려워집니다.**

### 2. **화면 전환 전에 데이터가 필요하니까**

- `EditBudgetViewController`는 초기부터 `salary`, `payday` 값을 보여줘야 하므로  
    **초기화 시점에 값이 반드시 필요**합니다.
    
- 따라서 **SettingsViewController가 CoreData에서 데이터를 읽고,  
    그 데이터를 EditBudgetViewController에 넘겨주는 구조**가 자연스럽습니다.

## 📌 예시 흐름 (코드 포함)

### CoreDataManager (간단 예시)

```swift
struct BudgetConfig {
    let salary: Int
    let payday: Int
}

class CoreDataManager {
    func fetchBudgetConfig() -> BudgetConfig {
        // 실제 CoreData fetch 코드...
        return BudgetConfig(salary: 3000000, payday: 25)
    }
}
```

## ✅ 코드 예시로 이해하기

### 1. `SettingsViewController`에서 이동 + 저장 처리

```swift
case "EditBudget":
    let config = CoreDataManager.shared.fetchBudgetConfig()
    let vc = EditBudgetViewController(currentSalary: config.salary, currentPayday: config.payday)

    vc.onSave { salary, payday in
        // 👉 여기에서 CoreData에 반영
        CoreDataManager.shared.updateBudgetConfig(salary: salary, payday: payday)
    }

    navigationController?.pushViewController(vc, animated: true)
```

---

### 2. `EditBudgetViewController`에서는?

```swift
// 사용자가 저장 버튼을 눌렀을 때
@objc private func didTapSave() {
    ...
    viewModel.updateSalary(salary)
    viewModel.updatePayday(payday)

    viewModel.save()  // 👉 내부에서 onSave?(salary, payday) 호출
    navigationController?.popViewController(animated: true)
}
```

### 3. `EditBudgetViewModel`에서는?

```swift
func save() {
    onSave?(salary, payday) // 👉 ViewController가 클로저를 연결해둠
}
```

## ✅ 왜 이렇게 분리하나요?

| 이유                 | 설명                                                             |
| ------------------ | -------------------------------------------------------------- |
| **단일 책임 원칙 (SRP)** | EditBudgetViewController는 UI 책임만, 데이터 저장은 상위 ViewController 책임 |
| **유연성**            | 나중에 CoreData가 아닌 서버 전송이 되어도 ViewController 코드는 그대로 유지 가능       |
| **테스트 용이성**        | ViewModel과 ViewController를 독립적으로 테스트 가능                        |
| **MVVM 유지**        | Model 변경은 ViewModel 또는 상위 구조에서 일어남                             |

---

## 💡 혹시 반대의 흐름도 가능한가요?

> `EditBudgetViewController`가 내부에서 CoreData를 fetch해도 되는가?

- 기술적으로는 가능합니다.
    
- 하지만 이 경우 **ViewController가 CoreData에 직접 의존하게 되며**,  
    테스트나 데이터 흐름 추적이 어려워집니다.
    
- 특히 ViewModel을 테스트하거나, 다른 저장소로 대체해야 할 때 곤란해집니다.

---

## ✅ 정리

|비교|SettingsViewController에서 fetch 후 전달|EditBudgetViewController 내부에서 fetch|
|---|---|---|
|책임 분리|✅ (ViewModel 순수 유지)|❌ (역할이 모호해짐)|
|테스트 용이성|✅|❌|
|데이터 흐름 추적|명확|흐려짐|
|권장 여부|✅ 매우 권장|❌ 권장하지 않음 (특별한 경우 제외)|

## 🔚 정리

> 🧠 **EditBudgetViewController는 데이터를 수정하는 UI만 담당하고,  
> 저장된 값은 onSave 클로저를 통해 상위 ViewController에서 CoreData에 반영**  
> ✅ 이 흐름이 MVVM 구조에서 가장 권장되는 방식입니다.
