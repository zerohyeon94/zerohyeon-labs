**JSON Web Token**의 약자.  
사용자/권한 같은 정보를 **JSON**으로 담고, 그것을 **서명**해서 위변조를 막는 **문자열 토큰**입니다. 보통 `aaaa.bbbb.cccc` 처럼 `.`로 구분된 **세 부분**으로 구성돼요.

1. **Header**: 알고리즘·토큰 타입 정보
    
2. **Payload(Claims)**: 실제 담긴 데이터 (예: 사용자 ID, 만료시간 등)
    
3. **Signature**: 위 두 부분을 비밀키(또는 공개키 방식)로 서명한 값

예시(요약):

```text
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.   // header (base64url)
eyJpc3MiOiJnb29nbGUiLCJzdWIiOiIxMjMiLCJhdWQiOiJmyApp...  // payload
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c              // signature
```

> **중요 포인트:** JWT는 **서명되었지 암호화가 아닙니다.** 따라서 민감한 개인정보(예: 실제 비밀번호)는 넣지 않습니다.

# 소셜 로그인과 JWT의 관계

대부분의 소셜 로그인은 **OAuth 2.0 / OpenID Connect(OIDC)** 흐름을 사용합니다.

- **ID 토큰**: 사용자의 “인증” 사실을 증명하는 토큰. 보통 **JWT**입니다. (Google, Apple, Kakao OIDC 등)
    
- **액세스 토큰**: API 호출 권한을 주는 토큰. 공급자에 따라 **JWT일 수도, 아닐 수도(불투명 토큰)** 있습니다.
    
- **리프레시 토큰**: 만료된 액세스 토큰을 새로 발급받을 때 사용 (보안상 더 엄격히 보관)

iOS에서 **Sign in with Apple**의 `identityToken`은 대표적인 **JWT**입니다. Google도 OIDC를 쓰면 ID 토큰이 JWT로 옵니다. Kakao도 OIDC로 설정하면 ID 토큰이 JWT예요.

# 서버는 무엇을 검증하나요?

서버에서 반드시 **서명 검증**과 **클레임 검사**를 합니다.

- **서명(Signature)**: 공급자 공개키(JWK)를 받아 **RS256/ES256** 등으로 검증
    
- **iss(발급자)**: 기대하는 발급자인지
    
- **aud(수신자)**: 내 앱의 클라이언트 ID인지
    
- **exp/iat/nbf**: 만료·발급 시간 유효한지
    
- **nonce**: 재사용 공격 방지(특히 모바일·Web 혼합 환경에서 중요)

> 일반적으로 **클라이언트(iOS)** 에서는 서명 검증까지 하지 않고, **서버**가 최종 검증자입니다.

# iOS 클라이언트에서의 실무 팁

- **어디에 보관?**
    
    - 짧게 쓰는 액세스 토큰: 메모리/Keychain.
        
    - 길게 쓰는 리프레시 토큰: **반드시 Keychain**.
        
- **만료 처리**: `exp`를 보고 만료 임박 시 **조용히 토큰 갱신** UX.
    
- **신뢰 경계**: 클라이언트의 “검증”은 **참고용**일 뿐, 보안 결정은 서버가 한다고 생각하세요.
    
- **네트워크 요청**: 액세스 토큰은 보통 `Authorization: Bearer <token>` 헤더로 전송.

# HS256 vs RS256/ES256

- **HS256**: 대칭키(서버 비밀키 공유). 간단하지만 **키 유출 위험**과 **검증 주체 분리 어려움**.
    
- **RS256/ES256**: **비대칭키**(공개키/개인키). 소셜 로그인·OIDC에서 표준. **권장**.

# JWT를 쓸 때의 장단점

**장점**

- 서버가 DB 조회 없이도 클레임 기반 인가 결정을 빠르게 내릴 수 있음
    
- 표준적(OIDC)이고, 다양한 언어/플랫폼 라이브러리 지원

**주의/단점**

- “자체 포함” 특성 때문에 **토큰 폐기(revoke)**가 쉽지 않음 → **짧은 만료 + 리프레시 토큰** 전략 권장
    
- **민감정보 노출 금지** (payload는 누구나 열람 가능)
    
- 클레임 난립/버전 관리 실패 시 스파게티화
