## 현재 권장 방향

1차 테스트 단계에서는 **Railway Hobby로 staging 환경을 만든다.**

권장 이유:

- 지금 목표는 운영 서버가 아니라 팀 테스트용 API base URL 확보이다.
- GitHub 연결 후 자동 빌드/배포가 가능해서 개발 흐름에 맞다.
- AWS Lightsail/EC2 방식은 추후 운영 후보로 좋지만, 지금 당장 적용하면 서버 관리와 배포 자동화 준비 시간이 늘어난다.
- Railway는 PostgreSQL/Redis 서비스를 같은 프로젝트 안에 빠르게 붙일 수 있다.
- 한 달 뒤 운영 배포 시점에 AWS Lightsail/EC2 또는 Railway Production 유지 중 다시 판단할 수 있다.

운영 배포 전환 시 검토:

- 테스트 중 Railway에서 비용이 과도하게 나오지 않는지 확인한다.
- 팀원이 Railway 로그/재배포/환경변수 관리에 접근할 수 있는지 확인한다.
- 실제 운영에는 DB 백업, 파일 저장소, 도메인, HTTPS, 장애 대응 기준을 분리해서 검토한다.
- AWS로 전환한다면 Docker Compose 기반으로 앱, PostgreSQL, Redis를 올리고 Nginx/Caddy, SSL, 백업 스크립트를 추가한다.

## Railway Hobby 적용 설계 초안

Railway 프로젝트 구성:

- `backend-app`
  - GitHub repo 연결
  - Dockerfile 경로: `docker/app/Dockerfile`
  - 24시간 테스트를 위해 serverless/sleep 기능은 비활성화
  - Healthcheck path: `/actuator/health` 또는 별도 `/health` 엔드포인트
- `postgres`
  - PostgreSQL 서비스
  - pgvector 확장 사용 가능 여부 확인
  - `ddl-auto: none`이므로 schema 생성/초기 데이터 주입 방식 필요
- `redis`
  - Railway Redis 서비스
  - Refresh Token, cache, rate limit 용도 확인

필요한 Railway 환경변수:

```text
APPS_ENV=staging
SERVER_PORT=${{ PORT }}
POSTGRES_HOST=<Railway PostgreSQL host 또는 reference variable>
POSTGRES_PORT=<Railway PostgreSQL port>
POSTGRES_DB=<Railway PostgreSQL database>
POSTGRES_USER=<Railway PostgreSQL user>
POSTGRES_PASSWORD=<Railway PostgreSQL password>
REDIS_HOST=<Railway Redis host 또는 reference variable>
REDIS_PORT=<Railway Redis port>
REDIS_PASSWORD=<Railway Redis password>
JWT_SECRET_KEY=<staging jwt secret>
JWT_ISSUER=haruup-staging
CLOVA_API_KEY=<Naver Clova Studio key>
FIREBASE_SERVICE_ACCOUNT_JSON=<Firebase service account JSON>
RAILWAY_DOCKERFILE_PATH=docker/app/Dockerfile
```

추가로 확인할 것:

- Railway는 기본적으로 `PORT` 환경변수 기준으로 public routing/healthcheck를 수행한다. 현재 앱은 `SERVER_PORT`를 사용하므로 `SERVER_PORT=${{ PORT }}`로 맞추거나 `application.yml`에서 `PORT`도 fallback으로 받도록 수정한다.
- `FIREBASE_SERVICE_ACCOUNT_JSON`은 멀티라인/JSON 문자열로 넣을 수 있는지 확인한다.
- 파일 업로드 테스트가 필요하면 Railway Volume을 붙이고 `file.upload-dir`를 볼륨 경로로 변경한다.
- 파일 업로드가 이번 테스트 범위가 아니라면 해당 기능은 known limitation으로 남긴다.
- CORS에 앱 테스트 도메인 또는 필요한 클라이언트 origin을 추가한다.
- 운영 배포 전에는 Swagger 공개 범위와 actuator 공개 범위를 다시 제한한다.

## AWS 운영 배포 후보 메모

운영 단계에서 AWS를 선택할 경우 예상 방향:

- AWS Lightsail 또는 EC2에 Docker Compose 배포
- 앱, PostgreSQL, Redis를 같은 서버에 올리는 단순 구조로 시작
- Nginx 또는 Caddy로 HTTPS 처리
- DB volume 백업 스크립트 추가
- 서버 방화벽, SSH key, 보안 그룹 설정
- GitHub Actions로 서버에 SSH 접속 후 `docker compose pull/up -d` 하는 자동 배포 고려

AWS를 운영 후보로 남기는 이유:

- 비용 예측이 쉽다.
- Docker Compose 구조를 그대로 살리기 좋다.
- 운영 자유도가 높다.
- 다만 지금 테스트 단계에서는 서버 관리 부담이 커서 후순위로 둔다.

## AWS 배포 구성 요소 역할

현재 `docker-compose.yml`에는 실제 API 서버뿐 아니라 QA 서버, DB, Redis, 모니터링 도구까지 함께 들어 있다. AWS에 그대로 올릴 수도 있지만, 처음부터 전부 올리면 서버 메모리와 운영 복잡도가 커진다. AWS 최소 배포에서는 필수 서비스만 먼저 남기고, QA/모니터링은 필요해지는 시점에 추가한다.

| Redis                 | 빠른 임시 저장소. 현재 코드에서는 refresh token 저장, cache, rate limit 등 인증/성능 관련 기능에 연결된다. DB처럼 장기 데이터를 보관하기보다 빠른 조회와 만료 처리가 목적이다.             | 필수에 가깝다. Redis 없이 일부 인증/캐시 기능이 깨질 수 있으므로 AWS 최소 구성에도 포함한다.                                  |
| redis-qa              | QA 서버가 운영 Redis와 토큰/캐시를 섞지 않도록 별도로 띄우는 QA용 Redis다.                                                                               | 초반 제외. `app-qa`를 제외하면 같이 제외한다.                                                              |
| Prometheus            | Spring Boot Actuator의 `/actuator/prometheus` 메트릭을 주기적으로 수집한다. CPU/메모리 자체보다 API, JVM, HTTP 요청, DB 관련 애플리케이션 메트릭 추적에 가깝다.          | 초반 제외 가능. 처음에는 `/health`, Docker logs, 서버 리소스 확인으로 시작하고 운영 안정화 시 추가한다.                      |
| Grafana               | Prometheus/Loki 데이터를 대시보드로 시각화한다. 장애 분석과 운영 지표 확인용 화면이다.                                                                         | 초반 제외 가능. Prometheus/Loki를 쓰기로 했을 때 함께 추가한다.                                                |
| Loki                  | 애플리케이션 로그를 수집/검색하는 로그 저장소다. 여러 컨테이너와 배포 이력이 많아질수록 유용하다.                                                                          | 초반 제외 가능. 처음에는 `docker compose logs` 또는 CloudWatch/서버 로그로 충분하다.                             |

정리하면 AWS 최초 배포에서 필요한 것은 다음 4개다.

```text
app + postgres + redis + reverse-proxy(Nginx 또는 Caddy)
```

현재 compose에 있는 `app-qa`, `redis-qa`, `prometheus`, `grafana`, `loki`는 처음 AWS 배포에서는 제외하고, 운영 중 필요성이 확인되면 단계적으로 붙이는 편이 안전하다.

## AWS 최소 배포 설정

### 서버 크기

최저비용 테스트/초기 운영:

- AWS Lightsail `Small-2GB Linux with public IPv4`
- 사양: 2 vCPU, 2 GB RAM, 60 GB SSD
- 용도: 하루업 API + PostgreSQL + Redis를 한 서버에 올리는 최소 구성
- 주의: Spring Boot JVM, PostgreSQL, Redis가 메모리를 나눠 써야 하므로 모니터링 도구까지 같은 서버에 올리기에는 빡빡하다.

운영 초반 권장:

- 4 GB RAM급 Lightsail 또는 EC2
- 이유: 사용자 트래픽, 배치, Clova API 호출, DB 캐시, Redis를 고려하면 2 GB는 여유가 작다.

운영 분리 후보:

- 앱 서버: Lightsail/EC2
- DB: Amazon RDS for PostgreSQL
- Redis: ElastiCache 또는 별도 Redis
- 파일 저장: S3
- 이유: 백업, 장애 복구, 저장소 안정성이 좋아진다. 다만 비용과 설정 난이도가 올라간다.

### 최소 컨테이너 구성

```text
haruup-app
haruup-postgres
haruup-redis
caddy 또는 nginx
```

각 컨테이너 역할:

- `haruup-app`: Spring Boot WAR 실행. 내부 포트 8080.
- `haruup-postgres`: PostgreSQL + pgvector. `postgres_data` volume 필요.
- `haruup-redis`: refresh token/cache/rate limit 저장. `redis_data` volume 권장.
- `caddy` 또는 `nginx`: 80/443 요청을 받아 `haruup-app:8080`으로 전달하고 HTTPS를 처리.

### 네트워크/방화벽

외부 공개:

- `80`: HTTP, 인증서 발급 및 HTTPS 리다이렉트
- `443`: HTTPS API 요청
- `22`: SSH. 가능하면 본인 IP에서만 접근 허용

외부 비공개:

- `8080`: Spring Boot 앱 포트. reverse proxy 내부에서만 접근
- `5432`: PostgreSQL. 외부 공개 금지
- `6379`: Redis. 외부 공개 금지
- `9090`, `3000`, `3100`: Prometheus/Grafana/Loki를 쓰지 않는다면 열지 않음

### 볼륨

최소로 필요한 영속 저장소:

```text
postgres_data  -> PostgreSQL 데이터
redis_data     -> Redis 데이터
uploads        -> 파일 업로드 저장소
caddy_data     -> Caddy 인증서/설정 데이터, Caddy 사용 시
```

주의:

- 현재 파일 업로드 경로는 `./uploads`다. AWS Docker 배포에서는 컨테이너 내부 임시 경로가 아니라 서버 volume에 연결해야 재배포 후에도 파일이 남는다.
- 운영에서는 업로드 파일을 서버 로컬 volume에 계속 두기보다 S3로 이전하는 것이 더 안전하다.

### 환경변수

AWS 서버에서도 Railway와 마찬가지로 `env.properties`를 커밋하지 않고 서버의 `.env` 또는 Docker Compose 환경변수로 주입한다.

```text
APPS_ENV=prod
SERVER_PORT=8080
POSTGRES_HOST=postgres
POSTGRES_PORT=5432
POSTGRES_DB=haruup
POSTGRES_USER=haruup_user
POSTGRES_PASSWORD=<strong-password>
REDIS_HOST=redis
REDIS_PORT=6379
REDIS_PASSWORD=<strong-password>
JWT_SECRET_KEY=<production-jwt-secret>
JWT_ISSUER=haruup
CLOVA_API_KEY=<Naver Clova Studio key>
FIREBASE_SERVICE_ACCOUNT_JSON=<Firebase service account JSON>
```

### 메모리 제한

2 GB 서버에 앱, DB, Redis를 같이 올릴 경우 JVM이 메모리를 과하게 잡지 않도록 제한해야 한다.

권장 시작값:

```text
Spring Boot app: -Xms256m -Xmx768m
PostgreSQL: shared_buffers 128MB~256MB 수준
Redis: maxmemory 128MB~256MB 수준
```

현재 앱 Dockerfile의 `JAVA_OPTS`는 `MaxRAMPercentage=75.0`이다. 컨테이너 메모리 제한 없이 2 GB 서버에서 실행하면 앱이 메모리를 크게 잡아 PostgreSQL/Redis와 충돌할 수 있으므로 AWS 배포용 compose에서는 앱 컨테이너 메모리 제한 또는 `-Xmx` 설정을 별도로 둔다.
